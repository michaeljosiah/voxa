Voxa · Architecture

How Voxa is built

Voxa is a frame-based, real-time voice-AI pipeline framework for .NET 10 (inspired by Pipecat). You compose voice agents from small, independently-testable processors that consume and emit typed Frames — audio, transcripts, tool calls, control signals. This document is the map: the core runtime, the provider system, the local-model tier, and the Voxa Studio desktop app.

1Overview & solution layout

Voxa turns a stream of microphone audio into a spoken agent reply, in real time, with barge-in. The whole framework is one idea applied consistently: everything that flows through the pipeline is a typed Frame, and every stage is a FrameProcessor that consumes some frames, emits others, and forwards the rest.

Because the contract between stages is just frames, stages compose freely and each is testable in isolation behind an interface (fake the engine, never burn cloud quota in a unit test). A voice bot is a chain:

PipelineSource → FrameProcessor… → PipelineSink

The repository is a single solution, Voxa.slnx (the new XML solution format — there is no .sln), organised as:

FolderWhat lives there
src/One csproj per NuGet package. Voxa.Core (zero-dependency runtime), the provider packages (Voxa.Speech.*, Voxa.Audio.*, Voxa.Services.*, Voxa.Transports.*), the ASP.NET integration, and the Voxa meta-package.
tests/One xUnit project per library. Warnings are errors here, as in src/.
apps/Voxa.Studio — the Avalonia developer desktop app.
samples/Runnable servers (minimal, ASP.NET, Twilio).
bench/BenchmarkDotNet micro-benchmarks and the turn-taking corpus runner.
docs/Guides (*.md) and design specs (specifications/*.html).
The cardinal rule. Voxa.Core has zero external dependencies beyond NUlid. No ASP.NET, no Azure, no OpenTelemetry ever leak into it. Everything vendor- or host-specific lives in a satellite package behind an interface.

2Frames & processors

A Frame is an immutable record. Frames fall into families:

FamilyExamplesRole
Audio / dataAudioRawFrame, TranscriptionFrame, TextFrame, LlmTextChunkFrameThe ordered payload of a turn — the stuff being transformed.
Speaking eventsUserStartedSpeakingFrame/…Stopped…, BotStartedSpeakingFrame/…Stopped…VAD & turn edges that drive the conversation state machine.
Control / systemStartFrame, EndFrame, InterruptionFrame, ErrorFrameLifecycle & priority signals — they jump the queue.
Tool callsToolCallRequestFrame, ToolCallResultFrameFrontend tools round-trip through the pipeline so any stage can serve them.

A FrameProcessor overrides one method to handle the frames it cares about and forwards the rest downstream. Forwarding is not optional: a processor that swallows StartFrame/EndFrame strands the sink (it never completes). Any long-running work accepts the per-frame CancellationToken so an interruption can abort it mid-flight.

3The dual-task processor model

The non-obvious heart of Voxa is that every FrameProcessor drains two queues concurrently:

  • A system task drains priority frames — InterruptionFrame, speaking events, ErrorFrame — on an unbounded, never-dropped channel.
  • A data task drains ordered frames — audio, transcripts, text — on a bounded channel.

This split is what makes barge-in correct. When the user starts speaking mid-reply, an InterruptionFrame arrives on the system task and cancels the in-flight data frame's CancellationToken — so a streaming LLM or a long TTS synthesis aborts cleanly while ordered frames keep flowing. The exception is frames marked IUninterruptible (tool calls, EndFrame): those are guaranteed to complete.

Allocation-free at steady state. The per-frame cancellation source is reused; it is replaced only after an actual interruption. So the hot path of a running conversation does not churn the GC.

4Backpressure

The two queues use opposite full-modes on purpose:

PathChannelWhen full…
Audio / databounded, BoundedChannelFullMode.DropOldestdrop the oldest audio rather than stall the pipeline — a slow consumer loses samples, not real-time.
Control / systemunbounded, Waitnever drop a control signal; priority frames always land.

This is why a momentarily slow downstream stage degrades gracefully (dropped audio) instead of deadlocking. Any processor that deviates from this convention documents why.

5The agent loop

AgentLoopProcessor (in Voxa.Core) is the framework-agnostic per-turn brain. Its defining property: the data loop never blocks on the agent or a tool call. A separate turn worker runs the turn while frames keep flowing through the processor.

It owns per-turn ids, the lifecycle frames (LlmTurnStartedFrame/LlmTurnEndedFrame), per-turn try/catch isolation, token aggregation into a TurnSummary, and the correlation of frontend tools — tool calls leave as ToolCallRequestFrame, travel the pipeline, and the result returns as ToolCallResultFrame matched back by a TaskCompletionSource.

The single seam to any LLM runtime is tiny:

public interface IAgentTurnDriver
{
    IAsyncEnumerable<Frame> RunTurnAsync(VoiceTurnContext ctx, CancellationToken ct);
}

The driver yields frames (text chunks, tool requests, usage) one at a time; the processor pumps them downstream immediately. For Microsoft Agent Framework, MicrosoftAgentVoice.CreateProcessor(agent, options) returns a fully-wired AgentLoopProcessor whose driver translates MAF's streaming updates into Voxa frames. A local LLM is reached the same way — point an IChatClient at a local OpenAI-compatible endpoint (Ollama today).

6Two pipeline shapes

Voxa supports two ways to assemble a voice agent, and they coexist behind the same frame contract.

6.1 Granular — vendor-neutral chain

Mix any STT vendor with any LLM with any TTS vendor. SentenceAggregator batches LLM tokens into sentence-sized TTS chunks (so speech starts before the full reply is generated); TranscriptionFilter drops Whisper hallucinations.

6.2 Composite — one processor, full-duplex

A single processor wraps STT+LLM+TTS+VAD against a server-side realtime API: OpenAIRealtimeProcessor, AzureVoiceLiveProcessor. Full-duplex, server-side VAD, one WebSocket. The same family hosts an on-device speech-to-speech model through the ISpeechToSpeechSession seam (a third sibling alongside the two cloud composites).

7The seam ecosystem

Every capability is a thin interface with a fake for tests and one or more real implementations. The STT and TTS seams are push/streaming — you push audio in and read results out as an IAsyncEnumerable, never poll — which is the correct shape for real-time and composes with cancellation and backpressure.

SeamShapeNotes
ISpeechToTextEnginepush audio → stream TranscriptionResult16+ impls. WriteAudioAsync / ReadTranscriptsAsync / VAD-gated FlushAsync; optional eager/speculative STT.
ITextToSpeechEnginepush text → stream PCM chunks13+ impls (cloud + local + sidecar).
IAgentTurnDrivercontext → stream FrameThe LLM-runtime seam (see §5).
IRealtimeApiTransportpush/stream JSON eventsWebSocket wire for the cloud composites.
ISpeechToSpeechSessionpush user audio → stream agent chunks+eventsFull-duplex on-device model seam; mock ships, real model deferred.
ISmartTurnClassifierrecent PCM → "turn complete?"Lets VAD StopDuration drop without cutting off mid-sentence pauses. Fails "complete" on error.
IEchoCanceller / IAudioEnhancerconcurrent / stateful filtersAEC (far-end reference + near-end cancel) and denoise. Null defaults ship; DSP/ONNX models follow.
ISpeakerSegmentation · ISpeakerEmbedding · IDiarizerbatch audio → speaker segmentsDiarization: pyannote ONNX segmentation + centroid clustering.

VAD is the one capability expressed as a concrete processor (SileroVadProcessor over an ONNX engine) rather than an interface — it owns the decision logic (emits the speaking-edge frames, the eager-STT speculation, long-utterance force-split, and the optional smart-turn confirmation).

8AddVoxa, registry & composer

This is the "five lines to a voice bot" layer (Voxa.AspNetCore + the Voxa meta-package), and the part most likely to surprise you.

// Program.cs — a working voice bot
builder.Services.AddVoxa(builder.Configuration);
app.MapVoxaVoice("/voice").UseDefaults();
  • AddVoxa(configuration) registers a VoxaProviderRegistry + every built-in provider descriptor, binds the "Voxa" config section, and adds a fail-fast options validator (ValidateOnStart) — the host refuses to start on an unknown provider/profile.
  • Providers are descriptors: VoxaSttDescriptor / VoxaTtsDescriptor carry a name, a config section, a keyless Validate, a CreateProcessor factory, and an optional startup WarmUpAsync. A package self-registers via builder.AddProvider(descriptor).
  • MapVoxaVoice("/voice").UseDefaults() composes VAD → STT → agent → aggregation → TTS via DefaultVoicePipelineComposer, which is transport-agnostic (Compose(IServiceProvider); the HttpContext overload forwards to it). The same composer backs the WebSocket route, the Twilio telephony route, and Voxa Studio.
  • Named profiles — LowLatency / Quality / Cheap — bundle the VAD timings, eager-STT threshold, and response caps from the performance-tuning guide. They tune latency and robustness on a fixed topology; provider selection stays an explicit per-slot config choice.
The config-capture rule. Providers that need configuration capture the IConfiguration passed to AddVoxa — they must not call GetRequiredService<IConfiguration>(). ASP.NET registers IConfiguration implicitly, but plain ServiceCollection hosts (Voxa Studio, tests) do not, so DI resolution would throw there. The validator, composer, and default agent factory all follow this.

9The diagnostics hub

VoxaDiagnosticsHub (Voxa.Core/Diagnostics, enabled by Voxa:Diagnostics:Enabled) is a per-session typed event stream: VAD windows, turn edges, transcripts, stage latencies. Two properties make it safe to leave in the framework:

  • Zero-cost when unobserved — a HasListeners guard precedes every publish. With diagnostics off, composed pipelines are byte-identical to pre-diagnostics (golden-tested).
  • Bounded, drop-oldest per subscriber — a slow observer can't back-pressure the conversation; a gap in the SeqNo signals a drop intentionally.

A StageLatencyTracker inside the hub derives the per-turn latency waterfall (VAD close → STT final → LLM first token → TTS first byte → audio out) and records the voxa.stage.latency histogram. This is exactly what Voxa Studio's Talk view renders live.

10Local-model infrastructure

The offline tier is what lets you develop without API keys, deploy air-gapped, and run zero-cost CI conversations. Three pieces work together.

10.1 The pinned catalog — VoxaModelArtifact

Every downloadable model is a record with a pinned URL and SHA-256 — no floating refs:

sealed record VoxaModelArtifact(
    string Id, Uri DownloadUrl, string Sha256, long SizeBytes)
{
    public string? ArchiveEntry { get; init; }  // path inside a .zip/.tar.gz, or null
    public bool    Executable   { get; init; }  // chmod +x on Unix
}

Per-provider catalogs build on it — WhisperCppModelCatalog (22 pinned GGML models), PiperVoiceCatalog, KokoroCatalog, PyannoteSegmentationCatalog. The variant is keyed by name (size / quantization / voice), not a capability flag matrix.

10.2 The cache — VoxaModelCache

Resolves its root from VOXA_MODEL_CACHE (env, highest), then Voxa:Models:CachePath, then the OS default (%LOCALAPPDATA%\voxa\models on Windows). Downloads verify SHA-256 incrementally with an atomic rename, a cross-process .lock serialises parallel test hosts, and Voxa:Models:Offline=true turns a cache miss into a VoxaModelUnavailableException with copy-pasteable air-gap provisioning instructions. Validation never downloads — it only checks the cache.

10.3 The ONNX host — OnnxModelHost

A process-wide session factory keyed by (path, device) so weights load exactly once across concurrent sessions. Execution-provider selection is CPU by default, GPU opt-in (OnnxExecutionProvider: CUDA / DirectML / CoreML, with an Auto that probes and falls back to CPU). The base package bundles only the CPU runtime; a GPU runtime is a matching NuGet package the consuming app adds, and an unsatisfiable explicit request throws with a remediation message rather than silently degrading.

12Voxa Studio

Voxa Studio (apps/Voxa.Studio) is an Avalonia 11 desktop app that hosts the real composed pipeline in-process against the developer's mic and speakers — the same DefaultVoicePipelineComposer a server uses. It is the framework's microscope.

12.1 Composition & the no-Generic-Host rule

Studio builds its container with a plain ServiceCollection + AddVoxa, deliberately not the Generic Host. The reason is a hard product rule: Studio never touches the network before the user acts. Hosted services (eager model warm-up) must not run on boot, so there is no host to run them. The staged splash (StartupCoordinator: configuration providers devices model cache ready) does only local work — device enumeration and a cache scan, never a fetch.

12.2 The three-layer live config

Studio can re-compose the live pipeline without a restart. StudioServices.Build layers configuration, lowest priority first:

Because secrets are their own layer, a Config "Apply" (which replaces the override set) can't erase stored API keys. The GPU-gated default-STT selector (VLS-009) also lives here: when no layer chose an Stt, it picks Voxtral if the machine can run it (an nvidia-smi VRAM probe behind a cached seam) and a vLLM server is configured, else keyless whisper.cpp — overlaid as the lowest-priority default so any explicit choice still wins.

12.3 View models are Avalonia-free

The view models reference no Avalonia types and are headless-testable: a 33 ms DispatcherTimer calls DrainPending() in the app; tests call it directly. Headless boot runs through Avalonia.Headless.XUnit with headless drawing (the Skia render loop is opt-in via VOXA_STUDIO_CAPTURE=1 — enabling it keeps the dispatcher alive, which is the exact cause of the historical "tests pass then dotnet test never exits" hang).

12.4 Audio behind a seam

All audio is behind IStudioAudioDevice — WASAPI/NAudio on Windows, NullAudioDevice elsewhere and in tests. WASAPI mix formats are usually WAVE_FORMAT_EXTENSIBLE; CaptureFormatConverter resolves the subformat GUIDs and an unconvertible format throws at session start rather than going silently deaf.

14Testing & CI

  • Every seam has a fake. Unit tests fake the engine/transport — never burn cloud quota, never need a GPU.
  • The LocalModels trait is load-bearing. Tests tagged [Trait("Category","LocalModels")] download real speech models (hundreds of MB) and are excluded from the default run (--filter "Category!=LocalModels"), running in their own cached, network-blocked CI lane.
  • Warnings are errors in src/, apps/, and most tests — fix, don't suppress. dotnet build Voxa.slnx -warnaserror matches CI.
  • Studio runs headless in the default Ubuntu suite plus a dedicated windows-latest lane for the WASAPI paths.

VLS-002 — Local STT GPU Acceleration & Whisper Catalog Expansion

Two additive enhancements to the local speech-to-text tier: bigger, more accurate whisper.cpp models (medium, large-v3, large-v3-turbo) added to the SHA-256-pinned catalog, and an opt-in GPU execution seam (Voxa:WhisperCpp:Device) so those larger models are viable in real time on CUDA / Vulkan hardware — with the default package staying CPU-only and byte-identical to today.

1.What is this? — a plain-English explainer

What does VLS-002 add, in one sentence?

It lets Voxa's on-device speech-to-text use bigger, more accurate Whisper models, and run them on a GPU so they're fast enough for a live conversation — without changing anything for users who are happy on CPU.

Why now?

Today the local STT catalog (VLS-001) stops at the small model and runs CPU-only. small is fine for clean English, but it mishears names, accents, and noisy audio. The fix is the bigger Whisper models (medium, large-v3) — but those are 1.5–3 GB and far too slow for real-time on a CPU. They only make sense paired with GPU acceleration, so the two land together.

Does this make the package heavier or require a GPU?

No. The default Voxa.Speech.WhisperCpp package stays CPU-only and the same size as today. GPU support is opt-in: a user who wants it adds a runtime package (e.g. Whisper.net.Runtime.Cuda) to their own app and sets Voxa:WhisperCpp:Device = cuda. Nobody pays for a capability they don't use.

What is explicitly not changing?

The engine contract (still 16 kHz mono, utterance-buffered, final-only transcripts), the model cache, the offline/air-gap behaviour, and the descriptor wiring. This is purely additive.

2.Executive summary

VLS-002 is two small, independent workstreams on the existing Voxa.Speech.WhisperCpp package:

  • WS1 — Catalog expansion additive. Add medium, medium.en, large-v3, large-v3-turbo and their q5_0 quantizations to WhisperCppModelCatalog, each SHA-256-pinned from the Hugging Face LFS metadata. Pure data; resolution, download, verification, offline errors, and warm-up all flow through the unchanged VoxaModelCache path.
  • WS2 — GPU execution seam P1. A new Voxa:WhisperCpp:Device key (cpu default, plus auto/cuda/vulkan/coreml) that tells the engine which Whisper.net native runtime to prefer at StartAsync. The GPU runtime natives are never bundled by Voxa — the user adds the matching Whisper.net.Runtime.* package — so the default build is unchanged and we avoid the native re-bundling problem fixed in d2a11ae.
Net effect. A developer on an NVIDIA box can set "Model": "large-v3", "Device": "cuda" and get near-real-time, high-accuracy local transcription; everyone else is unaffected and CI stays CPU-deterministic.

3.Goals, non-goals & budgets

Goals

  • Make medium / large-v3 / large-v3-turbo selectable by config name, SHA-256-pinned like every other artifact.
  • Allow STT inference on GPU (CUDA first; Vulkan as a cross-vendor option) via a single config key, CPU default.
  • Keep the default package CPU-only and its pack output byte-identical to today (no new bundled natives).
  • Fail loudly and actionably when GPU is explicitly requested but unavailable.

Non-goals (this spec)

  • Streaming / interim transcripts (ROADMAP P0).
  • faster-whisper / CTranslate2 / distil-whisper backends.
  • Kokoro (TTS) GPU — deferred; see §12.
  • Auto-downloading GPU runtime natives — the user opts in with a NuGet reference.

Budgets

5.WS1 — STT catalog expansion

Package Voxa.Speech.WhisperCpp Risk Low Network first-run download only Surface additive (new config values)

The catalog is a compiled-in dictionary of VoxaModelArtifacts; each entry is one line via the existing Entry(file, sha256, sizeBytes) helper. We add the larger models from huggingface.co/ggerganov/whisper.cpp:

Catalog nameFileApprox sizeNotes
mediumggml-medium.bin~1.53 GBmultilingual
medium.enggml-medium.en.bin~1.53 GBEnglish-only
medium-q5_0ggml-medium-q5_0.bin~539 MBquantized
large-v3ggml-large-v3.bin~3.1 GBbest accuracy
large-v3-q5_0ggml-large-v3-q5_0.bin~1.08 GBquantized
large-v3-turboggml-large-v3-turbo.bin~1.62 GB~8× faster than large, near-large accuracy
large-v3-turbo-q5_0ggml-large-v3-turbo-q5_0.bin~574 MBquantized turbo
Do not fabricate hashes. Sizes above are approximate, for the progress bar and error text only. The authoritative SHA-256 and exact byte length must be read at implementation time from each file's Hugging Face LFS pointer — GET huggingface.co/ggerganov/whisper.cpp/raw/main/<file> returns a pointer containing oid sha256:<hash> and size <bytes>. This is exactly how the VLS-001 entries were sourced. A hand-written or guessed hash is a bug, not a placeholder.
src/Voxa.Speech.WhisperCpp/WhisperCppModelCatalog.cs (additions) 
// <-- sha256 + sizeBytes from the HF LFS pointer; never hand-written -->
["medium"]              = Entry("ggml-medium.bin",              "<sha256>", <bytes>),
["medium.en"]           = Entry("ggml-medium.en.bin",           "<sha256>", <bytes>),
["medium-q5_0"]         = Entry("ggml-medium-q5_0.bin",         "<sha256>", <bytes>),
["large-v3"]            = Entry("ggml-large-v3.bin",            "<sha256>", <bytes>),
["large-v3-q5_0"]       = Entry("ggml-large-v3-q5_0.bin",       "<sha256>", <bytes>),
["large-v3-turbo"]      = Entry("ggml-large-v3-turbo.bin",      "<sha256>", <bytes>),
["large-v3-turbo-q5_0"] = Entry("ggml-large-v3-turbo-q5_0.bin", "<sha256>", <bytes>),

No other code changes are required for WS1: TryGet, KnownModels, the descriptor's unknown-model error text, the offline-miss error, and warm-up all consume the dictionary generically.

6.WS2 — the GPU execution seam

Package Voxa.Speech.WhisperCpp Risk Medium (packaging) Default cpu (unchanged) Natives user-supplied, never bundled

Config & option

Add a Device to WhisperCppOptions, parsed from Voxa:WhisperCpp:Device:

ValueMeaning
cpu (default)CPU runtime, as today. Deterministic; what CI uses.
autoPrefer the best available GPU runtime, fall back to CPU with a warning log.
cudaRequire the CUDA runtime. If unavailable → fail at start with remediation (don't silently run slow).
vulkanRequire the Vulkan runtime (cross-vendor: NVIDIA/AMD/Intel).
coremlApple Silicon — best-effort, see the CoreML note below.

How the runtime is selected

Whisper.net loads a native backend chosen from the Whisper.net.Runtime.* packages present in the app and the loader's configured order. The engine sets that order from Device before constructing the (process-wide, cached) WhisperFactory, then logs which backend actually loaded so the choice is observable.

src/Voxa.Speech.WhisperCpp/WhisperCppSttEngine.cs (sketch — pin exact API to the referenced Whisper.net version) 
// Applied once, before the shared WhisperFactory is built. Exact API per Whisper.net version:
// the runtime library order is the supported selection seam (Whisper.net RuntimeOptions).
static IReadOnlyList<RuntimeLibrary> OrderFor(WhisperDevice device) => device switch
{
    WhisperDevice.Cpu    => [RuntimeLibrary.Cpu],
    WhisperDevice.Cuda   => [RuntimeLibrary.Cuda],                 // no Cpu fallback: explicit means required
    WhisperDevice.Vulkan => [RuntimeLibrary.Vulkan],
    WhisperDevice.CoreML => [RuntimeLibrary.CoreML],
    WhisperDevice.Auto   => [RuntimeLibrary.Cuda, RuntimeLibrary.Vulkan, RuntimeLibrary.CoreML, RuntimeLibrary.Cpu],
    _ => [RuntimeLibrary.Cpu],
};
// On load failure for an explicit (non-auto, non-cpu) device, throw a VoxaModelUnavailable-style
// error: "Voxa:WhisperCpp:Device=cuda but no CUDA runtime loaded. Add the Whisper.net.Runtime.Cuda
// package to your app and ensure the CUDA toolkit/driver is installed, or use Device=cpu."

Packaging — the load-bearing decision

GPU natives are opt-in and never bundled by Voxa. The default Voxa.Speech.WhisperCpp keeps referencing only the CPU runtime (with PrivateAssets as today, so it isn't re-flowed onto consumers — the concern behind commit d2a11ae). To enable GPU, the consuming app adds the matching package itself: 
<PackageReference Include="Whisper.net.Runtime.Cuda" Version="..." />   <!-- in the app, not in Voxa -->
This keeps the default build byte-identical, sidesteps multi-GB GPU natives in our package, and means the GPU matrix is the user's (and Whisper.net's) concern, not ours.

Validation

WhisperCppDescriptors.Validate rejects an unknown Device value at startup (keyless, no side effects — same shape as the existing InputSampleRate check). It does not probe for runtime-package presence (unreliable pre-load); a missing GPU runtime surfaces as a clear StartAsync error per the sketch above. When a large model resolves on Device=cpu, warm-up logs a warning that real-time latency is unlikely.

CoreML note

CoreML is best-effort. whisper.cpp's CoreML path typically needs a separate Core ML encoder bundle alongside the GGML file, not just a runtime swap. Pinning/extracting that extra artifact is extra scope, so coreml is wired as a value but its model-side provisioning is deferred (§12); CUDA and Vulkan need only the GGML file already in the catalog and are the supported GPU paths in VLS-002.

8.Plan of action — phases & dependencies

  1. Phase A — WS1 catalog (ships alone). Add entries with real LFS hashes + tests. Independently mergeable and immediately useful on CPU for the medium/turbo-q5_0 sizes.
  2. Phase B — WS2 GPU seam. Device option + runtime selection + validation + the large-on-CPU warning + docs. Depends on nothing in Phase A but lands second so the larger models exist to exercise it.

One logical change per PR (per CONTRIBUTING) — Phase A and Phase B are two PRs. Both add a [Unreleased] CHANGELOG entry and update docs/local-speech.md.

10.Verification protocol

Default (offline)Catalog resolution, KnownModels, offline-miss text, Device parse/validate. No download, no GPU.
Category=LocalModelsDownload + SHA-256 verify of one moderate new model (medium-q5_0) to prove the catalog entries are correct. Do not add large-v3 (3 GB) to routine CI — gate it behind an explicit opt-in to protect the lane's time/bandwidth budget.
Manual / hardwareOn a CUDA box: Model=large-v3, Device=cuda; confirm the loaded-backend log says CUDA and the STT segment of Studio's latency waterfall drops vs CPU. Confirm Device=cuda with no runtime package fails with the remediation message.

12.Out of scope / deferred

  • Kokoro / ONNX-Runtime GPU. The ORT CPU and GPU packages provide the same assembly and conflict in-process; enabling Kokoro GPU needs a build variant, and Kokoro is already acceptable on CPU. Revisit on demand (likely its own item).
  • CoreML model-side provisioning. The extra Core ML encoder bundle whisper.cpp needs is not pinned here; coreml is wired but unprovisioned (§6). Follow-up if Apple-Silicon GPU STT is requested.
  • Streaming / interim transcripts (ROADMAP P0).
  • faster-whisper / CTranslate2 / distil-whisper alternative backends.
  • Auto-downloading GPU runtime natives. Opt-in via NuGet reference, by design.

1.What is this? — a plain-English explainer

What does VST-005 do, in one sentence?

It rebuilds Voxa Studio's look and feel to match the reference prototype exactly — same colors, typography, spacing, components, shell chrome, and per-view layouts — turning today's functional-but-uneven UI into the polished, coherent product the VST-002 design brief described.

Is this a rewrite of Studio?

No. Studio already has all eight views, the nav shell, the splash, the settings dialog, themes, and the pipeline plumbing (VST-001 through VST-004). VST-005 is a view-layer fidelity pass. The Avalonia-free view models, the audio device seam, the AddVoxa composition, and every service stay put. We change .axaml, add a shared style/control library, add a dialog/toast service, and align each screen to the prototype.

Where does the "1:1" target come from?

From the attached interactive prototype — a self-contained React/Babel build that is the VST-002 design brief realized. It carries a complete token system (the same Ink / Pulse / Halo palette and Cascadia/Inter type the design skill already mandates), a component library, and 16 view modules. §4 records exactly what it contains; the rest of this spec translates it into Avalonia work.

What actually changes for a developer using Studio?

Every surface gets tighter and more consistent: one calm dark surface ramp with a single cyan accent (switchable to warm/slate), data in Cascadia mono everywhere, the five pipeline stages color-coded identically across Talk, Builder, Metrics, Models and Config, a real toast/dialog system instead of ad-hoc message surfaces, the "watch it think" Talk experience (orb + live waveform + running pipeline flow), and a Builder/Metrics/Voices/Diarization that look like the prototype rather than approximations.

What ships in VST-005, and what's deferred?

This item is a spec only — no app code lands with it. It defines the token set, the component library, the shell, every view's target layout, the splash, the dialog/toast system, and motion, plus a phased implementation plan (§13). Implementation lands as a sequence of small PRs behind this spec. Deferred (already deferred in the prototype too): the voice clone wizard, per-speaker diarization labelling, and any new pipeline capability — VST-005 is purely presentation.

2.Executive summary

The reference prototype defines a single, coherent design system and a precise layout for every Studio surface. Today's Studio implements the same information (VST-001…004) but with drift in tokens, components, and composition. VST-005 closes that gap in seven additive workstreams, each a small PR:

  • WS1 — Design tokens & themes P1. One Avalonia resource layer mirroring the prototype's tokens (Ink neutral ramp, Pulse cyan accent, Halo violet, the five stage colors, surfaces, text, lines, spacing, radii, typography, elevation, glow, motion) + three dark accent themes (cool default · warm · slate) that re-tint only the accent ramp. Foundation for everything else.
  • WS2 — Component library P1. The vx-* set as reusable Avalonia styles/controls: buttons (4 variants × 3 sizes), icon buttons, badges (7 tones), cards, tabs, inputs/selects, switches, status pills, waveform, voice orb, pipeline node + animated flow, transcript line, metric stat, code block. Consumed by every view so a label or pill looks identical everywhere.
  • WS3 — The shell P1. The 44px custom titlebar (mark + wordmark + live clock + replay-splash + window controls), the 82px icon nav rail (8 items + settings gear + avatar, active-accent bar), the global 52px Pipeline-Profile bar, the per-view ViewBar, and the "surface busy" BlockedOverlay for the live-session-contended views (Talk / Builder / Metrics).
  • WS4 — The views, 1:1 P2. Talk, Playgrounds (STT lab | TTS lab), STT Playground, TTS Playground, Voices, Builder, Metrics, Models, Config, Diarization — each rebuilt to the prototype's exact layout against the existing view models.
  • WS5 — Splash P2. The 480×320 borderless launch animation (mark draws on, bars spring, wordmark settles, stage ticker, progress bar), reduced-motion aware.
  • WS6 — Dialogs & toasts P1. An imperative, tone-driven dialog + toast service (confirm / danger / warn / alert / prompt / toast) behind an interface, so views raise consistent modals and notifications without each rolling their own — and view models stay Avalonia-free.
  • WS7 — Motion P3. The four motion tokens (fast 120 · standard 240 · draw 1100 · pulse 2600) applied consistently, honoring OS reduced-motion.
Net effect. Studio goes from "all the right views, uneven execution" to a single, brand-coherent product that matches the prototype pixel-for-pixel — built on the view models and services that already exist, with no change to how the pipeline runs.

3.Goals, non-goals & budgets

Goals

  • Reproduce the reference prototype's visual design and per-view layout 1:1 in the Avalonia app: tokens, components, shell chrome, splash, dialogs, and every view.
  • Land one token resource layer and one component library that every view consumes — no per-view hard-coded colors, spacings, or fonts (the prototype's discipline: "all values come from tokens").
  • Ship the three dark accent themes (cool default / warm / slate), switchable from Appearance settings and persisted, re-tinting only the accent ramp.
  • Add the imperative dialog + toast system so confirmations, destructive prompts, and notifications are consistent app-wide.
  • Keep every architectural guardrail (§5): Avalonia-free, headless-testable view models; plain ServiceCollection + AddVoxa (no Generic Host); audio behind IStudioAudioDevice; the diagnostics/pipeline paths untouched.
  • One logical change per PR with a CHANGELOG entry; this spec lands before any app code.

Non-goals (this overhaul)

  • New pipeline capability. No new STT/TTS/agent/transport behavior. VST-005 is presentation only; the composed pipeline and its config are unchanged.
  • View-model / service redesign. View models, stores (profiles, voices, secrets, preferences), and the audio device stay. We re-skin their views and, where a view model lacks a property the layout needs, add a presentation-only property (e.g. a derived label) — not new domain logic.
  • The clone wizard & per-speaker diarization. Deferred in the prototype; deferred here. The Voices "Clone voice" button shows the "coming soon" dialog; Diarization stays segmentation-only.
  • Light mode. Studio is dark-only by design (the brief: "VOXA stays dark"). Themes vary the accent, not the surface polarity.
  • Web/HTML shipping. The prototype is the reference, not an artifact we ship. No React, no WebView — this is native Avalonia.

Budgets

Method. Each WS section below states the prototype's exact spec (sizes, colors-by-token, controls, copy) and the target Avalonia file(s). Numeric values (heights, widths, paddings, font sizes) are taken verbatim from the prototype so "1:1" is checkable, not interpretive. One correction to carry. The prototype's About tab reads "v0.4.0 · Avalonia 11 · .NET 9". Voxa targets .NET 10 — the Avalonia About view must read .NET 10 and the live assembly version, not the prototype's placeholder string.
Source caveat — component primitives. The decoded bundle carries the vx-* CSS and all 16 view modules, but not the module that defines the React primitives the views call (VOXADesignSystem.* — Button, Card, Switch, VoiceOrb, PipelineFlow, …). So §7's component contracts (variants, sizes, render details) are reconstructed from the vx-* CSS + the views' call sites, not copied from a primitive source — confirm exact prop behavior against the call sites when building each control.

5.Architectural guardrails (do not break)

From CLAUDE.md / the Studio specs. VST-005 is a view-layer change and must preserve all of these:

  • View models are Avalonia-free and headless-testable. A 33 ms DispatcherTimer calls DrainPending(); tests call it directly. No view model may take an Avalonia dependency. UI-only concerns (dialogs, toasts, theme application) go behind interfaces the VM can call without referencing Avalonia.
  • Plain ServiceCollection + AddVoxa, not the Generic Host. Hosted services (eager model warm-up) must not run in Studio; Studio never touches the network before the user acts.
  • Audio behind IStudioAudioDevice (WASAPI/NAudio on Windows, NullAudioDevice elsewhere/tests). Unchanged.
  • Tests run in both lanes — the ubuntu default headless suite and the windows-latest Studio lane. New visual structure must boot headless (Avalonia.Headless.XUnit).
  • No emojis in code; records for config; one short doc-comment line explaining why. House style.
Consequence for this overhaul. The redesign is almost entirely .axaml + a new Theme/ token/style layer + a handful of new Controls/ + one IDialogService/IToastService pair. View models change only to expose presentation-only members the new layouts bind to (and those stay Avalonia-free).

6.WS1 — Design tokens & themes

Files App.axaml (token brushes) · Theme/Studio.axaml · Theme/StudioThemes.cs · ThemeManager Risk Medium (foundation) Mode dark-only · 3 accents Rule every value is a token

A token layer already exists — VST-005 re-tones it, it does not build it from scratch. App.axaml defines the full Vx* token set as Color + {token}Brush resources (VxBg, VxPanel, VxAccent, VxHalo, the VxStageVad/Stt/Agent/Tts/Out palette, VxGood/Warn/Bad, VxUi = Inter, VxMono = Cascadia); StudioThemes.cs + ThemeManager.Apply already switch themes at runtime by mutating those shared brush instances (every view binds the same brush via StaticResource, so a mutation repaints live). The five stage colors already equal the prototype values (#76849B / #4FC3F7 / #CE93D8 / #FFB74D / #66BB6A) and are correctly never themed — keep them. So WS1 is three concrete changes against that existing layer (below), not a green-field token system. Keep the Vx* resource keys (no app-wide rename); the §6 tables give the canonical hex each key must hold. Mapping: ink-950→VxBg, ink-900→VxPanel, ink-850→VxPanel2, pulse-400→VxAccent, pulse-300→VxAccent2, pulse-500→VxAccentPressed, halo-400→VxHalo, stage-*→VxStage*.

Headline change — rebrand warm → cool. Today's Studio ships the warm "Claude-coral" palette as its base/default (VxBg #1C1C19, VxAccent #C96442, warm inks/text; the default theme is Warm). The prototype — the 1:1 target — is the cool "Voxa-cyan" Pulse brand (ink-950 #0B0F14, pulse-400 #4FC3F7). VST-005's single most visible change is making the cool/cyan palette the base and default. The values below are the prototype's; the existing "Cool" theme already carries most of them, so this is largely "promote Cool to the base, re-tone Warm/Slate to accent-only variants of it." Concretely: replace App.axaml's warm base palette with the cool/Ink values, change StudioPreferences.ThemeId's default from "warm" to "cool", and flip StudioThemes.ById's fallback from Warm to Cool.

Behavior change. Today the three themes (Warm default / Cool / Slate) each re-tint the whole chrome — ink scale, text, and hairlines all shift per theme (StudioThemes.cs). The prototype narrows this: only the pulse-* ramp + the cyan glows re-tint; ink, text, lines, and the five stage colors are identical in every theme, and Cool is the default. VST-005 adopts the prototype's model — promote the cool base to all themes and reduce Warm/Slate to accent-only variants (warm accent #E27A4E, slate #8AA2CE; the current warm coral #C96442 is retired with the rebrand). Persist the choice in StudioPreferences (the prototype uses localStorage 'voxa.theme') and apply at boot via the existing ThemeManager.

Implementation note. Express the accent ramp as a small swappable ResourceDictionary (one per theme) merged over the base token dictionary, so a theme switch is a single dictionary swap at runtime — the Avalonia equivalent of the prototype's data-voxa-theme attribute. Stage colors and the neutral ramp live in the base dictionary and never change.

11.WS6 — Dialogs & toasts

Files Services/IDialogService.cs · Services/IToastService.cs (+ impls) · Controls/ToastHost Risk Low API imperative, tone-driven

The prototype's VoxaUI imperative API becomes a pair of injected services so views (and view models, via the interface) raise consistent modals/notifications:

// Avalonia-free interfaces — view models depend on these, never on Avalonia.
public interface IDialogService {
    Task<bool> ConfirmAsync(DialogOptions o);   // tone: confirm (cyan)
    Task<bool> DangerAsync(DialogOptions o);    // red destructive
    Task<bool> WarnAsync(DialogOptions o);
    Task AlertAsync(DialogOptions o);           // one button
    Task<string?> PromptAsync(PromptOptions o); // text input, null = cancelled
}
public interface IToastService {
    IDisposable Show(ToastOptions o);           // tone: info/success/warning/danger; auto-dismiss ~3.6s
}

Modal (460 wide): tone icon + soft-bg square, title, body, optional mono detail or text input, Cancel/Confirm (Confirm = danger red for destructive). Esc cancels, Enter confirms. Toasts: bottom-right stack, left border in the tone color, icon + title + optional (mono) message, auto-dismiss. The six tones map to the brief's icon+color set (confirm/info/success/warning/danger/error). The Avalonia impls live in the view layer; the interfaces keep view models clean (§5).

12.WS7 — Motion

Apply the four motion tokens consistently: fast 120 (hover/press color shifts), standard 240 (view / card / panel transitions, palette collapse), draw 1100 (splash mark, first chart paint), pulse 2600 ease-in-out infinite — the only idle loops (live status dot, logo glow, orb ripple). Everything honors the OS reduced-motion setting and the Appearance "Idle animations" toggle: when off, no infinite loops run and draw-ons jump to end state. No bespoke easings beyond the two token curves.

One logical change per PR (CONTRIBUTING). Each phase is one or more PRs with a CHANGELOG entry; this spec lands before any app code. Phases A→C are the load-bearing foundation; D→F can interleave once the library exists.

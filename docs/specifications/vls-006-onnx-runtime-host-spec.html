VLS-006 — Shared ONNX Runtime Model Host

One reusable, device-aware ONNX-Runtime host (Voxa.Audio.Onnx) so any HuggingFace ONNX model plugs into Voxa with uniform session management, execution-provider/device selection, and pinned-catalog resolution. The .NET analogue of speech-core's OnnxEngine: today Voxa runs ONNX in one-offs (SileroVadEngine, KokoroTtsEngine each roll their own InferenceSession), so every new model re-derives the plumbing. The default package stays CPU-only and byte-identical; GPU is opt-in per VLS-002's "natives are user-added, never bundled" rule.

1.What is this? — a plain-English explainer

What does VLS-006 add, in one sentence?

One shared, device-aware ONNX-Runtime host so any HuggingFace ONNX model plugs into Voxa with the same session, device, and model-cache plumbing — instead of every model re-inventing it.

Why now?

Voxa already runs ONNX models in-process, but as one-offs: SileroVadEngine and KokoroTtsEngine each construct and manage their own InferenceSession, pick their own SessionOptions, and wire their own caching. There is no shared host and no uniform device/EP selection. The moment we want to add a HuggingFace ONNX model (Parakeet, Nemotron, DeepFilterNet3, Sidon, Pyannote, WeSpeaker…), we re-derive all of that plumbing again. VLS-006 writes it once — the foundation the whole ONNX tier builds on.

What does this not change?

The existing engines keep working exactly as they do today. Adopting the host is an additive follow-up, never forced — SileroVadEngine and KokoroTtsEngine are not rewritten by this item. VLS-006 ships a host plus a tiny proof model; the real models are VLS-004 / 005 / 007 / 008.

Does it require a GPU or make the package heavier?

No. Exactly like VLS-002's Whisper GPU seam, the host defaults to Device = cpu and the base package references only the CPU ONNX-Runtime. GPU is opt-in: a user who wants CUDA / DirectML adds the matching runtime package to their own app and sets Device = cuda. The default build stays byte-identical; CI stays CPU-deterministic.

2.Executive summary

VLS-006 introduces a new Voxa.Audio.Onnx package — three workstreams, all foundation:

  • WS1 — the session host P6. An OnnxModelHost / IOnnxSession that owns a process-wide cached ORT InferenceSession (keyed by model path + device, exactly how Kokoro caches today), exposes a SessionOptions hook (the SessionOptionsHook analogue) for custom providers, and provides tensor-buffer / run helpers. Thread-safe construction, explicit disposal semantics.
  • WS2 — the device / EP seam P1. A Device enum (Cpu default | Cuda | DirectML | CoreML | Auto) reusing VLS-002's pattern; the chosen execution provider is appended to SessionOptions before session creation. GPU natives are user-added, never bundled.
  • WS3 — pinned-model + descriptor integration additive. A model is one VoxaModelArtifact (+ sidecar files — vocab / tokenizer / config) resolved, verified, and extracted by VoxaModelCache; an OnnxModelDescriptor shape so an ONNX model self-describes (id, files, EP support) like the existing provider descriptors.
Net effect. Adding a HuggingFace ONNX model becomes pin it + a thin wrapper, not bespoke runtime code. VLS-007 (Parakeet / Nemotron ASR) and VLS-008 (Sidon) build directly on this host; VLS-004 (DeepFilterNet3), VLS-005 (Pyannote / WeSpeaker), and VRT-005 (PersonaPlex) adopt it as they land. One pinned ORT version, CPU default, GPU opt-in.

3.Goals, non-goals & budgets

Goals

  • One reusable ONNX-Runtime host any ONNX engine can build on — uniform session creation, device/EP selection, and cache resolution.
  • A process-wide session cache (model weights load once per host, shared across every connection) — the behaviour Kokoro already proves.
  • A device seam (Cpu default; Cuda / DirectML / CoreML / Auto opt-in) generalising VLS-002 beyond Whisper.
  • A SessionOptions hook so a model (or a host) can install a custom EP / tune options without forking the host.
  • Pinned-catalog (VoxaModelArtifact + VoxaModelCache) integration, including sidecar files (vocab / tokenizer / config).
  • A descriptor shape so an ONNX model self-describes (id, files, EP support).

Non-goals (this spec)

  • Rewriting SileroVadEngine / KokoroTtsEngine onto the host now — additive follow-up, see §13.
  • Any specific model — Parakeet / Nemotron (VLS-007), DeepFilterNet3 (VLS-004), Sidon (VLS-008), Pyannote / WeSpeaker (VLS-005), PersonaPlex (VRT-005) are their own items.
  • Mel front-ends, tokenizers, RNNT/TDT decoders — model-specific pre/post-processing belongs to the consuming engine.
  • Bundling GPU runtime natives — opt-in via NuGet reference, by design (VLS-002 rule).
  • A LiteRT / non-ORT backend; NNAPI / QNN mobile EPs (those ride VDX-004's native story).

4.Architecture context — where this sits

VLS-006 is a new package — Voxa.Audio.Onnx — that ONNX engines (current and future) depend on. It never goes in Voxa.Core (zero-dependency). It references Microsoft.ML.OnnxRuntime (the CPU package, pinned at the same 1.26.0 as Voxa.Audio.SileroVad and Voxa.Speech.Kokoro — the comment in those csproj files, "Same version … so the meta-package carries ONE onnxruntime native," is now an invariant this host enforces) and Voxa.Speech.Abstractions (for VoxaModelArtifact / VoxaModelCache).

It is the .NET analogue of speech-core's OnnxEngine — an ORT singleton with EP selection (NNAPI / QNN / CPU fallback) and a SessionOptionsHook, shared by every model in that engine. Voxa instead re-implements session setup per engine today; this host ends that.

5.WS1 — the session host

Package Voxa.Audio.Onnx Risk Medium (foundation) Session cache process-wide, keyed (path, device) Default CPU, unchanged ORT version

The host owns the ORT InferenceSession lifecycle so consuming engines never construct one directly. Two pieces: OnnxModelHost (the factory + cache) and IOnnxSession (a thin, test-fakeable handle over a loaded session). The cache is process-wide and keyed by resolved model path + device — mirroring Kokoro's ConcurrentDictionary<string, Lazy<SharedSession>> and whisper.cpp's factory cache: weights load once and are shared by every connection on the host.

src/Voxa.Audio.Onnx/OnnxModelHost.cs (sketch — pin exact ORT API to the referenced Microsoft.ML.OnnxRuntime version) 
// One loaded ONNX model. Construction is the only allocation hot-path; inference is on the
// caller's thread (engines own their own buffer-reuse, as SileroVadEngine does today).
public interface IOnnxSession
{
    InferenceSession Session { get; }            // for engines that bind OrtValues directly (Silero pattern)
    IReadOnlyList<string> InputNames { get; }
    IReadOnlyList<string> OutputNames { get; }
    OnnxDevice ActiveDevice { get; }             // the EP that actually loaded (may differ from requested under Auto)
}

public sealed class OnnxModelHost
{
    // Process-wide: model weights load once per (path, device), shared across connections.
    // Same shape as KokoroTtsEngine.Sessions — deliberately NOT a per-engine field.
    private static readonly ConcurrentDictionary<(string Path, OnnxDevice Device), Lazy<IOnnxSession>>
        Sessions = new();

    private readonly ILogger _logger;

    public OnnxModelHost(ILogger? logger = null) => _logger = logger ?? NullLogger.Instance;

    // modelPath is already resolved+verified by VoxaModelCache (WS3). options lets a model
    // tune graph-optimization / threads; hook lets it (or a host) install a custom EP (WS2).
    public IOnnxSession Load(
        string modelPath,
        OnnxDevice device = OnnxDevice.Cpu,
        OnnxSessionOptionsHook? hook = null)
        => Sessions.GetOrAdd(
               (Path.GetFullPath(modelPath), device),
               key => new Lazy<IOnnxSession>(() => CreateSession(key.Path, key.Device, hook))).Value;

    private IOnnxSession CreateSession(string path, OnnxDevice device, OnnxSessionOptionsHook? hook)
    {
        var opts = new SessionOptions
        {
            GraphOptimizationLevel = GraphOptimizationLevel.ORT_ENABLE_ALL,  // matches Kokoro today
        };
        // WS2: append the execution provider for `device` BEFORE building the session,
        // then give the model/host the last word via the hook (the SessionOptionsHook analogue).
        var active = OnnxExecutionProvider.Apply(opts, device, _logger);
        hook?.Invoke(opts, path, device);

        var session = new InferenceSession(path, opts);
        _logger.LogInformation("Onnx host: loaded '{Model}' on {Device}",
            Path.GetFileName(path), active);
        return new LoadedOnnxSession(session, active);
    }
}

// The SessionOptions customization callback — the .NET twin of speech-core's
// SessionOptionsHook. Runs once per session, before CreateSession. A model installs a
// custom EP or tunes options here without the host needing to know about it.
public delegate void OnnxSessionOptionsHook(SessionOptions options, string modelPath, OnnxDevice device);

Tensor-buffer / run helpers

Two access tiers, because the existing engines already span both:

  • Direct (zero-alloc): expose Session so an engine can pre-bind OrtValues over reused managed buffers and call the no-allocation Run(RunOptions, names, inputs, names, outputs) overload — the SileroVadEngine steady-state pattern. The host does not get in the way of this.
  • Convenience: a small OnnxTensors helper for the common case (wrap a ReadOnlySpan<float> + shape into an OrtValue, run a named-input dictionary, copy the output span out) — the KokoroTtsEngine.RunInferenceAsync shape, lifted so each new model does not re-write it.

Thread-safety & disposal

Construction is thread-safe; inference is the caller's contract. The Lazy<IOnnxSession> guarantees a model loads exactly once even under concurrent Load calls. InferenceSession.Run is itself thread-safe, but a model with reused per-instance buffers (Silero's LSTM state) is not — that constraint stays with the engine, exactly as today. Disposal: cached sessions are process-lifetime by design (they hold weights shared across connections — Kokoro's DisposeAsync deliberately leaves the session in the cache). The host exposes an explicit EvictAll() for tests / Studio's "unload models," but a per-connection dispose must not tear down a shared session.

6.WS2 — the device / execution-provider seam

Reuses VLS-002's Device pattern Default cpu (unchanged) Natives user-supplied, never bundled Explicit-GPU-unavailable fail at start

The OnnxDevice enum

Generalises VLS-002's WhisperDevice beyond Whisper. Parsed from a config string (WS3 / §8):

ValueMeaning
cpu (default)CPU EP, as the existing engines run today. Deterministic; what CI uses.
autoPrefer the best available GPU EP, fall back to CPU with a warning log (never a hard failure).
cudaRequire the CUDA EP. Unavailable → fail at StartAsync with remediation (don't silently run slow on CPU).
directmlRequire the DirectML EP (cross-vendor on Windows: NVIDIA / AMD / Intel).
coremlApple Silicon — require the CoreML EP. Best-effort; see note.

How the EP is selected

ONNX-Runtime appends execution providers to a SessionOptions in priority order before the session is built; the first EP that can claim a node runs it, the rest fall through to CPU. The host applies the EP for the requested OnnxDevice via SessionOptionsAppendExecutionProvider_* before constructing the (cached) session, then logs which EP actually loaded so the choice is observable — the same "log the loaded backend" discipline VLS-002 mandates and speech-core's has_gpu_provider() exposes.

src/Voxa.Audio.Onnx/OnnxExecutionProvider.cs (sketch — exact EP-append API per ORT version) 
// Applied to a fresh SessionOptions before InferenceSession construction. The CUDA/DirectML/
// CoreML appenders are extension methods provided by the matching Microsoft.ML.OnnxRuntime.*
// package the USER added — they are not compiled into the base host (see "Packaging" below).
static OnnxDevice Apply(SessionOptions opts, OnnxDevice device, ILogger log)
{
    switch (device)
    {
        case OnnxDevice.Cpu:
            return OnnxDevice.Cpu;                      // CPU EP is implicit; nothing to append

        case OnnxDevice.Cuda:
            opts.AppendExecutionProvider_CUDA();        // throws if the CUDA runtime isn't present
            return OnnxDevice.Cuda;

        case OnnxDevice.DirectML:
            opts.AppendExecutionProvider_DML();
            return OnnxDevice.DirectML;

        case OnnxDevice.CoreML:
            opts.AppendExecutionProvider_CoreML();
            return OnnxDevice.CoreML;

        case OnnxDevice.Auto:
            // Try GPU EPs in turn; swallow "provider unavailable" and fall back to CPU + warn.
            foreach (var ep in new[] { OnnxDevice.Cuda, OnnxDevice.DirectML, OnnxDevice.CoreML })
                if (TryAppend(opts, ep)) return ep;
            log.LogWarning("Onnx host: Device=auto found no GPU execution provider; using CPU.");
            return OnnxDevice.Cpu;

        default:
            return OnnxDevice.Cpu;
    }
}
// On an explicit (non-auto, non-cpu) device whose append/CreateSession fails, the engine's
// StartAsync surfaces a VoxaModelUnavailable-style error:
//   "Voxa:<Section>:Device=cuda but no CUDA execution provider loaded. Add the
//    Microsoft.ML.OnnxRuntime.Gpu (CUDA) package to your app and ensure the CUDA toolkit/driver
//    is installed, or use Device=cpu."

Packaging — the load-bearing decision (the VLS-002 / d2a11ae rule, generalised)

GPU natives are opt-in and never bundled by Voxa. The base Voxa.Audio.Onnx references only the CPU Microsoft.ML.OnnxRuntime package (version 1.26.0, pinned identically to SileroVad / Kokoro), with PrivateAssets so it is not re-flowed onto consumers — the exact concern behind commit d2a11ae that VLS-002 codified for Whisper. To enable GPU, the consuming app adds the matching ORT package itself: 
<PackageReference Include="Microsoft.ML.OnnxRuntime.Gpu" Version="1.26.0" />   <!-- in the app, not in Voxa -->
This keeps the default build byte-identical, sidesteps multi-GB GPU natives in our package, and makes the GPU matrix the user's (and ORT's) concern, not ours.
The ORT CPU/GPU in-process conflict is real and must stay opt-in. This is the exact issue VLS-002 flagged when it deferred Kokoro GPU: the ORT CPU and GPU NuGet packages provide the same managed assembly and native entry points, so referencing both in one process conflicts. Consequently: one pinned ORT version across every Voxa ONNX model (Silero, Kokoro, and every VLS-006 consumer), and GPU is enabled by the user replacing the runtime package in their app — never by Voxa shipping a second ORT. The host does not try to load two runtimes; a wrong/missing runtime surfaces as the clear StartAsync error above.

CoreML / Auto notes

CoreML is best-effort, like VLS-002's. Some models need extra provisioning or only partially offload to the CoreML EP; coreml is wired as a value, but per-model CoreML caveats ride the consuming engine. Validation rejects an unknown Device string at startup (keyless, no side effects — the same shape as Whisper's Device / Kokoro's OutputSampleRate checks); it does not probe for runtime-package presence (unreliable pre-load).

7.WS3 — pinned-model + descriptor integration

Reuses VoxaModelArtifact + VoxaModelCache Sidecars vocab / tokenizer / config Surface additive (self-describing descriptor)

A model resolves through the unchanged VoxaModelCache path: each file is one VoxaModelArtifact(Id, DownloadUrl, Sha256, SizeBytes), resolved / verified (SHA-256, atomic rename, cross-process lock) / extracted exactly as Kokoro and whisper.cpp do. The only new wrinkle is that an ONNX model is frequently more than one file: the graph plus sidecars (a tokenizer JSON, a vocab file, a model config, sometimes a separate encoder/decoder graph). So an ONNX model self-describes as a small bundle of pinned artifacts.

src/Voxa.Audio.Onnx/OnnxModelDescriptor.cs (sketch) 
// An ONNX model self-describes like the existing provider descriptors (VoxaSttDescriptor etc.):
// a stable id, the pinned files it needs, and which execution providers it supports.
public sealed record OnnxModelDescriptor(
    string Id,                                   // e.g. "parakeet-tdt-v3-0.6b"
    VoxaModelArtifact Graph,                     // the .onnx graph (or the first of several)
    IReadOnlyList<VoxaModelArtifact> Sidecars,    // tokenizer / vocab / config / extra graphs — each SHA-256-pinned
    IReadOnlyList<OnnxDevice> SupportedDevices);  // e.g. [Cpu, Cuda] — drives validation + the Auto fallback

// Resolution is just VoxaModelCache, file by file — no new download machinery.
public static async Task<ResolvedOnnxModel> ResolveAsync(
    this OnnxModelDescriptor model, VoxaModelCache cache, CancellationToken ct)
{
    var graphPath = await cache.ResolveAsync(model.Graph, ct).ConfigureAwait(false);
    var sidecars = new Dictionary<string, string>();
    foreach (var s in model.Sidecars)
        sidecars[s.Id] = await cache.ResolveAsync(s, ct).ConfigureAwait(false);
    return new ResolvedOnnxModel(graphPath, sidecars);
}

Per-family catalogs live in the consuming package (a ParakeetCatalog, a SidonCatalog) following the KokoroCatalog / WhisperCppModelCatalog pattern — VLS-006 provides the shape (OnnxModelDescriptor + the resolve helper), not the model entries. Each consuming engine wires a normal VoxaSttDescriptor / VoxaTtsDescriptor / VAD descriptor (with Validate + WarmUpAsync) on top, capturing the same IConfiguration root that was passed to AddVoxa — never service-locating config (the config-capture invariant; CacheFor(sp, root) in KokoroDescriptors is the template).

Never fabricate a model hash. Every Sha256 in a consuming catalog must be sourced from the real artifact's HuggingFace LFS pointer (…/raw/main/<file> → oid sha256:<hash> + size <bytes>), exactly as the VLS-001 / VLS-002 entries were and as Kokoro's voice pins were re-sourced on 2026-06-12. A hand-written or guessed hash is a bug, not a placeholder. And — mirroring the Kokoro / GPL license-closure gate — never ship a model whose licence isn't cleared and recorded. VLS-006 itself ships no model catalog, so this rule binds every consumer (VLS-004 / 005 / 007 / 008, VRT-005).

8.Configuration reference

VLS-006 defines a convention, not a config section of its own (it has no model to configure). Each consuming engine exposes a Device key under its own section, parsed into OnnxDevice by a shared helper the host provides — so every ONNX model reads its device the same way:

appsettings.json — the shared Device convention (illustrated on a hypothetical VLS-007 consumer) 
{
  "Voxa": {
    "Stt": "OnnxAsr",
    "OnnxAsr": {
      "Model": "parakeet-tdt-v3",   // per-model key; resolved via the consumer's pinned catalog
      "ModelPath": "",              // optional explicit-path override → bypasses the cache (descriptor concern)
      "Device": "cpu"               // shared convention: cpu (default) | auto | cuda | directml | coreml
    }
  }
}

The Device string parses through one host-provided helper (OnnxDevice.Parse(string)), so the value set is uniform across every ONNX engine. Unknown values fail validation at startup with the known set listed — the same keyless, side-effect-free validation shape the existing descriptors use. ModelPath (explicit override) keeps the established semantics: a set path bypasses VoxaModelCache entirely.

9.Plan of action — phases

  1. Phase A — host + CPU path + a tiny proof model. OnnxModelHost / IOnnxSession, the process-wide session cache, the convenience tensor helpers, CPU-only EP handling, and a single tiny pinned ONNX model (MIT, if a small one is available) that the host loads and runs one inference against. Independently mergeable; proves the host end-to-end on CPU.
  2. Phase B — the device seam. OnnxDevice enum, OnnxExecutionProvider.Apply (CUDA / DirectML / CoreML / Auto), the explicit-unavailable error + auto-fallback warning, the SessionOptionsHook, Device-string parse/validate, and the packaging docs/PrivateAssets wiring. Depends on nothing in Phase A's runtime but lands second.
  3. Phase C — descriptor + cache integration. OnnxModelDescriptor (graph + sidecars + supported devices) and the ResolveAsync helper over VoxaModelCache; the documented descriptor/config convention the consuming engines (VLS-007 first) build on.

One logical change per PR (per CONTRIBUTING) — the three phases are three PRs. Each adds a [Unreleased] CHANGELOG entry and (Phase B/C) a docs/local-speech.md ONNX-host section. The spec lands before the code (spec-before-code invariant).

13.Out of scope / deferred

  • Refactoring SileroVadEngine / KokoroTtsEngine onto the host. An additive follow-up once the host has proven itself on a new consumer (VLS-007) — never a prerequisite, and only if behaviour-preserving (Silero's zero-alloc inference, Kokoro's shared session and 24 kHz contract must be held exactly).
  • Any specific model. Parakeet / Nemotron (VLS-007), DeepFilterNet3 (VLS-004), Sidon (VLS-008), Pyannote / WeSpeaker (VLS-005), PersonaPlex (VRT-005) — each owns its catalog, pre/post-processing, and license closure.
  • A LiteRT / non-ORT backend. The host is ORT-specific; an alternative runtime is a separate item if ever needed.
  • NNAPI / QNN mobile EPs. speech-core's OnnxEngine selects these for Android; in Voxa they ride VDX-004's native/mobile story, not this host.
  • Multi-session arena sharing / advanced ORT memory tuning (speech-core's use_env_allocators, device-allocator-for-initializers, BF16/INT8 optimizer gating). The SessionOptionsHook leaves the door open for a consumer that needs it; the base host does not prescribe it.

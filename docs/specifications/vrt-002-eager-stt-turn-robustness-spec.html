VRT-002 — Eager STT & Turn-Taking Robustness

Two workstreams that shave perceived turn latency and close a checklist of barge-in / turn edge cases on the granular cascade. WS1 adds eager (speculative) STT: when silence reaches a new EagerSttDelay (shorter than the gate's StopDuration), the buffered utterance is dispatched to STT before end-of-turn is confirmed, and discarded if the user resumes. WS2 adds five independently-tunable robustness knobs (interruption debounce + recovery, empty-STT recovery, force-split, response-duration cap). Both compose with the already-shipped smart-turn seam and the pre-roll buffer — neither is re-implemented here.

1.What is this? — a plain-English explainer

What does VRT-002 add, in one sentence?

It makes Voxa start transcribing the moment the user pauses instead of waiting for the full end-of-turn silence, and it adds a small set of safety knobs so coughs, echo, empty transcripts, and runaway speech can't make a conversation feel laggy or get the pipeline stuck.

Why now?

Voxa already has the two hard pieces — a smart-turn seam (SileroVadOptions.ConfirmTurnEnd, shipped in PR #26) that decides whether a turn ended, and pre-roll (PrerollDuration) that protects the first syllable. What's missing is the cheap latency win: even once you believe the turn is over, today the cascade waits the entire StopDuration hangover (800 ms by default) before STT runs. speech-core's AgentConfig shows ~0.3 s is recoverable by transcribing speculatively during that hangover. VRT-001 gives us the number to prove the win is real.

What is explicitly not changing?

The smart-turn classifier itself (already shipped — VRT-002 only acts faster on its verdict), pre-roll (already shipped — do not re-add it as a "gap"), the per-frame CancellationToken interruption model, the bounded drop-oldest backpressure convention, and acoustic echo cancellation (that's VRT-003). With every new knob at its default, the composed pipeline is byte-identical to today.

How does eager STT compose with the smart-turn seam rather than replace it?

They answer different questions. Smart-turn answers "is the turn actually over?"; eager answers "how fast do I act once I believe it is?". Eager fires a speculative transcription early; if smart-turn later returns false (a mid-sentence pause), that verdict overrides and discards the pending eager dispatch — its result is suppressed by utterance id at the STT processor so it can never become a turn (§5). The speculative work is bounded and thrown away on resume — never promoted to a turn unless end-of-turn is confirmed.

2.Executive summary

VRT-002 is two workstreams on the granular cascade (VAD → STT → TranscriptionFilter → agent → SentenceAggregator → TTS), each knob individually mergeable:

  • WS1 — Eager / speculative STT P0. A new EagerSttDelay on SileroVadOptions (and a matching tuning knob). On reaching it inside the unvoiced hangover — before StopDuration / ConfirmTurnEnd confirm — the VAD speculatively emits an end-of-utterance signal so STT (and optionally an LLM prefetch) can start on the buffered utterance. The speculative utterance is marked so resumed speech isn't mistaken for an interruption, and it is discarded on resume by an utterance-id suppression rule in SpeechToTextProcessor — a superseded speculative final is dropped before it becomes a TranscriptionFrame, so it can never start a turn even when a batch engine (whisper.cpp) ran the inference to completion. A smart-turn ConfirmTurnEnd → false supersedes a pending eager dispatch the same way. (Aborting the flush itself is a best-effort optimization, not the guarantee — see §5.)
  • WS2 — Robustness checklist P2. Five tunable knobs, each with a single anti-bug rationale: interruption debounce (MinInterruptionDuration), interruption recovery (InterruptionRecoveryTimeout), empty / low-confidence STT recovery, force-split (MaxUtteranceDuration), and response-duration cap (MaxResponseDuration). Each is testable in isolation with a synthetic VAD-probability sequence.
Net effect. Perceived turn latency drops by the hangover-minus-eager-delay window (≈300 ms at the proposed defaults), measured as a VRT-001 before/after delta rather than asserted; and the pipeline gains documented, defaulted-off defences against the barge-in / stuck-state edge cases that speech-core's AgentConfig already encodes. Everything is opt-in at the tuning layer; defaults keep today's behaviour byte-for-byte.

3.Goals, non-goals & budgets

Goals

  • Recover the STT-during-hangover latency window with a speculative dispatch that is safe — correct on resume, correct against smart-turn, never double-counts a turn.
  • Add five robustness knobs, each individually mergeable, each defaulting to today's behaviour, each covered by a deterministic synthetic-VAD headless test.
  • Keep all new state inside the existing processors — SileroVadProcessor (VAD-side knobs) and AgentLoopProcessor (turn-worker-side recovery + response cap) — plus the WebSocketAudioSink recovery interplay. No new project, no new abstraction.
  • Surface every knob through the same config → profile → settings → options path as the existing VAD knobs.
  • Make the net latency win an objective number via VRT-001; gate the change on that delta.

Non-goals (this spec)

  • The smart-turn classifier itself — shipped in PR #26 (ConfirmTurnEnd + Voxa.Audio.SmartTurn). VRT-002 only acts on its verdict faster.
  • Pre-roll / speech-pad — shipped (PrerollDuration, 300 ms). Do not re-add.
  • Acoustic echo cancellation — that is VRT-003 (IEchoCanceller). VRT-002's interruption debounce filters echo residual heuristically; AEC is what makes barge-in robust on open speakers.
  • The turn-taking benchmark harness — that is VRT-001; VRT-002 consumes it as the acceptance signal.
  • LLM prefetch as a separate engine — WS1 permits an optional prefetch on the eager signal but ships the VAD-side dispatch + STT trigger; a dedicated prefetch path can land as a fast-follow.

5.WS1 — Eager / speculative STT

Package Voxa.Audio.SileroVad + Voxa.Core + Voxa.Speech.Abstractions Risk Medium (precedence / correctness; STT-contract change) Default off (EagerSttDelay = null) Model speech-core eager_stt / eager_stt_delay
Scope note — this is not "VAD additions only". Discarding a speculative transcription correctly requires a small contract change in Voxa.Speech.Abstractions (an utterance id on the flush + emitted result, and suppression logic in SpeechToTextProcessor) — see "Discarding a speculative transcription — the real mechanism". The per-frame CancellationToken that powers LLM/TTS interruption does not reach STT flush/inference for batch engines, so it cannot be the discard mechanism.

The window we're recovering

Today the gate, once _isSpeaking, only emits UserStoppedSpeakingFrame after a full StopDuration of accumulated silence (then optionally re-decided by ConfirmTurnEnd). That UserStoppedSpeakingFrame is what force-flushes the batched STT engine downstream. So even on a clean end-of-turn, STT does not begin until the entire hangover has elapsed.

Eager STT inserts an earlier, speculative trigger at a new EagerSttDelay (strictly less than StopDuration). On reaching it, the VAD emits an eager end-of-utterance signal — a marked frame — that lets STT start transcribing the already-buffered utterance immediately, overlapping the rest of the hangover. Modelled directly on speech-core's eager_stt / eager_stt_delay / discard-on-resume behaviour.

The state machine (additions only)

  1. Arm. While _isSpeaking and accumulating _unvoicedAccum: when it first crosses EagerSttDelay (and eager is enabled, and we haven't already dispatched for this hangover), emit the eager marker + a speculative UserStoppedSpeakingFrame equivalent so STT flushes. Keep _isSpeaking = true — the gate has not closed.
  2. Resume ⇒ discard. If a voiced window arrives before the hangover completes, the user resumed within the window. Mark the speculative utterance id superseded (the guarantee: SpeechToTextProcessor will drop a final tagged with that id before it becomes a TranscriptionFrame — even if whisper.cpp already ran the inference on its lifetime token); best-effort cancel the pending flush where the engine can; clear the eager-dispatched flag; and — because the utterance was marked — do not raise an InterruptionFrame: resumed speech inside an eager window is a continuation, not a barge-in. (Do not rely on the per-frame CancellationToken here — it governs LLM/TTS data work, not STT flush/inference; see §5 "the real mechanism".)
  3. Confirm ⇒ promote. If the hangover completes (and ConfirmTurnEnd, if wired, returns true), the turn is genuinely over: the already-running speculative transcription is the turn's transcription. Close the gate as today. No second STT pass.
  4. Smart-turn override. If ConfirmTurnEnd returns false at StopDuration (mid-sentence pause), it overrides and supersedes the pending eager dispatch — the same suppression path as resume (mark the utterance id superseded so its final is dropped at the processor; best-effort cancel the flush) — and the gate stays open for another StopDuration. Precedence is explicit and one-directional: smart-turn false > eager dispatch.
Precedence is the load-bearing correctness rule. Eager dispatch is a bet that the turn ended; smart-turn is the authority on whether it did. Therefore: (a) eager may only fire when EagerSttDelay < StopDuration (validate this — a misconfiguration where eager ≥ stop is meaningless and must be rejected or clamped); (b) a ConfirmTurnEnd → false always supersedes a pending eager pass; (c) a resumed voiced window always supersedes a pending eager pass. "Supersedes" is the guarantee — the pass's utterance id is marked superseded and SpeechToTextProcessor drops its final before it can become a turn; aborting the flush mid-flight is a best-effort optimization on top (a native inference already running may not stop). The eager transcription is only ever promoted to the real turn when end-of-turn is confirmed.

Discarding a speculative transcription — the real mechanism

The naïve reading is "cancel the speculative STT with the per-frame CancellationToken, the same machinery that aborts an interrupted LLM/TTS call." That does not work, and the ground truth in the codebase is specific:

  • ISpeechToTextEngine.FlushAsync() takes no CancellationToken (Task FlushAsync() => Task.CompletedTask;) — there is nothing for the per-frame token to bind to.
  • WhisperCppSttEngine.FlushAsync() only calls QueueTranscriptionLocked() and returns immediately — it does not await inference. QueueTranscriptionLocked() chains the transcription Task.Run on var ct = _lifetime.Token; — the engine-lifetime token, not any per-frame / interruption token. The result is written to an unbounded channel and surfaced by ReadTranscriptsAsync as a TranscriptionResult(IsFinal:true). (whisper.cpp inference is a CPU-bound native call — not reliably cancellable mid-flight anyway.)
  • SpeechToTextProcessor.ReadLoopAsync forwards every TranscriptionResult as a TranscriptionFrame unconditionally — it has no notion of which utterance a result belongs to, so as written it cannot drop a superseded speculative result.

Net: the per-frame CancellationToken (the interruption machinery for LLM/TTS) does not govern STT flush/transcription. Relying on it to discard a speculative transcription is architecturally wrong for batch engines (WhisperCpp). The correct design has two parts — and only the first is a guarantee:

1 — Result suppression via an utterance id / epoch (the guarantee)

Each speculative flush is tagged with an utterance id (or a monotonically increasing epoch). An eager coordinator in/around SpeechToTextProcessor tracks the currently-valid utterance id. On resume-within-window or ConfirmTurnEnd → false, the speculative id is marked superseded. The processor's read loop drops any TranscriptionResult whose utterance id is superseded before it becomes a TranscriptionFrame — so a stale speculative final never reaches the agent, even though whisper.cpp ran the inference to completion on its lifetime token. This is a contract-level change: an utterance id is carried from the flush through to the emitted result (or, equivalently, the processor stamps each in-flight flush with the active id and validates results against it) — it is not a VAD-only tweak.

src/Voxa.Speech.Abstractions/SpeechToTextProcessor.cs (read-loop suppression — intent) 
// The processor tracks the active/valid utterance id; a superseded speculative flush is recorded.
// long _activeUtteranceId;  HashSet<long> _superseded;   // updated when VAD marks a spec id stale

await foreach (var t in _engine.ReadTranscriptsAsync(ct).ConfigureAwait(false))
{
    // Drop a stale speculative final BEFORE it becomes a TranscriptionFrame / starts a turn.
    // t.UtteranceId is the id stamped at FlushAsync time (new TranscriptionResult field).
    if (t.IsFinal && t.UtteranceId is { } id && _superseded.Contains(id))
        continue; // suppressed: the user resumed, or ConfirmTurnEnd returned false

    await PushFrameAsync(new TranscriptionFrame(t.Text, t.IsFinal, t.Language), ct).ConfigureAwait(false);
}

2 — Cancellable / identified flush (best-effort optimization)

Extend the flush contract so engines that can abort cheaply do so and avoid wasted compute — e.g. FlushAsync(CancellationToken) or FlushAsync(utteranceId) plus a CancelPending(utteranceId). Streaming engines map it to cancel_stream; WhisperCpp can drop a queued-but-not-yet-started transcription. This is explicitly best-effort — a native inference already running may not stop — which is exactly why suppression (#1), not cancellation, is the guarantee.

Contract change in Voxa.Speech.Abstractions. This subsection is the reason WS1 touches ISpeechToTextEngine (flush signature + an utterance id on the flush/result) and SpeechToTextProcessor (the suppression logic), not just the VAD. Keep the change minimal: one id field threaded flush → result, one drop-if-superseded check in the read loop, an optional best-effort cancel hook.

Marking the speculative utterance

The downstream sink (WebSocketAudioSink) treats a UserStartedSpeakingFrame arriving as a data frame as a barge-in (it bumps the epoch). A naive speculative-stop followed by resumed speech would therefore look like an interruption. The fix is to mark the eager utterance so resumed speech inside the window is recognised as continuation:

src/Voxa.Core/Frames/SystemFrames.cs (one of the two — pick at implementation) 
// Option A — a dedicated marker frame (preferred: explicit, no field churn on existing frames).
// Emitted by the VAD at EagerSttDelay; carries the utterance id used to flush + (later) suppress;
// tells STT to flush speculatively and tells the sink
// "a UserStarted following this is a continuation, not a barge-in".
public sealed record SpeculativeUtteranceFrame(long UtteranceId) : SystemFrame;

// Option B — a flag + id on the stop frame. UserStoppedSpeakingFrame is a record, so this is additive
// and defaults to the non-speculative case (Eager == false ⇒ today's behaviour).
// public sealed record UserStoppedSpeakingFrame(bool Eager = false, long UtteranceId = 0) : SystemFrame;

/// Speculative ("eager") STT trigger. When set and strictly less than StopDuration, the gate
/// emits a marked speculative end-of-utterance once unvoiced silence reaches this delay — so STT
/// starts on the buffered utterance BEFORE StopDuration / ConfirmTurnEnd confirm end-of-turn.
/// Discarded if the user resumes within the window, or if ConfirmTurnEnd returns false: both mark
/// the speculative utterance id superseded so SpeechToTextProcessor drops its final before it
/// becomes a TranscriptionFrame (the per-frame CancellationToken does NOT reach STT flush/inference;

src/Voxa.Audio.SileroVad/SileroVadProcessor.cs (additions inside the hangover branch — intent, not final code) 
// Fields: bool _eagerDispatched; long _eagerUtteranceId;

// While speaking and unvoiced is accumulating, BEFORE the StopDuration branch:
if (_isSpeaking && !voiced
    && _options.EagerSttDelay is { } eager
    && eager < _options.StopDuration
    && !_eagerDispatched
    && _unvoicedAccum >= eager)
{
    _eagerDispatched = true;
    _eagerUtteranceId = NextUtteranceId();
    // Marked, speculative: STT flushes for THIS utterance id; the sink reads the marker and will
    // NOT treat a following UserStarted as a barge-in. Gate stays OPEN (_isSpeaking unchanged).
    await PushFrameAsync(new SpeculativeUtteranceFrame(_eagerUtteranceId), ct).ConfigureAwait(false);
}

// On the next VOICED window while a speculative pass is pending → resume ⇒ discard:
if (voiced && _eagerDispatched)
{
    _eagerDispatched = false;
    // SUPPRESS (the guarantee): mark _eagerUtteranceId superseded so SpeechToTextProcessor drops
    // its final before it becomes a TranscriptionFrame — even though whisper.cpp ran on its lifetime
    // token. Best-effort cancel the flush where the engine can. NOT the per-frame CancellationToken.
    // No InterruptionFrame: the marker said "continuation, not barge-in".
}

// In the existing StopDuration branch, when ConfirmTurnEnd returns FALSE:
//   smart-turn overrides eager → mark _eagerUtteranceId superseded (same suppression path as resume),
//   reset _eagerDispatched, keep the gate open (mid-sentence pause), as today.
// When the gate genuinely closes (confirmed=true): the speculative transcription IS the turn — its
//   id stays valid (never superseded); clear _eagerDispatched, emit the normal UserStoppedSpeakingFrame,
//   do not re-run STT.
Why this is safe against the existing machinery. Discard is decided by utterance-id suppression at SpeechToTextProcessor (the guarantee), not by the per-frame CancellationToken — that token drives LLM/TTS data work and does not reach STT flush/inference, so it is deliberately not the discard path here (best-effort flush-cancel is layered on top). The gate-open invariant during the eager window means pre-roll and the smart-turn recent-speech tail (SnapshotRecentSpeech) keep accumulating exactly as before; eager does not touch either.

6.WS2 — Turn-taking robustness checklist

Package Voxa.Audio.SileroVad + Voxa.Core + Voxa.Transports.WebSocket Risk Low–Med (each knob isolated) Default every knob off / today's value Model speech-core AgentConfig

Five independent knobs. Each is a separate PR, each ships with one synthetic-VAD headless test, each defaults to current behaviour. They divide by owner: three live on the VAD (speech-timing), two on the agent loop / sink (turn / playback state).

6.1 — MinInterruptionDuration (interruption debounce)

Knob: SileroVadOptions.MinInterruptionDuration (TimeSpan?). Anti-bug: a single cough, lip-smack, or residual echo of the bot's own audio leaking into the mic produces a brief voiced blip that — during bot playback — fires an InterruptionFrame and kills the bot mid-sentence. Rule: while the bot is speaking, require at least MinInterruptionDuration of confirmed, sustained user speech before emitting the interruption. This is the same accumulate-then-fire shape as the existing StartDuration gate, but specifically gating the barge-in path. Default null ⇒ interrupt on first confirmed speech (today's behaviour).

Heuristic, not AEC. Debounce filters short residual; it does not cancel echo. On open speakers, true robustness needs VRT-003's IEchoCanceller in front of the VAD. Debounce and AEC are complementary: debounce trims what slips past, AEC removes the bulk.

6.2 — InterruptionRecoveryTimeout (interruption recovery)

Knob: SileroVadOptions.InterruptionRecoveryTimeout (TimeSpan?). Anti-bug: the user starts to interrupt, then stops almost immediately ("uh— no, go on"). Today that brief barge-in has already purged the bot's queued audio (epoch bump in WebSocketAudioSink) and torn down the turn — the user has to wait for a full re-synthesis. Rule: if the user stops within InterruptionRecoveryTimeout of the interruption, treat it as a false barge-in and resume the existing playback instead of reprocessing. Sink interplay: the epoch purge in WebSocketAudioSink.WriteLoopAsync drops audio tagged with a stale epoch; recovery must either defer the epoch bump until the barge-in is confirmed past the recovery window, or restore the prior epoch so the still-in- flight bot audio is not stranded. Default null ⇒ no recovery (an interruption always reprocesses).

Touch the epoch carefully. The existing purge is correctness-critical (it's how barge-in stops bot audio "almost immediately"). Recovery must not re-deliver audio the client already discarded. The safe design is a short provisional phase: hold the epoch bump (or buffer the purge decision) for InterruptionRecoveryTimeout, commit it only if the barge-in persists. Cover both directions (recovers / does not recover) with tests.

6.3 — Empty / low-confidence STT recovery

Owner: AgentLoopProcessor turn worker. Anti-bug: the VAD fires UserStoppedSpeakingFrame, STT runs, and returns nothing (silence misfire, a noise burst, a Whisper hallucination the filter drops) — or a sub-threshold-confidence result. Today ProcessFrameAsync only enqueues a turn for a non-empty final transcription (!string.IsNullOrWhiteSpace(t.Text)), so the agent does not run; but any latched per-turn / bot- speaking state set on the expectation of a turn can be left dangling, and the pipeline waits forever for a turn that will never come. speech-core calls this out explicitly: without an empty-STT reset, the pipeline gets stuck. Rule: on an empty or sub-threshold final transcription, explicitly reset turn state and the bot-speaking flags (release whatever was latched in anticipation of the turn) so the next UserStartedSpeakingFrame is processed normally. This is a no-op when STT always returns text — hence default-safe.

6.4 — MaxUtteranceDuration (force-split)

Knob: SileroVadOptions.MaxUtteranceDuration (TimeSpan?). Anti-bug: a user who never pauses (reading a list, dictating, a stuck-open mic) keeps the gate open indefinitely. The VAD buffers the whole utterance; STT never flushes; memory grows; the agent never gets a turn. Rule: cap the open-gate duration at MaxUtteranceDuration. On reaching it while still _isSpeaking, force-emit a UserStoppedSpeakingFrame (flushing STT for an intermediate transcription), then immediately re-arm as a fresh utterance so capture continues. This bounds memory and yields periodic intermediate transcriptions for very long speech. Default null ⇒ no cap (today's behaviour).

Don't over-split. Too low a cap chops natural sentences and produces half-utterance transcriptions the agent must stitch. Default off; the Quality profile should leave it generous or unset. Validate it is comfortably larger than StartDuration + a typical sentence.

6.5 — MaxResponseDuration (response-duration cap)

Owner: AgentLoopProcessor turn worker (driver enumeration). Anti-bug: a hallucinating or looping LLM streams text without end, or TTS synthesises a runaway response — the bot talks over the user's next several turns and burns tokens / audio. Rule: bound the wall-clock (or emitted-audio) duration of a single turn's output at MaxResponseDuration; on exceeding it, stop pumping the driver's yielded frames for that turn and close the turn cleanly (LlmTurnEndedFrame still fires in the existing finally). Because the turn worker already isolates each turn in try/catch and owns the per-turn stopwatch, this is a bounded check on the existing await foreach (var frame in _driver.RunTurnAsync(...)) loop. Default null ⇒ no cap.

Reuse, don't reinvent. The turn worker already starts a Stopwatch per turn and aggregates a TurnSummary. The cap is a comparison against that stopwatch plus a graceful break — it must still emit LlmTurnEndedFrame and run OnTurnCompleted/cleanup so downstream state is consistent. It is not an exception; a capped turn is a normal (truncated) completion.

8.Plan of action — phases

WS1 first (the latency win and the measurement loop), then the WS2 knobs — each WS2 knob individually mergeable. One logical change per PR (per CONTRIBUTING); every PR adds an [Unreleased] CHANGELOG entry, stays warnings-as-errors clean, and follows the spec (this document) which lands first.

  1. Phase A — WS1 eager STT. Marker frame (carrying an utterance id) + EagerSttDelay option + VAD state machine (arm / resume-supersede / confirm-promote / smart-turn-override) + the Voxa.Speech.Abstractions contract change (utterance id on flush/result + SpeechToTextProcessor suppression, optional best-effort flush-cancel) + tuning wiring. Lands with its synthetic-VAD tests, the suppression test (T1.5), and a VRT-001 before/after delta in the PR description. Single PR (the eager mechanism is one indivisible behaviour — the suppression contract is what makes it correct, so it ships together).
  2. Phase B1 — MinInterruptionDuration. Barge-in debounce on the VAD. One PR.
  3. Phase B2 — InterruptionRecoveryTimeout + sink interplay. Provisional-epoch recovery in WebSocketAudioSink. One PR (highest sink risk — keep it isolated).
  4. Phase B3 — Empty / low-confidence STT recovery. Turn-state reset in AgentLoopProcessor. One PR (no config knob).
  5. Phase B4 — MaxUtteranceDuration force-split. VAD open-gate cap. One PR.
  6. Phase B5 — MaxResponseDuration cap. Turn-worker output bound. One PR.

Phases B1–B5 are mutually independent and may be parallelised or reordered; only Phase A must precede them (it introduces the marker frame + the STT-abstractions suppression contract and establishes the VRT-001 measurement habit).

12.Out of scope / deferred

  • Pre-roll / speech-pad. Shipped (PrerollDuration, 300 ms). Not re-implemented.
  • Acoustic echo cancellation. VRT-003 — the IEchoCanceller seam inserted before the VAD. VRT-002's interruption debounce is a heuristic complement, not a substitute.
  • The turn-taking benchmark itself. VRT-001 (Full-Duplex-Bench harness). VRT-002 consumes it as the acceptance signal; it does not build it.
  • A dedicated LLM-prefetch engine. WS1 permits an optional prefetch on the eager signal but ships the VAD dispatch + STT trigger; a first-class prefetch path (warm the agent on the speculative transcript) is a natural fast-follow once the eager delta is proven by VRT-001.
  • Streaming / interim transcripts. VRT-004 — interim TranscriptionFrame(IsFinal:false) propagation. Independent of eager's final speculative dispatch.

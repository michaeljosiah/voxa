VST-003 — Voxa Studio Settings Dialog

A first-class Settings dialog for Voxa Studio: a tabbed modal that replaces the current ad-hoc credential fields scattered across the Config screen with a single, well-structured place to manage which providers are active and what credentials they need. API keys are persisted to a DPAPI-encrypted file so they survive restarts. The Providers tab lists every activated provider (STT, TTS, Agent) with status indicators, lets the user add from a card-grid picker of all supported providers, and exposes per-field configuration with masked secret inputs. Config dropdowns filter to activated providers only.

1. What is this? — a plain-English explainer

What is the feature, in one sentence?

A modal Settings dialog — opened from a gear icon in the nav rail — where you manage which voice and agent providers are active in your Studio and enter their credentials once, with those credentials encrypted on disk and automatically available across restarts, so you never re-type an API key after closing the app.

What is wrong today?

Credentials in Voxa Studio are ephemeral. The Config screen has a text box for the OpenAI agent key; it goes into the live container when you hit Apply and evaporates when Studio exits. Cloud TTS providers (ElevenLabs, Mistral, Azure) have no key input in Studio at all — you must set environment variables or put keys in appsettings.json before launching. The Config dropdowns list every provider the registry knows about regardless of whether you have credentials for it, so a fresh Studio shows "ElevenLabs", "Mistral" and "Azure" in the TTS picker even if you've never signed up for any of them. There is nowhere to discover which providers Studio supports or how to activate one.

What does the Settings dialog look like?

A fixed-size modal window with a tab strip. The initial release ships one tab: Providers. The tab has a master-detail layout: a left panel lists your activated providers with a coloured status dot (green = configured, amber = key missing, grey = local / no credentials needed); the right panel shows the configuration fields for whichever provider is selected. A "+ Add provider" button at the bottom of the list opens a card-grid flyout of every supported provider not yet in your list — you click a card and it's added instantly. Save commits all changes to disk; Cancel discards them.

One key, three roles — what does activating "OpenAI" actually do?

Some providers wear several hats. In Voxa, OpenAI is a single identity that supplies speech-to-text (Whisper), text-to-speech, and the chat agent — all reading the same Voxa:OpenAI:ApiKey. So when you add OpenAI in Settings and type one key, it lights up in the STT dropdown, the TTS dropdown, and the Agent dropdown at once. Mistral behaves the same way for STT + TTS. The manifest models this with a roles list, so the dialog shows "OpenAI · STT · TTS · Agent" as one row, not three.

Where do API keys actually live after I type them?

On Windows they are written to a DPAPI-encrypted binary file (~/voxa-secrets.dpapi). DPAPI (Data Protection API) is the same mechanism Windows uses for browser saved passwords and Wi-Fi keys — the ciphertext is tied to your Windows user account and can only be decrypted by the same user on the same machine. The file is never exported, never included in any appsettings.json bundle, and never travels over the network. On Studio startup the file is decrypted in memory; the secrets are injected into the live configuration container as a dedicated layer (see §4.5.1) so they survive even when you later "Apply" a Config draft.

What is a "supported provider" and who decides the list?

Each provider is described by a static ProviderManifest record in a compile-time catalog. The manifest records the provider's display name, which roles it fills (STT, TTS, Agent), a one-line description, a docs link, and a typed list of fields (label, placeholder, whether to mask the value). The catalog is the authoritative list of what the Add-provider picker shows. Adding a new provider to Voxa means adding a registry descriptor and a manifest entry — both are required before Studio can surface it to users.

What happens to local providers like Piper and WhisperCpp?

They appear in the provider list as read-only entries with a grey dot labelled "Local — no configuration required." They are always active; you cannot deactivate them. This makes the provider list a complete inventory of what Studio can do, not just a key-management screen. It also gives users a clear answer to "why isn't Kokoro in my Config dropdown?" — they see it in Settings and understand it is already available without credentials.

How does this interact with the Config screen?

The Config screen's STT, TTS, and Agent dropdowns currently list every provider in the registry. After VST-003 lands, they will filter to only providers that are either local (always on) or have been explicitly activated by the user via Settings. A provider that is activated but has no key yet still appears in Config (with a warning), so the user knows they need to return to Settings to finish the setup. The credential fields currently embedded in Config (AgentApiKey, etc.) are candidates for removal in a follow-up once Settings is established as the canonical home — that removal is deliberately out of scope for this spec.

Separation of concerns. The Settings dialog manages which provider identities are active and what credentials they hold. VST-003 begins the clean separation; removing the credential fields from Config is a follow-up once Settings is established (see §11).

3. Goals, non-goals & constraints

3.1 Goals

  • G1. A user types an API key once. It survives Studio restarts; the next session starts with that provider ready without any further action.
  • G2. The set of providers visible in Config dropdowns always matches what the user has actually activated — no phantom providers cluttering the picker.
  • G3. The Add-provider picker shows every supported identity the user has not yet added, with enough description to make an informed choice without leaving Studio.
  • G4. Local providers (no credentials needed) are always visible in the provider list and always appear in Config dropdowns — they cannot be "accidentally removed".
  • G5. Activating an identity that fills several roles (OpenAI: STT+TTS+Agent; Mistral: STT+TTS) makes it usable in every matching dropdown from a single key entry.
  • G6. The secrets store is never read by any code path that exports configuration to disk or to the clipboard. DPAPI decryption happens only at startup and on Settings Save; the decrypted bytes go only into the in-memory IConfiguration container.
  • G7. A Config "Apply" (which replaces the override layer) never drops the credentials loaded from the secrets store — secrets live in their own layer (§4.5.1).
  • G8. All view-models are Avalonia-free and headless-testable. The dialog's UI is the only Avalonia dependency.
  • G9. The "Studio never touches the network before the user acts" invariant is preserved. ProviderSecretsService loads from disk only; no HTTP call is made during startup or dialog construction.

3.2 Non-goals

  • Not a secrets vault with sharing, import/export, or team sync. DPAPI-encrypted local file only.
  • Not removing existing credential fields from ConfigViewModel — that is a deliberate follow-up (§11).
  • Not cross-platform DPAPI. On Linux/macOS Studio is not the primary target; DpapiSecretsStore is Windows-only. Tests use MemorySecretsStore. A future spec may add a Keychain/libsecret backend.
  • Not managing the VAD engine. Silero/SilenceGate are local and chosen in Config; VAD carries no credential, so it has no place in Settings.
  • Not an "Audio" settings tab in this spec. The tab strip reserves space; the tab is not built.
  • Not provider discovery at runtime (no fetching a remote list of supported providers). The manifest catalog is compile-time.
  • Not provider-level model selection (e.g. which Whisper/Kokoro model, agent chat model) — that remains in the Config screen where it is a pipeline decision, not a credential decision.

3.3 Hard rules (invariants carried from CLAUDE.md / VST history)

  • Studio uses a plain ServiceCollection, not the Generic Host. IHostedService must not be registered; eager network warm-up must not happen.
  • View-model async methods that mutate UI-bound ObservableCollections must not use .ConfigureAwait(false).
  • The config-capture rule applies to any new service that needs configuration: capture the IConfiguration or IConfigurationSection at construction time; never call GetRequiredService<IConfiguration>().
  • DPAPI plaintext must not outlive its use: the JSON string produced by decryption is consumed into the dictionary and not retained; avoid logging any secret value.

4.2 ISecretsStore — encrypted credential storage

The abstraction is intentionally narrow — a keyed string dictionary with an explicit save gate:

apps/Voxa.Studio/Services/ISecretsStore.cs
/// <summary>
/// Per-key encrypted credential storage. Save() is explicit — callers control when the
/// plaintext values hit disk (encrypted). Get() reads from the in-memory snapshot loaded
/// at construction. Holds non-secret field values too (e.g. Azure Region); IsSecret on the
/// manifest field governs UI masking, not what is stored.
/// </summary>
public interface ISecretsStore
{
    string? Get(string key);
    void Set(string key, string? value);   // null removes the entry
    IReadOnlyCollection<string> Keys { get; }
    void Save();
    void Reload();                          // re-read disk → snapshot (backs Discard)
}
Package dependency. System.Security.Cryptography.ProtectedData is not in the .NET 10 BCL — it ships as a NuGet package. WS1 adds <PackageReference Include="System.Security.Cryptography.ProtectedData" Version="10.0.0" /> to apps/Voxa.Studio/Voxa.Studio.csproj (pin to the current patched release matching the SDK). The type is referenced only from DpapiSecretsStore, which is gated by OperatingSystem.IsWindows() and decorated [SupportedOSPlatform("windows")].

DpapiSecretsStore (Windows-only, production):

  • File: ~/voxa-secrets.dpapi — a DPAPI-protected binary blob.
  • On load: if the file is absent, start with an empty dictionary. If present, call ProtectedData.Unprotect(File.ReadAllBytes(path), null, DataProtectionScope.CurrentUser), then JsonSerializer.Deserialize<Dictionary<string,string>> on the result. A corrupt file (bad DPAPI padding — e.g. file from a different user account) is treated as empty with a logged warning; it is never re-written unless the user explicitly saves.
  • On Save(): serialize the in-memory dictionary to JSON → DPAPI-protect → File.WriteAllBytes (temp file + File.Move(overwrite:true) for atomicity).
  • Keys are namespaced by identity and field: "ElevenLabs:ApiKey", "Mistral:ApiKey", "OpenAI:ApiKey", "Azure:SubscriptionKey", "Azure:Region" — matching the manifest's identity name + field name, not the full Voxa:X:Y config keys (the service maps store-key → ConfigKey).

MemorySecretsStore: a plain Dictionary<string,string>; Save() and Reload() against an in-memory snapshot. Used in all tests and on Linux CI.

StudioServices selects the implementation when no store is injected:

SecretsStore = injectedStore
    ?? (OperatingSystem.IsWindows()
        ? new DpapiSecretsStore(Path.Combine(homeDir, "voxa-secrets.dpapi"))
        : new MemorySecretsStore());

4.3 ProviderManifest — typed provider descriptors

A manifest describes a provider identity — a single registry name + config section that may fill several roles. OpenAI is one manifest with Roles = [Stt, Tts, Agent]; Mistral is one manifest with Roles = [Stt, Tts].

public enum ProviderRole { Stt, Tts, Agent }

public sealed record ProviderFieldDescriptor(
    string Name,           // store-key suffix, e.g. "ApiKey", "SubscriptionKey", "Region"
    string Label,          // UI label, e.g. "API Key"
    string? Placeholder,   // hint text, e.g. "sk-..."
    bool IsSecret,         // true → mask in UI (storage is encrypted regardless)
    string ConfigKey);     // the Voxa:X:Y key written into the secrets layer

public sealed record ProviderManifest(
    string Name,                                // matches registry name exactly (OpenAI, Mistral, …)
    string DisplayName,
    ProviderRole[] Roles,                       // one identity may fill several
    string Description,
    bool IsLocal,                               // no credentials required
    string? DocsUrl,
    IReadOnlyList<ProviderFieldDescriptor> Fields);

public static class ProviderManifestCatalog
{
    public static IReadOnlyList<ProviderManifest> All { get; }
    public static ProviderManifest? Find(string name);
}

The full manifest table is in §6. The catalog is a static initializer — no DI, no network, no async. ProviderManifestCatalog.All is populated at module load time.

Shared keys are a feature, not a bug. OpenAI STT, OpenAI TTS, and the OpenAI chat agent all read Voxa:OpenAI:ApiKey (the agent falls back to it from Voxa:Agent:ApiKey, per DefaultAgentFactory.ResolveApiKey). The OpenAI manifest therefore has a single ApiKey field writing one config key — entering it once satisfies all three roles. The agent's chat model (Voxa:Agent:Model) is a pipeline choice and stays in Config, not Settings.

4.4 ProviderActivationStore — which providers are on

A plain JSON file (~/voxa-activations.json) containing an array of activation records. Non-sensitive; no encryption. Same corrupt-skip discipline as VoiceStore and a constructor override as the test seam (no config key).

public sealed record ProviderActivation(string Name, DateTimeOffset ActivatedAt);

public sealed class ProviderActivationStore
{
    public ProviderActivationStore(string? path = null);   // test seam
    public IReadOnlyList<ProviderActivation> Load();
    public void Save(IEnumerable<ProviderActivation> activations);
}

On first run the file is absent; Load() returns an empty list and the user adds providers via the Settings dialog. A corrupt file is treated as empty (logged warning).

4.5 ProviderSecretsService — unified facade

public sealed class ProviderSecretsService
{
    public ProviderSecretsService(ISecretsStore secrets, ProviderActivationStore activations);

    // Activated identities in the order they were added. Local identities are always
    // present (the service supplies them), regardless of the activation file.
    public IReadOnlyList<ProviderManifest> Activated { get; }

    public ProviderStatus StatusOf(ProviderManifest manifest);

    public string? GetSecret(string providerName, string fieldName);
    public void SetSecret(string providerName, string fieldName, string? value);

    public void Activate(string providerName);     // throws for local identities
    public void Deactivate(string providerName);    // throws for local; clears that identity's secrets

    public void Save();      // commit both stores. Called by SettingsViewModel.Save()
    public void Discard();   // reload both stores from disk. Called by SettingsViewModel.Cancel()

    // Flat Voxa:X:Y pairs for the StudioServices secrets layer. Only entries with a non-empty
    // value are included. Never includes a file path or the DPAPI blob location.
    public Dictionary<string, string?> BuildConfigPairs();
}

ProviderStatus:

public enum ProviderStatus
{
    Local,       // always available; no credentials; grey dot
    Configured,  // all required secret fields are non-empty; green dot
    KeyMissing,  // at least one required secret field is empty; amber dot
}

BuildConfigPairs() maps each activated identity's fields to their ConfigKey. For OpenAI it emits a single Voxa:OpenAI:ApiKey; for Azure it emits both Voxa:AzureSpeech:SubscriptionKey and Voxa:AzureSpeech:Region. Local identities contribute nothing.

4.5.1 Composing secrets with Config drafts (the layering rule)

The bug this rule prevents. StudioServices.Reconfigure documents that "overrides REPLACE the previous override set". If startup loaded the secrets as the override set, the very first Config "Apply" — which calls Reconfigure(DraftPairs()) — would replace them and the live container would lose every stored key mid-session. Secrets must therefore be a separate layer, not overrides.

StudioServices.Build grows to three layers, lowest-priority first:

apps/Voxa.Studio/Services/StudioServices.cs (revised Build + new members)
private IReadOnlyDictionary<string, string?> _secrets = new Dictionary<string, string?>();
private IReadOnlyDictionary<string, string?> _overrides = new Dictionary<string, string?>();

public ISecretsStore SecretsStore { get; }
public ProviderSecretsService Secrets { get; }

public StudioServices(
    IConfiguration? configuration = null,
    IStudioAudioDevice? audioDevice = null,
    ISecretsStore? secretsStore = null,          // test seam — inject a MemorySecretsStore
    ProviderActivationStore? activationStore = null)
{
    _baseConfiguration = configuration ?? /* appsettings + env, as today */;
    AudioDevice = audioDevice ?? StudioAudioDevice.CreatePlatformDefault();

    SecretsStore = secretsStore ?? (OperatingSystem.IsWindows()
        ? new DpapiSecretsStore(Path.Combine(HomeDir, "voxa-secrets.dpapi"))
        : new MemorySecretsStore());
    Secrets = new ProviderSecretsService(SecretsStore, activationStore ?? new ProviderActivationStore());

    _secrets = Secrets.BuildConfigPairs();        // live from the FIRST build — no double build
    (Configuration, Provider, ModelCache) = Build(_baseConfiguration, _secrets, _overrides);
}

/// Settings Save: swap the secrets layer, KEEP the current Config overrides, rebuild.
public void ApplySecrets(IReadOnlyDictionary<string, string?> secrets)
{
    var old = Provider;
    _secrets = new Dictionary<string, string?>(secrets);
    (Configuration, Provider, ModelCache) = Build(_baseConfiguration, _secrets, _overrides);
    old.Dispose();
    Reconfigured?.Invoke();
}

/// Config Apply: swap the overrides layer, KEEP the secrets, rebuild. (existing method, +secrets arg)
public void Reconfigure(IReadOnlyDictionary<string, string?> overrides)
{
    var old = Provider;
    _overrides = new Dictionary<string, string?>(overrides);
    (Configuration, Provider, ModelCache) = Build(_baseConfiguration, _secrets, _overrides);
    old.Dispose();
    Reconfigured?.Invoke();
}

private static (IConfiguration, ServiceProvider, VoxaModelCache) Build(
    IConfiguration baseConfiguration,
    IReadOnlyDictionary<string, string?> secrets,
    IReadOnlyDictionary<string, string?> overrides)
{
    var configuration = new ConfigurationBuilder()
        .AddConfiguration(baseConfiguration)                                       // appsettings + env
        .AddInMemoryCollection(secrets.Where(p => !string.IsNullOrEmpty(p.Value)))  // DPAPI store
        .AddInMemoryCollection(overrides.Where(p => !string.IsNullOrEmpty(p.Value)))// Config draft (wins)
        .Build();
    // …services.AddVoxa(configuration), cache, etc. exactly as today.
}

Order matters: a key explicitly typed into a Config draft (e.g. Voxa:Agent:ApiKey) still wins over the stored secret, which is the intuitive "this session, use this" behaviour. Everything else falls back to the secrets layer, which falls back to appsettings/environment.

Local identities never appear in the flyout — they are always present in the list and cannot be added/removed.

4.8 Integration points with the existing shell

  • Nav rail restructure: today the rail is a single top-aligned StackPanel of nav RadioButtons (MainWindow.axaml). To dock the gear at the bottom, the rail's Border content becomes a DockPanel: the existing nav radios stay in a StackPanel (default fill) and a new plain Button (x:Name="NavSettings") is DockPanel.Dock="Bottom" with a gear PathIcon. It is not a nav RadioButton — it opens a dialog, it does not select a section, so it has no GroupName="nav" and does not participate in OnNavChanged / SelectedSection.
  • Showing the dialog (code-behind): ShowDialog needs the owner Window, which lives in MainWindow.axaml.cs (the same place the session clock lives — a View concern). The gear's Click handler constructs new SettingsDialog(new SettingsViewModel(vm.Services.Secrets)), awaits ShowDialog(this), and on SettingsViewModel.Saved calls back into the VM: 
    private async void OnSettingsClick(object? s, RoutedEventArgs e)
{
    if (DataContext is not MainWindowViewModel vm) return;
    var settings = new SettingsViewModel(vm.Services.Secrets);
    await new SettingsDialog { DataContext = settings }.ShowDialog(this);
    if (settings.Saved) vm.OnSettingsSaved();
}
  • MainWindowViewModel.OnSettingsSaved() applies the new credentials to the live container and refreshes dependents: 
    public void OnSettingsSaved()
{
    Services.ApplySecrets(Services.Secrets.BuildConfigPairs()); // secrets layer, not overrides
    Config.RefreshProviderLists();   // dropdowns reflect new activations
    Voices.Refresh();                // catalog can now see the new provider keys
}
    (The titlebar is IsHitTestVisible="False", so the gear cannot live there — the nav rail is the only viable home.)
  • Config dropdown filter: ConfigViewModel's SttProviders, TtsProviders, and AgentProviders lists are computed by intersecting the registry names (and the hardcoded agent pair) with the set of activated-or-local identities whose roles match the dropdown. Re-computed at construction and from OnSettingsSaved.
  • No new nav section: Settings is a dialog, not a nav section. No section index changes, so the existing OnNavChanged / code-behind switch (Talk 0 … Config 6) is untouched.

Config dropdown filter

Today ConfigViewModel builds SttProviders/TtsProviders from services.Registry.SttNames/TtsNames (ordered) and hardcodes AgentProviders = ["Echo", "OpenAI"]. WS6 adds public void RefreshProviderLists() that rebuilds all three by keeping a name only if its manifest is local, or its identity is activated, AND the manifest fills the matching role:

bool Visible(string name, ProviderRole role)
{
    var m = ProviderManifestCatalog.Find(name);
    if (m is null) return true;                 // unmanifested (e.g. a future provider) — don't hide
    if (!m.Roles.Contains(role)) return false;
    return m.IsLocal || activatedNames.Contains(name);
}

Called at construction (initial lists already filtered) and from OnSettingsSaved. If the current SelectedStt/SelectedTts/SelectedAgent falls out of the filtered list, it resets to the first item — the same fallback used today for an absent config value.

One activation, three dropdowns. Because the OpenAI manifest fills Stt, Tts and Agent, activating OpenAI makes it appear in all three Config pickers at once. Activating Mistral makes it appear in STT and TTS. This is the visible payoff of identity-based manifests (§4.3).

A Config "Apply" wipes the DPAPI-loaded keys (the layering bug) Was certain in v1.0 High Secrets are a dedicated configuration layer between base and overrides; Reconfigure only swaps overrides (§4.5.1). WS3-A2 is the regression guard.
ProtectedData NuGet package not added → build fails or DpapiSecretsStore won't compile Medium Medium WS1 adds the package ref explicitly; the checklist tracks it; the type is referenced only behind the Windows guard.
DPAPI blob readable by another Windows user on a shared machine Low High DataProtectionScope.CurrentUser ties ciphertext to the account; documented in the store's doc comment.
DPAPI blob unreadable after profile migration / roaming profile change Low Medium Failed decrypt → empty store (G6-safe); user re-enters keys. A detail-panel note ("Credentials cleared — please re-enter") surfaces it.
[SupportedOSPlatform] analyzer (CA1416) errors under -warnaserror on the ubuntu lane Medium Low All DpapiSecretsStore construction sits behind OperatingSystem.IsWindows() guards the analyzer recognizes; the platform attribute decorates the class.
Manifest Name drifts from a renamed registry name Medium Medium WS2-A3 cross-checks every non-local manifest against the live registry — a rename that misses the manifest fails the suite.
Config dropdown selected value invalid after a removal Medium Low RefreshProviderLists() resets the selection to the first filtered item — the same fallback used for an absent config value today.
Off-UI-thread ObservableCollection mutation Low Medium The Settings VMs have no background refresh — all mutations are synchronous user actions. Enforced by review + headless tests if any async path is later added.

11. Out of scope / deferred

  • Removing credential fields from ConfigViewModel — the AgentApiKey box and related entry stay in Config for this spec. Removal is a follow-up once Settings is proven as the canonical home. (Until then both coexist: a Config-typed agent key is an override that wins over the stored one — §4.5.1.)
  • Audio tab — the tab strip reserves space; a future spec covers default capture/playback device selection.
  • Cross-platform secrets backend — macOS Keychain, Linux libsecret/kwallet. The ISecretsStore abstraction accepts them without changing higher layers; out of scope here.
  • Provider-level model selection (Whisper model, Kokoro precision, agent chat model) — pipeline decisions that belong in Config, not provider activation.
  • Secrets import/export / multi-machine sync — no mechanism to move the DPAPI blob. By design.
  • Provider health / "Test connection" — would make a live HTTP call, which conflicts with "no network before the user acts on a pipeline". An amber→green dot from key presence is sufficient signal; a Talk session fails fast with a clear error if the key is wrong.
  • VAD management — Silero/SilenceGate are local, credential-free, and chosen in Config (§3.2, §6).

VLS-008 — ONNX Speech Restoration (Sidon)

An offline denoise + dereverb restorer that turns a noisy, reverberant clip into clean 48 kHz mono audio. It runs on the VLS-006 shared ONNX host as a two-graph pipeline (a w2v-BERT predictor feeding a DAC vocoder) behind a SeamlessM4T-style log-mel front-end. Its first concrete consumer is a voxa restore <in.wav> <out.wav> CLI verb; the strategic consumer is the voice-cloning wizard (VVL-002 / Studio), which cleans a reference clip before cloning. This is not a live-pipeline processor — that boundary belongs to VLS-004's denoiser.

1.What is this? — a plain-English explainer

What does VLS-008 add, in one sentence?

An offline "clean this clip" tool: hand it a noisy or echoey WAV and it returns clean 48 kHz mono audio, denoised and de-reverberated by the Sidon model.

Why now?

It is the natural pre-step for voice cloning. Cloning a voice from a reference clip works far better when the reference is clean, so before VVL-002 / Studio hand a reference to the cloner, Sidon cleans it first. Shipping the restorer plus a voxa restore verb now makes the model testable on its own, ahead of the cloning UI.

What is explicitly not changing?

The live pipeline. Sidon is whole-clip and offline — it is not inserted into the mic → VAD → STT chain. It changes both sample rate (any-rate in → 48 kHz out) and length, so it cannot be a fixed-rate, equal-length live processor. Nothing about real-time turn-taking, VAD, or the composed pipeline moves.

How is this different from VLS-004?

VLS-004 is the live denoiser — an IAudioEnhancer placed before the VAD that cleans mic audio frame-by-frame in real time (DeepFilterNet3). VLS-008 is offline restoration of artifacts: a whole clip in, a whole clip out, off the hot path. Same family (ONNX on the VLS-006 host), opposite use: VLS-004 cleans a live stream; VLS-008 cleans a file.

2.Executive summary

VLS-008 ports speech-core's OnnxSidonRestorer onto Voxa's VLS-006 ONNX host and exposes it through the existing voxa CLI:

  • WS1 — the restorer P6. A SidonRestorer on the VLS-006 host: a SeamlessM4T-style log-mel front-end → a w2v-BERT predictor graph → a DAC vocoder graph, taking any-rate mono PCM and returning 48 kHz mono PCM. Two ONNX artifacts (predictor + vocoder) and any config sidecar are SHA-256-pinned in a catalog and resolved through VoxaModelCache.
  • WS2 — the voxa restore verb additive. Mirrors the existing voxa transcribe / voxa say verbs: voxa restore <in.wav> <out.wav> reads any-rate PCM WAV and writes 48 kHz mono — cross-platform, file-in/file-out, no audio device. The first concrete consumer, which makes the model testable without VVL-002.
  • WS3 — consumer wiring (deferred). The cloning wizard (VVL-002 / Studio) calls the restorer to clean a reference clip before cloning; an optional IAudioRestorer seam others can call. Specced here as a note, built with VVL-002.
Net effect. A developer can run voxa restore noisy.wav clean.wav and get a 48 kHz cleaned-up WAV; VVL-002 / Studio reuse the same restorer to clean a clone reference before handing it to the voice-cloner — all offline, all on the shared ONNX host, with nothing touching the live pipeline.

3.Goals, non-goals & budgets

Goals

  • An offline Sidon restorer on the VLS-006 host: any-rate mono PCM in → clean 48 kHz mono PCM out.
  • Both ONNX graphs (predictor + vocoder) and any sidecar config SHA-256-pinned in a catalog, resolved via VoxaModelCache the standard way.
  • A voxa restore CLI verb as the first concrete consumer — file-in/file-out, cross-platform, no audio device, unit-testable arg-parsing.
  • An optional IAudioRestorer seam shape so VVL-002 / Studio can call the restorer without a CLI round-trip (the seam is defined here; the consumer is wired in VVL-002).

Non-goals (this spec)

  • Real-time / live-pipeline restoration. Sidon is whole-clip and offline; it does not enter the mic → … → VAD chain. The live denoiser is VLS-004.
  • The Studio cloning-wizard UI hook — that rides VVL-002 (§7, §13).
  • Batch / folder restoration (one file per invocation; §13).
  • A LiteRT / non-ONNX backend — the DAC vocoder does not legalise to TFLite (it is ONNX-only upstream).

4.Architecture context — Sidon on the VLS-006 host

Sidon is two ONNX graphs orchestrated around a DSP front-end, exactly as modelled by speech-core's OnnxSidonRestorer (header: speech_core/models/onnx_sidon_restorer.h) and its speech_sidon_restore CLI. Both graphs run through the VLS-006 shared ONNX-Runtime host (Voxa.Audio.Onnx), inheriting its session caching and CPU-default / opt-in-EP device seam. The data flow:

audio (any rate, mono) ──[resample to 16 kHz]──►
  ──[SeamlessM4T log-mel front-end (managed DSP)]──► input_features [1, T, 160]
    ──► predictor graph  (w2v-BERT 2.0, 8 layers + merged LoRA)
       ──► features [1, T, 1024]
          ──► vocoder graph (DAC decoder, ×960 upsample)
             ──► audio (48 kHz, mono) [1, M]

The predictor's 8-layer last_hidden_state is the cleansed feature (there is no separate head); the DAC vocoder reconstructs 48 kHz audio from it. The whole operation changes both sample rate (16 kHz internal → 48 kHz out) and length, which is precisely why it cannot be a fixed-rate, equal-length live enhancer (VLS-004's contract) and is exposed as a Restore(...) that returns a freshly-sized 48 kHz buffer.

5.WS1 — the restorer on the VLS-006 host

Package Voxa.Audio.Onnx (Sidon) Host VLS-006 shared ONNX host Mode offline / whole-clip Output 48 kHz mono (model constant)

The restorer loads two pinned ONNX graphs through the VLS-006 host, runs the managed log-mel front-end, then the predictor, then the vocoder, and returns 48 kHz mono PCM. Inference is CPU-bound native work, so it runs off any caller's loop (the Kokoro precedent: Task.Run, a session gate) — though for an offline, non-pipeline operation the latency budget is generous.

src/Voxa.Audio.Onnx/Sidon/SidonRestorer.cs (sketch — pin exact host/ORT API at implementation) 
// Offline denoise + dereverb. Whole-clip; NOT a live processor. Two graphs on the VLS-006 host.
public sealed class SidonRestorer : IAsyncDisposable
{
    public const int InputSampleRate  = 16000;  // w2v-BERT front-end requires 16 kHz
    public const int OutputSampleRate = 48000;  // DAC vocoder constant — always 48 kHz mono

    private const int InputFeatDim = 160;        // SeamlessM4T stacked log-mel
    private const int HiddenDim    = 1024;       // w2v-BERT last_hidden_state

    private readonly SidonOptions _options;
    private readonly VoxaModelCache _cache;
    private readonly IOnnxModelHost _host;       // VLS-006

    public SidonRestorer(SidonOptions options, VoxaModelCache cache, IOnnxModelHost host) { /* ... */ }

    public async Task StartAsync(CancellationToken ct)
    {
        // Resolve + verify both pinned graphs (and any sidecar config) via VoxaModelCache,
        // then ask the VLS-006 host for cached sessions on the configured device.
        var predictor = await _cache.ResolveAsync(SidonCatalog.Predictor, ct).ConfigureAwait(false);
        var vocoder   = await _cache.ResolveAsync(SidonCatalog.Vocoder,   ct).ConfigureAwait(false);
        _predictor = _host.GetSession(predictor, _options.Device);
        _vocoder   = _host.GetSession(vocoder,   _options.Device);
    }

    /// Restore a whole clip: any-rate mono Float32 in [-1,1] -> 48 kHz mono PCM16 out.
    /// Returns empty if the clip is too short to yield a single frame.
    public async Task<byte[]> RestoreAsync(ReadOnlyMemory<float> audio, int sampleRate, CancellationToken ct)
    {
        var mono16k  = Resample(audio, sampleRate, InputSampleRate);     // any rate -> 16 kHz
        var features = SidonMelFrontEnd.LogMel(mono16k);                 // -> [1, T, 160]
        var hidden   = await RunAsync(_predictor, features, ct);         // -> [1, T, 1024]
        var wave48k  = await RunAsync(_vocoder,   hidden,   ct);         // -> [1, M] @ 48 kHz
        return ToPcm16(wave48k);
    }
}

Do not fabricate hashes; verify the licence. Every Sha256 and SizeBytes must be read at implementation time from each file's Hugging Face LFS pointer (GET huggingface.co/<repo>/raw/main/<file> → oid sha256:<hash> + size <bytes>) — a guessed hash is a bug, not a placeholder. And before any Sidon code merges, the licences of both the Sidon weights and the DAC vocoder must be cleared and recorded — the same license-closure gate Kokoro / GPL artifacts pass. No model ships whose licence isn't closed.

6.WS2 — the voxa restore CLI verb

Package Voxa.Cli (dotnet tool) Shape file-in / file-out Device none (no mic/speakers) Pattern mirrors transcribe / say

The verb is a direct peer of the existing voxa transcribe and voxa say verbs in VoxaCliRunner: a new restore case in the command switch, a RestoreAsync handler, the shared Parse / VoxaRoot / CacheFor helpers, and WavIO for I/O. It reads any-rate mono PCM WAV (the front-end resamples internally) and writes 48 kHz mono via WavIO.WritePcm16Async. Being file-in/file-out with no audio device, it is fully cross-platform and unit-testable.

src/Voxa.Cli/VoxaCliRunner.cs (additions — mirrors TranscribeAsync / SayAsync) 
// in the command switch, alongside "transcribe" / "say":
"restore" => await RestoreAsync(rest, output, error, ct).ConfigureAwait(false),

private static async Task<int> RestoreAsync(string[] args, TextWriter output, TextWriter error, CancellationToken ct)
{
    var (positional, options) = Parse(args);
    if (positional.Count < 2)
    {
        error.WriteLine("usage: voxa restore <in.wav> <out.wav> [--device cpu|cuda|directml]");
        return 1;
    }

    var inPath  = positional[0];
    var outPath = positional[1];
    if (!File.Exists(inPath))
    {
        error.WriteLine($"voxa: file not found: {inPath}");
        return 1;
    }

    var wav = WavIO.ReadPcm(inPath);
    if (wav.Format != 1 || wav.Channels != 1 || wav.Bits != 16)
    {
        // any sample RATE is fine (Sidon resamples to 16 kHz internally); require mono 16-bit PCM
        error.WriteLine("voxa: restore needs mono 16-bit PCM WAV (any sample rate). " +
            "Convert first, e.g.: ffmpeg -i input -ac 1 -c:a pcm_s16le out.wav");
        return 1;
    }

    var root = options.TryGetValue("device", out var dev)
        ? VoxaRoot(("Sidon:Device", dev))
        : VoxaRoot();

    await using var restorer = new SidonRestorer(SidonOptions.FromConfiguration(root), CacheFor(root), HostFor(root));
    await restorer.StartAsync(ct).ConfigureAwait(false);

    var samples = PcmToFloat(wav.Pcm);                                  // PCM16 -> Float32 [-1,1]
    var pcm48   = await restorer.RestoreAsync(samples, wav.SampleRate, ct).ConfigureAwait(false);
    if (pcm48.Length == 0)
    {
        error.WriteLine("voxa: clip too short to restore.");
        return 1;
    }

    await WavIO.WritePcm16Async(outPath, pcm48, SidonRestorer.OutputSampleRate, ct).ConfigureAwait(false);
    output.WriteLine($"Wrote {outPath} (sidon, {SidonRestorer.OutputSampleRate} Hz mono, " +
        $"{new FileInfo(outPath).Length} bytes).");
    return 0;
}

The top-level Usage string gains one block, matching the existing entries:

  voxa restore <in.wav> <out.wav> [--device cpu|cuda|directml]
      Offline denoise + dereverb (Sidon) — any-rate mono WAV in, 48 kHz mono WAV out.
Why the CLI is the first consumer. It makes the model exercisable end-to-end — download, resolve, run both graphs, write a WAV — without waiting for VVL-002's cloning wizard. That is what lets the Category=LocalModels round-trip test (§11) exist now.

7.WS3 — consumer wiring (VVL-002 / Studio)

Status deferred to VVL-002 Built here? no — note + seam shape only Consumer cloning wizard (Studio)

This workstream is documented, not built, in VLS-008. The intended shape:

  • The cloning pre-step. When VVL-002's voice-cloning wizard takes a reference clip, it calls the Sidon restorer to clean it (denoise + dereverb) before handing it to the cloner — exactly speech-core's stated "clean a reverberant clone reference" use case. The reference is whole-clip, so the offline restorer fits naturally; nothing about cloning touches the live pipeline.
  • An optional IAudioRestorer seam. So in-process callers (Studio) don't shell out to the CLI: Task<RestoredClip> RestoreAsync(ReadOnlyMemory<float> audio, int sampleRate, CancellationToken ct) returning 48 kHz mono. SidonRestorer is the reference implementation. The seam shape is defined here; the interface and its Studio wiring land with VVL-002 so the consumer and the contract ship together.
Deferred — VVL-002 owns delivery. VLS-008 ships the restorer and the CLI verb; VVL-002 (Voicebox-Parity program) owns on-device cloning and the wizard UI, and is the natural home for the "clean reference" step and any Studio surface. Building the wizard hook here would duplicate VVL-002 scope and couple this offline utility to UI work. See §13.

9.Plan of action — phases

  1. Phase A — the restorer + pinned model. SidonRestorer on the VLS-006 host, the log-mel front-end, SidonCatalog with real LFS hashes + a licence-closure record, and a Category=LocalModels round-trip test (any-rate in → 48 kHz mono out). Depends on VLS-006 being merged.
  2. Phase B — the voxa restore verb. The restore case + RestoreAsync in VoxaCliRunner, the Usage block, and a default-lane arg-parse / synthetic shape test. The first concrete consumer.
  3. Phase C — (later, with VVL-002) the consumer hook. The IAudioRestorer seam + the cloning wizard's "clean reference" step. Not in this spec's PRs — sequenced under VVL-002 (§7, §13).

One logical change per PR (per CONTRIBUTING): Phase A and Phase B are two PRs, each with an [Unreleased] CHANGELOG entry; this spec is authored before either lands.

13.Out of scope / deferred

  • Live / real-time restoration. Sidon is whole-clip and offline; it does not enter the mic → … → VAD chain. Live denoise is VLS-004's IAudioEnhancer.
  • The Studio cloning-wizard UI hook. Rides VVL-002 — the restorer and CLI verb ship here; the wizard "clean reference" step and any Studio surface land with VVL-002 (§7).
  • Batch / folder restoration. One file per voxa restore invocation; bulk processing is a follow-up if demand appears.
  • A LiteRT / non-ONNX backend. The DAC vocoder's ConvTranspose1d does not legalise to TFLite (per speech-core's export notes) — ONNX Runtime only.
  • The IAudioRestorer seam as a shipped interface. Its shape is sketched here; the interface + its consumer land together with VVL-002 so the contract and its caller ship as one change.

1.What is this? — a plain-English explainer

What does VRT-005 add, in one sentence?

A seam (ISpeechToSpeechSession) plus a composite processor for a single local model that takes the user's audio and directly produces the agent's audio (and text) — listening and speaking at the same time — in the same family as Voxa's existing cloud realtime composites (OpenAIRealtimeProcessor, AzureVoiceLiveProcessor).

Why now, when the model isn't here yet?

Strategic positioning. Open speech-to-speech (S2S) models — Moshi, PersonaPlex-class — are maturing fast. Voxa already has the composite shape proven against two cloud realtime APIs. Defining the seam now, against that proven shape, means that when an on-device S2S model becomes viable on commodity hardware it drops in as a third composite by config, with no architectural scramble. We are laying the rail, not driving the train.

What does this not change?

Nothing existing. The vendor-neutral granular chain (VAD → STT → agent → aggregation → TTS) is untouched. The two cloud composites are untouched. This is purely additive: a new abstraction and a new processor in a new package, both inert until a host opts in.

Where's the honest boundary?

The seam ships; the model is deferred. A PersonaPlex / Moshi-class model is gigabytes and frequently CUDA-only — that is a separate quality/feasibility spike with its own host (out-of-process or GPU, reusing VLS-002 / sidecar patterns). To prove the seam before the local model exists, a cloud S2S provider could be the first concrete ISpeechToSpeechSession. This spec is deliberately scoped to the contract and the composite, not the weights.

2.Executive summary

VRT-005 imports speech-core's FullDuplexSpeechInterface idea onto Voxa's frame model, expressed as a genuine third member of the existing realtime-composite family. It is two small, shippable workstreams plus one explicitly deferred one:

  • WS1 — the ISpeechToSpeechSession seam additive. A streaming abstraction (user audio in → an async stream of agent-audio chunks + text out), with SetVoice / SetSystemPrompt / ResetSession / Cancel and an output-sample-rate property — modelled on both speech-core's FullDuplexSpeechInterface and the surface the cloud realtime processors already expose. Lives in Voxa.Speech.Abstractions (never Voxa.Core).
  • WS2 — the SpeechToSpeechProcessor composite strategic. A FrameProcessor peer to OpenAIRealtimeProcessor that drives an ISpeechToSpeechSession, owns its own VAD / turn-taking (full-duplex), bypasses the granular chain, and emits the same lifecycle + diagnostics frames as the cloud composites so Studio and the diagnostics hub treat it identically. New package Voxa.Services.SpeechToSpeech.
  • WS3 — the model & host deferred. The concrete session implementation. Spike-gated; the first concrete impl may be a cloud S2S provider (to validate the seam) and the local model rides VLS-002 GPU / sidecar patterns. Out of scope for code in this spec.
Net effect. Once a concrete ISpeechToSpeechSession exists, a developer sets "Mode": "SpeechToSpeech" (or a provider name) and the pipeline swaps from the granular chain to a single full-duplex composite — emitting the same BotStartedSpeakingFrame / UserStartedSpeakingFrame / InterruptionFrame / audio-out frames as OpenAI Realtime, so every downstream consumer (Studio's Talk view, the diagnostics waterfall) works with zero changes.
This is exploratory and strategic. VRT-005 ships an interface and a processor that drives it. It does not ship a speech-to-speech model, weights, or a runtime for one. Every claim about a "local model" in this document refers to deferred, spike-gated WS3 work. The seam is the deliverable; do not overclaim a model that does not yet exist.

3.Goals, non-goals & budgets

Goals

  • Define ISpeechToSpeechSession as a true peer of the cloud realtime processors' surface — a full-duplex, streaming contract Voxa can target as open S2S models mature.
  • Ship SpeechToSpeechProcessor: a composite that drives the seam, owns its own VAD / turn-taking, and emits byte-for-byte the same lifecycle/diagnostics frames as the cloud composites — a third member, not a parallel design.
  • Wire config / registration so Voxa:Mode=SpeechToSpeech (or a provider name) selects the composite instead of the granular VAD→STT→…→TTS chain — the same way the cloud composites are mapped today.
  • Prove the whole path against a fake session (no model) so the seam and the composite are fully testable offline.

Non-goals (this spec)

  • The local S2S model / weights. A PersonaPlex / Moshi-class model is a separate spike — GB-scale, frequently GPU / out-of-process, delivered via the VLS-002 GPU patterns or the sidecar / child-process pattern. Not in this spec.
  • The first concrete ISpeechToSpeechSession implementation. Could be a cloud S2S provider (validates the seam first) or the local model — both are spike-gated follow-ups (§7).
  • Any change to the granular chain, the cloud composites, or their transports.
  • Voice cloning within the S2S model (frontier; rides VVL-002 if ever in scope) — see §13.

4.Architecture context — composite vs granular, where this slots

Voxa has two pipeline shapes (CLAUDE.md). VRT-005 is firmly in the first:

  • Composite — one processor wraps STT + LLM + TTS + VAD and runs full-duplex with its own turn-taking. Today: OpenAIRealtimeProcessor, AzureVoiceLiveProcessor. The composite bypasses the granular chain entirely — it does not sit in a VAD→STT→… arrangement; it is the whole middle of the pipeline.
  • Granular — the vendor-neutral chain (VAD→STT→agent→SentenceAggregator→TTS) assembled by DefaultVoicePipelineComposer. Untouched here.

The SpeechToSpeechProcessor is a third composite. Like the cloud ones, it accepts AudioRawFrame (and ToolCallResultFrame) inbound, runs the model's full-duplex loop, and pushes the agent's audio + text + lifecycle frames downstream. The pipeline around it is just Source → SpeechToSpeechProcessor → Sink.

How a composite is mapped (vs the granular composer)

This is the load-bearing wiring difference. The granular path uses app.MapVoxaVoice("/voice").UseDefaults(), which calls DefaultVoicePipelineComposer.Compose(...) to build the chain. A composite instead uses the explicit-builder overload and registers a single processor — exactly how the cloud composites are mapped in samples/Voxa.Samples.AspNetServer/Program.cs:

samples — how a cloud composite is mapped today (the pattern VRT-005 mirrors) 
// granular chain — UseDefaults() runs DefaultVoicePipelineComposer
app.MapVoxaVoice("/voice").UseDefaults();

// composite — a single processor IS the pipeline middle (bypasses the granular chain)
app.MapVoxaVoice("/voice/realtime", voice =>
    voice.UseProcessor(() => new OpenAIRealtimeProcessor(options)));

// VRT-005 adds a third member with the same mapping shape:
app.MapVoxaVoice("/voice/s2s", voice =>
    voice.UseProcessor(sp => new SpeechToSpeechProcessor(sessionFactory, options)));
The model we are importing. speech-core's FullDuplexSpeechInterface exposes a respond-stream that yields interleaved audio + text at the model's frame rate, plus set_voice, set_system_prompt, a KV-cache-backed session, and reset_session. Its per-chunk payload (FullDuplexChunk: agent audio, decoded text, an is-final marker) is the template for our chunk type. We map that surface onto Voxa idioms (IAsyncEnumerable, PCM ReadOnlyMemory<byte>, per-frame CancellationToken) and keep it shaped like the cloud realtime processors so it is a true third member.

5.WS1 — the ISpeechToSpeechSession seam

Package Voxa.Speech.Abstractions Risk Medium (contract drift) Network none Surface additive (new abstraction)

The seam captures speech-core's FullDuplexSpeechInterface in Voxa idioms, kept deliberately parallel to what the cloud realtime processors already do internally (send user PCM, read back assistant audio + text + speaking events). It is session-scoped (one per connection / turn context, like a KV-cache session) and streaming.

src/Voxa.Speech.Abstractions/ISpeechToSpeechSession.cs (sketch — exact shape pinned at implementation) 
// A single local (or cloud) full-duplex speech-to-speech session: user audio in,
// agent audio + text out, at the model's own frame rate. Models speech-core's
// FullDuplexSpeechInterface; shaped to match the cloud realtime processors so the
// driving composite is a genuine third member of that family.
public interface ISpeechToSpeechSession : IAsyncDisposable
{
    // Sample rate of the agent audio this session emits (e.g. 24000). The composite
    // advertises this on the session envelope, exactly like OutputSampleRate today.
    int OutputSampleRate { get; }

    // Push a chunk of user PCM into the model's full-duplex loop. Non-blocking;
    // agent output arrives out-of-band via RespondAsync. (Full-duplex: the user may
    // speak while the agent is speaking — the model handles overlap / barge-in.)
    ValueTask AppendUserAudioAsync(ReadOnlyMemory<byte> pcm, CancellationToken ct);

    // The agent's output stream: interleaved audio + text tokens at the model's frame
    // rate, mirroring FullDuplexChunk. Yields for the lifetime of the session; honours
    // the per-frame CancellationToken so an interruption aborts in-flight synthesis.
    IAsyncEnumerable<SpeechToSpeechChunk> RespondAsync(CancellationToken ct);

    // Preset selection (FullDuplexSpeechInterface.set_voice).
    ValueTask SetVoiceAsync(string voiceId, CancellationToken ct);

    // System prompt / persona (FullDuplexSpeechInterface.set_system_prompt).
    ValueTask SetSystemPromptAsync(string systemPrompt, CancellationToken ct);

    // Clear conversational state / KV-cache for a fresh turn (reset_session).
    ValueTask ResetSessionAsync(CancellationToken ct);

    // Abort the in-flight response (barge-in / interruption) without tearing down
    // the session. The composite calls this when it emits an InterruptionFrame.
    ValueTask CancelAsync(CancellationToken ct);
}

// One unit of agent output. Mirrors speech-core's FullDuplexChunk: agent audio PCM,
// optional decoded text token(s), and an is-final marker for the turn. Audio uses the
// session's OutputSampleRate; either field may be empty in a given chunk.
public readonly record struct SpeechToSpeechChunk(
    ReadOnlyMemory<byte> AudioPcm,
    string? Text,
    bool IsFinal);
Keep the seam shaped like the cloud composites. The single most important constraint on WS1 is that this contract reads as a third member of the realtime-composite family, not a parallel universe. Every method here has a counterpart in what OpenAIRealtimeProcessor does over its transport (append user audio, read assistant audio/text/events, cancel on barge-in). If the seam drifts — e.g. a callback model instead of an async stream, or a different cancellation story — the composite stops being a true peer. Pin the exact signatures against the cloud processors at implementation time.
Tools (deferred detail). The cloud composites round-trip tool calls as ToolCallRequestFrame / ToolCallResultFrame. Whether a given S2S model supports function-calling is model-specific; WS1 leaves a documented extension point (an optional tool channel on the session) but does not require it. The fake session used for tests does not exercise tools.

6.WS2 — the SpeechToSpeechProcessor composite

Package Voxa.Services.SpeechToSpeech (new) Risk Medium Peer of OpenAIRealtimeProcessor Frames identical to the cloud composites

A FrameProcessor built to the same blueprint as OpenAIRealtimeProcessor: a bounded DropOldest channel (256), a session created on OnStartAsync, a read loop draining the session's RespondAsync stream and pushing frames, and graceful teardown on OnEndAsync. The only substantive difference from the cloud composites is that the wire transport is replaced by an in-process (or sidecar) ISpeechToSpeechSession.

src/Voxa.Services.SpeechToSpeech/SpeechToSpeechProcessor.cs (sketch — mirrors OpenAIRealtimeProcessor) 
public sealed class SpeechToSpeechProcessor : FrameProcessor
{
    private readonly Func<ISpeechToSpeechSession> _sessionFactory;
    private readonly SpeechToSpeechOptions _options;
    private ISpeechToSpeechSession? _session;
    private Task? _readLoop;
    private volatile bool _botSpeaking;

    public SpeechToSpeechProcessor(
        Func<ISpeechToSpeechSession> sessionFactory,
        SpeechToSpeechOptions options,
        ILogger<SpeechToSpeechProcessor>? logger = null)
        : base("SpeechToSpeech", new BoundedChannelOptions(256)
        {
            FullMode = BoundedChannelFullMode.DropOldest, SingleReader = true,
        })
    { /* store fields, default logger to NullLogger */ }

    protected override async ValueTask OnStartAsync(StartFrame frame, CancellationToken ct)
    {
        _session = _sessionFactory();
        await _session.SetVoiceAsync(_options.Voice, ct).ConfigureAwait(false);
        if (_options.SystemPrompt is { } sys)
            await _session.SetSystemPromptAsync(sys, ct).ConfigureAwait(false);
        _readLoop = Task.Run(() => ReadResponseLoopAsync(ct));
    }

    protected override async ValueTask ProcessFrameAsync(Frame frame, CancellationToken ct)
    {
        if (_session is not null && frame is AudioRawFrame audio)
        {
            await _session.AppendUserAudioAsync(audio.Pcm, ct).ConfigureAwait(false);
            return; // consumed; agent audio returns via the read loop
        }
        // Forward everything else (StartFrame, EndFrame, …) downstream so the sink completes.
        await PushFrameAsync(frame, ct).ConfigureAwait(false);
    }

    private async Task ReadResponseLoopAsync(CancellationToken ct)
    {
        var session = _session;
        if (session is null) return;
        try
        {
            await foreach (var chunk in session.RespondAsync(ct).ConfigureAwait(false))
            {
                // Emit the SAME lifecycle frames as the cloud composites so Studio /
                // diagnostics treat this identically. Speaking edges flip _botSpeaking.
                if (!_botSpeaking && !chunk.AudioPcm.IsEmpty)
                {
                    _botSpeaking = true;
                    await PushFrameAsync(new BotStartedSpeakingFrame(), ct).ConfigureAwait(false);
                }
                if (chunk.Text is { Length: > 0 } text)
                    await PushFrameAsync(new LlmTextChunkFrame(text), ct).ConfigureAwait(false);
                if (!chunk.AudioPcm.IsEmpty)
                    await PushFrameAsync(new AudioRawFrame(chunk.AudioPcm, session.OutputSampleRate), ct).ConfigureAwait(false);
                if (chunk.IsFinal && _botSpeaking)
                {
                    _botSpeaking = false;
                    await PushFrameAsync(new BotStoppedSpeakingFrame(), ct).ConfigureAwait(false);
                }
            }
        }
        catch (OperationCanceledException) { /* shutdown */ }
        catch (Exception ex)
        {
            await PushErrorAsync($"SpeechToSpeech read loop failed: {ex.Message}", ex, ct).ConfigureAwait(false);
        }
    }

    protected override async ValueTask OnEndAsync(EndFrame frame, CancellationToken ct)
    { /* DisposeAsync the session; await the read loop, swallowing on teardown */ }
}

Frame parity — the hard requirement

The composite MUST emit the same frame vocabulary the cloud composites do, so anything observing the pipeline (Studio's Talk view, the diagnostics hub, a custom sink) cannot tell a third-party realtime API from a local S2S model:

FrameWhenParity source
UserStartedSpeakingFrame / UserStoppedSpeakingFrameThe session's VAD / turn-taker confirms user speech edges (full-duplex, owned by the model)Both cloud composites
BotStartedSpeakingFrame / BotStoppedSpeakingFrameAgent audio starts / the turn goes finalBoth cloud composites (drives _botSpeaking)
AudioRawFrame (out)Each agent audio chunk, tagged with OutputSampleRateBoth cloud composites
LlmTextChunkFrame / TranscriptionFrameDecoded agent text tokens / user transcript, if the model surfaces themOpenAIRealtimeProcessor
InterruptionFrameBarge-in confirmed — the composite also calls session.CancelAsyncBoth cloud composites
ToolCallRequestFrame / ToolCallResultFrameOnly if the model supports tools (optional; see §5)Both cloud composites
ErrorFrame (upstream)Read-loop failure, via PushErrorAsyncBoth cloud composites
Invariants honoured. Forward unhandled frames downstream (notably StartFrame / EndFrame, or the sink never completes); accept the per-frame CancellationToken on the append + read-loop work so an interruption aborts in-flight synthesis; bounded DropOldest on the audio path, matching the backpressure convention. Any diagnostics tap added later sits behind a HasListeners guard so an unobserved pipeline is byte-identical — exactly as the granular composer and cloud composites do.

Selection / registration

Config selects the composite over the granular chain. Two equivalent routes, matching how the cloud composites are reached today:

  • Explicit map (available immediately): MapVoxaVoice("/voice/s2s", voice => voice.UseProcessor(...)) — a single processor, bypassing UseDefaults() / the granular composer.
  • Config-driven (small addition): a Voxa:Mode key (default Granular; SpeechToSpeech selects the composite) or a registered provider name, so UseDefaults() can short-circuit to the composite when configured. The session factory is resolved from a descriptor in the new package, registered the same self-describing way as VAD / STT / TTS providers. Config-capture rule: the descriptor captures the IConfiguration passed to AddVoxa — it never calls GetRequiredService<IConfiguration>() (that breaks the plain-ServiceCollection hosts: Studio, tests, CLI).

7.WS3 — the deferred model & host deferred / spike

This section is deferred and exploratory. Nothing here is in scope for VRT-005's code. It records the intended path so the seam in WS1/WS2 is designed to accommodate it — not so it ships now.

A concrete ISpeechToSpeechSession is the moment a real model enters the picture. Two plausible first implementations, in order of likely sequencing:

Option A — a cloud S2S provider validates the seam first

The cheapest way to prove ISpeechToSpeechSession + SpeechToSpeechProcessor end-to-end before any local model exists is to back the session with a cloud S2S endpoint. This exercises the full streaming contract (append audio → receive audio + text → barge-in cancel) against a real model with none of the GPU / hosting burden — and confirms the frame parity the composite promises. It is the recommended first concrete impl.

Option B — the local PersonaPlex / Moshi-class model

The strategic target, and the reason the seam exists. Honest constraints:

  • GB-scale, often CUDA-only. A PersonaPlex / Moshi-class model is gigabytes and typically needs a GPU. It does not fit the in-process, CPU-default posture of the local speech tier.
  • Host out-of-process or on GPU. Reuse the VLS-002 GPU patterns (opt-in runtime, user-supplied natives, fail-loud when the device is unavailable) and/or the PiperProcessHost-style sidecar / child-process pattern for an isolated, possibly Python, runtime. The session abstraction is host-agnostic: in-process, sidecar, and GPU all sit behind the same ISpeechToSpeechSession.
  • SHA-256-pinned, cached, offline. Whatever the weights are, they are pinned via VoxaModelArtifact and resolved through VoxaModelCache; downloads are trait-gated LocalModels tests, never the default lane. GPL / heavy / native deps stay behind a process boundary with a license-closure gate test (mirror the Kokoro one).
  • Spike-gated on feasibility. Whether such a model is viable in real time on commodity hardware is an open question (§12, R1). The spike answers that before any host code is written.
Sequencing. WS1 + WS2 (this spec) land first against a fake session. WS3 is a separate, later effort — Option A (cloud) to validate, then Option B (local) when a model proves viable. Treat the local model as unbudgeted and exploratory.

8.Configuration reference

Until a concrete session ships (WS3), these keys describe how the composite would be selected and tuned. The shape mirrors OpenAIRealtimeOptions (voice, system prompt, sample rates).

appsettings.json — selecting the S2S composite (illustrative; concrete provider deferred) 
{
  "Voxa": {
    // NEW (small). "Granular" (default) | "SpeechToSpeech".
    // SpeechToSpeech swaps the VAD→STT→…→TTS chain for the single composite.
    "Mode": "SpeechToSpeech",

    "SpeechToSpeech": {
      "Provider": "<cloud-or-local-session-name>",  // deferred — resolved from a descriptor (WS3)
      "Voice": "default",                            // → ISpeechToSpeechSession.SetVoiceAsync
      "SystemPrompt": "You are a helpful voice assistant.", // → SetSystemPromptAsync
      "OutputSampleRate": 24000                      // advertised on the session envelope
    }
  }
}

When Mode is absent or Granular, behaviour is exactly as today — the granular composer runs and these keys are inert. The cloud composites remain reachable via their own explicit MapVoxaVoice routes regardless of Mode.

9.Plan of action — phases & dependencies

  1. Phase A — WS1 seam. ISpeechToSpeechSession + SpeechToSpeechChunk in Voxa.Speech.Abstractions, pinned against the cloud processors' surface. No behaviour; independently mergeable.
  2. Phase B — WS2 composite + fake session. The new Voxa.Services.SpeechToSpeech package: SpeechToSpeechProcessor, SpeechToSpeechOptions, a FakeSpeechToSpeechSession (test double), and the lifecycle / frame-parity tests. Selection wiring (Voxa:Mode + descriptor). Depends on Phase A.
  3. Phase C — WS3 concrete session. deferred / spike Option A (cloud S2S) to validate the seam, then Option B (local model + GPU / sidecar host). A separate, spike-gated effort with its own spec(s); not part of VRT-005's code.

One logical change per PR (CONTRIBUTING) — Phase A and Phase B are separate PRs. Each adds an [Unreleased] CHANGELOG entry; the spec is authored before the code (this document). Phase C gates on the feasibility spike and is sequenced independently.

13.Out of scope / deferred

  • The S2S model weights + host deferred. A PersonaPlex / Moshi-class model is GB-scale and often GPU-only; it is a separate, spike-gated effort with its own spec(s), riding VLS-002 GPU / sidecar patterns. Not in VRT-005's code.
  • The first concrete ISpeechToSpeechSession. Cloud (Option A) or local (Option B) — both follow-ups; VRT-005 ships the seam + composite + a fake session only.
  • Voice cloning within the S2S model. A frontier capability; if ever in scope it rides VVL-002 (Voicebox-Parity), not this seam.
  • Changes to the granular chain or the cloud composites. Explicitly untouched.
  • Anything that requires the model to exist. Latency budgets, GPU matrices, streaming-quality tuning — all belong to the deferred WS3 spike, not here.

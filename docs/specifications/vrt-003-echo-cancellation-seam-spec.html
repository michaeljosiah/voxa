VRT-003 — Acoustic Echo Cancellation Seam (IEchoCanceller)

1.What is this? — a plain-English explainer

What does VRT-003 add, in one sentence?

It adds the plug (a named, swappable seam — IEchoCanceller) that a real acoustic echo canceller drops into, so a user can interrupt the bot while it is speaking out loud on speakers — but it deliberately ships without a DSP, defaulting to a do-nothing passthrough.

Why now?

Voxa is half-duplex today: while the bot is talking, the microphone is effectively gated so the bot doesn't hear (and try to transcribe) its own voice coming back through the speakers (ROADMAP P1). That gate is the thing that blocks true barge-in over a speaker. The only safe way to lift it is to first subtract the bot's audio from the mic signal — which is exactly what an acoustic echo canceller does. So before we can do barge-in-over-speakers well, we need the seam for echo cancellation in the right place in the chain. This is that foundation.

Does this make the pipeline heavier, or change anything for today's users?

No. The default is NullEchoCanceller — a passthrough that returns the mic audio unchanged and ignores the reference feed. With no acoustic echo canceller configured the composer inserts nothing, so the composed pipeline is byte-identical to today and golden-tested to prove it. You only pay for echo cancellation if you opt in via Voxa:Aec:Engine.

What ships in VRT-003, and what is deferred?

Ships: the IEchoCanceller interface (in a new Voxa.Audio.Abstractions), the EchoCancellerProcessor that runs it before the VAD, the plumbing that feeds the bot's TTS audio in as the far-end reference, the NullEchoCanceller default, and a TryGetAec registry slot + descriptor so an implementation drops in by config. Deferred: a production echo-cancellation DSP (WebRTC APM, SpeexDSP via P/Invoke, or a managed AEC) — that is a separate follow-up item once a license-clean .NET binding is chosen. VRT-003 builds the socket; the DSP is the bulb you screw in later.

What does it explicitly not change?

It does not change the existing interruption / barge-in policy (that's VRT-002's interruption-debounce and recovery), and it does not by itself remove the half-duplex gate — it makes lifting that gate possible and describes the migration (§7). The audio frame contract, the bounded-channel backpressure convention, and the diagnostics taps are untouched.

2.Executive summary

VRT-003 imports speech-core's EchoCancellerInterface + auto far-end-feed pattern onto Voxa's frame/processor model. speech-core's chain is mic → AEC → enhance → VAD → STT with the canceller auto-fed the playback (far-end) signal; Voxa's DefaultVoicePipelineComposer.Compose builds VAD → STT → … → TTS, so the AEC slots in as a new first stage before the VAD. Three small workstreams, all opt-in:

  • WS1 — the seam additive. An IEchoCanceller abstraction in a new zero-extra-dependency Voxa.Audio.Abstractions package: FeedReference(farEndPcm) (the TTS / bot playback), CancelEcho(nearEndPcm) → pcm (the mic), Reset(), and a sample-rate contract. The seam stays deliberately simple — buffering, frame alignment, and rate conversion between the far-end and near-end streams are the implementation's responsibility, exactly as in speech-core.
  • WS2 — the processor + reference feed P1. An EchoCancellerProcessor placed before the VAD that runs CancelEcho on each AudioRawFrame, with the bot's TTS audio auto-tapped and fed in via FeedReference. A NullEchoCanceller default keeps the chain byte-identical when nothing is configured; a TryGetAec registry slot + a descriptor let a real implementation drop in via Voxa:Aec:Engine.
  • WS3 — barge-in interplay P2 (design, not policy). With an acoustic echo canceller present, the half-duplex gate can be lifted for true barge-in; VRT-002's MinInterruptionDuration debounce filters whatever residual echo the canceller doesn't fully remove. VRT-003 describes the migration (AEC opt-in → gate optional) but ships neither the gate change nor the debounce.
Net effect. A real acoustic echo canceller drops in purely by configuration; with none configured the default is a passthrough and the composed pipeline is byte-identical to today. VRT-003 is the half-duplex → full-duplex foundation, sequenced (in program item M2) alongside VRT-002 whose barge-in robustness this makes viable on speakers.

3.Goals, non-goals & budgets

Goals

  • Define IEchoCanceller as a stable, minimal seam (in Voxa.Audio.Abstractions) mirroring speech-core's EchoCancellerInterface: feed far-end, cancel near-end, reset, sample-rate contract.
  • Insert an EchoCancellerProcessor before the VAD in DefaultVoicePipelineComposer and auto-feed it the bot-audio (TTS) stream as the far-end reference.
  • Ship a NullEchoCanceller passthrough default so the chain is unchanged and byte-identical when no acoustic echo canceller is configured.
  • Add a TryGetAec registry slot + a descriptor so a real implementation resolves from Voxa:Aec:Engine, the same self-describing way VAD providers resolve via TryGetVad.
  • Honour every cross-cutting invariant: forward StartFrame/EndFrame; bounded DropOldest on the audio path; diagnostics byte-identical-when-off; config-capture; one-PR-per-change with a CHANGELOG entry and this spec landed before the code.

Non-goals (this spec)

  • Shipping a production AEC DSP. No WebRTC APM / SpeexDSP / managed echo canceller binding lands here — that is a separate follow-up item once a license-clean .NET binding is selected (§9, §13).
  • Shipping the barge-in policy. The interruption-debounce, interruption-recovery, and empty-STT recovery knobs are VRT-002. VRT-003 only makes that policy robust on speakers; it does not implement it.
  • Removing the half-duplex gate. VRT-003 makes lifting it possible and specifies the migration (§7); the gate change itself is gated on a real acoustic echo canceller existing and is out of scope here.
  • Speech enhancement / denoise. That is the sibling VLS-004 IAudioEnhancer stage, which composes after the AEC in the same insertion region (§4). Not specced here.
  • Multi-rate resampling helpers as a shared utility — if a real implementation needs them and they don't already exist, that's its concern (§13).

4.Architecture context — the chain & the insertion point

speech-core's on-device chain is mic → AEC → enhance → VAD → STT, with the echo canceller given the playback signal automatically as its far-end reference. Voxa's granular chain is built by DefaultVoicePipelineComposer.Compose(IServiceProvider) as VAD → STT → TranscriptionFilter → agent → SentenceAggregator → TTS. The acoustic echo canceller is therefore a new first stage, inserted before the VAD block, and the same insertion region is where VLS-004's enhancer will later sit:

            ┌─────────────────────── far-end reference (bot audio) ───────────────────────┐
            │                                                                              │
 mic ──▶ [ AEC ] ──▶ [ enhance (VLS-004) ] ──▶ VAD ──▶ STT ──▶ filter ──▶ agent ──▶ aggregator ──▶ TTS ─┘
         VRT-003        (separate item)      └──────────────── unchanged granular chain ─────────────┘

The insertion is the first thing the composer adds to parts, ahead of the existing VAD block (the if (!string.Equals(o.Vad.Engine, "None", …)) region). Because every later stage forwards unhandled frames downstream, prepending a stage that emits the same AudioRawFrame shape it consumes is transparent to them. The far-end reference is the bot-audio (TTS-output) stream, tapped and routed to the canceller's FeedReference — modelled directly on speech-core's auto far-end feed.

5.WS1 — the IEchoCanceller seam

Package Voxa.Audio.Abstractions (new) Risk Low Network none Surface additive (new interface + null default)

The seam mirrors speech-core's EchoCancellerInterface on the smallest possible surface. Two concurrent inputs — a far-end reference (what is being played out the speakers, i.e. the bot's TTS audio) and a near-end capture (the raw mic) — and one output: the near-end with the far-end's echo removed. The implementation owns the hard parts; the seam stays trivial.

src/Voxa.Audio.Abstractions/IEchoCanceller.cs (new — sketch) 
namespace Voxa.Audio;

/// <summary>
/// Acoustic echo canceller seam (VRT-003): subtracts the far-end playback signal (the bot's TTS
/// audio) from the near-end mic capture so the user can barge in over speakers without the bot
/// hearing itself. Models speech-core's EchoCancellerInterface. Implementations OWN buffering,
/// frame alignment, and any rate conversion between far-end and near-end — the seam stays simple.
/// Default impl is <see cref="NullEchoCanceller"/> (passthrough); a real DSP is a follow-up item.
/// </summary>
public interface IEchoCanceller
{
    /// <summary>The PCM sample rate this canceller is configured for (near-end / output rate, Hz).
    /// The processor advertises the same rate the chain runs at; far-end at a different rate is the
    /// implementation's problem (it resamples internally).</summary>
    int SampleRate { get; }

    /// <summary>Feed the far-end reference — a chunk of the audio being played to the user
    /// (bot/TTS PCM). Called from the TTS / writer thread, CONCURRENTLY with <see cref="CancelEcho"/>.
    /// MUST be non-blocking and allocation-light: it runs on the synthesis hot path. The impl buffers
    /// and time-aligns this against the near-end internally; passing a different rate is allowed.</summary>
    void FeedReference(ReadOnlyMemory<byte> farEndPcm);

    /// <summary>Remove the far-end echo from one near-end (mic) chunk and return the cleaned PCM.
    /// Called from the audio thread, once per <c>AudioRawFrame</c>. May return the input unchanged
    /// (passthrough) or a same-length cleaned buffer; it must NOT change the sample count semantics
    /// the caller relies on for the frame's duration.</summary>
    ReadOnlyMemory<byte> CancelEcho(ReadOnlyMemory<byte> nearEndPcm);

    /// <summary>Reset all adaptive/filter state and drop buffered far-end. Called on session start,
    /// on an interruption epoch, and when the bot stops speaking, so stale echo state never bleeds
    /// into a fresh turn.</summary>
    void Reset();
}

/// <summary>The default. Returns the mic audio untouched and ignores the reference feed, so the
/// composed pipeline is byte-identical to one built before this seam existed. Used whenever
/// Voxa:Aec:Engine is unset or "None".</summary>
public sealed class NullEchoCanceller : IEchoCanceller
{
    public static readonly NullEchoCanceller Instance = new();
    public int SampleRate => 0;                                    // rate-agnostic; it never inspects audio
    public void FeedReference(ReadOnlyMemory<byte> farEndPcm) { }  // no-op
    public ReadOnlyMemory<byte> CancelEcho(ReadOnlyMemory<byte> nearEndPcm) => nearEndPcm; // passthrough
    public void Reset() { }
}
The seam stays simple on purpose. Far-end and near-end arrive on different threads, at different chunk sizes, and possibly at different sample rates (TTS-output rate vs mic rate). VRT-003 deliberately does not try to solve alignment or resampling in the seam — those are pushed into the implementation, precisely as speech-core does. A real IEchoCanceller internally keeps a far-end ring buffer, time-aligns it to the near-end, resamples as needed, and runs its adaptive filter. The contract the framework guarantees is only: FeedReference may be called concurrently with CancelEcho; both are on hot paths; the impl owns thread-safety of its own buffers.
Where this lives — never in Voxa.Core. Per the program's cross-cutting invariants, Voxa.Core stays zero-dependency (NUlid only). The seam therefore lives in a new Voxa.Audio.Abstractions package (a peer to Voxa.Speech.Abstractions). It references the Core frame/processor types it needs (for WS2's processor) and nothing else — no ASP.NET, Azure, ONNX, or OpenTelemetry. The reference-impl DSP, when it lands, goes in a further opt-in Voxa.Audio.Aec.* package, never here.

6.WS2 — the EchoCancellerProcessor + reference feed wiring

Package Voxa.Audio.Abstractions + Voxa.AspNetCore Risk Medium (far-end plumbing) Default NullEchoCanceller (passthrough) Insertion before the VAD block

The processor

A FrameProcessor placed before the VAD. On each downstream AudioRawFrame it runs CancelEcho and forwards a frame carrying the cleaned PCM (or, for NullEchoCanceller, the very same buffer). Every other frame — StartFrame, EndFrame, system frames — is forwarded unchanged. It resets the canceller on session start and on an interruption epoch so adaptive state never carries stale echo into a new turn.

src/Voxa.Audio.Abstractions/EchoCancellerProcessor.cs (new — sketch) 
public sealed class EchoCancellerProcessor : FrameProcessor
{
    private readonly IEchoCanceller _aec;

    public EchoCancellerProcessor(IEchoCanceller aec) : base("EchoCanceller")
        => _aec = aec;

    protected override ValueTask OnStartAsync(StartFrame frame, CancellationToken ct)
    {
        _aec.Reset();
        return base.OnStartAsync(frame, ct);   // forward StartFrame downstream — non-negotiable
    }

    protected override async ValueTask ProcessFrameAsync(Frame frame, CancellationToken ct)
    {
        switch (frame)
        {
            case AudioRawFrame audio:
                // Near-end mic in → cleaned PCM out. Same rate/channels; only the samples change.
                var cleaned = _aec.CancelEcho(audio.Pcm);
                await PushFrameAsync(audio with { Pcm = cleaned }, ct).ConfigureAwait(false);
                break;

            case InterruptionFrame:           // barge-in epoch: drop stale far-end + filter state
                _aec.Reset();
                await base.ProcessFrameAsync(frame, ct).ConfigureAwait(false);  // forward it on
                break;

            default:
                await base.ProcessFrameAsync(frame, ct).ConfigureAwait(false);  // forward everything else
                break;
        }
    }
}

Feeding the far-end reference (the bot-audio tap)

The canceller needs the audio that is about to be played to the user — the bot's TTS output, the far-end signal. In Voxa that is the bot-audio stream that flows downstream toward the transport: in the WebSocket host it is exactly what WebSocketAudioSink serializes as binary audio (and already epoch-purges on barge-in, VPS-001). The wiring routes a copy of each outbound bot AudioRawFrame into the canceller's FeedReference as it is produced — speech-core's "auto far-end feed."

Two viable plumbing shapes; the spec recommends the first as the smallest, most transport-agnostic change:

  • (Recommended) A shared per-session IEchoCanceller instance + a far-end tap processor. The composer constructs one canceller for the session, hands it to the EchoCancellerProcessor (near-end, before the VAD) and to a tiny tap on the bot-audio path that calls FeedReference on each outbound AudioRawFrame before it reaches the sink. The tap forwards the frame unchanged — it only observes. Because FeedReference and CancelEcho are contractually concurrent-safe, the two sides can run on their own threads. This keeps the sink (WebSocketAudioSink / Studio's audio device) unmodified and works for any transport.
  • (Alternative) The sink feeds the reference directly. A host whose sink already owns the bot-audio buffer could call FeedReference from there. Rejected as the default because it couples the seam to each transport and would need touching every sink; the tap-processor approach is transport-agnostic, matching the composer's "transport-agnostic since VST-001 WS0" design.
Timing note (why "during synthesis"). The reference must be fed as the bot audio is produced/played, not after the turn — an echo canceller can only subtract what it has already seen on the far-end. The tap therefore sits on the live outbound bot-audio path and feeds frame-by-frame. The verification protocol (§11) asserts exactly this: a fake canceller confirms FeedReference is called during synthesis, interleaved with CancelEcho, not batched at the end.

The null default — byte-identical when unconfigured

When Voxa:Aec:Engine is unset or None, the composer inserts no AEC stage and no far-end tap at all — not even a NullEchoCanceller-wrapped passthrough. (The passthrough exists for hosts that resolve IEchoCanceller explicitly and for tests; the default composition simply omits the stage.) This is the same discipline as the diagnostics taps, which are inserted "only when enabled, so a production pipeline with diagnostics off composes byte-identically." The composer change is guarded the same way:

src/Voxa.AspNetCore/DefaultVoicePipelineComposer.cs (insertion — before the VAD block) 
// 0. Acoustic echo cancellation (VRT-003). Inserted only when configured, so the default chain
//    is byte-identical to today. Far-end (bot audio) is auto-fed via a tap on the outbound path.
if (!string.Equals(o.Aec.Engine, "None", StringComparison.OrdinalIgnoreCase))
{
    if (_registry.TryGetAec(o.Aec.Engine, out var aecDesc))
    {
        var aecSettings = new VoxaAecSettings(SampleRate: inputSampleRate);
        parts.Add(sp => aecDesc.CreateProcessor(sp, aecSettings));   // before the VAD
        // …and register the far-end tap for this session's shared canceller (see WS2 plumbing).
    }
    else
    {
        _logger.LogWarning(
            "Voxa:Aec:Engine '{Engine}' not found in registry; running without echo cancellation.",
            o.Aec.Engine);
    }
}
// 1. VAD … (existing block, unchanged) …

The registry slot & descriptor

The registry grows an AEC slot in the identical shape to its VAD slot (TryGetVad → CreateProcessor(sp, VoxaVadSettings)), and the abstractions grow a VoxaAecDescriptor peer of VoxaVadDescriptor. A provider package describes itself once; the meta-package's AddVoxa registers it; the composer materialises it from the config name. No new resolution model.

src/Voxa.Speech.Abstractions/VoxaProviderDescriptor.cs (additions)  ·  src/Voxa.AspNetCore/VoxaProviderRegistry.cs (additions)

Drop-in story. A future Voxa.Audio.Aec.WebRtc (or SpeexDSP) package ships a VoxaAecDescriptor named e.g. WebRtc, its CreateProcessor wiring a real IEchoCanceller into an EchoCancellerProcessor. A user adds the package and sets Voxa:Aec:Engine = WebRtc — no other change. Until then, None is the default and the chain is unchanged.

7.WS3 — interplay with half-duplex gating & barge-in

Scope design / migration only Policy owner VRT-002 (debounce / recovery) Gate today half-duplex (ROADMAP P1) Ships here nothing — the path forward

Today Voxa is half-duplex: while the bot is speaking, the mic path is gated so the bot does not transcribe its own voice returning through the speakers. That gate is what blocks true barge-in over a speaker (it works fine with headphones / a near-perfect mic, where there is no acoustic loop). The point of VRT-003 is to make lifting that gate safe:

  1. Without an acoustic echo canceller (today, and the VRT-003 default): the half-duplex gate stays. Lifting it would feed the bot's own audio into STT and trigger phantom interruptions. NullEchoCanceller changes nothing; behaviour is exactly as before.
  2. With a real acoustic echo canceller configured: the mic signal reaching the VAD has the bot's audio subtracted, so the VAD/STT see (mostly) only the user. The gate can then be made optional — opening the path to genuine talk-over-the-bot barge-in.
  3. Residual echo is mopped up by VRT-002's debounce: no echo canceller is perfect; some residual leaks through, especially right after the bot starts or stops. VRT-002's MinInterruptionDuration requires a minimum span of confirmed user speech during bot playback before emitting an InterruptionFrame, which filters that residual (and coughs/clicks). VRT-002's interruption-recovery then resumes playback if the user stops within the recovery window. VRT-003 supplies the clean(er) signal; VRT-002 supplies the judgement.

Migration: AEC opt-in → gate optional

StageState
Now (VRT-003 ships)Seam + null default + wiring land. Gate unchanged. Voxa:Aec:Engine = None. Pipeline byte-identical.
+ a real DSP (follow-up item)A Voxa.Audio.Aec.* package ships an IEchoCanceller. Users opt in via Voxa:Aec:Engine; the canceller cleans the mic. Gate still on by default — AEC alone is the conservative win (the bot stops hearing itself).
+ gate-lift knob (VRT-002 territory)A profile/tuning flag (e.g. FullDuplexBargeIn) makes the half-duplex gate optional when an AEC is active, deferring interruption decisions to VRT-002's debounce/recovery. Measured by VRT-001's barge-in category before it becomes a default.
Why the existing epoch-purge matters here. WebSocketAudioSink already bumps an interruption epoch and drops queued bot audio on barge-in (VPS-001). That is the output side of barge-in — stop playing the now-stale bot audio fast. VRT-003 + VRT-002 are the input side — decide, from a cleaned mic, that the user really did barge in. The two compose: the AEC cleans the signal, VRT-002 debounces the decision, InterruptionFrame fires, and the existing purge stops the speaker almost immediately. The EchoCancellerProcessor also Reset()s on that same interruption epoch so the next turn starts with clean filter state.

9.Plan of action — phases & dependencies

  1. Phase A — the seam + null default (ships alone). New Voxa.Audio.Abstractions package with IEchoCanceller + NullEchoCanceller + EchoCancellerProcessor, plus the VoxaAecDescriptor/VoxaAecSettings records and the TryGetAec registry slot. Pure additive; nothing is wired into the default composition yet. Independently mergeable.
  2. Phase B — composer wiring + far-end feed + config/validator. Insert the AEC stage before the VAD, register the bot-audio far-end tap, add the Voxa:Aec option + fail-fast validator. Guarded so None inserts nothing — the passthrough-off golden must stay green. This is where the byte-identical invariant is enforced.
  3. Follow-up item (NOT this spec) — a real DSP binding. A separate Voxa.Audio.Aec.* package binding one echo-cancellation DSP, gated on choosing a license-clean .NET path. Candidate notes:
    • WebRTC Audio Processing Module (APM). The cleanest cross-platform AEC story and a permissive license, but needs a .NET binding / P/Invoke wrapper around a native build — bundling native binaries is the same re-bundling concern flagged for the Whisper GPU runtimes (VLS-002), so it follows the pinned-native pattern (VoxaModelArtifact / process-boundary discipline) and is opt-in.
    • SpeexDSP echo canceller via P/Invoke. Smaller and simpler; license is permissive (BSD-style) but the cancellation quality is older/weaker than WebRTC APM. Same native-bundling caveat.
    • A future managed AEC. Avoids native bundling entirely; none mature today — revisit on demand.
    Each candidate carries a license-closure gate test (mirroring the Kokoro GPL gate) and stays behind a process or package boundary per the cross-cutting invariants.

One logical change per PR (per CONTRIBUTING): Phase A and Phase B are two PRs; the DSP binding is its own later item with its own spec. Each PR adds an [Unreleased] CHANGELOG entry; this docs/specifications/* spec lands before the code.

12.Risk register

IDRiskSevMitigation
R1Golden drift — the inserted stage (or an accidentally-inserted passthrough) changes the default composition, breaking the byte-identical guarantee.HighComposer inserts nothing when Engine == None (not even a NullEchoCanceller wrapper). The passthrough-off golden (T2.5) extends the existing diagnostics-off golden and gates every PR.
R2Frame alignment / sample-rate mismatch between far-end (TTS rate) and near-end (mic rate) corrupts cancellation.MedPushed into the implementation, by design — the seam contract makes buffering/alignment/resampling the impl's responsibility (speech-core does exactly this). The framework only guarantees concurrent-safe feed/cancel. The null default is rate-agnostic, so VRT-003 itself carries no alignment risk.
R3Reference-feed thread-safety: FeedReference runs on the TTS/writer thread concurrently with CancelEcho on the audio thread; a blocking or allocating feed stalls synthesis.MedThe seam doc-comment mandates a non-blocking, allocation-light FeedReference; the fake-AEC test exercises concurrent feed/cancel. The tap only observes and forwards the frame, so the outbound bot-audio path is never stalled by the tap itself.
R4Far-end tap retains bot audio in memory (a second copy of the whole session).MedThe tap forwards the existing frame and calls FeedReference synchronously; it never buffers. Any buffering is the canceller's, which is bounded ring-buffer state it owns and Reset()s per turn.
R5Adaptive filter state bleeds across turns (stale echo applied to a new utterance).LowEchoCancellerProcessor calls Reset() on StartFrame and on every interruption epoch; the contract documents Reset semantics.
R6Users expect echo cancellation to work out of the box and are surprised the default does nothing.LowSpec, config docs, and the None default value make explicit that VRT-003 ships the seam, not a DSP; the drop-in story and the follow-up DSP item are documented (§6, §9).

13.Out of scope / deferred

  • A production AEC DSP. WebRTC APM / SpeexDSP-via-P/Invoke / a managed canceller — a separate follow-up item with its own spec, gated on a license-clean .NET binding and following the pinned-native / process-boundary discipline (§9).
  • The barge-in policy. Interruption-debounce (MinInterruptionDuration), interruption-recovery, empty-STT recovery — owned by VRT-002. VRT-003 only makes them robust on speakers.
  • Lifting the half-duplex gate. The full-duplex gate-optional knob is described as the migration (§7) but not shipped; it rides on a real AEC existing and on VRT-002, and is measured by VRT-001's barge-in category.
  • Speech enhancement / denoise (IAudioEnhancer) — the sibling VLS-004 stage that composes after the AEC in the same insertion region. Specced separately.
  • Shared multi-rate resampling helpers. If a real IEchoCanceller needs resampling and a helper doesn't already exist in the tree, providing one is that item's concern — VRT-003 keeps the seam minimal and does not add framework-level DSP utilities.
  • A Studio surface for AEC. Per the program's product boundary, all logic is Core; Studio would at most expose the toggle later — not part of this item.

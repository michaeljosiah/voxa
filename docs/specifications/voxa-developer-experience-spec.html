1.Executive summary

Voxa's runtime is now fast (VPS-001) and its processor model is sound — but using it requires expertise the target audience doesn't have yet. A developer who wants "a voice endpoint on my ASP.NET app" today must: pick 5–7 NuGet packages out of 14, hand-write configuration readers, construct engines with the right sample rates, know that VAD goes before STT and the aggregator goes before TTS, hand-roll per-connection chat history with BuildMessages/OnTurnCompleted closures, learn 11 tuning knobs to get acceptable latency, and then write ~500 lines of browser JavaScript (AudioWorklet capture, gap-free PCM scheduling) for which the only reference is the sample's embedded script. The working minimum today is ~47–58 lines of C# per endpoint plus ~500 lines of JS; the sample server's ReadXxxOptions helpers alone are 52 lines.

This spec collapses that to the ROADMAP P5 target — a working voice bot in five lines plus one config block:

var builder = WebApplication.CreateBuilder(args);
builder.Services.AddVoxa(builder.Configuration);          // reads the "Voxa" config section
var app = builder.Build();
app.UseWebSockets();
app.MapVoxaVoice("/voice").UseDefaults();                 // VAD → STT → agent → aggregator → TTS
app.Run();

…and a working browser client in five more:

import { VoxaClient } from "@voxa/client";
const voice = new VoxaClient({ url: "wss://localhost:5001/voice" });
voice.on("transcription", e => console.log("you:", e.text));
voice.on("text", e => console.log("bot:", e.text));
await voice.connect();                                    // mic, playback, barge-in flush — handled

Five deliverables, in dependency order:

  1. A typed configuration model with fail-fast validation. VoxaOptions binds the "Voxa" section ("Stt": "OpenAI", "Tts": "ElevenLabs", "Profile": "LowLatency", per-provider blocks). Misconfiguration fails at startup with a message naming the exact missing key — never a mid-call null reference.
  2. AddVoxa() + a provider registry. Provider packages describe themselves with descriptor objects (OpenAISpeech.SttDescriptor); AddVoxa resolves "Stt": "OpenAI" against the registry. IHttpClientFactory integration replaces VoxaHttp.Shared for hosts that need proxies/custom handlers (the promise in docs/performance-tuning.md).
  3. Named profiles. "LowLatency" / "Quality" / "Cheap" bundle the VPS-001 tuning knobs (VAD hangover, eager first chunk, buffer sizes) so users get a tuned pipeline without learning any of them. Explicit config always wins over the profile.
  4. UseDefaults() composition + the Voxa meta-package + a project template. One package reference, one dotnet new voxa-server, sensible default chain with built-in per-connection conversation memory.
  5. @voxa/client on npm with a generated wire contract. The TypeScript protocol types are generated from the C# WireMessages DTOs with a CI drift gate, so client and server cannot diverge silently. The client ships the missing client half of barge-in (ROADMAP P2): playback-buffer flush on the {"type":"interruption"} envelope.

Expected outcome: server boilerplate per endpoint drops from ~47–58 statements to ≤ 6; packages to discover drops from 5–7 to 1; client code drops from ~500 hand-rolled lines to ≤ 10; a misconfigured key is a startup error, not a silent runtime failure; and barge-in finally works end-to-end because the official client flushes its playback buffer. Every claim is enforced by a compiled acceptance fixture, golden-file tests, and a wire-contract drift gate.

2.2 Non-goals

  • No new speech providers (Deepgram, Cartesia, WhisperCpp, Piper) — ROADMAP P0/P6; the registry built here is where they will plug in.
  • No session resilience / reconnect tokens — ROADMAP P6. The JS client gets a clean disconnected event and a manual connect() retry path, nothing more.
  • No runtime control envelope ({"type":"configure"}) — ROADMAP P7. The additive session envelope in WS4 is server→client announce only.
  • No typed frontend tools / source-generated tool schemas — ROADMAP P6. @voxa/client exposes raw toolCall/sendToolResult passthrough only.
  • No changes to the frame model, processor lifecycle, or anything VPS-001 optimized. This spec is composition and packaging around the runtime, not the runtime.
  • No Realtime/VoiceLive support in UseDefaults() v1 — the default chain is the chained (STT→agent→TTS) pipeline. Realtime composition stays on the explicit MapVoxaVoice(pattern, configure) overload (see §11).

2.3 Hard rules for the implementer

Read this before writing any code.
  • Every workstream lands as its own commit (or small commit series) with its tests in the same commit. Never batch two workstreams in one commit.
  • Run dotnet build Voxa.sln -warnaserror and dotnet test after each workstream. All 13 existing test projects must stay green. The JS package additionally gates on npm test + npm run build.
  • The existing programming model must keep working unchanged. MapVoxaVoice(pattern, configure), manual Pipeline.Build(), and direct engine construction are the escape hatches that make UseDefaults() safe to opinionate. No existing public API is removed or behaviorally changed except where marked BREAKING (there is exactly one, WS7's WireTypes refactor, and it is internal-only).
  • Defaults must be production-plausible, not demo-plausible. Every default chosen by UseDefaults()/profiles must be one you would defend in a production review (e.g. conversation memory is bounded, errors close the socket with a proper status).
  • Wire-protocol additions are additive only: existing envelope bytes must remain byte-for-byte identical (the VPS-001 WS3 wire-compat tests are the guard — they must pass unmodified).
  • Configuration precedence is exactly: explicit config > profile > built-in default. Encode it in one place (WS3's resolver) and test it; do not re-derive it ad hoc per knob.
  • Keep ConfigureAwait(false) on every await in library code (existing convention).

    public string? Stt { get; set; }

    public string? Tts { get; set; }

    public VoxaAgentOptions Agent { get; set; } = new();
    public VoxaAggregatorOptions Aggregator { get; set; } = new();

WS1.1 — Validation: fail at startup, name the key

Register with ValidateOnStart() so every error below aborts host startup, not the first call:

src/Voxa.AspNetCore/Configuration/VoxaOptionsValidator.cs (new file)
internal sealed class VoxaOptionsValidator : IValidateOptions<VoxaOptions>
{
    private readonly VoxaProviderRegistry _registry;   // WS2
    private readonly IConfiguration _configuration;

    public VoxaOptionsValidator(VoxaProviderRegistry registry, IConfiguration configuration)
    { _registry = registry; _configuration = configuration; }

    public ValidateOptionsResult Validate(string? name, VoxaOptions o)
    {
        var errors = new List<string>();

        if (!VoxaProfiles.IsKnown(o.Profile))
            errors.Add($"Voxa:Profile '{o.Profile}' is not a known profile. " +
                       $"Valid values: {string.Join(", ", VoxaProfiles.Names)}.");

        ValidateProvider(errors, "Voxa:Stt", o.Stt, _registry.SttNames);
        ValidateProvider(errors, "Voxa:Tts", o.Tts, _registry.TtsNames);

        // Each selected provider's descriptor validates its own config sub-section
        // (e.g. OpenAI requires "Voxa:OpenAI:ApiKey"). See VoxaProviderDescriptor.Validate.
        if (o.Stt is not null && _registry.TryGetStt(o.Stt, out var stt))
            errors.AddRange(stt.Validate(_configuration.GetSection(VoxaOptions.SectionName)));
        if (o.Tts is not null && _registry.TryGetTts(o.Tts, out var tts))
            errors.AddRange(tts.Validate(_configuration.GetSection(VoxaOptions.SectionName)));

        if (o.Vad.Engine is not ("Silero" or "SilenceGate" or "None"))
            errors.Add($"Voxa:Vad:Engine '{o.Vad.Engine}' is invalid. Valid values: Silero, SilenceGate, None.");

        return errors.Count == 0
            ? ValidateOptionsResult.Success
            : ValidateOptionsResult.Fail(errors);
    }

    private static void ValidateProvider(List<string> errors, string key, string? value, IReadOnlyCollection<string> known)
    {
        if (value is null)
            errors.Add($"{key} is required when using UseDefaults(). " +
                       $"Registered providers: {(known.Count == 0 ? "(none — did you reference the Voxa meta-package or register descriptors?)" : string.Join(", ", known))}.");
        else if (!known.Contains(value, StringComparer.OrdinalIgnoreCase))
            errors.Add($"{key} '{value}' is not a registered provider. Registered: {string.Join(", ", known)}.");
    }
}
Error-message bar: every validation error must answer three questions in one sentence: which key, what's wrong, what are the valid values. The unit tests assert message content, not just failure — these strings are the product.
Validation scope: VoxaOptionsValidator runs only when UseDefaults() is in play conceptually, but options validation is global. Hosts that call AddVoxa purely for engine DI and compose manually must not be forced to set Stt/Tts. Resolution: the Stt/Tts-required checks run lazily — UseDefaults() (WS4) triggers them by requesting IOptions<VoxaOptions> through a wrapper service VoxaDefaultsGuard that performs the required-field checks at first map/first request; the structural checks (unknown profile, unknown provider name when one is given, provider sub-section validity) stay in the eager ValidateOnStart path. Test both modes.

Acceptance criteria: binding tests (full section → options graph, TimeSpan-as-ms ints honored); validation tests for: missing Stt with empty registry, unknown provider with suggestions list, unknown profile, missing ApiKey in a selected provider's block, manual-composition host with no Stt/Tts passing startup; all messages assert the three-question bar.

WS2.1 — Provider descriptors (in Speech.Abstractions — data only, no DI references)

A descriptor is a record of: a name, the config sub-section it owns, preferred sample rates, a validation function, and a per-connection processor factory. System.IServiceProvider is BCL; the only new package reference for Voxa.Speech.Abstractions is Microsoft.Extensions.Configuration.Abstractions (tiny, already transitively everywhere in ASP.NET hosts).

src/Voxa.Speech.Abstractions/VoxaProviderDescriptor.cs (new file)
using Microsoft.Extensions.Configuration;

namespace Voxa.Speech;

/// <summary>
/// Self-description of an STT provider for config-driven composition ("Voxa:Stt": "OpenAI").
/// Lives in abstractions so provider packages can describe themselves without referencing
/// DI or ASP.NET. The factory runs once per connection (engines are per-session stateful).
/// </summary>

    string ConfigSection,                         // sub-section under "Voxa", e.g. "OpenAI", "AzureSpeech"

    Func<IServiceProvider, IConfigurationSection, SpeechToTextProcessor> CreateProcessor);

/// <summary>TTS twin of <see cref="VoxaSttDescriptor"/>.</summary>
public sealed record VoxaTtsDescriptor(
    string Name,
    string ConfigSection,
    int OutputSampleRate,
    Func<IConfigurationSection, IReadOnlyList<string>> Validate,
    Func<IServiceProvider, IConfigurationSection, TextToSpeechProcessor> CreateProcessor);

public sealed class DefaultVoicePipelineComposer
{
    // ctor: IOptions<VoxaOptions>, VoxaProviderRegistry, VoxaTuningResolver, IConfiguration, ILogger

    /// <summary>Returns the per-connection composition + the rates for the session envelope.</summary>
    public ComposedVoice Compose(HttpContext httpContext)
    {
        var o = _options.Value;
        var tuning = _resolver.Resolve(o);
        var root = _configuration.GetSection(VoxaOptions.SectionName);

        _registry.TryGetStt(o.Stt!, out var stt);    // guaranteed by VoxaDefaultsGuard (WS1)
        _registry.TryGetTts(o.Tts!, out var tts);

        var parts = new List<Func<HttpContext, FrameProcessor>>();

        // 1. VAD — Silero descriptor if registered, else SilenceGate fallback with a one-time warning.
        if (o.Vad.Engine != "None")
            parts.Add(ctx => CreateVad(ctx.RequestServices, o.Vad.Engine, tuning, stt.PreferredInputSampleRate));

        // 2. STT (provider factory) → 3. hallucination filter.
        parts.Add(ctx => stt.CreateProcessor(ctx.RequestServices, root));
        parts.Add(_ => new TranscriptionFilter());

        // 4. Agent with built-in conversation memory (WS4.3).
        parts.Add(ctx => CreateAgentProcessor(ctx, o.Agent));

        // 5. Aggregator (profile-tuned) → 6. TTS.
        parts.Add(_ => new SentenceAggregator
        {
            EagerFirstChunkMinChars = tuning.EagerFirstChunkMinChars,
            MaxBufferChars = tuning.MaxBufferChars,
        });
        parts.Add(ctx => tts.CreateProcessor(ctx.RequestServices, root));

        return new ComposedVoice(
            Parts: parts,
            InputSampleRate: stt.PreferredInputSampleRate,
            OutputSampleRate: tts.OutputSampleRate);
    }
}

WS8.3 — Implementation requirements (the hard-won details from the sample, made library-grade)

  • Worklet without a second file. The AudioWorklet processor source is embedded as a string and loaded via URL.createObjectURL(new Blob([src], {type:"text/javascript"})) — consumers must not have to serve recorder-worklet.js next to their bundle (the #1 integration papercut with the sample). Port the sample worklet's logic: fixed-size Float32 buffering, clamp to [-1,1], Int16 conversion, transferable postMessage (zero-copy).
  • Resampling reality. Request new AudioContext({ sampleRate: inputSampleRate }}); browsers honor it where supported. Where the context comes up at a different rate (Safari quirk), do linear-interpolation downsampling in the worklet (the sample punts on this; the library must not). Unit-test the resampler math in isolation (pure function).
  • Playback queue. Maintain nextStartTime; for each binary frame decode Int16→Float32 into an AudioBuffer at outputSampleRate, schedule at max(ctx.currentTime, nextStartTime), advance by buffer duration. Track all live AudioBufferSourceNodes in a Set, pruned on ended.
  • Barge-in flush (G-04 — the point of this package). On {"type":"interruption"}: stop(0) every node in the live set, clear it, reset nextStartTime = ctx.currentTime, then emit the interruption event to the app. Combined with the server's epoch purge (VPS-001 WS4), bot audio now stops < 100 ms after the user speaks, end-to-end.
  • Session envelope. On {"type":"session"} reconfigure outputSampleRate for subsequent decoding and (if the mic context rate differs from inputSampleRate) the resampler target. Arrives before any audio by server ordering (ingested pre-conversation); handle late/never (fallback options) for old servers.
  • Autoplay policy. connect() must be documented as requiring a user gesture; if the playback context starts suspended, call resume() and surface VoxaConnectError("audio-context") on failure rather than playing silence (the silent-failure mode every first-time WebAudio user hits).
  • Binary frames are audio, text frames are JSON — exactly the server contract; unknown envelope types are ignored with a console.debug (forward compatibility).

11.Out of scope / deferred (with rationale)

  • Realtime/VoiceLive in UseDefaults(). A "Voxa:Mode": "Realtime" switch is the obvious v2 (the composer would emit a single realtime processor instead of the chain), but the realtime services have materially different config (endpoint URIs, turn-detection blocks, tools) and no registry shape yet. Deferred until the descriptor pattern proves itself; the explicit overload covers it today.
  • Auto-reconnect / resume tokens in @voxa/client — ROADMAP P6 (session resilience). The clean disconnected event is the seam; resume needs the server-side hello token first.
  • {"type":"configure"} runtime control envelope — ROADMAP P7. The session envelope is one-way announce; two-way negotiation is a separate design.
  • Typed frontend tools (voice.UseTool("show_chart", (string city) => ...)) — ROADMAP P6; requires schema source-generation, orthogonal to this spec's packaging goals.
  • Provider auto-discovery via assembly scanning. Considered for "reference a provider package and it just works" — rejected: reflection-based scanning is trimming-hostile, order-nondeterministic, and saves one line that the meta-package already saves for beginners. Explicit descriptors keep à-la-carte hosts honest.
  • Profiles mutating model choice ("Cheap" forcing gpt-4o-mini) — rejected in WS3; billing-relevant decisions stay explicit. Docs carry the guidance instead.
  • A Flutter/React-Native client — the AONIK adapter (ROADMAP P3) owns the mobile story; @voxa/client's protocol layer (generated TS) is reusable there but the audio layer is platform-specific.
  • dotnet new voxa-client (standalone SPA template) — the server template ships a working page; a dedicated frontend template adds maintenance surface without unlocking new users yet.

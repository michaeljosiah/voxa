VDX-009 — Behavioral conversation evals

Voxa's unit tests are excellent at processors-in-isolation and dishonest about conversations: the barge-in dead path, the aggregator tail-drop, and the delegation context loss all shipped past a green suite and were caught by a human talking to the bot. This spec adds the missing layer — declarative conversation scenarios executed against a real composed pipeline, asserting on what the bot actually did (turn timing, spoken text, interruptions, tool calls, background delegation) with optional LLM-judged content checks. Runnable from xUnit, the CLI, and CI.

Finals alone can never exercise barge-in (the loop's interruption path triggers on the speech-onset frame) or VRT-006's interim-fed gate, which would have made the starter pack's flagship scenarios silently vacuous in the default CI lane.

1. Problem

Every processor is testable in isolation behind an interface — that is the repo's core convention, and it works. What it cannot catch is emergent conversation behavior: the frames all flow, every unit passes, and yet an interrupted answer resumes from the next sentence, or an unpunctuated reply tail never reaches the TTS, or a delegated task's result gets spoken into the wrong turn. These are properties of the composition under time, and today the only instrument that measures them is a developer with a microphone. That instrument does not run in CI, does not regress-test, and was the actual discovery mechanism for the last three shipped bugs.

2. Design overview

scenario.yaml ──► EvalRunner
                    │  builds the host: plain ServiceCollection + AddVoxa(config)
                    │  + DefaultVoicePipelineComposer.Compose(sp)   (the Studio precedent)
                    ▼
   ┌──────────── composed pipeline ────────────┐
   │  EvalAudioSource ─► VAD ─► STT ─► agent ─► aggregator ─► TTS ─► EvalSink │
   └────────────────────────────────────────────┘
        ▲                                            │
        │ user turns:                                ▼
        │  • text  (inject transcription — fast lane) EvalRecorder ◄── VoxaDiagnosticsHub
        │  • audio (TTS-synthesized / .wav fixture,      │ ordered, timestamped event log
        │    played at real-time or accelerated cadence) ▼
        └────────────────────────────── assertions: events + budgets + judge

Three deliberate choices:

  • In-process by default. The runner hosts the composed pipeline the way Voxa Studio does (plain ServiceCollection, no Generic Host, no eager warm-up) — deterministic, no ports, no serialization variables. A remote mode over the WebSocket transport is a later workstream (§8 WS4), valuable for full-stack client-protocol coverage but not the foundation.
  • The diagnostics hub is the witness. Scenarios assert against the hub's typed, sequenced, timestamped events (VAD windows, turn edges, transcripts, stage latencies, interruption gate decisions, background-task lifecycle) plus sink-captured frames. No new instrumentation channel: if a behavior matters enough to eval, it matters enough to appear in diagnostics — gaps found while writing scenarios are hub features first.
  • Two input lanes, one schema. mode: text bypasses audio and STT but injects a complete synthetic utterance, not just a final: UserStartedSpeakingFrame → optional interims → UserStoppedSpeakingFrame → final TranscriptionFrame, with configurable gaps. The speaking edges are load-bearing, not decoration — the agent loop's barge-in path triggers on UserStartedSpeakingFrame and the VRT-006 gate consumes interims, so a finals-only lane could never exercise the interruption scenarios this spec exists to guard (no models, CI-fast, exercises turn logic, agent loop, aggregation, TTS via a fake engine). mode: audio synthesizes each user turn with a local TTS voice (or plays a .wav fixture) into the real capture chain at a controlled cadence — the full pipeline, VAD and STT producing the edges naturally, model-gated in CI.

3. Scenario schema

name: backchannel does not kill the answer
mode: text                      # text | audio
pipeline:
  profile: LowLatency           # named profile, or a config fragment under `voxa:`
  agent: scripted               # scripted driver (below) | host-registered driver
script:                         # the scripted IAgentTurnDriver: deterministic bot lines
  - match: "story"
    reply: "Once upon a time, in a kingdom by the sea, there lived a very patient queen."

turns:
  - user: "tell me a story"
    expect:
      - event: turn.started          # LlmTurnStartedFrame
        within_ms: 1000
      - event: bot.text
        contains: "kingdom"
  - user: "uh-huh"
    send_after: { event: bot.speaking.started, delay_ms: 200 }   # over the bot
    expect:
      - no_event: interruption       # gate must swallow it (VRT-006)
        for_ms: 1500
      - event: bot.speaking.stopped  # the answer completes
  - user: "what did you have for breakfast?"
    expect:
      - event: response
        judge: "politely declines or deflects; does not invent having eaten"
ElementSemantics
user / wav / dtmf Turn input: text (synthesized in audio mode; in text mode injected as speaking-edge → interims → stopped-edge → final), a fixture file, or DTMF digits (telephony serializers)
interims / edge_gap_ms Text-mode shaping: an optional list of interim texts emitted between the speaking edges (defaults to none), and the gap between injected frames (default 50 ms). Barge-in gating scenarios (VRT-006 min-words) set interims: explicitly to reproduce word-by-word arrival
send_after Schedule the send relative to an earlier event (+optional delay) or a pure delay — this is how barge-in scenarios say "while the bot is talking"
expect list Matched in order; unmatched interleaved events are skipped, so scenarios don't break when incidental events appear. Each entry: an event name, optional within_ms (budget anchored at the turn's send), optional content check
no_event … for_ms Negative assertion window — the only way to test suppression behaviors (backchannel gating, mute policies, relevance-gated silence from VDX-008)
contains / matches / judge Substring / regex / LLM-judged criterion against the event's text payload
calls For tool.requested: function name + args-subset match

Event names map 1:1 to hub event types — current ones plus those specced by VRT-006/VRT-007/VDX-008 (the runner fails a scenario referencing an event name with no hub counterpart, so the mapping table in the package README stays honest as specs land): turn.started/ended, transcript.interim/final, bot.text, bot.speaking.started/stopped, interruption, tool.requested/completed, background.task.accepted/completed/spoken/dropped (VDX-008), response (the aggregated per-turn bot text — a runner-derived convenience event).

Format: YAML via YamlDotNet, confined to Voxa.Evals (Voxa.Core stays dependency-clean); the loader also accepts the same shape as JSON for hosts that ban YAML. Records bind the schema; unknown keys are startup errors, not silent ignores.

4. The judge

  • Judged criteria run through Microsoft.Extensions.AI.IChatClient — any provider; the reference config is a local Ollama endpoint so the whole harness runs offline. One round-trip per criterion over the scenario's conversation-so-far.
  • The judge prompt is fixed and versioned in the package: strict-but-fair, explicitly tolerant of STT artifacts (homophones, split/merged words) in audio mode, binary verdict + one-line reason (the reason lands in the failure message).
  • Verdicts are cached per (criterion, conversation) hash within a run — repeated assertions are free.
  • No judge configured ⇒ scenarios using judge: fail with "judge not configured", never silently pass.

5. Execution surfaces

  • xUnit: EvalScenarioTheoryData("scenarios/") + a VoxaEvalRunner.RunAsync(path) assertion helper — each scenario is a test case with the scenario name as the display name; first failed expectation fails the scenario (fail-fast; a failed turn leaves the conversation in an unknown state) with the recorder's event timeline dumped in the failure output.
  • CLI: voxa evals run <dir> [--mode text|audio] [--filter name] in Voxa.Cli — table summary, non-zero exit on failure, --timeline writes the per-scenario event log for debugging.
  • CI lanes (honest gating):
    • mode: text + scripted driver + fake TTS ⇒ default suite — no network, no models, no keys. This is the tier that guards regressions like PR #100's.
    • mode: audio (local TTS/STT models) ⇒ Category=LocalModels lane.
    • judge: criteria ⇒ Category=EvalJudge, enabled by VOXA_EVALS_JUDGE_ENDPOINT; skipped-with-reason when absent. Never in the default lane.

6. Determinism & time

  • The scripted driver (script: block) is the default agent for Voxa's own scenarios — deterministic bot lines with optional per-line delays (reply_delay_ms, token_interval_ms) to reproduce streaming pacing. Real drivers (an actual LLM) are for host-side evals like Ada's.
  • Audio cadence is wall-clock real-time by default (VAD and turn logic are time-based); a time_scale knob accelerates playback and proportionally scales the pipeline's VAD durations via config so scaled runs stay self-consistent — marked experimental, off in CI.
  • Budgets (within_ms) on shared CI runners are noisy: scenario budgets are ceilings for catching hangs (seconds), not latency SLOs (the stage-latency histogram + bench project own those). Repo convention holds: no fixed sleeps in the runner — every wait is condition-based with a deadline.

7. Non-goals

  • Latency benchmarking. bench/Voxa.Benchmarks owns performance measurement; evals assert behavior with generous ceilings.
  • Judging audio quality (prosody, naturalness). Text-level content only; bot audio in audio mode may optionally be re-transcribed for assertions, not scored aesthetically.
  • Load/soak testing. One session per scenario.
  • A recorder UI. Studio integration (record a live Talk session into a scenario skeleton) is an attractive follow-up, explicitly out of v1.

8. Workstreams

  • WS1 — Core runner: schema records + YAML/JSON loader; in-process host builder (ServiceCollection + AddVoxa + composer); EvalRecorder (hub subscriber + sink capture, ordered timeline); text-mode synthetic-utterance injection (speaking edges + optional interims + final — asserted by a runner test that a text-mode turn does trigger the loop's barge-in path); scripted driver; ordered matcher with budgets + no_event windows; xUnit surface. Ships with a starter pack: the barge-in/backchannel, tail-flush, and VDX-008 delegation scenarios that re-derive this quarter's three shipped bugs.
  • WS2 — Audio mode: synth lane (local TTS voice per scenario), wav fixtures, real-time cadence player on the capture seam; LocalModels-gated scenario set.
  • WS3 — Judge + CLI: IChatClient judge with cache + versioned prompt; voxa evals run; EvalJudge trait/env gating.
  • WS4 — Remote mode (stretch): drive a running host over the WebSocket transport using the versioned client protocol — full-stack coverage incl. serialization; DTMF turns for telephony serializers.
  • WS5 — Adoption: Ada scenario suite for M10 (delegation ack, result-relevance silence, barge-in during delegation); docs page docs/evals.md; CHANGELOG; CI wiring.

9. Verification

  • The starter-pack scenarios fail on deliberately re-broken code: revert PR #100's loop emission on a scratch branch ⇒ the barge-in scenario fails; disable the aggregator tail flush ⇒ the tail scenario fails. (Run once during development as the harness's own honesty check — documented in the PR, not kept as permanent mutation tests.)
  • Runner unit tests: matcher ordering/skip semantics, budget anchoring, negative windows, unknown-key rejection, judge-absent failure.
  • Default suite stays offline: a CI assertion greps the text-mode scenario set for mode: audio/judge: leakage into the default lane.
  • Docs commands run as written (repo convention).

10. Risks

RiskMitigation
Flaky time-based assertions on CI runners Budgets are hang-catching ceilings; ordered-skip matching tolerates incidental events; no fixed sleeps
Scenario schema sprawl Schema is records + fail-fast on unknown keys; new event names require a hub event to exist first
Judge nondeterminism Binary verdict + low temperature + versioned prompt; judged criteria live outside the default lane; failures print the judge's reason for fast triage
In-process mode masks transport bugs Acknowledged scope; WS4 remote mode covers the wire; unit tests already cover serializers

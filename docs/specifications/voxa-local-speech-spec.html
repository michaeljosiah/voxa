VLS-001 — Voxa Local Speech Tier Specification

Local/offline speech: Voxa.Speech.WhisperCpp (speech-to-text on your own CPU via whisper.cpp), Voxa.Speech.Piper (fast text-to-speech via Piper), and Voxa.Speech.Kokoro (high-quality text-to-speech via Kokoro-82M), with a shared model cache that downloads and verifies models on first run — so a Voxa voice bot can be developed, tested in CI, and deployed air-gapped with zero API keys and zero cloud spend.

1.What is this? — a plain-English explainer

What is the feature, in one sentence?

Three new plug-in packages that let a Voxa voice bot hear and speak using AI models running on your own machine, instead of calling a cloud service like OpenAI or Azure.

How does Voxa work today?

A Voxa voice bot has three "brains": one that turns your speech into text (STT), one that decides what to say (the agent / LLM), and one that turns the reply back into audio (TTS). Today the STT and TTS brains are all cloud services — OpenAI, Azure, ElevenLabs, Mistral. Every one of them needs an account, an API key, a network connection, and a credit card. Even running the project's own test suite end-to-end costs real money and needs real secrets.

What changes?

We add three new options to the existing provider menu. Set "Stt": "WhisperCpp" and Voxa transcribes speech with whisper.cpp — OpenAI's open-source Whisper model running locally on the CPU. Set "Tts": "Piper" and Voxa speaks with Piper — a fast, natural-sounding open-source voice that also runs locally. Prefer a richer voice? Set "Tts": "Kokoro" for Kokoro — a small open model whose speech quality rivals commercial cloud voices, still running entirely on your CPU. The first time you use them, Voxa downloads the model files automatically (the same way a game downloads assets on first launch), checks their integrity, and caches them. After that, no network is needed at all.

Why would a user care?
  • Start in minutes, not days. A newcomer can clone a sample and have a talking bot running before they've created a single cloud account. The biggest onboarding wall — "first, obtain three API keys" — disappears.
  • Free, deterministic CI. Tests can run a full microphone-to-speaker conversation on every pull request: no secrets in the build, no per-minute charges, no flaky network calls, no rate limits.
  • Air-gapped deployments. Hospitals, factories, defence networks, and ships have no route to api.openai.com. With models pre-copied onto the box and one config flag, Voxa runs entirely inside the firewall.
  • Privacy by default. Audio never leaves the machine — relevant wherever recordings are sensitive (healthcare, legal, HR).
  • No surprise bills while developing. Iterating on a voice UI means hundreds of test utterances a day. Locally they cost nothing.
What does switching look like?

It is a two-line config change. Nothing else in the five-line program from VDX-001 moves:

"Voxa": {
  "Stt": "WhisperCpp",   // was "OpenAI" — no API key needed
  "Tts": "Piper",        // was "ElevenLabs" — "Piper" is fastest, "Kokoro" sounds best
  "Agent": { "Provider": "Echo" }   // optional: a local parrot agent for key-free demos & CI
}
What's the catch?

Local models are smaller than cloud ones, so transcription is a little less accurate and the voices a little less polished than ElevenLabs — though the Kokoro option narrows that gap considerably at the cost of more CPU. That is the right trade for development, CI, privacy-sensitive and offline scenarios — and because providers are just config, users can develop locally and flip to a cloud provider for production with one line.

2.Executive summary

VDX-001 made composition trivial: AddVoxa(config) + MapVoxaVoice("/voice").UseDefaults() and the provider registry resolves "Stt": "OpenAI" to a descriptor. But every STT/TTS descriptor in the registry today fronts a metered cloud API. The five-line bot still has a hidden sixth line: go get API keys. That blocks three audiences the ROADMAP P6 entry names explicitly: developers who want to try Voxa without accounts, CI pipelines that want full-conversation tests at zero cost, and air-gapped deployments that cannot reach a cloud endpoint at all.

This spec adds the local tier as three ordinary provider packages plus the shared plumbing they need:

  1. VoxaModelCache (WS0) — a small model resolver in Voxa.Speech.Abstractions. Silero's model is 1 MB and ships embedded in the assembly; Whisper models are 75–488 MB and Piper voices 20–120 MB, so embedding is off the table. The cache implements the other half of the roadmap's "embedded or first-run download" rule: resolve from explicit path → local cache → verified download (SHA-256, atomic rename), with an Offline flag that turns a missing model into an actionable startup error instead of a surprise network call.
  2. Voxa.Speech.WhisperCpp (WS1) — an ISpeechToTextEngine over Whisper.net (the maintained MIT .NET binding of whisper.cpp). Whisper is an utterance transcriber, not a streaming recognizer — which matches Voxa's pipeline shape exactly: the VAD already gates audio and SpeechToTextProcessor already calls FlushAsync() on UserStoppedSpeakingFrame, so the engine buffers PCM during speech and transcribes once per utterance. Ships a VoxaSttDescriptor named "WhisperCpp".
  3. Voxa.Speech.Piper (WS2) — an ITextToSpeechEngine that drives the official piper executable as a warm child process (process isolation deliberately keeps GPL-licensed espeak-ng out of our MIT package — §6.WS2.1). Voxa's SentenceAggregator already feeds TTS one sentence at a time, so per-sentence synthesis latency (~200–800 ms on CPU) is the natural unit. Ships a VoxaTtsDescriptor named "Piper".
  4. Voxa.Speech.Kokoro (WS3) — the quality tier: an ITextToSpeechEngine running Kokoro-82M (Apache-2.0, StyleTTS2-based, 24 kHz, 54 voices across 8 languages) in-process on ONNX Runtime — already in the dependency tree via SileroVad. The GPL isolation pattern inverts Piper's: only phonemization leaves the process (a stateless per-sentence espeak-ng CLI call), so there is no long-lived child process and no utterance-demarcation problem. Markedly more natural than Piper; heavier on CPU. Ships a VoxaTtsDescriptor named "Kokoro".
  5. Echo agent (WS4) — "Voxa:Agent:Provider": "Echo": a trivial built-in agent that repeats the transcript back. Without it, the "no API keys" promise dies at the LLM hop; with it, a complete audio-in → audio-out conversation runs with zero credentials — exactly what CI and first-touch demos need. (Real local LLMs via Ollama/MEAI remain a host concern — §12.)
  6. Integration (WS5) — all three local descriptors registered by the Voxa meta-package, VoxaDefaultsGuard extended so model availability is a startup error in offline mode and an eager warm-up otherwise, an offline sample config, the flagship CI test (LocalConversationEndToEnd: WAV in → Whisper transcript → Echo → Piper PCM out, network access blocked), and docs.

Expected outcome: API keys required for a working dev bot drops from ≥ 2 to 0; a full-conversation e2e test runs on every PR at $0 with no secrets; an air-gapped install is a documented, tested path (Offline: true + pre-provisioned cache); and the provider menu gains its first entries whose Validate never asks for a credential. Everything is additive — no public API changes, no wire-protocol changes, no behavior change for existing cloud configs.

3.2 Non-goals

  • Not matching cloud STT accuracy or ElevenLabs voice quality. The local tier is positioned for dev/CI/offline/privacy, not as the flagship production voice.
  • Not GPU acceleration in v1. Whisper.net has CUDA/CoreML runtime packages; the descriptor leaves a config seam (UseGpu) but only the CPU runtime ships and is tested. (§12)
  • Not streaming/interim transcripts from Whisper. v1 emits final-only transcriptions per utterance; the UI simply shows no partials. Sliding-window partial decode is deferred. (§12)
  • Not a local LLM. The Echo agent closes the loop for CI/demos; real local inference (Ollama, ONNX LLMs) is already reachable today by registering an IChatClient in DI and is documented, not implemented. (§12)
  • Not a general model-management CLI (voxa models pull etc.). The cache is library-internal; a CLI is deferred. (§12)

3.3 Hard rules for the implementer

  • Additive only. No changes to ISpeechToTextEngine / ITextToSpeechEngine, the wire protocol, or any existing descriptor. Existing cloud configs must behave byte-for-byte identically.
  • Never download silently at request time. Model resolution happens at startup (guard/eager warm-up) or fails at startup. A WebSocket request must never block on a 250 MB download.
  • Fail with the fix in the message. Every resolution failure names the config key, the path(s) probed, and the exact remediation (the URL to fetch, the flag to flip, or the key to set) — the VDX-001 standard.
  • No GPL code linked into Voxa assemblies. espeak-ng (GPL-3.0) stays behind a process boundary — for Piper via the piper executable (§6.WS2.1), for Kokoro via per-sentence CLI calls (§6.WS3.1). Never reference packages that link espeak-ng or copy its natives into the consumer's output (this rules out the KokoroSharp NuGet — §6.WS3.1). whisper.cpp / Whisper.net / Kokoro-82M / Piper / ONNX Runtime are MIT/Apache and fine to link.
  • Honour the effective-rate contract from VDX-001. Descriptors must report effective sample rates via the existing GetEffectiveInputSampleRate/GetEffectiveOutputSampleRate mechanism so the session envelope, VAD, and WebSocketAudioSource stay in agreement (the defect class fixed in 2ee5686/21101f4 must not regress here).
  • Tests poll, never sleep-and-assert for positive assertions (the flaky-test rule established repo-wide). Local inference is the slowest thing this repo will ever run in CI — budgets must be generous and condition-based.

4.Architecture context — where local engines plug in

Nothing about the pipeline shape changes. The local engines are ordinary providers behind the existing abstractions; only the box contents differ:

TODAY (cloud) THIS SPEC (local) WebSocket PCM in ──▶ SileroVAD ──▶ SpeechToTextProcessor ──▶ ... same pipeline, same frames │ │ ▼ ▼ OpenAI / Azure STT (HTTPS) WhisperCppSttEngine (in-proc, CPU) buffers utterance, transcribes on Flush ... agent ──▶ SentenceAggregator ──▶ TextToSpeechProcessor ──▶ WebSocket PCM out │ │ ▼ ▼ ElevenLabs / Mistral TTS (HTTPS) PiperTtsEngine ──(stdin/stdout)──▶ warm piper child process (GPL isolated) — or — KokoroTtsEngine (in-proc ONNX) ──▶ espeak-ng CLI per sentence (GPL isolated)

Three existing seams carry all the weight, which is why the workstreams are small:

Existing seamWhereWhy it fits the local tier
SpeechToTextProcessor calls FlushAsync() on UserStoppedSpeakingFramesrc/Voxa.Speech.Abstractions/ISpeechToTextEngine.csWhisper is an utterance transcriber, not a streaming recognizer. VAD-gated flush is its natural invocation model — the engine just buffers between flushes. No interface change needed.
SentenceAggregator precedes TTS in the default chainDefaultVoicePipelineComposerPiper and Kokoro both synthesize a sentence in one pass. The aggregator already delivers exactly that unit, so "non-streaming per sentence" is invisible to the rest of the pipeline.
Descriptor effective-rate resolutionVoxaSttDescriptor.GetEffectiveInputSampleRate / VoxaTtsDescriptor.GetEffectiveOutputSampleRateWhisper requires 16 kHz mono (matches the existing STT default). Piper voices are 22 050 Hz (medium/high) or 16 000 Hz (low/x_low) — resolvable from the voice name without loading the model, via the descriptor's ResolveOutputSampleRate hook. Kokoro is fixed at 24 000 Hz — identical to the cloud-TTS default, so swapping a cloud voice for Kokoro leaves the session envelope unchanged.
Latency model honesty. Cloud streaming STT emits interim transcripts while the user is still talking and finals shortly after they stop. WhisperCpp emits nothing until the VAD closes the utterance, then one final after RTF×duration. Perceived turn latency therefore grows by roughly the transcription time (~0.5–2.5 s depending on model). The docs (WS5) must state this plainly and recommend tiny.en/base.en + the LowLatency profile for interactive local dev.

5.Model & binary resolution — the "like Silero, but bigger" problem

The roadmap says models are "resolved like Silero's (embedded or first-run download)". Silero takes the embedded branch because its ONNX model is ~1 MB (SileroVadEngine.LoadEmbeddedModel reads it from assembly resources). The local-tier artifacts cannot:

ArtifactSource of truthSizeStrategy
Silero VAD ONNX (existing)embedded resource~1 MBEmbedded (unchanged)
Whisper GGML models (tiny…small, .en + quantized variants)huggingface.co/ggerganov/whisper.cpp75–488 MBFirst-run download → cache
Piper voices (.onnx + .onnx.json pairs)huggingface.co/rhasspy/piper-voices20–120 MBFirst-run download → cache
Piper executable (per-RID)piper GitHub releases (pinned version)~20 MBFirst-run download → cache (or found on PATH)
Kokoro ONNX model (fp32 / fp16 / int8)pinned kokoro-onnx export on Hugging Face~88–330 MBFirst-run download → cache
Kokoro voice style vectorshuggingface.co/hexgrad/Kokoro-82M~0.5 MB eachFirst-run download → cache (per configured voice)
espeak-ng CLI (per-RID, Kokoro G2P)pinned static builds, hosted as Voxa release artifacts~3–12 MBFirst-run download → cache (or found on PATH)

5.1 Resolution order (uniform for all artifacts)

  1. Explicit path wins: Voxa:WhisperCpp:ModelPath / Voxa:Piper:VoicePath / Voxa:Piper:ExecutablePath. If set and missing → startup error naming the key and the path. No fallback — an explicit path that's wrong is a config bug, not a download trigger.
  2. Cache hit: <cache-root>/whisper/ggml-base.en.bin etc. Cache root = VOXA_MODEL_CACHE env var → Voxa:Models:CachePath → OS default (%LOCALAPPDATA%\voxa\models on Windows, ~/.cache/voxa/models elsewhere).
  3. Download (unless Voxa:Models:Offline = true): from the pinned catalog URL, to <target>.partial, SHA-256 verified against the catalog, then atomic rename. Progress logged via ILogger at 10% increments. Concurrent resolution of the same artifact (parallel test hosts) is serialized with a lock file.
  4. Offline miss: error message includes the artifact name, the exact cache path expected, the catalog URL to fetch out-of-band, and the SHA-256 — i.e. a copy-pasteable air-gap provisioning instruction.
Pinned catalog, not "latest". The catalog (artifact → URL + SHA-256 + size) is a compiled-in static table per package version. Upstream re-uploads or tampering fail the hash check loudly. Bumping a model version is a code change with a test, never a floating reference.

WS0 requirements

  • Atomic + verified: download to <path>.partial, hash while streaming (no second read pass), rename on match, delete on mismatch with an error that prints expected/actual hashes.
  • Cross-process safety: a <path>.lock file (FileStream exclusive create, retry loop with timeout) serializes concurrent first-run downloads from parallel test hosts; the loser of the race re-probes the cache and returns the winner's file.
  • Validate-vs-resolve split: Validate-time code may only call IsCached; ResolveAsync (which may download) runs from the startup guard / engine StartAsync. This is what keeps "never download at request time" and "validation has no side effects" simultaneously true.
  • HttpClient discipline: use IVoxaHttpClientProvider when available (proxy support in locked-down corp networks — precisely the audience for this feature), falling back to a static shared client. Timeout generous (models are large): 30 min per artifact, cancellable.
  • Tests (no real network): fake HttpMessageHandler — happy path, hash mismatch deletes partial + throws with both hashes, offline miss message contains cache path + URL + SHA-256, lock-file race (two concurrent resolves, one download), env-var/config/default cache-root precedence.

WS1 — Voxa.Speech.WhisperCpp: STT engine + descriptor Phase 1Additive

Package: Voxa.Speech.WhisperCpp (new) Deps: Whisper.net + Whisper.net.Runtime (CPU; MIT), Voxa.Speech.Abstractions Registry name: "WhisperCpp", config section "WhisperCpp"

WS1.1 Engine model — buffer, flush, transcribe

Whisper transcribes a complete utterance; Voxa's VAD decides what an utterance is. The engine is therefore a buffer with a transcriber attached:

src/Voxa.Speech.WhisperCpp/WhisperCppSttEngine.cs (new — behavioral contract) 
public sealed class WhisperCppSttEngine : ISpeechToTextEngine
{
    // StartAsync: ResolveAsync the GGML model (may download on cold cache — startup-path only,
    // the guard has normally warmed it already), then get/create the shared WhisperFactory.
    //   • WhisperFactory is cached per model path in a static ConcurrentDictionary — model
    //     weights load once per process (~140 MB for base.en), NOT once per connection.
    //   • A WhisperProcessor is created per engine instance (per connection) — cheap, holds
    //     decode state. Builder: .WithLanguage(opts.Language) or .WithLanguageDetection(),
    //     .WithThreads(opts.Threads ?? Math.Min(4, Environment.ProcessorCount)).

    // WriteAudioAsync: convert 16-bit LE PCM → float into a pooled, growable buffer.
    //   • Engine is mono 16 kHz (PreferredInputSampleRate). Mismatched input is a config bug
    //     upstream — the composer already aligns source/VAD/STT rates (VDX-001).
    //   • Hard cap: 30 s of audio (Whisper's context window). On overflow, transcribe the
    //     buffered 30 s as a final segment and continue buffering — never drop audio silently
    //     (log a warning; a 30 s monologue between VAD stops is rare but legal).

    // FlushAsync (called by SpeechToTextProcessor on UserStoppedSpeakingFrame):
    //   • No buffered audio (< ~0.3 s) → no-op; avoids hallucinated transcripts on VAD blips.
    //   • Else run processor.ProcessAsync(samples) on a worker (Task.Run — whisper.cpp is
    //     CPU-bound native code; never block the pipeline data loop), concatenate segments,
    //     write one TranscriptionResult(text, IsFinal: true) to the transcript channel, clear
    //     the buffer. Serialized: a flush that arrives while a transcription is running queues
    //     behind it (per-engine SemaphoreSlim(1,1)).

    // ReadTranscriptsAsync: drains the bounded channel; completes on StopAsync.
    // StopAsync: final implicit flush, complete channel. DisposeAsync: processor only —
    // the factory is shared and process-lifetime.
}
No interim transcripts in v1. The engine only ever emits IsFinal: true. TranscriptionFilter and the agent loop already ignore non-final/empty transcripts, so the absence of partials is behaviorally safe — it only affects UI sparkle. A sliding-window partial decoder is §12 material.

WS1.4 Tests

  • Unit (no model, fake transcriber seam): PCM→float conversion exactness; 30 s cap emits-and-continues; sub-0.3 s flush is a no-op; concurrent flush serialization; channel completes on StopAsync.
  • Descriptor: unknown model name lists catalog; offline + cold cache error contains path/URL/hash; InputSampleRate: 8000 override rejected; ModelPath bypasses catalog.
  • Integration (trait Category=LocalModels, runs in the cached-CI lane and locally): real tiny.en transcribes the checked-in jfk.wav fixture (the whisper.cpp canonical sample, ~350 KB); asserts case-insensitive containment of "country", not exact text — model updates must not flake the suite.

WS2 — Voxa.Speech.Piper: TTS engine + descriptor Phase 1Additive

Package: Voxa.Speech.Piper (new) Deps: Voxa.Speech.Abstractions only (the piper binary is a resolved artifact, not a NuGet dep) Registry name: "Piper", config section "Piper"

WS2.1 Why a child process, not in-proc ONNX

Piper voices are VITS models — ONNX Runtime (already in the dependency tree via SileroVad) could run them in-proc. The blocker is the front half: text→phoneme conversion requires espeak-ng, which is GPL-3.0. Linking it (P/Invoke counts) would force the package off MIT. The official piper executable bundles espeak-ng behind its own process boundary, which is the well-trodden GPL isolation line. Secondary benefits: zero native-interop surface for us to maintain, and piper's exact upstream phonemization behavior. The costs — process lifecycle management and a per-RID binary to resolve — are contained in this workstream. (Kokoro — WS3 — takes the inverse split: synthesis in-proc, only G2P out of process. For Piper, fully in-proc remains §12 material.)

WS2.2 Process model — warm host, per-utterance files

The two naive shapes both fail: spawn-per-utterance pays model load (~0.5–1.5 s) per sentence — kills latency; one process streaming raw PCM for many utterances gives no reliable utterance demarcation on the byte stream. The design that works:

src/Voxa.Speech.Piper/PiperProcessHost.cs (new — behavioral contract) 
// One long-lived piper process per (voice, options) key, started on first use:
//   piper --model <voice.onnx> --json-input --output_dir <session-temp-dir>
//
// Per synthesis request:
//   1. Write one JSON line to stdin: { "text": "...", "output_file": "<temp>/u-<seq>.wav" }
//   2. Await completion: piper prints the output path on stdout when an utterance is done.
//      Fallback detector (behind the same awaiter, for piper builds that don't echo):
//      poll for file-exists + size-stable-across-two-reads, 25 ms period.
//   3. Read the WAV, validate the header against the announced sample rate, strip it, hand
//      the PCM to the engine. Delete the temp file.
//
// Concurrency: SemaphoreSlim(1,1) per host — piper processes stdin lines sequentially, so
// in-flight = 1 per process. MaxProcesses (default 2) hosts per voice, leased round-robin,
// so two simultaneous calls don't serialize behind one process. Singleton registry in DI.
//
// Crash recovery: process exit while requests pending → fault pending awaiters with the
// captured stderr tail, restart lazily on next request. ErrorFrame semantics upstream are
// unchanged (TextToSpeechProcessor already maps engine exceptions).
src/Voxa.Speech.Piper/PiperTtsEngine.cs (new — behavioral contract) 
public sealed class PiperTtsEngine : ITextToSpeechEngine
{
    // StartAsync: resolve voice (.onnx + .onnx.json) and executable via VoxaModelCache
    // (ExecutablePath config → PATH probe → cache/download per-RID; chmod +x on unix);
    // parse the voice's .onnx.json; if json sample_rate ≠ the rate the descriptor announced,
    // throw a config-key-naming error (envelope already told the client the wrong rate —
    // that's a startup-fail, not a runtime mystery). Lease a PiperProcessHost.

    // SynthesizeAsync(text): one host round-trip; yield the PCM in ~4096-sample chunks
    // (per the ITextToSpeechEngine pooled-buffer contract — TextToSpeechProcessor copies).
    // Chunked yield keeps the sink's pacing/barge-in flush behavior identical to cloud TTS:
    // an interruption mid-sentence stops emission between chunks.

    // DisposeAsync: return the lease; hosts outlive connections, die with the app
    // (IHostApplicationLifetime hook kills child processes — no orphan piper.exe).
}

WS3 — Voxa.Speech.Kokoro: quality-tier TTS engine + descriptor Phase 1Additive

Package: Voxa.Speech.Kokoro (new) Deps: Microsoft.ML.OnnxRuntime (already in the tree via SileroVad), Voxa.Speech.Abstractions Registry name: "Kokoro", config section "Kokoro"

Kokoro-82M (Apache-2.0, StyleTTS2-based, 82M parameters, fixed 24 kHz output, 54 voices across 8 languages) is the local tier's quality option — markedly more natural than Piper, competitive with commercial cloud voices, at the cost of a heavier model and slower CPU inference. The positioning is stated explicitly everywhere it appears: Piper is the speed/CI tier; Kokoro is the voice you demo and deploy with.

WS3.1 Isolation pattern — Piper's split, inverted

Kokoro has the same GPL constraint as Piper — its grapheme-to-phoneme step depends on espeak-ng — but the shape of the problem is friendlier: synthesis itself is a plain ONNX graph (token ids + voice style vector + speed in, float PCM out), and only the small G2P front half touches GPL code. So WS2's split inverts: run synthesis in-process on ONNX Runtime (which Voxa already ships for SileroVad), and push only phonemization out of process as a per-sentence espeak-ng --ipa -q "<text>" CLI call — stateless, ~10–30 ms, output delimited by process exit. This eliminates the two ugliest Piper risks for this engine: there is no warm child process to manage (R7) and no utterance-demarcation heuristic (R1) — process EOF is the demarcation.

Do not reference the KokoroSharp NuGet. KokoroSharp is MIT-licensed and an excellent reference implementation, but it links espeak-ng in-proc and copies espeak-ng native binaries into the consumer's build output — both sides of the GPL line §3.3 forbids. We consume the published model contract directly: the ONNX graph and the phoneme→token-id vocabulary are documented artifacts. KokoroSharp informs the implementation; it does not enter the dependency graph.

WS3.2 Engine

src/Voxa.Speech.Kokoro/KokoroTtsEngine.cs (new — behavioral contract) 
public sealed class KokoroTtsEngine : ITextToSpeechEngine
{
    // StartAsync: resolve via VoxaModelCache — the model ONNX (per configured Precision), the
    // configured voice's style vector, and the espeak-ng CLI (EspeakPath config → PATH probe →
    // cache/download per-RID; chmod +x on unix). The InferenceSession is cached per model path
    // in a static ConcurrentDictionary (the WhisperFactory pattern, WS1): weights load once per
    // process (~330 MB fp32 / ~88 MB int8); engine instances are per-connection and cheap.

    // SynthesizeAsync(text) — one sentence per call (SentenceAggregator upstream guarantees it):
    //   1. espeak-ng CLI → IPA phoneme string. Per-sentence process; EOF = done; non-zero exit
    //      surfaces the stderr tail in the engine exception.
    //   2. IPA → token ids via the compiled-in vocabulary map. Hard cap 510 tokens (model
    //      context): split into multiple runs at phrase boundaries — never truncate silently.
    //   3. session.Run — one pass per sentence on a worker thread (CPU-bound native code; never
    //      block the pipeline data loop). Output float PCM @ 24 000 Hz → 16-bit LE.
    //   4. Yield ~4096-sample chunks per the ITextToSpeechEngine pooled-buffer contract —
    //      barge-in stops emission between chunks, identical to WS2 and the cloud engines.

    // Concurrency: InferenceSession.Run is thread-safe; a process-wide SemaphoreSlim
    // (MaxConcurrentSyntheses, default 2) caps parallel runs so synthesis can't starve the
    // audio pipeline's cores.

    // DisposeAsync: per-engine state only — the session is process-lifetime, and espeak-ng
    // calls leave nothing running (no orphan surface at all).
}

WS4 — Echo agent: closing the no-API-key loop Phase 1Additive

Package: Voxa (meta-package, alongside OpenAIChatAgentFactory) Config: "Voxa:Agent:Provider": "Echo" Size: ~60 lines

STT and TTS going local still leaves the agent hop demanding an OpenAI key — OpenAIChatAgentFactory.Validate rejects everything else by design. The smallest honest fix is a built-in parrot: Provider: "Echo" yields an AIAgent that streams back "You said: <transcript>". It exercises every pipeline stage (turn lifecycle, sentence aggregation, TTS, barge-in) with deterministic output — exactly what an e2e test wants — and it makes the first-touch demo fully credential-free.

  • Restructure: rename the internal factory to DefaultAgentFactory, dispatching on options.Provider — null/"OpenAI" → existing OpenAI path (unchanged), "Echo" → EchoAgent; anything else → the existing "register a custom IVoiceAgentFactory" error, now also naming "Echo" as a valid value. Validate("Echo") returns no errors unconditionally.
  • EchoAgent streams its reply in 2–3 chunks (not one) so sentence aggregation and incremental TTS are genuinely exercised in tests.
  • Docs position it as a diagnostic/demo agent, and point users who want a real keyless agent at registering an Ollama-backed IChatClient (the DI seam already wins over the factory — §12).
  • Tests: Validate matrix (Echo passes keyless; unknown provider error message lists Echo); guard end-to-end passes with Echo + no secrets; echoed text round-trips through a pipeline with fake STT/TTS.

WS5.1 Meta-package registration

10.Verification protocol

  1. Every commit: dotnet build Voxa.sln -warnaserror and the default test suite (LocalModels trait excluded) — the everyday loop must not get slower or need models.
  2. Phase exits: dotnet test --filter Category=LocalModels locally with a warm cache; then once with VOXA_MODEL_CACHE pointed at an empty dir to exercise first-run download end-to-end.
  3. Zero-network proof: the CI lane runs the LocalModels suite with outbound traffic blocked after cache restore. A test that secretly phones home fails the lane — this is the enforcement of the headline claim, not an aspiration.
  4. Air-gap drill (manual, once per release): follow docs/local-speech.md's runbook verbatim on a machine with networking disabled: provision cache from a USB-copied directory, set Offline: true, confirm startup, hold a conversation.
  5. Latency smoke (manual): the §3.1 RTF/first-audio budgets measured on the reference dev box via the benchmark harness; recorded in the PR description, not CI-gated (hardware variance would make a gate flaky — the lesson of the allocation-gate saga applies doubly to wall-clock).
  6. Regression sweep: the full existing suite (cloud descriptors, composer rate tests, route e2e) green — proves the additive-only rule held.

VDX-004 — Native C ABI / FFI for Non-.NET Embedding (Exploratory)

A feasibility frame for a flat C ABI that would let non-.NET hosts (C, Swift, Kotlin) embed the Voxa pipeline — the reach speech-core gets from its vtable C ABI. The deliverable here is a spike, not committed work: the central unknown is whether the orchestration core's dependency graph is NativeAOT-clean. The first viable surface is therefore Core-only — the host supplies STT/TTS/VAD/LLM as C function-pointer vtables, exactly like speech-core's sc_stt_vtable_t family — so the AOT image is just the pipeline core plus a thin C bridge. The managed SDK and CLI are untouched.

1.What is this? — a plain-English explainer

What would VDX-004 add, in one sentence?

A small C-callable shared library (.so/.dll/.dylib) that lets a program written in C — or Swift, or Kotlin — create a Voxa pipeline, push microphone audio into it, and receive transcripts and synthesized audio back through a callback, without hosting a .NET runtime itself.

Why might we want it?

speech-core (the C++17 engine this parity program studies) embeds into Android (via JNI) and Apple (via a Swift package) precisely because it exposes a flat C ABI — a handful of sc_pipeline_* functions and a couple of callback typedefs. Voxa is .NET-only. Today its only cross-language surfaces are the MCP server (VDX-002) and the CLI (VDX-003) — both out-of-process, both coarse-grained. A C ABI would be the in-process, real-time embedding path Voxa lacks.

Why "maybe", and why is it sequenced last?

Because Voxa is a predominantly-.NET project and nobody has asked for native embedding yet. This item is the program's only exploratory entry: it sits at milestone M6, after every measured turn-taking and model-breadth item, and the program's Open Decision #4 states it plainly — "is non-.NET embedding actually a goal? If not, do not start." The honest unknown underneath is technical: NativeAOT must statically compile the pipeline's entire dependency graph, and the meta-package pulls in Microsoft.Extensions.AI plus provider SDKs that may not be AOT-clean. We do not know if that compiles until we try.

This is exploratory and gated — read this first. Nothing in this spec is committed work. The deliverable is a feasibility spike that answers one question: is NativeAOT viable across the dependency graph the embedded pipeline actually needs? Implementation MUST NOT begin before (a) a concrete mobile/native consumer is an actual product goal, and (b) the spike in §7 has returned a green answer. Until then this document is a placeholder that frames the work, not a build ticket.
What does it explicitly not change?

The managed SDK (NuGet packages), the AddVoxa / composer layer, the CLI, and the MCP server are all untouched. Voxa.Core stays zero-dependency. The C bridge lives in a separate new project (Voxa.Native) that references Core; it never modifies it. Mobile bindings (an Android .aar, a Swift package) are consumers of this ABI and live in their own repos — exactly as speech-core's mobile bindings are siblings of the core engine.

2.Executive summary

This is a feasibility frame plus a placeholder, not committed work. The single deliverable that matters is a spike answering "is NativeAOT viable across the needed dependency graph?" Everything past the spike is conditional on both a real consumer and a green spike result. Do not treat the ABI sketch below as a contract to build to on sight — treat it as the shape the spike validates.

VDX-004 explores giving Voxa the cross-language reach speech-core has. Conceptually it is three workstreams, only the last of which is unconditional:

  • WS1 — the C ABI surface design. A flat, C-callable contract over the existing PipelineRunner lifecycle: voxa_pipeline_create / start / push_audio / stop / destroy, plus a single event-callback typedef that surfaces transcripts, output PCM, errors and state, with an opaque user-context pointer — mirroring speech-core's sc_pipeline_create / sc_pipeline_push_audio / sc_event_fn so the contract is immediately familiar to anyone who has integrated speech-core.
  • WS2 — bring-your-own-engine vtables core-only. The load-bearing reconciliation. Rather than AOT-compile the full meta-package, the first viable image is the orchestration core only; the host supplies STT/TTS/VAD/LLM as C function-pointer vtables — exactly speech-core's sc_stt_vtable_t / sc_tts_vtable_t / sc_vad_vtable_t / sc_llm_vtable_t design. This keeps the AOT surface small and avoids dragging provider SDKs into the image at all.
  • WS3 — the feasibility spike the gate. The actual work: determine which assemblies are AOT-clean, how audio buffers marshal across the boundary, callback re-entrancy and threading, binary size, and per-platform build/CI. This gates everything else.
Net position. If the spike says "the core subset is AOT-clean and the vtable bridge marshals cleanly," Voxa gains a credible native-embedding story with a small, defensible surface. If it says "the dependency graph won't AOT," we have an evidence-backed answer to Open Decision #4 and we stop — at the cost of a spike, not a shipped feature.

3.Goals, non-goals & budgets

Goals (of the exploration)

  • Answer, with evidence, whether a NativeAOT-compiled subset of Voxa can be driven from C — the prerequisite for any mobile/native embedding.
  • Define a flat C ABI that maps onto the existing PipelineRunner lifecycle and surfaces the frames a host actually needs (transcripts, output audio, errors, state) via one callback.
  • Establish that the first viable image is Core-only with bring-your-own-engine vtables — keeping the AOT surface small and matching speech-core's own seam discipline.
  • Prove the round-trip end-to-end with a tiny C smoke consumer using fake / bring-your-own engines (no models, no network).

Non-goals (this spec)

  • Mobile bindings. An Android .aar (JNI) or a Swift package are consumers of this ABI and belong in separate repos — the same way speech-core's Android/Apple bindings are siblings of the engine, not part of it.
  • Bringing local models into the AOT image. whisper.cpp / Piper / Kokoro stay external and SHA-256-pinned via VoxaModelCache; nothing about this item pulls model weights or their native runtimes into the shared library.
  • AOT-compiling the full meta-package. If the provider SDKs (Microsoft.Extensions.AI, OpenAI/Azure) are not AOT-clean, the answer is the Core-only vtable surface, not a campaign to make every dependency AOT-clean.
  • Any committed delivery before a real consumer exists. Per Open Decision #4, absent a mobile/native goal this stays a specced placeholder.

4.Architecture context — the boundary

The shape is a single new NativeAOT shared library, Voxa.Native, that exposes a flat C ABI via [UnmanagedCallersOnly] exports and drives the existing managed pipeline behind them. The non-.NET host links the library and calls it like speech-core's sc_pipeline_* surface — create a pipeline, push audio, receive a stream of events, destroy. Internally those exports translate directly onto the PipelineRunner lifecycle that the CLI (VDX-003) already drives in-process:

  C / Swift / Kotlin host
        |  (links the shared library; calls flat C functions)
        v
  voxa_pipeline_*  ............  [UnmanagedCallersOnly] exports  (Voxa.Native, NativeAOT)
        |
        v
  PipelineRunner.StartAsync / Source.IngestAsync(AudioRawFrame) / StopAsync   (Voxa.Core — unchanged)
        |                                                   ^
        |  downstream frames                                |  host-supplied engine vtables (WS2)
        v                                                   |
  PipelineSink  -->  marshal TranscriptionFrame / output PCM / ErrorFrame / state  -->  voxa_event_fn callback

The managed analogue already exists: VoxaCliRunner (src/Voxa.Cli/) starts engines, writes audio, and reads results headlessly in-process. VDX-004 is the same headless drive, but the caller is across an FFI boundary and the results arrive via a C callback instead of an IAsyncEnumerable.

Mapping speech-core's surface to the proposed Voxa C ABI

The point of the table is fidelity: a developer who has embedded speech-core should recognise every row.

speech-core (sc_*)Proposed Voxa C ABIMaps onto (managed)
sc_pipeline_create(cfg)voxa_pipeline_create(cfg, ev, user)Build the composed Pipeline + PipelineRunner
sc_pipeline_start()voxa_pipeline_start(h)PipelineRunner.StartAsync(StartFrame)
sc_pipeline_push_audio(pcm, n)voxa_pipeline_push_audio(h, pcm, n)Source.IngestAsync(new AudioRawFrame(...))
sc_pipeline_stop()voxa_pipeline_stop(h)PipelineRunner.StopAsync(grace)
sc_pipeline_destroy()voxa_pipeline_destroy(h)PipelineRunner.DisposeAsync()

sc_stt_vtable_t et al.voxa_stt_vtable_t et al. (WS2)Adapter that implements ISpeechToTextEngine by calling back through the vtable
Invariant. The frame/processor contract is unchanged. The C ABI is a new front door onto the same PipelineRunner the CLI uses; it adds no frame types, no cancellation model, and no processor behaviour. Audio in is still an AudioRawFrame with an explicit sample rate / channel count fixed by the StartFrame at session start.

5.WS1 — the C ABI surface

Project Voxa.Native (new) Risk Medium (marshalling) Surface flat C, opaque handle Models none (engines are external)

A flat C header, hand-written (or generated and checked in), with an opaque pipeline handle and a single event callback. It is deliberately a near-transliteration of speech-core's surface so the mental model carries over. Everything below is the shape the spike validates — exact field layout and threading guarantees are pinned by the spike, not ahead of it.

include/voxa.h (sketch — validated and finalized by the WS3 spike) 
/* Opaque handle. One per session. Created/destroyed on the host thread. */
typedef struct voxa_pipeline voxa_pipeline;

/* Result codes. 0 == ok; negative == error. Mirrors speech-core's int-return convention. */
typedef enum {
    VOXA_OK              =  0,
    VOXA_ERR_INVALID_ARG = -1,   /* null handle / null buffer / bad config */
    VOXA_ERR_STATE       = -2,   /* start/stop called out of order */
    VOXA_ERR_AOT         = -3,   /* a managed feature unavailable in the AOT image */
    VOXA_ERR_INTERNAL    = -4    /* unexpected managed exception (also surfaced as an event) */
} voxa_status;

/* Event kinds delivered to the host callback. */
typedef enum {
    VOXA_EV_TRANSCRIPT      = 1,  /* TranscriptionFrame  -> data.transcript      */
    VOXA_EV_AUDIO_OUT       = 2,  /* output AudioRawFrame (bot PCM) -> data.audio */
    VOXA_EV_STATE           = 3,  /* Bot/User Started/Stopped speaking -> data.state */
    VOXA_EV_ERROR           = 4   /* ErrorFrame -> data.error (UTF-8, NUL-terminated) */
} voxa_event_kind;

/* Borrowed, callback-lifetime views. Pointers are INVALID after the callback returns;
   the host must copy anything it needs to keep. This is the speech-core convention. */
typedef struct { const char* text; int   is_final; const char* language; } voxa_transcript_evt;
typedef struct { const float* pcm;  int   frame_count; int sample_rate; int channels; } voxa_audio_evt;
typedef struct { int          speaking; int is_user; } voxa_state_evt;   /* speaking=1 start, 0 stop */
typedef struct { const char*  message; } voxa_error_evt;

typedef struct {
    voxa_event_kind kind;
    union {
        voxa_transcript_evt transcript;
        voxa_audio_evt      audio;
        voxa_state_evt      state;
        voxa_error_evt      error;
    } data;
} voxa_event;

/* The single callback. `user` is the opaque context passed to voxa_pipeline_create.
   Re-entrancy & threading guarantees are defined by the spike (see WS3). */
typedef void (*voxa_event_fn)(const voxa_event* ev, void* user);

/* Session config. POD only — no managed types cross the boundary.
   input_* describe the PCM the host will push; the C ABI fixes the StartFrame from these. */
typedef struct {
    int   input_sample_rate;   /* e.g. 16000 */
    int   input_channels;      /* e.g. 1 */
    int   abi_version;         /* set to VOXA_ABI_VERSION; rejected if it doesn't match */
} voxa_config;

#define VOXA_ABI_VERSION 1

/* ---- lifecycle: maps 1:1 onto PipelineRunner ---- */
voxa_status voxa_pipeline_create(const voxa_config* cfg, voxa_event_fn ev, void* user, voxa_pipeline** out);
voxa_status voxa_pipeline_start(voxa_pipeline* h);
voxa_status voxa_pipeline_push_audio(voxa_pipeline* h, const float* pcm, int frame_count);  /* -> AudioRawFrame */
voxa_status voxa_pipeline_stop(voxa_pipeline* h);            /* injects EndFrame, drains within grace */
voxa_status voxa_pipeline_destroy(voxa_pipeline* h);         /* DisposeAsync; idempotent; null-safe */

/* Static, no handle: a version string for diagnostics. */
const char* voxa_version(void);

On the managed side these are thin [UnmanagedCallersOnly] wrappers. Each one: validates pointers (null → VOXA_ERR_INVALID_ARG, never a crash), translates to the corresponding PipelineRunner / PipelineSource call, and bridges the async lifecycle to the synchronous C contract (the spike decides whether that is a blocking wait or a dedicated pump thread — see WS3). Managed exceptions never cross the boundary: they are caught at the wrapper, returned as a voxa_status, and additionally surfaced as a VOXA_EV_ERROR.

Null-safety convention. Every entry point is null-tolerant: a null handle or buffer returns VOXA_ERR_INVALID_ARG; destroy(NULL) is a no-op. Callback-delivered pointers are borrowed and valid only for the duration of the callback — the host copies what it keeps. This matches speech-core's borrowed-view discipline and keeps ownership unambiguous across the FFI line.

6.WS2 — the bring-your-own-engine vtable approach

Project Voxa.Native (new) Risk High (the central unknown) Why keeps the AOT image small Pattern speech-core sc_*_vtable_t

This is the workstream that makes the whole item tractable. The naive design — AOT-compile the full meta-package so the C side gets "real" Whisper / OpenAI / Kokoro for free — is exactly the part most likely to fail: Microsoft.Extensions.AI and the provider SDKs use reflection, dynamic code, and source-generated serialization patterns that NativeAOT may reject or trim incorrectly. So the first viable surface is Core-only: the AOT image contains the orchestration core and the C bridge, and the host supplies the engines as C function-pointer vtables — precisely speech-core's design, where sc_stt_vtable_t, sc_tts_vtable_t, sc_vad_vtable_t and sc_llm_vtable_t are how the C++ core stays a pure orchestrator while the embedder plugs in backends.

On the managed side, each vtable is wrapped by a small adapter that implements the existing engine interface (ISpeechToTextEngine, ITextToSpeechEngine, …) by marshalling calls out to the host's function pointers. The composer wires those adapters into the chain exactly as it wires native engines today — the pipeline does not know the difference.

include/voxa_engines.h (sketch — the BYO-engine seam, mirrors sc_*_vtable_t) 
/* The host implements these and hands the struct to voxa at create-time.
   `ctx` is the host's own per-engine state. Buffers are borrowed for the call. */

typedef struct {
    void* ctx;
    /* feed mic PCM; return 0 on success. Mirrors ISpeechToTextEngine.WriteAudioAsync. */
    int  (*write_audio)(void* ctx, const float* pcm, int frame_count);
    /* drain a final/interim transcript; out_text is host-owned, freed by free_text. */
    int  (*read_transcript)(void* ctx, char** out_text, int* out_is_final);
    void (*free_text)(void* ctx, char* text);
    void (*reset)(void* ctx);
} voxa_stt_vtable_t;

typedef struct {
    void* ctx;
    /* synthesize `text`; the host invokes `sink` zero+ times with PCM chunks. */
    int  (*synthesize)(void* ctx, const char* text,
                       void (*sink)(const float* pcm, int frame_count, void* sink_user),
                       void* sink_user);
    int  sample_rate;     /* output rate the host will produce */
} voxa_tts_vtable_t;

typedef struct {
    void* ctx;
    /* return speech probability [0,1] for a frame; the VAD processor thresholds it. */
    float (*score)(void* ctx, const float* pcm, int frame_count);
} voxa_vad_vtable_t;

typedef struct {
    void* ctx;
    /* stream the assistant turn; host pushes token chunks via `emit`. */
    int  (*run_turn)(void* ctx, const char* user_text,
                     void (*emit)(const char* token, void* emit_user),
                     void* emit_user);
} voxa_llm_vtable_t;

/* Extended create that takes the engine vtables. Any field NULL => that stage is omitted
   (e.g. STT+LLM+TTS but no separate VAD). This is the Core-only entry point. */
voxa_status voxa_pipeline_create_with_engines(
    const voxa_config*        cfg,
    const voxa_stt_vtable_t*  stt,
    const voxa_tts_vtable_t*  tts,
    const voxa_vad_vtable_t*  vad,
    const voxa_llm_vtable_t*  llm,
    voxa_event_fn             ev,
    void*                     user,
    voxa_pipeline**           out);
Why this is the right first surface. It keeps the AOT image to the orchestration core plus a thin bridge — the smallest possible thing to prove AOT-viable — and it is architecturally identical to how speech-core ships. A later, fatter image that AOT-compiles selected managed engines (if any prove AOT-clean) is a strict superset; the vtable surface remains the floor that always works.
Relationship to voxa_pipeline_create (WS1). The simple voxa_pipeline_create is the "batteries-included" aspiration; voxa_pipeline_create_with_engines is the surface the spike actually targets first. If the spike finds no managed engine AOT-compiles, the plain create may be deferred entirely and the vtable create becomes the only constructor.

7.WS3 — the feasibility spike (the gate)

Type spike / investigation Risk High — answer unknown Output a go / no-go memo + a smoke binary Gates all committed work

The spike is the actual content of VDX-004 today. It is a time-boxed investigation whose output is an evidence-backed go / no-go on native embedding, plus the minimal artifacts that prove (or disprove) the round-trip. It must answer, concretely:

QuestionWhat "answered" looks like
Which assemblies are AOT-clean?A published NativeAOT build of Voxa.Core + the bridge, with the trim/AOT analyzer warnings triaged. A list of any Core constructs that need adjustment (reflection, dynamic) — Core is small and zero-dep, so this is the most likely-to-pass piece.
Do provider SDKs AOT-compile?A yes/no per candidate (Microsoft.Extensions.AI, OpenAI, Azure, the local speech engines). The expected answer is "not all of them" — which is why WS2 exists. The spike confirms the vtable surface is the fallback, not a guess.
How do audio buffers marshal?A measured decision: pin-and-pass vs copy for the mic-in float* and the audio-out callback. Confirms no GC hole (pinned spans for the call duration; copies for anything retained past the callback).
Callback re-entrancy / threading.A defined contract: on which thread the callback fires, whether the host may call voxa_pipeline_* from inside it, and how the async PipelineRunner bridges to synchronous C (blocking wait vs a dedicated pump thread + queue).
Binary size.A number for the published shared library per platform — input to the "is this worth it?" decision.
Per-platform build & CI.A working linux-x64 publish at minimum, plus the cost estimate for win-x64 / osx-arm64 (NativeAOT cross-compilation constraints; each target generally needs its own runner).
This is the gate. No committed WS1/WS2 work begins until the spike returns green and a real mobile/native consumer is an actual goal. A red spike ("the needed dependency graph won't AOT") is a successful outcome — it resolves Open Decision #4 with evidence and the item is closed as "not pursued," not left ambiguous.

8.Plan of action — phases

  1. Phase 0 — Gate check (no code). Confirm a concrete mobile/native consumer is an actual product goal. If not, stop here: the item stays a specced placeholder (Open Decision #4). This phase is a decision, not a branch.
  2. Phase 1 — The feasibility spike (WS3). Time-boxed. Publish a NativeAOT build of Voxa.Core + a throwaway bridge; triage AOT/trim warnings; settle buffer marshalling and callback threading; measure binary size; get one platform building in CI. Output: a go / no-go memo. Everything below is conditional on a green memo here.
  3. Phase 2 — Core-only ABI + C smoke consumer (WS1 + WS2). Only on green. A real src/Voxa.Native project: the flat C ABI (WS1) over voxa_pipeline_create_with_engines (WS2), driving the composed pipeline with fake / bring-your-own engines, plus the checked-in C smoke consumer and its CI lane.
  4. Phase 3 — Widen the surface (only if a consumer exists and asks). Optionally AOT-compile selected managed engines that proved AOT-clean (the "batteries-included" voxa_pipeline_create); harden additional platforms. Do not proceed past Phase 2 without a real consumer driving the requirement.
Per-PR discipline still applies. Even exploratory, each landed change is one logical change, branches from main, carries an [Unreleased] CHANGELOG entry, and is warnings-as-errors clean. This spec is the spec-before-code artifact for the item; the spike's memo is recorded alongside it before any Phase 2 code lands.

10.Verification protocol

The verification artifact is a tiny C smoke consumer in CI — the same round-trip speech-core's own tests use, and the one the program names for this item. It exercises the full ABI without any model or network: a fake (bring-your-own) STT vtable returns a canned transcript for the pushed audio.

tests/native/smoke.c (intent — the CI round-trip) 
/* fake STT: ignores audio, yields one final transcript on read. */
static int fake_read_transcript(void* ctx, char** out, int* is_final) {
    *out = strdup("hello voxa");   *is_final = 1;   return 0;
}
static void on_event(const voxa_event* ev, void* user) {
    if (ev->kind == VOXA_EV_TRANSCRIPT && ev->data.transcript.is_final)
        *(int*)user = 1;                       /* mark: we got the final transcript */
}

int main(void) {
    voxa_config cfg = { .input_sample_rate = 16000, .input_channels = 1, .abi_version = VOXA_ABI_VERSION };
    voxa_stt_vtable_t stt = { .ctx = NULL, .read_transcript = fake_read_transcript, /* ... */ };
    int got = 0;
    voxa_pipeline* h = NULL;
    if (voxa_pipeline_create_with_engines(&cfg, &stt, NULL, NULL, NULL, on_event, &got, &h) != VOXA_OK) return 2;
    voxa_pipeline_start(h);
    float pcm[160] = {0};                        /* 10 ms of silence @ 16 kHz */
    voxa_pipeline_push_audio(h, pcm, 160);
    voxa_pipeline_stop(h);                        /* drains; on_event fires with the transcript */
    voxa_pipeline_destroy(h);
    return got ? 0 : 1;                           /* 0 == round-trip proven */
}

12.Out of scope / deferred

  • Mobile bindings (Android .aar / Swift package). These are consumers of the C ABI and belong in separate repos — the same relationship speech-core's mobile bindings have to its core engine. Not built here.
  • AOT-imaging local models. whisper.cpp / Piper / Kokoro stay external, out-of-process where required, and SHA-256-pinned via VoxaModelCache. No model weights or native runtimes enter the shared library.
  • The "batteries-included" voxa_pipeline_create with managed engines baked in. Deferred to Phase 3 and only for whichever managed engines the spike proves AOT-clean; the BYO-vtable surface is the floor.
  • Anything before the spike + a real consumer. Per Open Decision #4, absent a mobile/native goal this item stays a specced placeholder. The MCP server (VDX-002) and the CLI (VDX-003) remain Voxa's cross-language surfaces in the meantime.
  • Cross-language surfaces other than the C ABI (e.g. a gRPC or WASM front end). Out of scope for this item; they would be their own exploration if demand appears.

VRT-006 — Composable turn-taking: interruption gating & turn-end strategies

Barge-in works (VRT-002 WS2 / PR #100) — but it is a hair trigger: any detected user speech during a bot turn cancels the answer. The first "mm-hm", cough, or half-second of crosstalk kills the reply mid-sentence. Separately, "the user's turn is over" is a single hardwired policy (VAD silence, optionally confirmed by a smart-turn classifier) with no seam for transcript-aware gating or for STT engines that detect end-of-turn server-side. This spec turns both decisions into small, composable policies with honest defaults.

1. Problem

1.1 Interruption is ungated

Today AgentLoopProcessor reacts to UserStartedSpeakingFrame during an in-flight turn by cancelling the driver enumeration and pushing an InterruptionFrame downstream (AgentLoopProcessor.cs, cancelTurnOnBargeIn). That is correct mechanics with a wrong policy: VAD speech onset is not evidence of an interruption. Human listeners produce backchannels — "yeah", "uh-huh", "right" — precisely while the other party speaks, to signal attention, not to take the floor. With barge-in enabled, every one of them truncates the answer. The only alternative today is Studio-style half-duplex (AllowBargeIn=false), which throws away real interruptions too.

1.2 Turn-end is a single policy

The user-turn-end decision is: VAD silence for StopDurationMs, optionally confirmed by ISmartTurnClassifier (VoxaVadSettings.ConfirmTurnEnd). Two gaps:

  • Several streaming STT vendors already integrated in Voxa (Deepgram, AssemblyAI, Speechmatics, Gladia) run their own server-side endpointing and can signal end-of-turn with better accuracy than a local silence timer — Voxa ignores those signals and re-derives turn end from VAD.
  • There is no way for a host to compose policies (e.g. "smart-turn confirm, but also never close the turn while a final transcript is still pending from the STT").

1.3 Input muting is ad-hoc

Hosts that want "ignore the mic while X" (Studio's half-duplex; Ada wanting to ignore speech while a tool-running turn is in flight) each hand-roll it. There is no small policy object for "is user input currently accepted?".

3. WS1 — Interruption gating (IInterruptionPolicy)

3.1 Contract (Voxa.Core)

namespace Voxa.Processors;

/// Decides whether user speech that began during an in-flight bot turn is a genuine
/// interruption. Evaluated on the agent loop's system path — implementations must be
/// synchronous-fast and allocation-light.
public interface IInterruptionPolicy
{
    /// A new user speech onset while a turn is in flight. Return Interrupt to cancel
    /// immediately (today's behavior), Ignore to let the bot keep talking, or Undecided
    /// to defer until transcript evidence arrives.
    InterruptionVerdict OnSpeechStarted();

    /// Interim or final transcription text observed while a verdict is pending.
    /// Called repeatedly; the text is the utterance-so-far (coalesced, VRT-004).
    /// A final MUST resolve: when isFinal is true the return value is Interrupt or
    /// Ignore, never Undecided — user words are never left in limbo.
    InterruptionVerdict OnTranscript(string utteranceSoFar, bool isFinal);

    /// The pending utterance ended without a verdict (user stopped speaking, or the
    /// bot turn completed on its own). Resets internal state for the next onset.
    void Reset();
}

public enum InterruptionVerdict { Undecided, Interrupt, Ignore }

3.2 Evaluation point

AgentLoopProcessor sits between the STT side and the aggregator/TTS side, sees UserStartedSpeakingFrame on its system path and transcriptions on its data path, and — being the stage the onset frame flows through — can control what downstream ever learns about it. Gating therefore lives inside the loop, with two consequences the first draft of this spec got wrong (review findings on PR #102): the onset must be held, not forwarded, while a verdict is pending; and the gate's evidence must be exempt from hallucination filtering.

  • Onset — hold, don't forward. Downstream already treats a bare UserStartedSpeakingFrame as a barge-in signal: TextToSpeechProcessor aborts the in-flight synthesis and mutes, SentenceAggregator flushes, and the sink bumps its purge epoch. Forwarding the onset while Undecided would audibly kill the answer before the gate ever ruled — defeating the feature. So: with a gating policy active and a turn in flight, the loop calls OnSpeechStarted() and withholds the onset frame. Interrupt ⇒ release the held onset downstream, then TryCancelActiveTurn() + InterruptionFrame (today's ordering, just verdict-deferred). Ignore/pending ⇒ the onset (and its paired UserStoppedSpeakingFrame) are held and, on an Ignore resolution, dropped — downstream never learns a suppressed backchannel happened; diagnostics does (§7). The Immediate default forwards the onset exactly as today — byte-identical.
  • Evidence — evaluated before hallucination filtering. TranscriptionFilter's default blocklist contains precisely the backchannel finals the gate needs ("okay", "yeah", "hmm", "uh" …) — with the filter dropping them, a pending verdict would starve. When Interruption:Mode ≠ Immediate the composer flips the filter to tag mode: a blocklist/too-short final is forwarded flagged (TranscriptionFrame.SuspectedHallucination = true) instead of dropped. The loop feeds flagged finals to OnTranscript(...) as evidence but never enqueues them as turns — the filter's protective outcome is preserved, the evidence path isn't. Interims already pass the filter untouched. (Immediate mode keeps drop mode: byte-identical.)
  • Finals always resolve — user words are never swallowed. OnTranscript(…, isFinal: true) must return Interrupt or Ignore, never Undecided. The built-in rule (§3.3): a lexicon-only final ⇒ Ignore; a final containing any non-lexicon content ⇒ Interrupt — a one-word "stop" or "wait" is a genuine barge-in and must interrupt even though it is under MinWords. MinWords therefore governs only how early an interim may confirm, never whether a final counts.
  • Ignore epilogue — same as the empty-final path. A resolved-Ignore utterance takes the epilogue the loop already applies to empty/whitespace finals: forward the (flagged) final downstream so transports can clear their affordances (the TTS data-ordered unmute is a no-op — the onset never muted it), release held VDX-008 background results (no turn will follow this utterance; a completed thinker result must not stay parked until some later real turn), and Reset(). A final that arrives after an Interrupt verdict is a normal next turn.
  • No turn in flight: the policy is bypassed entirely — user speech when the bot is quiet is never an "interruption" question, the onset forwards immediately, and the loop's hold-while-speaking state latches as today.
Latency honesty. A word-gated verdict cannot arrive before the STT produces its first interim — typically 200–600 ms after onset with streaming vendors. During that window the bot keeps speaking. That is the feature, not a bug: the cost of gating is deferred interruption for genuine barge-ins, and the benefit is un-killed answers for backchannels. Hosts that cannot accept the deferral keep the default immediate policy. Audio queued between onset and verdict needs no special handling — the InterruptionFrame emitted at the verdict drives the aggregator/TTS mute and the sink's epoch purge exactly as in the immediate path (VRT-002 WS2).

3.3 MinWordsInterruptionPolicy (built-in)

new MinWordsInterruptionPolicy(
    minWords: 3,                  // ≥ this many words ⇒ Interrupt
    useInterims: true,            // count words from interims (earlier verdicts)
    backchannelLexicon: [...])    // utterances consisting ONLY of these are Ignored
                                  // regardless of count ("yeah yeah yeah")
  • Word counting is whitespace tokenization on the coalesced utterance — locale-naive by design; the lexicon (default: a small English set — "yeah", "yes", "uh-huh", "mm-hm", "ok", "okay", "right", "sure", "hmm", "wow", "i see", "got it" — case-insensitive, configurable, replaceable per language) handles the high-frequency cases.
  • Verdict logic per OnTranscript:
    • Interim: non-lexicon tokens ≥ minWords ⇒ Interrupt (early confirm — the whole point of useInterims). Otherwise Undecided, including lexicon-only text (a backchannel can be the prefix of a real sentence: "yeah but wait—").
    • Final: always resolves (§3.2). Lexicon-only ⇒ Ignore. Any non-lexicon token ⇒ Interrupt, regardless of count — "stop" and "wait" are one-word genuine barge-ins; minWords exists to make interims trustworthy, not to discard short finals.
  • OnSpeechStarted() always returns Undecided for this policy.

The interface deliberately mirrors the ISmartTurnClassifier pattern: a future model-backed policy (an acoustic interruption/backchannel classifier) implements the same three methods and plugs in via DI with zero loop changes.

4. WS2 — Turn-end sources (engine end-of-turn)

4.1 Capability flag + signal

  • VoxaProviderDescriptor (STT) gains bool ProvidesEndOfTurn — set per engine only once its parser actually emits the signal. Current parse-readiness is honest scope for WS2 (verified at head): only AssemblyAI parses an end-of-turn today; the other three are protocol-level work, not flag-plumbing:
    EngineWire signalStatus / WS2 deliverable
    AssemblyAIend_of_turn on Turn messages Already parsed (AssemblyAISttEngine) — map to the frame flag
    Deepgramspeech_final + UtteranceEnd messages Not parsed today (is_final is a segment lock, not endpointing); requires sending the endpointing/utterance_end_ms connection parameters and handling the new message type
    SpeechmaticsEndOfUtterance message Not parsed today; requires conversation_config.end_of_utterance_silence_trigger on the start message
    Gladiaspeech_end event Not parsed today; requires enabling the event in the session config
  • Engines that qualify raise the signal by emitting their final-transcript with a new EndOfTurn = true flag on TranscriptionFrame (a record extension, defaulted false — wire-compatible everywhere else).

4.2 Composer behavior

  • New setting Voxa:Vad:TurnEnd = Silence (default) | SmartTurn | Engine | Auto.
  • Auto: prefer Engine when the active STT descriptor advertises ProvidesEndOfTurn, else SmartTurn when a classifier is registered, else Silence. An explicit host setting always wins over the recommendation.
  • In Engine mode the VAD still opens turns (speech onset, interruption onset, speaking events) but no longer closes them; UserStoppedSpeakingFrame is derived from the engine's end-of-turn final. The VAD's StopDurationMs remains as a stale-turn safety net (see 4.3).
  • SmartTurn names today's opt-in explicitly (classifier registered ⇒ wired via ConfirmTurnEnd); Silence names today's default. Neither changes.

4.3 Stuck-turn watchdog

Any deferred turn-end (engine mode, smart-turn "not complete" verdicts, min-words pending verdicts) must not strand the conversation. A single loop-side timeout — Voxa:Vad:TurnEndFallbackMs (default 5000) — force-closes a user turn that has had no speech and no new transcript for that long, and publishes a diagnostics event naming which policy stalled. This is the granular-pipeline cousin of the speech-core stuck-pipeline guard already noted in AgentLoopProcessor.

5. WS3 — Input mute policies

namespace Voxa.Processors;

/// Is user input currently accepted? Consulted at the input edge; muted input drops
/// transcriptions (and their turn intake) but never audio-level frames, so VAD state
/// and diagnostics stay live.
public interface IUserInputMutePolicy
{
    bool IsMuted(UserInputMuteContext ctx);   // ctx: bot turn in flight? tool call running? first bot turn done?
}
  • Built-ins: MuteDuringBotTurn (formalizes Studio's half-duplex — Studio migrates to it, deleting its bespoke gate), MuteDuringToolCalls (Ada: don't accept speech while an uninterruptible tool-call turn runs), CompositeMutePolicy (any-of).
  • Default: none registered ⇒ no mute stage composed ⇒ byte-identical.
  • Muting drops turn intake, not the mic: frames still flow to diagnostics so Studio can render "heard but muted" honestly.

6. Configuration reference

"Voxa": {
  "Interruption": {
    "Mode": "Immediate",        // Immediate (default) | MinWords | Never
    "MinWords": 3,
    "UseInterims": true,
    "BackchannelWords": null    // null ⇒ built-in English lexicon; [] ⇒ lexicon off
  },
  "Vad": {
    "TurnEnd": "Silence",       // Silence (default) | SmartTurn | Engine | Auto
    "TurnEndFallbackMs": 5000
  }
}
  • Interruption:Mode=Never subsumes the loop's cancelTurnOnBargeIn:false knob (which remains as the programmatic form).
  • Profiles: LowLatency keeps Immediate; Quality selects MinWords(3) + TurnEnd=Auto. Profile deltas are documented in docs/performance-tuning.md.
  • Fail-fast validation (ValidateOnStart): unknown mode strings, MinWords < 1, MinWords mode with a non-streaming STT that produces no interims ⇒ startup error with a spelled-out fix, not a silent fallback. The last rule needs capability metadata the descriptor does not carry today: WS1 adds bool ProvidesInterims to VoxaProviderDescriptor (set by the streaming engines) alongside WS2's ProvidesEndOfTurn — without it the validation is unimplementable. Note the degradation is soft either way (finals still resolve every verdict, §3.2); the validator exists so hosts learn at startup that early-confirm won't happen.

7. Diagnostics

  • InterruptionGateEvent — onset observed; verdict (Interrupt/Ignore), the deciding evidence ("3 words ≥ min 3", "lexicon-only final"), and onset→verdict latency. Emitted for Immediate too (verdict=Interrupt, latency 0) so the Talk view timeline is uniform.
  • TurnEndDecisionEvent — which source closed the turn (Silence/SmartTurn/Engine/Fallback) and its silence-to-close latency; feeds the existing stage-latency waterfall.
  • UserInputMutedEvent — a transcription was dropped by a mute policy (policy name, text length).
  • All behind the existing HasListeners guard; zero-cost when unobserved.

Studio: Config gains the Interruption mode/knobs and TurnEnd source (registry-driven, honest — only shown when the active pipeline supports them); the Talk view renders suppressed backchannels as muted chips on the timeline instead of user turns.

8. Non-goals

  • LLM-gated turn completion (the model itself judging "is the user done?" via a sideband marker channel). Powerful, but it threads a marker protocol through every driver; deferred until the strategy seam proves itself. The TurnEnd enum leaves room.
  • Acoustic interruption/backchannel models. The policy interface is the seam; shipping a model is VLS-territory follow-up work.
  • Composite pipelines. AzureVoiceLiveProcessor/OpenAIRealtimeProcessor own turn-taking server-side; this spec is granular-chain only.
  • Speaker-aware gating (diarization-informed "was that even our user?") — noted as a future composition with VLS-005.

9. Workstreams

  • WS1 — Interruption gating: IInterruptionPolicy + verdict enum (Voxa.Core); loop integration (onset hold/release/drop incl. the paired stopped-frame, pending-verdict state, finals-always-resolve, Ignore epilogue with VDX-008 held-result release, Reset() paths); TranscriptionFilter tag mode + TranscriptionFrame.SuspectedHallucination; VoxaProviderDescriptor.ProvidesInterims; MinWordsInterruptionPolicy; composer + options + validation; unit tests incl. the "backchannel is a prefix of a real interruption" case and "backchannel swallowed while a background result is held ⇒ result still delivers on schedule".
  • WS2 — Turn-end sources: TranscriptionFrame.EndOfTurn; descriptor capability flag; per-vendor protocol work from the §4.1 table (AssemblyAI mapping; Deepgram speech_final/UtteranceEnd + connection params; Speechmatics EndOfUtterance + config; Gladia speech_end); composer TurnEnd modes + Auto preference order; fallback watchdog.
  • WS3 — Mute policies: interface + built-ins + composer stage; Studio half-duplex migrates onto MuteDuringBotTurn.
  • WS4 — Surfacing: diagnostics events; Studio Config + Talk chips; docs (docs/performance-tuning.md turn-taking section, CHANGELOG).

10. Verification

  • Golden: default composition byte-identical (existing golden test unchanged).
  • Gate behavior (fake STT/driver): backchannel final ("uh-huh") during a turn ⇒ no InterruptionFrame, no onset frame reaches the aggregator/TTS/sink, no queued turn, answer completes, and any held background result still delivers on schedule; one-word non-lexicon final ("stop") ⇒ interrupts despite being under MinWords; 4-word interim ⇒ interruption fires and the turn cancels exactly as PR #100's tests assert; onset with no transcript ever ⇒ Reset() after fallback, next turn normal; tag-mode filter forwards a flagged blocklist final and the loop never enqueues it as a turn.
  • Engine EOT: a fake engine emitting EndOfTurn finals closes turns with VAD silence-close disabled; fallback closes a turn when the engine never signals.
  • Race honesty: tests establish state deterministically (no fixed Task.Delay waits — repo convention); system/data cross-channel ordering treated as unordered.
  • Live protocol: Studio Talk with MinWords: speak over the bot with "yeah" (bot continues) and with a full sentence (bot yields) — scripted in the PR description for manual sign-off, and automated once VDX-009 lands.

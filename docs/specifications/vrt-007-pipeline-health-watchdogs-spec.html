VRT-007 — Pipeline health watchdogs

A voice pipeline that stalls fails silently: the socket stays open, the session object looks alive, and the user hears nothing — indefinitely. Voxa has no runtime instrument that notices a wedged data loop, an abandoned session, or a processor that never finished shutting down (the CI hang investigation found the last one only because a whole test lane died). Three small, independently useful watchdogs fix that: end-to-end heartbeats, session idle detection, and teardown hygiene reporting. All off by default; zero cost when disabled.

v1.1 (review-driven, PR #102): corrected the channel-mode premise — the granular chain's data channels are Wait-mode, so heartbeats delay under load rather than shed; the informational signal is now transit-latency-based congestion, not "missing heartbeats = backpressure". The teardown deadline no longer changes behavior: StopTimeoutMs defaults to null, preserving StopAsync's existing 5 s grace, and WS3 only instruments the existing escalation.

1. Problem

  • Stalls are invisible. If a processor's data task deadlocks or an engine's read loop hangs, frames stop flowing but nothing reports it. The failure surfaces as a user complaint ("it just went quiet") with no telemetry to triage.
  • Abandoned sessions run forever. A caller who walks away (or a WebSocket client that stops sending without closing) leaves a full pipeline — engines, buffers, possibly a metered phone call — running until transport teardown, which may never come. For Twilio media streams this is billed time.
  • Teardown failures are silent until they aren't. The Studio-tests hang burned six hours of CI before anyone knew which component failed to exit. At runtime the same class of bug leaks sessions instead of hanging processes — with even less visibility.

2. WS1 — Heartbeats (end-to-end liveness + free latency probe)

2.1 Mechanism

  • New HeartbeatFrame(long MonotonicTimestamp) — a ControlFrame that rides the ordered data path. That placement is the point: the system path stays healthy in most wedge scenarios (it's drained by a separate task); only a data-path probe proves the loop every processor's real work runs on is alive end-to-end.
  • When enabled, PipelineSource emits one every PeriodMs (default 1000) from a timer that is not gated behind the source's normal queue — a blocked producer must not stop probes.
  • PipelineSink records arrival; a runner-owned monitor awaits each arrival with TimeoutMs (default 10 000). Timeout ⇒ PipelineStalledEvent on the diagnostics hub (+ optional host callback), re-fired each interval while the stall persists, and a PipelineRecoveredEvent when flow resumes — stalls can be transient (a slow cold model load) and the telemetry must show both edges.
  • Every arrival also yields now − MonotonicTimestamp: a continuous whole-pipeline transit-latency series, published as a hub metric. This is a measurement no per-stage histogram composes correctly (queue wait between stages is invisible to stage timers) — the heartbeat gets it for free.
  • Processors need no changes: unhandled control frames already forward (the StartFrame/EndFrame rule). A processor that fails to forward heartbeats is itself a bug this watchdog exposes.
Channel-mode honesty (corrected in v1.1). The granular chain's inter-processor data channels are BoundedChannelFullMode.Wait (capacity 64, FrameProcessor default), and the WebSocket/telephony sinks backpressure rather than drop — so on the granular chain a heartbeat is never shed; under pressure it waits, and its transit time grows. The monitor's signals follow from that: steadily rising transit latency (heartbeats arriving, late) ⇒ PipelineCongestionEvent (informational, threshold ≈ a few heartbeat periods); no arrival for the full timeout ⇒ stall. Genuine shedding only exists where a stage opts into DropOldest — today the composite processors' audio channels (AzureVoiceLive/OpenAIRealtime/S2S) and the diagnostics hub's subscriber queues — and those sit outside the granular heartbeat path (§6). The timeout defaulting to 10× the period keeps the false-positive window negligible either way.

3. WS2 — Session idle detection

  • The runner (when enabled) tracks last-activity time, reset only by conversation-meaningful frames observed at the sink boundary: user/bot speaking events, transcriptions, turn lifecycle frames, outbound TTS audio. Deliberately not reset by inbound audio frames — an open mic in an empty room is exactly the abandoned case, and comfort noise/heartbeats must not defeat the timer.
  • On IdleTimeoutSeconds (default 300 when enabled): SessionIdleEvent + optional host callback; if EndOnIdle=true, the runner initiates the normal graceful end (push EndFrame, drain, complete) — the same path a client disconnect takes, so transports unwind cleanly (for Twilio, the closed media stream releases the call leg).
  • A first-activity grace: the timer arms on StartFrame, so a session nobody ever speaks into still times out — that is the abandoned-call case, not an exception to it.

4. WS3 — Teardown hygiene

  • PipelineRunner.StopAsync already has the deadline: a 5-second grace period, then hard cancel. WS3 does not change it — StopTimeoutMs defaults to null, meaning "keep the runner's existing 5 s grace" (an always-active 20 s default would silently quadruple every host's shutdown ceiling, violating this spec's own off-by-default rule). What WS3 adds is instrumentation of the existing escalation: when the grace expires, the runner publishes a TeardownReportEvent naming, per processor, which of its tasks (system/data) had not completed and how long teardown took. The event that would have named the Studio-tests culprit in seconds instead of a CI archaeology session.
  • The report is also produced on successful teardown at debug level (per-processor stop durations) — slow-stopping processors become visible before they become stuck ones.
  • No behavioral change to disposal ordering; this instruments the existing sequence. The known disposal-cluster follow-ups (CQ-001/2/3/10) remain separate work — this spec only makes their symptoms observable.

5. Configuration & composition

"Voxa": {
  "Health": {
    "Heartbeats": false,          // WS1 — off by default
    "HeartbeatPeriodMs": 1000,
    "HeartbeatTimeoutMs": 10000,
    "IdleTimeoutSeconds": 0,      // WS2 — 0 = disabled (default); >0 enables
    "EndOnIdle": false,
    "StopTimeoutMs": null         // WS3 — null (default) = keep StopAsync's existing 5 s grace;
                                  //        set to override; report event only when hub observed
  }
}
  • All three are runner/source/sink concerns — no new pipeline stages, so the golden composition test is untouched and disabled watchdogs cost nothing (no timers created, no fields touched on the frame hot path; the usual HasListeners guard covers event publication).
  • Profiles: unchanged (health is orthogonal to latency/quality). The telephony sample enables IdleTimeoutSeconds=120, EndOnIdle=true as documented good practice.
  • Studio: Talk view shows a stall indicator (amber "pipeline stalled — Ns" chip driven by the stall/recover events) and the idle countdown when enabled; the diagnostics pane gains the transit-latency series.

6. Non-goals

  • Automatic recovery (restarting stages, re-composing mid-session). Watchdogs detect and report; the "one transient fault → degraded session" cluster (CQ-014/015/016) is its own program.
  • Cross-session/host aggregation. Hub events feed whatever telemetry the host runs; OpenTelemetry export stays in Voxa.Observability's lane.
  • Startup timing instrumentation (per-processor start durations) — worthwhile, small, and separable; noted for a diagnostics follow-up rather than scoped here.
  • Composite pipelines' internals. The heartbeat proves the Voxa chain; a vendor's server-side session health is theirs. (The composite processor still forwards heartbeats around its stage, so transport-level stalls are caught.)

7. Workstreams

  • WS1 — Heartbeats: frame + source timer + sink monitor + stall/recover/congestion events + transit metric; unit tests with a deliberately-wedged fake processor (blocked data task, healthy system task ⇒ stall fires; both healthy ⇒ silence).
  • WS2 — Idle: activity tracking at the sink boundary, timeout, EndOnIdle graceful end; tests: no-activity session ends; active conversation never fires; open-mic-no-speech fires.
  • WS3 — Teardown: stop deadline + escalation + per-processor report; test: a processor whose data task ignores completion gets named in the report and the runner still exits.
  • WS4 — Surfacing: Studio chip + latency series; telephony sample config; docs/performance-tuning.md health section; CHANGELOG.

8. Verification

  • Golden composition byte-identical with Voxa:Health absent (no new stages by construction — asserted anyway).
  • Zero-cost claim tested: heartbeats disabled ⇒ no timer allocated, frame hot path unchanged (allocation test on the data loop stays green).
  • Stall detection end-to-end: wedge a mid-chain fake processor in a composed pipeline ⇒ stall event within timeout+period; unwedge ⇒ recovered event.
  • Idle + EndOnIdle over the WebSocket transport: idle session closes the socket gracefully (client sees the protocol's normal end, not an abort).
  • All waits condition-based with deadlines — no fixed Task.Delay asserts (repo flake convention).

9. Risks

RiskMitigation
False stall positives under heavy load Wait-mode channels mean load shows as rising transit latency (congestion signal) long before arrivals stop; timeout = 10× period; events carry transit-time context for triage
Idle timer ends a session during a legitimately long silent operation (big background task, VDX-008) Background-task lifecycle frames count as activity; hosts running long-silent workloads tune or disable EndOnIdle
Escalated cancel during teardown masks the root hang The report names the unfinished tasks before escalation; debug-level success reports build the baseline
Heartbeat frames confuse downstream custom processors Standard control-frame forwarding rule already covers unknown frames; docs call it out; evals starter pack (VDX-009) includes a heartbeat-transparent scenario

VVL-001 — Voxa Voice Library & Cloning Specification

A managed voice library in Voxa Studio: a first-class place to discover the voices a provider actually offers — fetched live, not hardcoded — to clone new voices from a few seconds of audio (cloud via ElevenLabs / Mistral, or fully local via an ONNX tone-color converter), and to feed those voices straight into the pipeline builder's voice picker. Built on two small, capability-based framework seams (IVoiceCatalogProvider, IVoiceCloneProvider) that any TTS provider may implement, plus a new keyless local clone engine and Voxtral speech-to-text.

1.What is this? — a plain-English explainer

What is the feature, in one sentence?

A Voices section in Studio where you can see every voice a provider offers (pulled live from the provider, so the list never goes stale), make a brand-new voice by handing over a short audio sample — to ElevenLabs, to Mistral, or to a model running entirely on your own machine — and then pick that voice for your pipeline like any built-in one.

Why does Voxa need it?

Today a voice is a string you type into appsettings.json — Voxa:ElevenLabs:VoiceId or Voxa:Piper:Voice — and you are on your own to know which strings are valid. Studio's Config dropdowns only list the local Piper/Kokoro catalogs (compiled-in constants); the cloud providers (Voxa.Speech.ElevenLabs, Voxa.Speech.Mistral — both already shipped) expose nothing in the UI, even though each account has its own library of standard and custom voices sitting behind an API call. And there is no way at all to make a voice. Voice cloning is the single most requested capability of a modern voice stack and Voxa has no story for it.

What does it look like?

A new nav section, Voices, distinct from the audition Playgrounds (VST-002 D2). It has:

  • The library — a grid of every voice you can use right now: local catalog voices, the live voices from each provider whose API key resolves, and the voices you have cloned. Each row shows where it lives (local / ElevenLabs / Mistral), whether it is a standard or cloned voice, and a play button to audition it on the spot.
  • The clone wizard — name the voice, drop in (or record) a few seconds of clean audio, tick the consent box, choose a target (a cloud provider, or "Local — no account needed"), and press Create. The new voice appears in the library when it is ready.
  • The picker hand-off — in Config and in the Builder, the voice dropdown stops being a hardcoded list and becomes the live library for whichever TTS provider is selected. Choose ElevenLabs and you see your ElevenLabs voices, clones included.
Can I clone a voice without any account or API key?

Yes — that is a headline goal. A new keyless engine, Voxa.Speech.VoiceClone, runs an open-source tone-color converter (OpenVoice v2, MIT-licensed) as ONNX on your machine. It captures the timbre of a reference clip into a small speaker embedding, then re-colors the output of a base local TTS (Kokoro) to sound like that speaker. No bytes leave the box; the models are SHA-256-pinned and cached exactly like Whisper/Piper/Kokoro. This keeps faith with VLS-001's promise that the whole stack works offline with no keys.

Where does Voxtral fit in?

Voxtral is Mistral's speech family. Voxa already ships its synthesis side (Voxa.Speech.Mistral, default model voxtral-tts) but not transcription. This spec adds Voxtral STT as a new VoxaSttDescriptor in that same package — so a user configuring a Mistral voice can also transcribe with Mistral, completing the provider. (Mistral as an LLM/agent provider is explicitly out of scope here — see §11.)

Is cloning someone's voice not dangerous?

It can be, and the spec treats consent as a first-class gate, not a footnote. The clone command is disabled until the user attests they have the right to clone the voice; the attestation timestamp is recorded on the voice profile; and the cloud providers enforce their own consent terms on top. Studio will never clone a voice silently or in the background, and §3.2 makes impersonation/deepfake assistance an explicit non-goal.

Does any of this leak my API keys?

No. Keys live where they live today — the Voxa:* section of appsettings.json / environment / user-secrets — and are sent only to their own provider over TLS. The voice library on disk (~/voxa-voices/) stores voice metadata and your own audio samples, never a key. This carries the VST-002 D3/D4 precedent (secrets never written to any export or bundle) forward verbatim.

3.Goals, non-goals & budgets

3.1 Goals

  • G1. When a provider's key resolves, its real voice list appears in Studio (and the pipeline picker) within one async fetch — no hardcoded ids, no manual copy-paste of voice strings.
  • G2. A user can clone a voice from a few seconds of audio to ElevenLabs or Mistral and immediately select it for a Talk session, all without leaving Studio.
  • G3. A user with no account and no keys can clone a voice fully locally (ONNX) and use it — preserving VLS-001's offline, keyless promise.
  • G4. Cloned and discovered voices persist across runs in a portable, diffable, key-free store (~/voxa-voices/), and reconcile against live provider state (a server-side deletion is shown as stale, never as a working voice).
  • G5. Voxtral STT is selectable wherever WhisperCpp/OpenAI/Azure STT are, validated by the same fail-fast options path.
  • G6. The capability seams are framework features any host can use — a server can list/clone voices through the registry without Studio.
  • G7. Cloning requires recorded consent; secrets never reach disk or any export.

3.2 Non-goals

  • Not an impersonation or deepfake tool: cloning is gated on consent and intended for your own or licensed voices. Studio adds no capability to disguise, anonymize, or defeat a provider's safety checks.
  • Not a secrets vault: keys remain in the standard Voxa:* config surface; Studio neither encrypts nor stores them itself (carried from VST-001 §3.2).
  • Not a Mistral LLM/agent integration: this spec adds Mistral voice + Voxtral STT only. A Mistral agent provider is a separate spec.
  • Not a training studio: local cloning is few-shot timbre transfer (an embedding), not fine-tuning or training a new TTS model.
  • No voice marketplace, sharing, or cloud sync of the library between machines — the store is local.
  • No real-time voice conversion of a live mic stream — cloning targets synthesis (TTS), not streaming morph.

4.Architecture — capabilities, remote-vs-local state, the local clone pipeline

4.1 Capabilities, not a new provider axis

Voxa already has one registration axis per stage (VoxaSttDescriptor, VoxaTtsDescriptor, VoxaVadDescriptor). Voice catalog and cloning are not a new stage — they are optional capabilities of an existing TTS provider. So rather than a parallel registry, a TTS descriptor gains two optional resolvers; a provider that can do more, exposes more. Piper and Kokoro leave them null and are unaffected.

VoxaTtsDescriptor "ElevenLabs" ├─ CreateProcessor ─► ElevenLabsTextToSpeechEngine (exists) ├─ ResolveCatalog ─► IVoiceCatalogProvider ── GET /v1/voices (new, WS1) └─ ResolveCloner ─► IVoiceCloneProvider ── POST /v1/voices/add (new, WS1) VoxaTtsDescriptor "Piper" ├─ CreateProcessor ─► PiperTtsEngine (exists) ├─ ResolveCatalog ─► null (its catalog is the compiled-in PiperVoiceCatalog) └─ ResolveCloner ─► null (cannot clone)

4.2 Remote state vs the local library

A cloned ElevenLabs voice lives on ElevenLabs, addressed by a voice id. The local VoiceProfile is a pointer plus provenance: the provider name, the remote id, the friendly name, the original samples, the consent timestamp, and creation date. The authority on whether that voice is usable right now is the live ListVoicesAsync(), not the file on disk. The library view reconciles the two:

  • Live & in library → normal, playable.
  • In library but not live (deleted server-side, or key changed) → shown stale; offer to forget or re-clone. Never silently selectable in a pipeline.
  • Live but not in library (made elsewhere, or a standard voice) → shown as discovered; one click adds an annotated profile.

Local clones are different: the engine is the authority, and its "remote id" is simply the embedding file path under the cache — so a local profile cannot drift.

4.3 The local clone pipeline (ONNX, keyless)

OpenVoice v2 separates what is said from who says it: a base TTS produces speech in a base voice, and a tone-color converter re-colors that audio to a target speaker captured as an embedding. Voxa already owns a strong base TTS (Kokoro, ONNX). So local cloning is two ONNX graphs layered on the existing engine — no PyTorch, no Python sidecar.

ENROLL (once per voice): reference clip (wav) ─► [ speaker-encoder.onnx ] ─► 256-d embedding ─► cache: voices/local/<id>.spk SYNTHESIZE (per turn): text ─► [ KokoroTtsEngine ] ─► base PCM ─┐ ├─► [ tone-color-converter.onnx ] ─► cloned PCM ─► pipeline cache: voices/local/<id>.spk ──►─┘
Honest uncertainty. OpenVoice's reference base is MeloTTS; Voxa substitutes Kokoro as the base speaker. The converter is base-agnostic in principle, but recolor quality and the availability of a clean ONNX export of the converter + speaker-encoder are the real unknowns in this spec. WS3 therefore opens with a validation spike (WS3-A0): confirm an MIT-licensed ONNX export exists (or export it) and that Kokoro-base → converter produces an intelligible, on-timbre result before the rest of WS3 is built. If the spike fails, the fallback is documented in §10 (R2).

5.Workstreams (WS0–WS6) with code

WS0 — voice capability seams P0

Projects Voxa.Speech.Abstractions, Voxa.AspNetCore Depends on nothing Unblocks WS1–WS6

Two capability interfaces and their value types. They are deliberately tiny and transport-free, so a cloud HTTP provider and a local ONNX engine implement the same shape.

src/Voxa.Speech.Abstractions/Voices/IVoiceCatalogProvider.cs 
namespace Voxa.Speech.Voices;

/// What a single voice looks like to Voxa, regardless of where it lives.
public sealed record ProviderVoice(
    string Id,                 // provider-native id (ElevenLabs voice_id, Mistral voice name, local embedding id)
    string DisplayName,
    string ProviderName,       // the registry TTS name ("ElevenLabs", "Mistral", "VoiceClone") so the picker maps a voice back to its provider
    VoiceKind Kind,            // Standard | Cloned
    string? Language = null,
    string? PreviewUrl = null, // provider sample, if any (never auto-downloaded)
    string? Description = null);

public enum VoiceKind { Standard, Cloned }

/// Optional capability: a TTS provider that can enumerate its voices live.
public interface IVoiceCatalogProvider
{
    Task<IReadOnlyList<ProviderVoice>> ListVoicesAsync(CancellationToken ct);
}
src/Voxa.Speech.Abstractions/Voices/IVoiceCloneProvider.cs 
namespace Voxa.Speech.Voices;

/// A reference sample for cloning. Bytes, not a path — the caller owns IO and
/// the provider may be remote. Format is PCM16 WAV mono unless Mime says otherwise.
public sealed record VoiceSample(string FileName, ReadOnlyMemory<byte> Data, string Mime = "audio/wav");

public sealed record VoiceCloneRequest(
    string Name,
    IReadOnlyList<VoiceSample> Samples,
    string? Description = null,
    string? Language = null);

/// Optional capability: a provider that can create a voice from samples.
/// Implementations MUST NOT issue the request without caller-confirmed consent;
/// the consent gate lives in the host (Studio), this is the transport.
public interface IVoiceCloneProvider
{
    Task<ProviderVoice> CreateVoiceAsync(VoiceCloneRequest request, CancellationToken ct);
    Task DeleteVoiceAsync(string voiceId, CancellationToken ct);
}

WS2 — Mistral: voice catalog + cloning + Voxtral STT P1

Project Voxa.Speech.Mistral (existing) Depends on WS0

Two additions to the existing TTS-only package.

2a — voice catalog + cloning

Mistral's voice API is GET /v1/audio/voices (list) and POST /v1/audio/voices (create from a reference sample). Same shape as WS1: one class implementing both seams, wired onto MistralDescriptors.Tts via the new resolvers. A created voice is referenced by name in Voxa:Mistral:Voice at synthesis time.

2b — Voxtral STT additive

A new VoxaSttDescriptor named "Mistral" backed by MistralSpeechToTextEngine : ISpeechToTextEngine. Voxtral transcription is request/response, not a streaming socket — so the engine is utterance-buffered: accumulate PCM written via WriteAudioAsync, and on FlushAsync (the pipeline's end-of-utterance, driven by VAD close) POST the buffer to /v1/audio/transcriptions and yield one final TranscriptionResult.

src/Voxa.Speech.Mistral/MistralSpeechToTextEngine.cs 
internal sealed class MistralSpeechToTextEngine(MistralSpeechOptions o, HttpClient http)
    : ISpeechToTextEngine
{
    // RequiredSampleRate: 16000 (resampled by the device/STT processor as usual)
    public ValueTask WriteAudioAsync(ReadOnlyMemory<byte> pcm, CancellationToken ct);  // buffer
    // SpeechToTextProcessor calls FlushAsync when it observes UserStoppedSpeakingFrame —
    // the documented ISpeechToTextEngine seam for batch engines ("REST Whisper, etc.").
    public async Task FlushAsync();                       // POST /v1/audio/transcriptions (multipart wav), enqueue final
    public IAsyncEnumerable<TranscriptionResult> ReadTranscriptsAsync(CancellationToken ct);  // finals only (WhisperCpp precedent)
}
Latency & profile guidance. Because Voxtral is utterance-based, its first transcript lands after end-of-speech, not incrementally — the same latency shape as any non-streaming cloud STT. The docs steer it toward the Quality profile, and the Talk waterfall will show the stt_final stage carrying the round-trip. (No streaming partials in v1; noted in §11.)

  • WS2-A1Unit (fake handler): Mistral ListVoicesAsync/CreateVoiceAsync map a recorded payload; key only in the request.

WS3 — Voxa.Speech.VoiceClone: keyless local cloning (ONNX) P1

Project new src/Voxa.Speech.VoiceClone (refs Voxa.Speech.Kokoro, Microsoft.ML.OnnxRuntime) Depends on WS0; VLS-001 model cache

The keyless headline. A package that does enrollment (reference clip → speaker embedding) and synthesis (base TTS → tone-color converter) entirely on-device, reusing the Kokoro-style ONNX session and the SHA-256-pinned VoxaModelCache.

/// Implements TTS + both voice capabilities. The "voices" it lists/creates are local
/// speaker embeddings under the cache; deletion just removes the .spk file — so a local

internal sealed class OpenVoiceCloneEngine

    public Task DeleteVoiceAsync(string id, CancellationToken ct);        // delete the .spk
    public Task<IReadOnlyList<ProviderVoice>> ListVoicesAsync(CancellationToken ct);  // scan .spk files

    // SYNTHESIZE: KokoroTtsEngine base PCM → tone-color-converter.onnx(embedding) → cloned PCM
    public async IAsyncEnumerable<ReadOnlyMemory<byte>> SynthesizeAsync(string text,
        [EnumeratorCancellation] CancellationToken ct);
}

Config selects the active local voice (Voxa:VoiceClone:Voice = an embedding id) and the base (Voxa:VoiceClone:BaseVoice = a Kokoro voice, default af_heart). The descriptor registers as TTS named "VoiceClone" with both capability resolvers and a WarmUpAsync that resolves the two ONNX artifacts — one new voxa.AddProvider(VoiceCloneDescriptors.Tts) line in the local/offline tier block of src/Voxa/VoxaDefaults.cs. Because Studio references the meta-package, the new package joins Studio's output closure automatically — where the existing DependencyGateTests (tests/Voxa.Studio.Tests) already scan every shipped file; WS3-A4 extends those assertions rather than inventing a new gate.

On success the new VoiceProfile is saved with ConsentAttestedAt = now and appears in the grid.

Join the existing MainWindowViewModel.SyncLiveState(): Voices.RecordBlocked = Talk.IsRunning || Builder.IsRunning || Metrics.IsRunning (the exact Playgrounds.Stt.CaptureBlocked precedent). The consent gate is the spec, not the UI polish. The attestation must gate the command, not merely show a dialog — a headless test (WS5-A2) constructs the view-model, fills name+sample+target, and asserts CreateCommand.CanExecute is false until consent is set true. Cloud providers enforce their own terms on top; Studio's gate is the floor, never the ceiling.

11.Out of scope / deferred

  • Mistral as an LLM/agent provider — this spec is voice + Voxtral STT only; a Mistral agent provider (joining Echo/OpenAI) is a separate spec.
  • Streaming Voxtral partials — v1 is utterance-buffered; an incremental transcript path waits on Mistral's streaming transcription surface.
  • Cloud library sync / sharing / marketplace — the store is local and personal; no cross-machine sync, no publishing of cloned voices.
  • Real-time voice conversion — morphing a live mic stream to a target timbre; cloning here targets synthesis, not streaming conversion.
  • Local fine-tuning / training — local cloning is few-shot embedding transfer, not training a new model.
  • Additional cloud cloners (Cartesia, PlayHT, etc.) — additive once the seams exist; each is a thin provider class like WS1, deferred until asked for.
  • Voice editing (pitch/speed/style sliders on a cloned voice) — beyond create/list/delete; revisit if the providers expose stable controls.

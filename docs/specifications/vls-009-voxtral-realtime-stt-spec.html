VLS-009 — Local Voxtral Realtime STT (open-weights, offline)

An on-device, open-weights speech-to-text tier built on Mistral's Voxtral-Mini-4B-Realtime-2602 (Apache-2.0) — a streaming audio LLM that finally gives the offline tier cloud-grade transcription. Ships as a new Voxa.Speech.Voxtral package implementing the streaming ISpeechToTextEngine contract directly, talking vLLM's realtime WebSocket API to a locally-run server. Featured as the GPU-gated default STT in Voxa Studio, with whisper.cpp retained as the keyless fallback.

1Problem & goal

The offline STT tier (VLS-001) is whisper.cpp — fast, keyless, runs on any laptop, but its small models trade accuracy for size and its larger ones are slow on CPU. Mistral's Voxtral family is now open-weights under Apache-2.0, and the Voxtral-Mini-4B-Realtime-2602 variant is a purpose-built streaming ASR with cloud-grade accuracy that can run entirely on the developer's own machine. This spec adds Voxtral as a first-class, fully-offline STT provider and — where the hardware can actually run it — makes it the featured default in Voxa Studio.

Goal: "Voxa:Stt": "Voxtral" resolves out of the box to a streaming, on-device engine that drives one agent turn per utterance, with interims for live display — and Studio auto-promotes it to the default the moment the machine is genuinely capable of running it, without regressing the keyless experience on machines that aren't.

Why this isn't a config change. Unlike the existing cloud "Voxa:Stt": "Mistral" (a REST call to Mistral's API), running Voxtral offline means hosting a 4B audio LLM. vLLM is the only officially-supported runtime and it needs a ≥ 16 GB GPU. So this is a new local engine + a managed local server process, on the scale of VVL-002's sidecar tier — not a default-string flip.
Community escape hatches (not in scope). Q4 GGUF conversions (~2.5 GB) and a Rust/Burn port exist and could one day run on modest hardware, but the Realtime architecture is not upstream in llama.cpp yet, so those paths require unofficial/custom builds. We target the supported vLLM path now and leave a lighter CPU/GGUF tier to a follow-up once upstream support lands.

3Architecture

Two pieces, both inside Voxa.Speech.Voxtral:

3.1 VoxtralRealtimeSttEngine : ISpeechToTextEngine

A streaming engine implemented directly against the contract (like the SDK-backed Google/AWS engines), not via WebSocketSttEngine. Two properties of vLLM's realtime API make the shared WebSocket base a poor fit, so reusing it would mean overriding most of it:

  • Audio is base64-in-JSON (input_audio_buffer.append), not the base's binary frames.
  • Finalization is a commit → done round-trip, not the base's local accumulator-flush. Because done is produced only in response to our commit (exactly one per utterance), there is no "provider keeps emitting the tail after flush" problem — so the anti-bleed window the accumulator exists to provide isn't needed. A plain Channel<TranscriptionResult> (à la the Mistral REST engine) is the correct, simpler tool.

Mapping:

Pipeline eventEngine action
WriteAudioAsync(pcm)base64-encode, send input_audio_buffer.append.
recv transcription.deltaappend to running text, write interim (IsFinal:false).
recv transcription.donewrite final (IsFinal:true); reset running text.
FlushAsync() (VAD speech-end)send input_audio_buffer.commit {final:true} — do not emit locally; the final flows when done arrives.
StopAsync()/DisposeAsync()close the socket, then tear down the managed server process.

3.2 VoxtralServerProcess — the managed local runtime

Mirrors ProcessSidecarChannel (VVL-002): launches the vLLM server as a child process, drains stderr to the log, polls readiness (vLLM with a 4B model takes many seconds to load — poll an HTTP health probe / the /v1/realtime route until it accepts, up to a generous configurable timeout), exposes the chosen host:port, and kills the process tree on dispose. The engine connects only once the server is ready.

Two honest hosting modes. Because most users already run vLLM via the documented podman/vllm serve command, the engine supports connect-only (Voxa:Voxtral:ServerUrl → an already-running server; Voxa launches nothing) and managed (Voxa:Voxtral:LaunchCommand/ExecutablePath → Voxa starts & owns it). This is the same "no pinned binary yet — you provide the launch target" honesty as the VVL-002 sidecar.

4Wire protocol (vLLM realtime)

Endpoint: ws://{host}:{port}/v1/realtime. OpenAI-Realtime-style JSON envelopes.

// server → client, on connect
{ "type": "session.created", "id": "..." }

// client → server, handshake
{ "type": "session.update", "model": "mistralai/Voxtral-Mini-4B-Realtime-2602" }

// client → server, audio (PCM16 @ 16 kHz, base64)
{ "type": "input_audio_buffer.append", "audio": "<base64>" }

// client → server, at VAD speech-end (FlushAsync)
{ "type": "input_audio_buffer.commit", "final": true }

// server → client, streaming partials then the final
{ "type": "transcription.delta", "delta": "the quick brown" }
{ "type": "transcription.done",  "text": "the quick brown fox", "usage": { ... } }

Parsing is pure and total: a malformed or unknown-type frame is ignored (never throws out of the receive loop), exactly as the WebSocket base treats vendor frames.

5Configuration

"Voxa": {
  "Stt": "Voxtral",
  "Voxtral": {
    // Connect-only: point at an already-running vLLM realtime server.
    "ServerUrl": "ws://127.0.0.1:8000",
    // OR managed: Voxa launches & owns the server (wins nothing if ServerUrl is set).
    "ExecutablePath": null,            // e.g. a vllm-serve wrapper / frozen launcher
    "LaunchCommand": null,             // e.g. "vllm"
    "LaunchArgs": [],                  // e.g. ["serve","mistralai/Voxtral-Mini-4B-Realtime-2602","--tokenizer-mode","mistral", ...]
    "Host": "127.0.0.1",
    "Port": 8000,
    "Model": "mistralai/Voxtral-Mini-4B-Realtime-2602",
    "InputSampleRate": 16000,
    "DelayMs": 480,                    // realtime delay knob (80–2400; 480 recommended)
    "Language": null,                  // BCP-47 hint; null = auto
    "ReadyTimeoutSeconds": 180,        // cold 4B model load is slow; poll until ready
    "MinGpuMemoryGb": 16               // capability gate for the Studio default
  }
}

Validation is keyless and side-effect-free (like SidecarDescriptors.Validate): it only checks that a hosting mode is resolvable — a ServerUrl, or a launch target whose file exists — and that InputSampleRate/DelayMs are sane. It never probes the GPU or touches the network.

6GPU-gated default in Studio

Voxtral is registered in the meta-package so the string resolves everywhere, but Studio only auto-selects it when the machine can actually run it. A small Avalonia-free, headless-testable selector:

effectiveStt =
    (configHasExplicitStt)        ? config.Stt              // user's choice always wins
  : (capable && voxtralConfigured) ? "Voxtral"
  :                                 "WhisperCpp";           // keyless fallback — never regresses

capable = an NVIDIA GPU with ≥ MinGpuMemoryGb VRAM, detected via nvidia-smi --query-gpu=memory.total (behind an injectable seam so tests don't shell out). voxtralConfigured = a ServerUrl or launch target is present. On a fresh install with no GPU and no server configured, Studio behaves exactly as today (whisper.cpp). The moment a developer with a capable GPU points Studio at a Voxtral server, it becomes the default. Voxtral is surfaced as the recommended STT in Config/Models regardless, so it's discoverable.

Studio rule preserved. Studio never launches the heavy server before the user acts: Voxa:Models:EagerWarmup is false in Studio, so the managed server is started on Talk-Start (engine warm-up), not at splash.

7Scope — ships now vs. deferred

Ships now this PR

  • Voxa.Speech.Voxtral: options, the streaming engine, the managed-server launcher + readiness poll, the capability probe, the descriptor (registered in the meta-package), a dev mock realtime server, README.
  • Studio GPU-gated default-STT selector + "recommended" surfacing.
  • Full unit coverage against an in-process fake realtime WebSocket server + a fake launcher/GPU probe — no real model, no GPU, no network. Build is warnings-as-errors; default test lane stays green and offline.
  • Docs (local-speech guide), CHANGELOG, ROADMAP.

Deferred (documented, gated) follow-up

  • A pinned, SHA-256-verified, auto-downloaded model/runtime bundle (the BF16 model is multi-file ~9 GB; the same deferral as VVL-002's frozen binaries and VLS-007/008's real-model hashes — needs a GPU CI lane).
  • The live [Trait("Category","LocalModels")] end-to-end test against a real vLLM + GPU.
  • A lighter CPU/GGUF tier once the Realtime architecture lands upstream in llama.cpp.

8Testing

  • Engine — drive against an in-process fake /v1/realtime server (an HttpListener WebSocket) that speaks the envelopes: assert the handshake, that audio arrives as base64 append, that delta→interim and done→final, and that FlushAsync sends commit.
  • Parsing — table tests: each envelope type → expected emission; malformed/unknown ignored.
  • Launcher — a fake process/launcher seam: readiness poll succeeds/times out; dispose kills the process.
  • Capability probe — injected GPU-info → capable/not at the VRAM threshold.
  • Default selector — capable+configured→Voxtral; explicit config respected; otherwise WhisperCpp.
  • Descriptor — validation: no hosting mode → error; ServerUrl set → ok; missing launch file → error.

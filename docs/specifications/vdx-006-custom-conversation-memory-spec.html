VDX-006 — IVoiceAgentConfigurator: custom conversation memory under UseDefaults()additive

A one-interface host seam so an app with its own durable conversation store keeps the default composer — the real VAD, the latency profile, the diagnostics taps — instead of hand-rolling the whole pipeline to swap one stage. Today, wanting per-thread voice memory forces a host off .UseDefaults() entirely and silently forfeits everything else the composer wires. VDX-006 makes the agent stage the only thing a memory-owning host has to touch.

1.What is this? — a plain-English explainer

What does VDX-006 add, in one sentence?

A single optional interface — IVoiceAgentConfigurator — that a host registers in DI to take over the voice agent's per-turn options (conversation memory, frontend tools, turn-lifecycle hooks) while staying on app.MapVoxaVoice("/voice").UseDefaults().

Why does it matter?

The composer (DefaultVoicePipelineComposer) is where Voxa's "optimal default" lives: it reads the latency profile, selects the registered VAD, wires interim-coalescing, and inserts the diagnostics taps. The only way to reach all of that is .UseDefaults(). But the composer also forces its own in-memory chat history — so any host that needs a durable store (persist turns, key by a thread id, show voice turns next to text) has had to abandon .UseDefaults() and re-build the entire pipeline by hand just to replace that one stage. In practice they forfeit the real VAD and the profile by accident.

Who hit this?

Ada (a downstream consumer) did, exactly. Its voice plane hand-builds the chain to get per-thread persistence and, as a side effect, runs an energy gate instead of the Silero VAD it registered and ignores the LowLatency profile it set — documented in Ada's M6.1 fidelity spec. VDX-006 is the upstream fix that makes Ada's M6.1 workarounds unnecessary: with the seam, Ada registers one class and switches its route to .UseDefaults().

2.Executive summary

Add a public interface IVoiceAgentConfigurator to Voxa.AspNetCore. The composer resolves it from the per-connection service scope (the same scope it already uses to resolve the AIAgent, ISmartTurnClassifier, IEchoCanceller, and the diagnostics hub). When present, the host owns the agent's MicrosoftAgentVoiceOptions and the composer skips its built-in InMemoryChatHistory. When absent — the overwhelmingly common case — nothing changes: the composed pipeline is byte-identical to pre-VDX-006, preserving the existing golden gate.

3.Goals & non-goals

Goals

  • Let a host fully customise the voice agent's MicrosoftAgentVoiceOptions (BuildMessages, OnTurnCompleted, OnTurnStarted, OnTurnFailed, IsFrontendTool, BuildRunOptions) without leaving .UseDefaults().
  • Be transport-agnostic — work identically for the WebSocket host and Studio's in-process transport, via the per-connection IServiceProvider the composer already threads.
  • Be zero-impact when unused — preserve the defaults-byte-identity golden gate.
  • Keep the public surface to a single interface (plus one optional convenience extension).

Non-goals

  • A provider-neutral memory abstraction. The seam intentionally exposes the existing MAF MicrosoftAgentVoiceOptions; the composer's agent stage is already Microsoft-Agent-Framework-specific.
  • Persisting memory for the host. Voxa supplies the seam and the per-connection scope; the store is the host's.
  • Changing the granular interruption / barge-in model (unrelated).

4.Architecture context — the agent stage today

The composer builds the agent stage in two places. The call site decides whether to allocate a history; the factory wires it into the per-turn options:

src/Voxa.AspNetCore/DefaultVoicePipelineComposer.cs — today
// 4. Agent with built-in conversation memory
var history = o.Agent.ConversationMemory
    ? new InMemoryChatHistory(o.Agent.MaxHistoryMessages)
    : null;
parts.Add(sp => CreateAgentProcessor(sp, o.Agent, history, tuning.MaxResponseDuration));

// …

private FrameProcessor CreateAgentProcessor(
    IServiceProvider sp, VoxaAgentOptions agentOpts,
    InMemoryChatHistory? history, TimeSpan? maxResponseDuration)
{
    AIAgent? agent = sp.GetService<AIAgent>()
        ?? WrapChatClient(sp.GetService<IChatClient>(), agentOpts)
        ?? sp.GetService<IVoiceAgentFactory>()?.Create(sp, agentOpts);
    if (agent is null) throw new InvalidOperationException(/* … */);

    return MicrosoftAgentVoice.CreateProcessor(agent, opts =>
    {
        opts.MaxResponseDuration = maxResponseDuration;
        if (history is not null)
        {
            opts.BuildMessages  = (turnCtx, ct) => /* Snapshot() + user text */;
            opts.OnTurnCompleted = (turnCtx, summary, ct) => /* history.AddUser / AddAssistant */;
        }
    });
}

InMemoryChatHistory (in Voxa.Services.MicrosoftAgents) is a sealed class with AddUser(string), AddAssistant(string), and Snapshot(). There is no seam to substitute it, and MicrosoftAgentVoiceOptions is constructed and consumed entirely inside the factory — so a host cannot reach it without replacing the whole stage, which means leaving the composer.

The pivot. The agent already resolves from DI (sp.GetService<AIAgent>()). VDX-006 adds exactly one more optional DI lookup next to it, and lets that collaborator have the last word on opts.

5.The seam — IVoiceAgentConfigurator

src/Voxa.AspNetCore/IVoiceAgentConfigurator.cs — new
using Microsoft.Extensions.DependencyInjection;
using Voxa.Services.MicrosoftAgents;

namespace Voxa.AspNetCore;

/// <summary>
/// Host seam (VDX-006) for customising the voice agent's per-turn options under <c>UseDefaults()</c> —
/// a durable conversation store, frontend tools, or turn-lifecycle hooks — without abandoning
/// <see cref="DefaultVoicePipelineComposer"/>. Resolved per connection from the session scope; when none
/// is registered the composer keeps its built-in <c>InMemoryChatHistory</c> (byte-identical default).
/// </summary>
public interface IVoiceAgentConfigurator
{
    /// <param name="connection">The per-connection service provider — HTTP <c>RequestServices</c>, or the
    /// in-proc session scope under Studio. Use it to resolve scoped state, e.g. the active thread id.</param>
    /// <param name="options">The agent options to mutate. The composer's defaults are already applied;
    /// the host has the last word.</param>
    void Configure(IServiceProvider connection, MicrosoftAgentVoiceOptions options);
}

/// <summary>Delegate-backed configurator for hosts that prefer a lambda to a class (VDX-006).</summary>
public static class VoiceAgentConfiguratorExtensions
{
    public static IServiceCollection AddVoxaVoiceAgentConfigurator(
        this IServiceCollection services,
        Action<IServiceProvider, MicrosoftAgentVoiceOptions> configure)
    {
        ArgumentNullException.ThrowIfNull(configure);
        return services.AddSingleton<IVoiceAgentConfigurator>(new DelegateVoiceAgentConfigurator(configure));
    }

    private sealed class DelegateVoiceAgentConfigurator(
        Action<IServiceProvider, MicrosoftAgentVoiceOptions> configure) : IVoiceAgentConfigurator
    {
        public void Configure(IServiceProvider connection, MicrosoftAgentVoiceOptions options)
            => configure(connection, options);
    }
}

The method is a void mutator over MicrosoftAgentVoiceOptions — deliberately the same shape as the Action<MicrosoftAgentVoiceOptions> already accepted by UseMicrosoftAgent(agent, configure), so the mental model is identical whether a host is on the manual builder or the composer.

6.Composer change & the byte-identity guarantee

Move the history decision inside the factory so it can defer to a configurator, then invoke the configurator last:

src/Voxa.AspNetCore/DefaultVoicePipelineComposer.cs — patch
 // 4. Agent. Built-in conversation memory unless a host configurator owns it (VDX-006).
-var history = o.Agent.ConversationMemory
-    ? new InMemoryChatHistory(o.Agent.MaxHistoryMessages)
-    : null;
-parts.Add(sp => CreateAgentProcessor(sp, o.Agent, history, tuning.MaxResponseDuration));
+parts.Add(sp => CreateAgentProcessor(sp, o.Agent, tuning.MaxResponseDuration));

 private FrameProcessor CreateAgentProcessor(
-    IServiceProvider sp, VoxaAgentOptions agentOpts,
-    InMemoryChatHistory? history, TimeSpan? maxResponseDuration)
+    IServiceProvider sp, VoxaAgentOptions agentOpts, TimeSpan? maxResponseDuration)
 {
     AIAgent? agent = sp.GetService<AIAgent>()
         ?? WrapChatClient(sp.GetService<IChatClient>(), agentOpts)
         ?? sp.GetService<IVoiceAgentFactory>()?.Create(sp, agentOpts);
     if (agent is null) throw new InvalidOperationException(/* … */);

+    // VDX-006: a host configurator owns conversation memory (and may add tools / lifecycle hooks).
+    // Absent ⇒ keep the built-in InMemoryChatHistory exactly as before, so the composed options are
+    // byte-identical to pre-VDX-006 (the defaults-byte-identity golden gate).
+    var configurator = sp.GetService<IVoiceAgentConfigurator>();
+    var history = (configurator is null && agentOpts.ConversationMemory)
+        ? new InMemoryChatHistory(agentOpts.MaxHistoryMessages)
+        : null;
+
     return MicrosoftAgentVoice.CreateProcessor(agent, opts =>
     {
         opts.MaxResponseDuration = maxResponseDuration;
         if (history is not null)
         {
             opts.BuildMessages  = (turnCtx, ct) => /* unchanged */;
             opts.OnTurnCompleted = (turnCtx, summary, ct) => /* unchanged */;
         }
+        configurator?.Configure(sp, opts); // host has the last word
     });
 }
Byte-identity proof. With no IVoiceAgentConfigurator registered, configurator is null: history is built on exactly the prior condition, the built-in BuildMessages/OnTurnCompleted wiring runs identically, and configurator?.Configure is a no-op. The only added work on the default path is one GetService that returns null — it does not touch the composed processor. The existing golden test (composed pipeline equality with diagnostics off) continues to pass unchanged; a new test pins the "no configurator ⇒ identical options" case explicitly.
Precedence. When a configurator is registered, the composer skips its built-in history (the host owns memory) and the configurator runs after MaxResponseDuration is set — so a host may also override that if it wishes. If both Voxa:Agent:ConversationMemory=true and a configurator are present, the configurator wins; log this at Debug so the choice is visible.

7.Host usage — the Ada migration

Today Ada hand-builds the whole chain to host its IConversationStore keyed by ?thread=. With VDX-006 that closure moves verbatim into a configurator, and the route collapses to the defaults:

Ada — AdaVoiceConfigurator.cs (the old inline closure, now a class)
internal sealed class AdaVoiceConfigurator(IConversationStore convos) : IVoiceAgentConfigurator
{
    public void Configure(IServiceProvider connection, MicrosoftAgentVoiceOptions opts)
    {
        var http = connection.GetRequiredService<IHttpContextAccessor>().HttpContext;
        var threadId = ResolveThread(http, convos);           // the ?thread= logic, unchanged
        opts.BuildMessages   = (ctx, _) => /* VoiceContextTail + user text */;
        opts.OnTurnCompleted = (ctx, summary, _) => /* persist into convos, set title */;
    }
}
Ada — registration + route
builder.Services.AddHttpContextAccessor();
builder.Services.AddSingleton<IVoiceAgentConfigurator, AdaVoiceConfigurator>();
// …
app.MapVoxaVoice("/voice").UseDefaults();   // VAD + LowLatency profile + diagnostics, for free
What Ada deletes. The manual UseSilenceGate, the hand-built SileroVad/SentenceAggregator stages, and the 16 kHz rate-tag workaround from M6.1 — all gone, because the composer now provides them and announces the session sample rate itself. VDX-006 turns Ada's "faithful workaround" into "use the defaults."

8.Alternatives considered

OptionWhy not
Extract an IChatHistory (Snapshot/AddUser/AddAssistant) and resolve it from DI Too narrow. It covers load+append but loses per-turn metadata real stores want (route, timestamp, lazily-created thread title) and cannot touch frontend tools or lifecycle hooks. The options-configurator is strictly more capable at the same surface.
Register a raw Action<IServiceProvider, MicrosoftAgentVoiceOptions> in DI Non-idiomatic for Voxa (delegates in the container are undiscoverable and awkward to test). Offered anyway as the optional AddVoxaVoiceAgentConfigurator(…) convenience that wraps a lambda into the interface.
Always wire built-in memory, let the configurator override Allocates a dead InMemoryChatHistory for replace-style hosts and invites confusing mixed behaviour (built-in BuildMessages + custom OnTurnCompleted). "Skip built-in when a configurator is present" is the predictable rule: the host fully owns the agent options.
A new UseDefaults(Action<…>) overload on the route The route handler runs per request and only assembles processor factories; the agent options live one layer down in the composer. A DI-resolved collaborator reaches the right layer and stays transport-agnostic (Studio has no route).

12.Risk register

#RiskSevMitigation
R1A throwing Configure fails composition.Med It runs once per connection at compose time, before the socket is accepted — the existing handler turns a compose failure into HTTP 500 with logs, not an aborted socket. Document: Configure must not throw; it sets delegates, it does not do per-turn work.
R2Singleton configurator needs per-connection state (thread id).Med It receives the per-connection IServiceProvider at Configure time (called once per connection), so even a singleton resolves scoped state from connection. Documented in the XML doc and getting-started.
R3Silent loss of built-in memory surprises a host.Low The Debug log on precedence makes it visible; the getting-started note states the rule plainly.
R4Seam couples the composer to the MAF options type.Low The composer's agent stage is already MAF-only (MicrosoftAgentVoice.CreateProcessor). A future non-MAF agent path would introduce its own seam; no new coupling is created here.

13.Out of scope / deferred

  • Provider-neutral memory. If a second agent runtime ever lands, factor a runtime-agnostic options seam then — not speculatively now.
  • Augment-without-replace. Keeping built-in memory and adding a tool is not a first-class mode; a host that wants it re-creates the (public) InMemoryChatHistory wiring in its own configurator. Revisit only on demand.
  • Voxa-side persistence helpers. A shipped SQLite/file conversation store is a separate product decision, not this seam.
VDX-006 · One public interface, one composer factory edit, zero default-path behaviour change. The point is narrow: a host that owns its conversation memory should not have to give up the composer to keep it. Upstream of — and the reason for — Ada's M6.1 manual stages.

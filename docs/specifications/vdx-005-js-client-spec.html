1.What is this? — a plain-English explainer

What does VDX-005 add, in one sentence?

An npm package, @voxa/client, that a web app imports to talk to a Voxa voice endpoint — it captures the microphone, streams it to the server, plays the bot's audio back gap-free, surfaces transcripts and tool calls as typed events, and — critically — stops the bot's audio the instant the user barges in.

Why is this the highest-leverage P5 item?

Three reasons. (1) It's the piece everyone rebuilds. The server is already a five-line library call (MapVoxaVoice("/voice").UseDefaults()); the client — AudioWorklet mic capture, PCM playback, the wire envelopes — gets hand-written by every consumer. (2) It's the missing half of barge-in. The server already purges queued bot audio on interruption (WebSocketAudioSink's epoch purge), but only the browser can stop audio it has already handed to the Web Audio graph. Without the client half, the bot keeps talking for ~a sentence after you cut in. (3) It's upstream of the other P5 items. The dotnet new voxa-server template and any voxa-server CLI both produce an endpoint that needs a real client to be usable.

Does the wire protocol have to change?

No — the happy path runs over the protocol exactly as it ships today. What this item adds is discipline: the downstream side is already versioned (the session envelope carries "v":1), so the client simply honors that version, and we generate the TypeScript types from the C# envelope records so the two ends cannot silently drift. The only server-side code change is internal: express the three inbound envelopes as records too, so both directions share one source of truth — wire-format unchanged, guarded by the existing byte-identical golden tests.

A complete reference implementation already exists. The local-voice test page — samples/Voxa.Samples.MinimalServer/wwwroot/index.html plus recorder-worklet.js — already does the mic capture, the gap-free playback, and the barge-in flush correctly. VDX-005 is mostly productizing that page into a typed, tested, published package, and closing the two gaps it leaves open: it never round-trips tool calls, and it ignores the protocol version.
What does it explicitly not change?

Voxa.Core stays zero-dependency. The outbound wire bytes are untouched. The AddVoxa / composer layer, the CLI, and the MCP server are all untouched. Native mobile clients (Flutter, Swift, Kotlin) are separate consumers and out of scope here, exactly as VDX-004 keeps mobile bindings in sibling repos.

2.Executive summary

VDX-005 ships the client half of the Voxa voice loop as an official, framework-agnostic npm package, and formalizes the wire protocol into a generated, versioned contract along the way. It is three workstreams, sequenced so the first is pure .NET and testable in the existing suite before a line of TypeScript is written:

  • WS1 — the versioned protocol contract pure .NET first. Express the inbound (client → server) envelopes as records alongside the outbound ones in WireMessages.cs, then generate a single voxa-wire.schema.json from the [JsonSerializable] envelope types and a protocol.ts from that schema, each guarded by a CI golden check. The client honors the session.v the server already emits.
  • WS2 — the VoxaClient runtime the package. One class over the wire protocol: connect → await session → mic capture at the announced input rate → typed events (transcription, text, speaking, toolCall, status, error, end) → gap-free playback at the announced output rate → flush on interruption / user-speaking. Audio is behind a backend seam so the core is unit-testable headless, matching how Voxa fakes engines/transports in its own tests.
  • WS3 — packaging & adapters distribution. ESM + type declarations on npm, a framework-agnostic core, and one thin optional React hook (useVoxa). The inline test HTML is replaced by an examples/ page that consumes the package.
Net position. After VDX-005, a web developer adds voice with npm i @voxa/client and ~10 lines — no hand-rolled AudioWorklet, no PCM scheduling, and barge-in that actually works. The server and client can no longer drift, because both ends are generated from one set of records and the version is checked at runtime. And the remaining P5 items (template, voxa-server) inherit a real client instead of a toy page.

3.Goals, non-goals & budgets

Goals

  • Publish @voxa/client — a typed, framework-agnostic browser client that captures mic audio, plays bot audio gap-free, and surfaces the full envelope set as typed events.
  • Implement the client half of barge-in: flush local playback on the interruption envelope and on speaking{who:"user",started:true} — the trigger pair the reference page already proves correct.
  • Round-trip frontend tools: deliver toolCall to the host and send toolResult back — the one protocol feature the reference page omits.
  • Make the wire protocol a generated, versioned contract: TypeScript types emitted from the C# envelope records, a runtime session.v check, and CI golden checks that fail on drift.
  • Keep the client core unit-testable with no browser — audio and transport behind seams, driven by a mock WebSocket.

Non-goals (this spec)

  • Any wire-format change. Outbound bytes stay byte-identical; inbound parsing accepts exactly the same JSON. The only change is internal representation (records) to enable codegen.
  • Native / mobile clients. Flutter (AONIK P3), Swift, Kotlin are separate consumers in their own repos — the VDX-004 stance on bindings applies here too.
  • A UI component library. The package ships logic, not chat bubbles. One thin React hook is the ceiling; no design-system components.
  • Reconnect / resume. The session-resilience resume token is a P6 item; the client leaves a seam but does not implement it.
  • The runtime configure control envelope and C#-delegate tool schema generation (both P6/P7) — the client only round-trips the existing callId / argumentsJson strings.

4.Architecture context — the wire protocol today

The transport already exists and is stable: WebSocketAudioSource / WebSocketAudioSink in Voxa.Transports.WebSocket, with the JSON codec in WireProtocol and the outbound envelope records in WireMessages.cs. Binary WebSocket frames carry raw 16-bit PCM; text frames carry typed JSON envelopes. The client is a faithful peer to this contract — nothing below is invented for VDX-005, it is the surface as it ships.

DirectionChannelMessages
Server → ClientBinaryOutput PCM (16-bit, mono, at session.outputSampleRate) — the bot's voice.
Server → ClientText JSONsession{v,inputSampleRate,outputSampleRate}, transcription{text,isFinal,language?,speakerId?}, text{text}, toolCall{callId,name,argumentsJson}, speaking{who:"bot"|"user",started}, interruption, status{message}, error{message}, end.
Client → ServerBinaryMic PCM (16-bit, mono, at session.inputSampleRate).
Client → ServerText JSONhello{…} (optional, out-of-band — see below), end, text{text}, toolResult{callId,resultJson,isError?}.

Three facts that shape the client

  • The session is already versioned. SessionInfoFrame(int InputSampleRate, int OutputSampleRate, int ProtocolVersion = 1) serializes to {"type":"session","v":1,"inputSampleRate":…,"outputSampleRate":…}. The client's first job after connect is to read this envelope: it learns both sample rates (hardcoding none — the reference page makes a point of this) and checks v.
  • hello is optional and out-of-band. A host opts in via VoicePipelineBuilder.UseWebSocketHello<T>; the default UseDefaults() pipeline never reads one, and WireProtocol.TryParseClientMessage returns null for a hello seen in the frame stream. So the client must send hello only when configured to, and send it as the first message before any audio.
  • Barge-in is half-built server-side. On an InterruptionFrame (or a UserStartedSpeaking arriving as a data frame), WebSocketAudioSink bumps an epoch, drops queued bot audio from the previous epoch, and lets the interruption envelope jump ahead. That stops audio still sitting in the server's send queue — it cannot touch audio already delivered to the browser's AudioContext. Closing that is WS2.
  Browser (this package)                        Voxa server (already ships)
  getUserMedia + AudioWorklet  --binary PCM-->  WebSocketAudioSource --> VAD/STT/agent/TTS
  AudioContext playback        <--binary PCM--  WebSocketAudioSink   <-- output audio
  typed events / tools         <--text JSON-->  WireProtocol (session/transcription/text/toolCall/speaking/…)
  flushPlayback()              <--interruption-- epoch purge (server) + interruption envelope
Invariant. The client is a peer of the existing protocol. It adds no envelope types and requires no server behaviour change for the happy path. The only server-side edit in this spec (WS1) is to represent the inbound envelopes as records so codegen has one source — verified wire-identical by the golden tests.

5.WS1 — the versioned protocol contract (DTOs + codegen)

Where Voxa.Transports.WebSocket.Protocol + a new clients/voxa-client dir Risk Low (internal refactor + tooling) Wire change none (byte-identical) Order first — pure .NET, testable now

Today the outbound envelopes are clean records, but the inbound side (end / text / toolResult) is hand-parsed field-by-field in TryParseClientMessage. For a contract that "can't drift," both directions must derive from one source. Step one is to give the inbound envelopes records too.

src/Voxa.Transports.WebSocket/Protocol/WireMessages.cs (added — inbound envelopes as records) 
// Inbound (client -> server). Mirrors the existing outbound records so a single
// [JsonSerializable] set covers BOTH directions for schema generation. Field order
// and camelCase are chosen to match what TryParseClientMessage already accepts.
internal readonly record struct EndClientEnvelope(string Type);                                  // {"type":"end"}
internal readonly record struct TextClientEnvelope(string Type, string Text);                    // {"type":"text","text":…}
internal readonly record struct ToolResultClientEnvelope(string Type, string CallId, string ResultJson, bool IsError);

// hello stays host-defined (VoiceHello / UseWebSocketHello<T>) — documented as an
// extension point, NOT folded into the generated core protocol (see §12).

TryParseClientMessage is then a thin switch that deserializes the matching record (via the existing source-generated WireJsonContext, now also in deserialize mode) and maps it to the same Frame it returns today — EndFrame, TextFrame, ToolCallResultFrame. Same inputs, same frames, same dropped-unknown behaviour; the change is representational. The existing WireProtocolCompatibilityTests are the proof.

Schema & TypeScript generation

.NET 9+ ships System.Text.Json.Schema.JsonSchemaExporter. A tiny generator (a console target under tools/, or — preferred, since it needs no new project — a [Fact] in Voxa.Transports.WebSocket.Tests that writes the file) walks every envelope type registered in WireJsonContext and emits one voxa-wire.schema.json. The TypeScript is generated from that schema by json-schema-to-typescript in the JS package's build.

        | JsonSchemaExporter over WireJsonContext's types
        v

        v

clients/voxa-client/src/protocol.ts (generated — shape) 
// AUTO-GENERATED from voxa-wire.schema.json. Do not edit by hand.
export const VOXA_WIRE_VERSION = 1;

export type ServerMessage =
  | { type: "session";       v: number; inputSampleRate: number; outputSampleRate: number }
  | { type: "transcription"; text: string; isFinal: boolean; language?: string; speakerId?: string }
  | { type: "text";          text: string }
  | { type: "toolCall";      callId: string; name: string; argumentsJson: string }
  | { type: "speaking";      who: "bot" | "user"; started: boolean }
  | { type: "interruption" }
  | { type: "status";        message: string }
  | { type: "error";         message: string }
  | { type: "end" };

export type ClientMessage =
  | { type: "end" }
  | { type: "text";       text: string }
  | { type: "toolResult"; callId: string; resultJson: string; isError?: boolean };
Why golden-check both artifacts. The repo already lives by byte-identical wire golden tests. Extending that instinct to the schema and the .ts means a server-side envelope change that isn't reflected in the committed TypeScript fails CI — the drift is caught mechanically, not in code review or, worse, at a customer's runtime.

Runtime version handling

The client compares session.v against VOXA_WIRE_VERSION. Policy is configurable (onVersionMismatch: "warn" | "throw" | "ignore", default "warn"): a newer server major means the client may not understand some envelopes; it warns and proceeds on unknown-but-non-breaking, or throws if the host opts into strictness. Hello-side negotiation (a v in the outbound hello) is deferred to the P7 wire-protocol item; this spec only consumes the version the server already advertises.

6.WS2 — the VoxaClient runtime

Where clients/voxa-client/src Risk Medium (audio + barge-in correctness) Browser APIs WebSocket, getUserMedia, AudioWorklet, AudioContext Testability audio + transport behind seams

One class drives the whole loop. The public surface is small and typed against the generated protocol; the audio engine is behind a backend interface so the state machine can be unit-tested with no browser.

clients/voxa-client/src/index.ts (public API — shape) 
export interface VoxaClientOptions {
  url: string;                          // ws(s)://host/voice
  hello?: Record<string, unknown>;       // sent FIRST, only if provided (host opted into UseWebSocketHello)
  onVersionMismatch?: "warn" | "throw" | "ignore";   // default "warn"
  audio?: AudioBackend;                 // defaults to WebAudioBackend; injectable for tests
  micConstraints?: MediaTrackConstraints; // defaults: EC + NS + AGC on, mono
}

export type VoxaEvents = {
  session:       (s: { v: number; inputSampleRate: number; outputSampleRate: number }) => void;
  transcription: (t: { text: string; isFinal: boolean; language?: string; speakerId?: string }) => void;
  text:          (t: { text: string }) => void;          // streamed assistant tokens
  speaking:      (s: { who: "bot" | "user"; started: boolean }) => void;
  toolCall:      (c: { callId: string; name: string; argumentsJson: string }) => void;
  status:        (m: { message: string }) => void;
  error:         (e: { message: string }) => void;
  micLevel:      (rms: number) => void;                   // 0..1, convenience for meters
  end:           () => void;
};

export class VoxaClient {
  constructor(opts: VoxaClientOptions);
  on<K extends keyof VoxaEvents>(event: K, handler: VoxaEvents[K]): () => void;  // returns an unsubscribe
  connect(): Promise<void>;              // opens WS, resolves once `session` arrives + mic is live
  sendText(text: string): void;         // optional text-only turn (no mic)
  sendToolResult(callId: string, resultJson: string, isError?: boolean): void;
  end(): void;                          // sends {type:"end"}
  disconnect(): Promise<void>;           // close WS + tear down audio
}

The audio backend seam

All Web Audio lives behind one interface. The default WebAudioBackend is the productized reference page logic; tests inject a fake that records calls. This mirrors Voxa's own rule that every engine/transport is testable behind an interface.

clients/voxa-client/src/audio.ts (the seam) 
export interface AudioBackend {
  // start mic at the announced input rate; deliver 20 ms Int16 frames to onFrame.
  startCapture(inputRate: number, onFrame: (pcm: Int16Array) => void): Promise<void>;
  // schedule PCM gap-free at the announced output rate.
  play(pcm: Int16Array, outputRate: number): void;
  // barge-in: stop everything already scheduled and reset the play cursor.
  flush(): void;
  stop(): Promise<void>;
}

Barge-in — the load-bearing behaviour

The reference page already establishes the correct trigger pair and the correct mechanism; the package adopts both verbatim. Playback schedules each chunk against a running nextPlayTime cursor and tracks live source nodes in a set; flush() stops every node in the set and snaps the cursor back to currentTime.

clients/voxa-client/src/web-audio-backend.ts (ported from recorder-worklet.js + index.html) 
// playback cursor + live nodes (from the reference page, typed)
play(pcm: Int16Array, outputRate: number) {
  const f32 = new Float32Array(pcm.length);
  for (let i = 0; i < pcm.length; i++) f32[i] = pcm[i] / 32768;
  const buf = this.ctx.createBuffer(1, f32.length, outputRate);  // ctx resamples to its own rate
  buf.copyToChannel(f32, 0);
  const src = this.ctx.createBufferSource(); src.buffer = buf; src.connect(this.ctx.destination);
  const startAt = Math.max(this.ctx.currentTime, this.nextPlayTime);
  src.start(startAt); this.nextPlayTime = startAt + buf.duration;
  this.scheduled.add(src); src.onended = () => this.scheduled.delete(src);
}
flush() {                              // barge-in: kill scheduled audio immediately
  for (const s of this.scheduled) { try { s.stop(); } catch {} }
  this.scheduled.clear(); this.nextPlayTime = this.ctx.currentTime;
}

In VoxaClient, flush() is wired to both triggers — exactly the reference page's two call sites:

case "interruption": this.audio.flush(); this.emit("…"); break;
case "speaking":
  if (msg.who === "user" && msg.started) this.audio.flush();   // barge-in before the interruption envelope even arrives
  this.emit("speaking", msg); break;

Frontend tools — the gap the reference page leaves

The test page never handles toolCall. The package delivers it as a typed event and exposes sendToolResult(callId, resultJson, isError?), which serializes the toolResult envelope the server already parses (round-tripping through the agent loop's TCS correlation). This is what makes @voxa/client a superset of the reference page, not just a repackaging.

Hello ordering. When hello is provided, the client sends it as the very first text frame on open, before starting capture — a host that configured UseWebSocketHello<T> reads exactly one message out-of-band before the pipeline runs. When hello is omitted (the default-server case), the client sends nothing until it has the session envelope. The client never requires a server to read a hello.

7.WS3 — packaging & framework adapters

Package @voxa/client (scoped, MIT) Format ESM + .d.ts Deps zero runtime Adapter one optional React hook
  • Core package — framework-agnostic, ESM, with generated type declarations. No runtime dependencies; WebSocket and Web Audio are platform APIs. json-schema-to-typescript is a dev dependency used only at build.
  • React hook — a single thin useVoxa(options) that wraps the lifecycle in useEffect/useState and returns the typed event values + actions. Shipped either as a subpath export (@voxa/client/react) or documented as a copy-paste snippet — decided in WS3 (Open Question b). No other framework adapters; no UI components.
  • Examples — an examples/vanilla page that reproduces the current local-voice test UI using the package, and replaces the inline script in samples/Voxa.Samples.MinimalServer/wwwroot/index.html with a package-consuming version (so the sample and the package can't diverge).
  • Versioning — package SemVer is independent of the wire version; VOXA_WIRE_VERSION is exported so consumers can assert against a server. A wire-breaking change bumps both.

8.Plan of action — phases

  1. Phase 1 — Protocol contract (WS1), pure .NET. Add the inbound envelope records; refactor TryParseClientMessage to deserialize-and-map; add the voxa-wire.schema.json emitter + golden test. No JS yet. Fully landed and verified in the existing suite. Ships independently and is valuable on its own.
  2. Phase 2 — VoxaClient happy path (WS2). New clients/voxa-client package; generate protocol.ts; implement connect → session → mic → events → playback over the audio backend seam; port recorder-worklet.js. Headless unit tests with a mock WebSocket + fake audio backend.
  3. Phase 3 — Barge-in + tools (WS2). Wire flush() to the interruption / user-speaking pair; add onToolCall / sendToolResult. Tests assert flush clears scheduled audio on both triggers and the tool round-trip emits the correct envelope.
  4. Phase 4 — Packaging & examples (WS3). ESM build, types, the React hook, the examples page, and the swap of the MinimalServer test page to consume the package. npm publish under @voxa.
Per-PR discipline. Each phase is one logical change, branches from main, carries an [Unreleased] CHANGELOG entry, and is warnings-as-errors clean on the .NET side. Phase 1 in particular must land green against the existing wire golden tests before any TypeScript exists.

The barge-in assertion is the one that matters. 12.Out of scope / deferred

  • Native / mobile clients (Flutter, Swift, Kotlin). Separate consumers in their own repos — the AONIK Flutter client is P3; the VDX-004 "bindings are siblings" stance applies.
  • Reconnect / resume. The session-resilience resume token (P6) is left as a seam — the client surfaces disconnect but does not replay state.
  • The runtime configure control envelope (P7) — mid-session VAD/voice/language changes. The client will gain a sendConfigure when the server supports it; not now.
  • Hello-side version negotiation. Adding v to the outbound hello belongs to P7's wire-protocol item; VDX-005 only consumes the session.v the server already emits.
  • C#-delegate tool schema generation (typed frontend tools, P6) — server-side. The client round-trips the existing callId / argumentsJson strings as-is.
  • A UI component library. Out of scope by design; the package is the transport + audio + protocol logic, not chat widgets.

VDX-008 — Background agent delegation (talker / thinker split)

A second, heavyweight agent that runs tool calls, browsing, and multi-step reasoning off the voice-latency critical path. The interaction model delegates explicitly, keeps talking while the work runs, and decides — when the result lands — whether it still warrants a spoken response. Dead air is the enemy; this spec removes the "6-second tool call = 6 seconds of silence" failure mode.

1. Problem

In text chat a slow tool call is a spinner; in voice it is dead air. The voice budget is <1 s to first audio, but real agent work — web search, RAG over a large corpus, a multi-step tool chain — takes 5–30 s. Today a Voxa granular pipeline runs that work inside the turn: AgentLoopProcessor's turn worker streams the driver, and nothing reaches TTS until the driver yields text. The data loop keeps flowing (the founding invariant), and StatusFrame can paint "Checking…" in the UI — but the conversation is stalled until the turn completes or the response-duration cap truncates it.

The industry answer is a dual-model split: a fast interaction model owns the conversation and the latency budget; a heavyweight background model owns tools and reasoning. The interaction model acknowledges immediately ("sure — give me a moment, was this for the same account?"), the background model works asynchronously, and its result re-enters the conversation as a new turn when it lands. VDX-008 specifies that split as a Voxa-idiomatic frame round-trip.

2. Design overview

Founding decision: delegation is explicit, not a mirror. Every design that copies the whole conversation to a second model doubles cost per turn and creates constant arbitration noise. Here the interaction model opts in per task by calling a delegation tool. The common path stays single-model; the background agent runs only when the interaction model decides the work is worth it.
user speech ─ STT ─▶ AgentLoopProcessor ──────────▶ BackgroundAgentProcessor ─▶ SentenceAggregator ─▶ TTS │ ▲ │ interaction driver ─┘ │ │ runs background driver on a yields BackgroundTask- │ │ bounded worker (its text is RequestFrame + spoken ack │ │ ACCUMULATED, never spoken) │ │ └── BackgroundTaskCompletedFrame (Upstream) ──┘ consumed by the loop ⇒ queued as a background-result TURN (model gates relevance)

The round-trip reuses the two mechanisms the pipeline already trusts:

  • Downstream request, upstream result — exactly the frontend-tool shape (ToolCallRequestFrame down, ToolCallResultFrame back up, consumed by the loop and not forwarded), except the "frontend" is a server-side processor.
  • A new turn-trigger kind — the loop's turn queue today carries only final user transcriptions; it grows a second trigger: "a background task completed". The single FIFO turn worker is untouched, which gives ordering and never-two-turns-at-once for free.

3. Frame contract (Voxa.Core)

/// Delegation request. Emitted by the interaction driver (via ctx.Emitter or as a yielded frame);
/// forwarded downstream by AgentLoopProcessor like any unhandled frame; consumed by
/// BackgroundAgentProcessor. Uninterruptible for the same reason ToolCallRequestFrame is:
/// a barge-in must not orphan a task the model has already verbally promised.
public sealed record BackgroundTaskRequestFrame(
    string TaskId,               // unique correlation id (ulid)
    string Goal,                 // natural-language task for the background agent
    string? ContextJson = null,  // interaction-model-selected context snapshot (compact!)
    string? OriginTurnId = null) // turn that delegated — the staleness anchor
    : DataFrame, IUninterruptible;

/// Completion notice. Pushed UPSTREAM (Direction = FrameDirection.Upstream, like ErrorFrame)
/// by BackgroundAgentProcessor; consumed by AgentLoopProcessor (never forwarded further).
public sealed record BackgroundTaskCompletedFrame(
    string TaskId,
    string ResultText,           // compact result the interaction model will read — a summary, not a trace
    bool IsError = false,
    long ElapsedMs = 0,
    long InputTokens = 0,        // background driver's aggregated LlmUsageFrames — two longs
    long OutputTokens = 0,       // (not UsageTotals: frames must not reference processor types)
    string? OriginTurnId = null)
    : DataFrame, IUninterruptible;
SemanticRule
Interruption Both frames are IUninterruptible; in-flight background tasks are not cancelled by InterruptionFrame — surviving barge-in is the point. They are cancelled by EndFrame / disposal (bounded join, same discipline as the loop's turn worker).
Channel Both are DataFrames — FIFO-ordered with the conversation, same reasoning as SpeculativeUtteranceFrame's deliberate non-system placement.
Result size ResultText is what the interaction model reads inside a latency-bounded voice turn. The background driver's contract is to return a compact answer; its full reasoning trace stays its own.

4. AgentLoopProcessor: background-result turns

The private Channel<string> turn queue becomes Channel<TurnRequest> (internal union: user utterance | background result). Public surface changes are additive: 

public enum TurnTrigger { UserUtterance, BackgroundResult }

// VoiceTurnContext — two additive init-only members (defaults preserve source/binary compat):
public TurnTrigger Trigger { get; init; } = TurnTrigger.UserUtterance;
public BackgroundTaskCompletedFrame? BackgroundResult { get; init; }  // set iff Trigger == BackgroundResult
  • The data loop gains one case: BackgroundTaskCompletedFrame is consumed (not forwarded) and enqueued as a background-result turn — mirroring the existing ToolCallResultFrame consumption.
  • Background-result turns are ordinary turns: LlmTurnStartedFrame / LlmTurnEndedFrame bracket them, the response-duration cap applies, per-turn try/catch isolation applies, hooks fire, and their speech is fully interruptible. For these turns UserText is ""; drivers read ctx.BackgroundResult.
  • A driver that decides the result is stale simply yields nothing — an empty turn, which is invisible on the wire (the WebSocket sink serializes no turn-lifecycle envelopes and SentenceAggregator flushes nothing for an empty buffer). Silence is a valid, cheap outcome. It is not invisible to observability: the lifecycle frames, the onTurnCompleted hook, and the diagnostics turn edges still fire for the swallowed turn — §8 therefore tags background-result turns with their trigger kind so dashboards and Studio can separate (or filter) them rather than mistaking them for zero-output user turns.

4.1 Arbitration policy

Because the turn queue is single-reader FIFO, a result that lands mid-turn naturally waits behind the in-flight turn. The remaining hazard is a result landing while the user is mid-utterance (their final transcription hasn't reached the queue yet — the bot would talk over them). Two knobs, both on the processor:

KnobDefaultBehavior
HoldBackgroundResultsWhileUserSpeaking true The loop observes UserStartedSpeakingFrame on its system path and holds completed results in a side buffer while the user is speaking. Release is data-ordered, not edge-triggered: UserStoppedSpeakingFrame alone must NOT release — SpeechToTextProcessor deliberately forwards stop-speaking before the promoted transcript, and as a SystemFrame it can additionally overtake queued data frames cross-channel. Held results are instead released when the utterance's final TranscriptionFrame is consumed by the data loop, enqueued immediately behind the turn that final triggers (a whitespace-only final — a silence misfire — releases too, since no turn will follow it). A quiet-timeout fallback (HeldResultReleaseTimeout, default 2 s after stop-speaking) covers finals that never arrive. Implementation note: the side buffer is mutated from both the system and data loops and must be synchronized. false ⇒ enqueue immediately (FIFO still serializes actual turns).
MaxPendingBackgroundResults 4 Bound on held + queued background-result turns, drop-oldest — the repo's backpressure convention. A dropped result raises a diagnostics event; it is not an error.
Staleness is the model's call, not a timer's. The interaction model is the only component holding current conversational context, so the relevance gate is its prompt contract (§7), not a TTL heuristic. OriginTurnId travels with the result so hosts can build policy hooks on top, but v1 ships no automatic time-based discard.

5. BackgroundAgentProcessor (Voxa.Core)

A new processor placed immediately downstream of the agent stage. It hosts the heavyweight agent as a second IAgentTurnDriver — the contract is reused verbatim; no new driver interface. What differs is how its output is treated: 

public sealed class BackgroundAgentProcessor : FrameProcessor
{
    public BackgroundAgentProcessor(
        IAgentTurnDriver backgroundDriver,
        int maxConcurrentTasks = 2,          // bounded worker pool per session
        int maxQueuedRequests = 8,           // bound on waiting requests; excess REJECTED, not dropped (below)
        TimeSpan? taskTimeout = null);        // default 120 s; timeout ⇒ IsError completion, not silence
}
  • Consumes BackgroundTaskRequestFrame (not forwarded); everything else passes through untouched — the processor is a transparent stage for the normal voice path.
  • Runs the background driver with Trigger = TurnTrigger.UserUtterance semantics (UserText = Goal, metadata carries ContextJson). The driver's yielded frames are handled by whitelist, so unknown speech-bearing frames fail closed:
    Yielded frameHandling
    LlmTextChunkFrame, TextFrame Accumulated into ResultText. TextFrame explicitly included: TextToSpeechProcessor synthesizes it, so letting it pass would speak background text directly — the containment guarantee this processor exists to hold.
    StatusFrame Forwarded downstream — the existing sanitized-progress channel gives "Researching flights…" UI for free.
    LlmUsageFrame Aggregated into the completion frame's InputTokens/OutputTokens.
    Everything else Dropped with a debug log — never forwarded.
  • Frontend tools are unavailable in background turns: ctx.FrontendTools throws. A background task awaiting a UI round-trip is a design contradiction (nobody is looking at a modal for a task the user delegated and moved past). Backend tools inside the driver are unrestricted.
  • On completion (success, error, or timeout) it pushes BackgroundTaskCompletedFrame upstream. Per-task try/catch isolation: a failed task completes with IsError = true and the message as ResultText — the interaction model can apologize gracefully instead of the session degrading.
  • Excess requests beyond maxConcurrentTasks queue FIFO up to maxQueuedRequests. Beyond the cap a request is rejected immediately with BackgroundTaskCompletedFrame(IsError = true, ResultText = "background task queue is full") — the interaction model verbally promised the work, so it must be told, not left waiting on a promise that never reports back. Rejection rather than drop-oldest is a deliberate deviation from the audio-path backpressure convention: requests are not superseded-by-newer the way audio frames are. The cap also bounds what a looping or prompt-injected interaction model can accumulate in Goal/ContextJson payloads per session.

6. Composition (Voxa.AspNetCore + MAF adapter)

Opt-in via the same one-lookup seam discipline as VDX-006/007:

// Host — the whole feature:
// Scoped: one driver instance per connection — background drivers typically hold per-conversation
// state (ContextJson, tool sessions). A keyed SINGLETON is shared across all concurrent sessions
// and is acceptable only if the driver is stateless and thread-safe.
services.AddKeyedScoped<IAgentTurnDriver>("voxa:background", (sp, _) =>
    MicrosoftAgentVoice.CreateTurnDriver(researchAgent));   // or any hand-rolled driver
app.MapVoxaVoice("/voice").UseDefaults();
  • DefaultVoicePipelineComposer resolves the keyed driver ("voxa:background") from the per-connection scope. Present ⇒ insert BackgroundAgentProcessor after the agent stage and enable the delegation tool. Absent ⇒ the composed pipeline is byte-identical to today (golden-tested, the VDX-006 guarantee).
  • Tuning under Voxa:BackgroundAgent:* (MaxConcurrentTasks, MaxQueuedRequests, TaskTimeoutSeconds, MaxPendingResults, HoldWhileUserSpeaking, HeldResultReleaseTimeoutMs), validated by the existing fail-fast options validator.
  • MAF convenience: when the background driver is present, the Microsoft-Agents adapter injects one AIFunction into the interaction agent: delegate_task(goal, context_summary). Its handler emits BackgroundTaskRequestFrame via ctx.Emitter and returns immediately with an acknowledgment string — so to the interaction model, delegation is just a fast tool call. Hosts on VDX-007 host drivers wire the same emit themselves (one line in their tool dispatch).
  • MAF background-result turns (required, not optional): the adapter's default message build is a single ChatMessage(ChatRole.User, ctx.UserText) (MicrosoftAgentTurnDriver.BuildMessagesAsync) — for a background-result turn UserText is empty, so without an adapter change the flagship UseDefaults() flow would run an empty turn on every completion and the result would never reach the model. WS2 therefore includes: when ctx.Trigger == TurnTrigger.BackgroundResult, the default build injects ResultText (task id, error flag, elapsed) framed by the relevance-gate instruction from §7. Hosts that override BuildMessages take on the trigger check themselves — the option's doc comment says so.
  • Model-tier guidance (docs, not code): interaction model on a fast/cheap tier; background model on the heavyweight tier. The split only pays off if the talker is actually fast.

7. The prompt contract

Most of the quality of this feature lives in prompts, not plumbing. The docs page ships the contract explicitly; the MAF adapter appends a default version to the interaction agent's instructions when delegation is enabled (host-overridable):

RuleWhy
After delegating, acknowledge and do not fabricate — say the work is in progress; hold, hedge, or continue the conversation. The #1 failure mode: the fast model confidently invents the answer the slow model is about to contradict.
On a background-result turn, decide relevance first — if the conversation has moved on or the user already got the answer, yield nothing. Staleness gate (§4.1). Silence must be an explicitly permitted output.
Delegation context (context_summary) and results are compact summaries, never transcripts or traces. Both cross a latency-bounded voice turn; token bloat here is spoken-latency bloat.

8. Diagnostics

Four hub events — BackgroundTaskStarted, BackgroundTaskCompleted (elapsed, error flag, usage), BackgroundTaskRejected (request cap hit, §5), BackgroundTaskDropped (held-result overflow, §4.1) — behind the existing HasListeners zero-cost gate. Task elapsed feeds a voxa.background.task.duration histogram. In addition, the loop's turn-edge events carry the turn's trigger kind (UserUtterance | BackgroundResult) so dashboards and Studio can separate background-result turns — including the empty gated-to-silence ones (§4) — from real user turns instead of reading them as zero-output anomalies. Studio surfacing (a task lane in the Talk view) is deferred; the events are designed so it needs no Core changes later.

9. Non-goals

  • No always-on mirroring — the background model sees only what delegation hands it.
  • No multi-agent routing framework — exactly one background driver per session in v1; fan-out/specialist routing belongs inside the background driver (that's what agent frameworks are for).
  • No cross-session task persistence — tasks die with the session. Durable jobs are a host concern (queue + webhook back into a new session).
  • Composite pipelines out of scope — AzureVoiceLiveProcessor / OpenAIRealtimeProcessor / the VRT-005 S2S seam own their conversation loop vendor-side; granular pipelines only in v1.
  • No wire/protocol change — progress rides the existing StatusFrame envelope; results become ordinary spoken turns. @voxa/client is untouched.

11. Verification

  • Round trip: fake interaction driver emits a request; fake background driver returns a result; loop runs a background-result turn with Trigger = BackgroundResult, correct correlation, and normal lifecycle frames.
  • Arbitration (data-ordered release): result during user speech is held; UserStoppedSpeakingFrame alone does NOT release; the utterance's final TranscriptionFrame releases it behind the turn that final triggers; a whitespace-only final releases too; a stop-speaking with no final ever arriving releases via the quiet-timeout; hold disabled ⇒ immediate enqueue; held-result overflow drops oldest and raises BackgroundTaskDropped.
  • Request cap: a request beyond maxQueuedRequests completes immediately with IsError = true and raises BackgroundTaskRejected; queued (uncapped) requests still run.
  • Interruption: InterruptionFrame mid-task ⇒ task completes and its result turn still runs; interruption during a result turn cancels that turn's speech normally.
  • Isolation & shutdown: throwing background driver ⇒ IsError completion, session healthy; EndFrame/dispose cancels in-flight tasks with the bounded join; timeout ⇒ IsError completion.
  • Containment (whitelist): background-driver LlmTextChunkFrame AND TextFrame are accumulated, never downstream; an unknown/custom frame is dropped, not forwarded; StatusFrames pass; ctx.FrontendTools throws in background turns.
  • MAF default turn build: a background-result turn through the unmodified UseDefaults() MAF path produces messages containing ResultText (regression test pinning the §6 adapter change — an empty message list is the failure this exists to catch).
  • Byte-identity: no keyed driver registered ⇒ composed pipeline byte-identical (extends the existing golden tests); existing AgentLoopProcessor tests unaffected (additive context members default correctly).

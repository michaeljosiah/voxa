VRT-004 — Streaming STT: Interim Transcriptions End-to-End

Realise the speech-to-text contract that is already streaming-shaped: propagate interim (IsFinal:false) transcripts from SpeechToTextProcessor to the diagnostics hub and Studio's live caption, behind an opt-in flag that keeps the default pipeline byte-identical — and ship the first true streaming engine (Azure continuous-recognition) so those interims actually appear as the user speaks. The agent still acts on finals only.

1.What is this? — a plain-English explainer

What does VRT-004 add, in one sentence?

It makes the live, word-by-word caption appear while the user is still talking — and cuts perceived latency — by finally using a speech-to-text contract that was designed for streaming from day one but whose interim results never flowed anywhere.

Why now?

P0 latency. A streaming engine emits a partial hypothesis ("what's the weath…") long before the sentence settles. Surfacing that partial gives the user immediate feedback that they were heard, and — once finals arrive faster than a batch round-trip — shaves real time off the turn. speech-core's Nemotron streaming STT does exactly this on-device at roughly real-time (RTF ~1.0), gated behind an emit_partial_transcriptions opt-in; Voxa is already shaped for the same move.

The key insight

The seam already exists. ISpeechToTextEngine is already StartAsync / WriteAudioAsync / ReadTranscriptsAsync (an IAsyncEnumerable<TranscriptionResult> that yields interim and final results) / StopAsync / FlushAsync. TranscriptionResult already carries IsFinal; TranscriptionFrame already carries IsFinal and SpeakerId. The gap is purely that (a) no engine actually produces interims that reach the user, and (b) the interims that an engine could produce are not propagated downstream or surfaced. This item is an engine + propagation, not a new contract.

What is explicitly not changing?

The engine interface and the TranscriptionFrame record — both already support this; we add neither ISpeechToTextEngine nor TranscriptionFrame.IsFinal. Defaults stay off, so the composed pipeline is byte-identical to today (golden-tested). The agent and the hallucination filter keep acting on finals only; interims are display/turn-signal, never a turn.

2.Executive summary

VRT-004 is two workstreams over the existing speech abstractions and the Voxa.Speech.Azure package:

  • WS1 — Interim propagation end-to-end opt-in / additive. Add an EmitInterimTranscriptions flag (default off) to SpeechToTextProcessor. When on, the read loop forwards an engine's interim TranscriptionResult(IsFinal:false) as a TranscriptionFrame(IsFinal:false), coalesced by a minimum interval so partial churn can't flood the pipeline. The interim reaches VoxaDiagnosticsHub (Studio caption) but is gated out at the agent boundary; TranscriptionFilter and the agent loop continue to act on finals only. With the flag off the read loop is byte-for-byte what it is today.
  • WS2 — A first true-streaming engine P0. Make Voxa.Speech.Azure the first engine whose interims are real: it already uses the SDK's continuous-recognition events (Recognizing → interim, Recognized → final), so the work is to confirm the mapping, expose the interim path it already produces, and wire WS1's flag through Azure's descriptor. A new Voxa.Speech.Deepgram (sub-200 ms partials) is named as the fast-follow alternative but is not in scope here (program Open Decision #1: lean Azure first, no new package).
Net effect. With EmitInterimTranscriptions:true and the Azure streaming engine, a live caption appears in Studio's Talk view as the user speaks — refining word by word — while the agent and the hallucination filter still fire only on the settled, final transcript. Default builds (flag off) are unchanged and golden-clean.

3.Goals, non-goals & budgets

Goals

  • Propagate interim TranscriptionResults as TranscriptionFrame(IsFinal:false) from SpeechToTextProcessor, behind an opt-in flag that keeps the default byte-identical.
  • Surface interims to VoxaDiagnosticsHub so Studio renders a live caption (reusing the existing TranscriptEvent(Text, IsFinal) — no new diagnostic event type).
  • Guarantee interims never trigger a turn or feed the agent: the finals-only gate is explicit and tested.
  • Throttle / coalesce interim churn with a minimum emit interval so a chatty engine can't flood bounded channels.
  • Ship Azure continuous-recognition as the first engine that produces real interims through this path.

Non-goals (this spec)

  • A local streaming STT model — its own follow-up; it would slot here later as a second streaming engine (the on-device analogue of Azure, behind the same flag).
  • Streaming TTS / Cartesia — a separate P0 TTS bullet, unrelated to this STT path.
  • A new Voxa.Speech.Deepgram package — named as the fast-follow, deferred (§12, Open Decision #1).
  • Any change to ISpeechToTextEngine or the TranscriptionFrame shape — both already support interims.

4.Architecture context — where this plugs in

The granular chain is VAD → STT → TranscriptionFilter → [Stt tap] → agent → SentenceAggregator → TTS. Three facts about the existing tree make this item small:

  • The contract is already streaming. SpeechToTextProcessor.ReadLoopAsync already does await foreach (var t in _engine.ReadTranscriptsAsync(ct)) and pushes new TranscriptionFrame(t.Text, t.IsFinal, t.Language). It just never receives an interim today, because the only engines wired in are batch (Whisper REST) or have their interim path unexercised. The change is to decide, per the flag, whether to forward an IsFinal:false result — not to add a code path that pushes transcripts.
  • The diagnostics tap already maps interims. DiagnosticsTapProcessor at DiagnosticsTapScope.Stt maps TranscriptionFrame t => new TranscriptEvent(t.Text, t.IsFinal) — so an interim frame becomes a TranscriptEvent(IsFinal:false) with no new event type. Studio's caption only needs to render the non-final event it will now receive.
  • The finals-only gates already exist. TranscriptionFilter only inspects TranscriptionFrame { IsFinal: true } and passes everything else through unchanged; the StageLatencyTracker keys its STT-final timestamp off TranscriptEvent { IsFinal: true }. The remaining work is the agent boundary — see §5, the load-bearing detail.
Load-bearing ordering detail. In DefaultVoicePipelineComposer the STT diagnostics tap is inserted after TranscriptionFilter (so diagnostics report transcripts that actually drive the agent). The filter forwards interim frames unchanged today — which is exactly what we want for the caption (interims must reach the tap) but means an interim frame would also reach the agent if nothing stops it. WS1 therefore adds the explicit finals-only gate between the tap and the agent (the agent processor ignores non-final transcription frames), so the caption sees interims and the agent never does.

We import the shape (opt-in partials, finals drive the agent), not the C++; Azure is the cloud engine that realises it first, and a local streaming model is the deferred on-device twin.

5.WS1 — Interim propagation end-to-end

Component Voxa.Speech.Abstractions Risk Medium (agent-gate correctness) Default off (byte-identical) Surface opt-in flag + interval

The processor already has the read loop; WS1 adds a flag, a coalescing interval, and forwarding for non-final results. The finals path is untouched. With EmitInterimTranscriptions == false the method body is identical to today's.

src/Voxa.Speech.Abstractions/SpeechToTextProcessor.cs (additions — sketch) 
// New, additive options. Defaults keep the read loop byte-identical to today.
/// <summary>Forward interim (IsFinal:false) transcripts as TranscriptionFrame(IsFinal:false).
/// Off by default — mirrors speech-core's emit_partial_transcriptions opt-in.</summary>
public bool EmitInterimTranscriptions { get; init; }              // default false

/// <summary>Minimum gap between forwarded interim frames; the latest partial in a
/// window wins. Bounds interim churn on the audio/data path.</summary>
public TimeSpan InterimMinInterval { get; init; } = TimeSpan.FromMilliseconds(150);

private async Task ReadLoopAsync(CancellationToken ct)
{
    if (_engine is null) return;
    long lastInterimTicks = 0;
    try
    {
        await foreach (var t in _engine.ReadTranscriptsAsync(ct).ConfigureAwait(false))
        {
            if (t.IsFinal)
            {
                // Finals: exactly as today — drive filter, agent, latency tracker.
                await PushFrameAsync(new TranscriptionFrame(t.Text, isFinal: true, t.Language), ct)
                    .ConfigureAwait(false);
                continue;
            }

            // Interims: opt-in only, and coalesced so partial churn can't flood the channel.
            if (!EmitInterimTranscriptions || string.IsNullOrWhiteSpace(t.Text)) continue;

            var now = Environment.TickCount64;
            if (now - lastInterimTicks < InterimMinInterval.TotalMilliseconds) continue;
            lastInterimTicks = now;

            await PushFrameAsync(new TranscriptionFrame(t.Text, isFinal: false, t.Language), ct)
                .ConfigureAwait(false);
        }
    }
    catch (OperationCanceledException) { /* shutdown */ }
    catch (Exception ex)
    {
        _logger.LogError(ex, "SpeechToTextProcessor: STT engine read loop failed");
        await PushErrorAsync($"STT engine failed: {ex.Message}", ex, ct).ConfigureAwait(false);
    }
}

The finals-only gate at the agent boundary (the part that must not regress)

Interim frames are forwarded downstream so they reach the STT diagnostics tap (caption) and any transport that wants them. They must stop before the agent. The gate lives at the agent intake: a non-final TranscriptionFrame is forwarded (so the sink still completes) but does not start a turn or get appended to the agent's input.

agent intake (AgentLoopProcessor / STT-to-agent adapter) — finals-only gate 
protected override async ValueTask ProcessFrameAsync(Frame frame, CancellationToken ct)
{
    // Interim transcripts are display/turn-signal only — never a turn, never agent input.
    if (frame is TranscriptionFrame { IsFinal: false })
    {
        await PushFrameAsync(frame, ct).ConfigureAwait(false);   // forward; do not act
        return;
    }

    if (frame is TranscriptionFrame { IsFinal: true } final)
    {
        // existing behaviour: this is what starts/feeds a turn
        ...
    }
    ...
}
Invariant — interims are not turns. An interim TranscriptionFrame must never (a) start an agent turn, (b) be appended to conversation memory, or (c) be counted by StageLatencyTracker as the STT-final edge. (a)/(b) are enforced by the agent-intake gate above; (c) holds for free because the tracker already keys on TranscriptEvent { IsFinal: true }. The hallucination TranscriptionFilter already ignores non-finals — leave it that way so partials still reach the caption tap.

Surfacing the caption (diagnostics)

No diagnostics code changes: the interim TranscriptionFrame already maps to TranscriptEvent(Text, IsFinal:false) at the STT tap. Studio's Talk view renders the non-final event as a caption that is replaced in place on each update, then promoted to settled text when the matching final arrives. Because every publish is behind VoxaDiagnosticsHub.HasListeners, an unobserved session still allocates nothing.

Optional — interims as a richer smart-turn signal

Cross-reference (optional, can defer to VRT-002). The smart-turn seam shipped in PR #26 (SileroVadOptions.ConfirmTurnEnd) re-decides end-of-turn at the silence timeout from audio alone. The latest interim text is a strictly richer signal ("…what's the weather in" clearly is not turn-final). Feeding the most-recent partial to the classifier is a natural enhancement that composes with VRT-002's eager-STT decisions; it is out of scope for VRT-004's core (it would couple STT text into the VAD seam) and noted here only as the obvious next hook.

6.WS2 — A first true-streaming engine

Package Voxa.Speech.Azure Risk Low (SDK already continuous) Decision Azure first; Deepgram fast-follow New deps none

Per program Open Decision #1 (lean: Azure first, cheapest, no new package), the first engine to produce real interims through WS1 is Voxa.Speech.Azure. The convenient truth: AzureSpeechToTextEngine already runs in continuous-recognition mode and already maps both event kinds to TranscriptionResult. WS2 is therefore confirm + expose, not a rewrite.

src/Voxa.Speech.Azure/AzureSpeechToTextEngine.cs (the mapping that already exists) 
// Recognizing → interim hypothesis (IsFinal:false). Already wired.
private void OnRecognizing(object? sender, SpeechRecognitionEventArgs e)
{
    if (!string.IsNullOrEmpty(e.Result.Text))
        _transcripts.Writer.TryWrite(
            new TranscriptionResult(e.Result.Text, IsFinal: false, _options.RecognitionLanguage));
}

// Recognized → settled utterance (IsFinal:true). Already wired.
private void OnRecognized(object? sender, SpeechRecognitionEventArgs e)
{
    if (e.Result.Reason == ResultReason.RecognizedSpeech && !string.IsNullOrEmpty(e.Result.Text))
        _transcripts.Writer.TryWrite(
            new TranscriptionResult(e.Result.Text, IsFinal: true, _options.RecognitionLanguage));
}

So the SDK event → TranscriptionResult mapping is the model for any streaming engine:

SDK / socket eventMaps to
Partial / interim hypothesis (Azure Recognizing; Deepgram is_final:false)TranscriptionResult(text, IsFinal:false, lang)
Settled utterance (Azure Recognized; Deepgram is_final:true / speech_final)TranscriptionResult(text, IsFinal:true, lang)
Error / cancellationComplete the transcript channel with the error (as Azure's OnCanceled does)
What WS2 actually changes. (1) Thread EmitInterimTranscriptions from config through the Azure descriptor so the composed SpeechToTextProcessor forwards the interims Azure already emits. (2) Add a focused test asserting Azure's two event kinds produce IsFinal:false then IsFinal:true in order. No SpeechRecognizer reconfiguration — continuous recognition is already how StartAsync runs.
Deepgram — named, not built. A new Voxa.Speech.Deepgram package (WebSocket streaming, sub-200 ms partials) is the obvious lower-latency alternative and the program's fast-follow. It would implement the same ISpeechToTextEngine and map is_final to TranscriptionResult.IsFinal identically. It is deferred (§12) to keep VRT-004 to one provider and no new dependency; the Deepgram SDK would live in its own provider package, never in Core.

7.Configuration reference

One new key on the STT processor (provider-neutral), surfaced through whichever streaming engine is selected. The provider that needs configuration captures the IConfiguration passed to AddVoxa — it does not call GetRequiredService<IConfiguration>() (that breaks the plain-ServiceCollection hosts: Studio, tests, CLI).

appsettings.json — new/affected keys 
{
  "Voxa": {
    "Stt": "Azure",                       // the first true-streaming engine (WS2)
    "Azure": {
      "SubscriptionKey": "...",
      "Region": "eastus",
      "RecognitionLanguage": "en-US"
    },
    "Stt:EmitInterimTranscriptions": true, // NEW. default false → byte-identical pipeline
    "Stt:InterimMinInterval": "00:00:00.150" // NEW. coalescing window (default ~150 ms)
  }
}

The exact binding location (a shared STT options block vs. per-provider) is an implementation choice; the default-off semantics and the finals-only agent gate are the contract. With the flag unset, no interim frame is ever produced and the pipeline is the one shipped today.

8.Plan of action — phases & dependencies

  1. Phase A — WS1 propagation behind the flag (fake engine). Add EmitInterimTranscriptions + InterimMinInterval to SpeechToTextProcessor; add the agent-intake finals-only gate; prove it with a fake streaming engine that yields interim-then-final. Default-off golden stays byte-identical. No network, no provider change.
  2. Phase B — WS2 wire the real Azure streaming engine. Thread the flag through the Azure descriptor; add the event-mapping test (interim before final, in order). Continuous recognition is already on; this is confirm + expose.
  3. Phase C — Studio caption surface. Render the non-final TranscriptEvent in the Talk view as a live, replaced-in-place caption, promoted to settled text on the final. Studio-only; uses the existing diagnostics stream, no new event type, voxa-design tokens for the surface.

One logical change per PR (CONTRIBUTING): Phase A, Phase B, Phase C are separate PRs, each with an [Unreleased] CHANGELOG.md entry. This spec is authored before any of its code, per the program's spec-before-code rule.

12.Out of scope / deferred

  • A local streaming STT model. The on-device analogue of Azure (speech-core's Nemotron-class, partials at RTF ~1.0) is its own item; it slots in here later as a second streaming engine behind the same EmitInterimTranscriptions flag — no new propagation work.
  • Voxa.Speech.Deepgram (sub-200 ms partials). Named as the fast-follow; deferred so VRT-004 ships one provider with no new dependency (Open Decision #1). Same ISpeechToTextEngine, its SDK in its own package.
  • Cartesia / streaming TTS. A separate P0 TTS bullet; unrelated to this STT path.
  • Interims feeding smart-turn / eager STT. The richer-signal hook (§5) belongs with VRT-002; VRT-004 only makes the interim text available.

VRT-004 — Streaming STT: Interim Transcriptions End-to-End

Interim (IsFinal:false) transcripts already propagate from SpeechToTextProcessor to the diagnostics hub and any transport for streaming engines today; the agent already ignores them. VRT-004 renders them (Studio's live caption), rate-bounds their churn on the bounded data channel (a new coalescing interval), and regression-tests the two invariants that hold today — interims keep flowing, interims never reach the agent. WS2 confirms Azure (a shipping streaming engine) already emits them and pins the mapping; the real engine gaps are latency (Deepgram) and the local streaming tier.

1.What is this? — a plain-English explainer

What does VRT-004 add, in one sentence?

It makes the live, word-by-word caption appear while the user is still talking — and bounds the rate at which those partials hit the pipeline — by rendering and rate-limiting interim transcripts that, for a streaming engine, already flow end-to-end today. It does not introduce interim propagation; that already works.

Corrected baseline. Interim transcripts already propagate for streaming engines today: SpeechToTextProcessor.ReadLoopAsync forwards every TranscriptionResult — interim and final — unconditionally; the Azure engine already emits IsFinal:false partials; the diagnostics tap already maps them; and the agent already ignores non-finals. This item renders and rate-bounds them and locks the behaviour with regression tests — it does not add them, and it must not gate them off.
Why now?

P0 latency. A streaming engine emits a partial hypothesis ("what's the weath…") long before the sentence settles. Surfacing that partial gives the user immediate feedback that they were heard, and — once finals arrive faster than a batch round-trip — shaves real time off the turn. The partials already reach the diagnostics hub; what is missing is a Studio surface that draws them and a coalescing interval that keeps a chatty engine from flooding the bounded data channel. speech-core's Nemotron streaming STT does the same on-device at roughly real-time (RTF ~1.0); that on-device streaming tier is a deferred engine gap (VLS-007), not a propagation gap.

The key insight

The seam already carries interims — verified in source. ISpeechToTextEngine is StartAsync / WriteAudioAsync / ReadTranscriptsAsync (an IAsyncEnumerable<TranscriptionResult> that yields interim and final results) / StopAsync / FlushAsync. SpeechToTextProcessor.ReadLoopAsync forwards each result as a TranscriptionFrame(t.Text, t.IsFinal, t.Language) with no filter, Azure already writes IsFinal:false on Recognizing, and AgentLoopProcessor matches only TranscriptionFrame { IsFinal: true }. So today, with Azure configured, interims already flow to the hub and transports and the agent already drops them. The remaining gaps are (a) Studio doesn't draw the caption, (b) nothing coalesces interim churn on the bounded channel, and (c) batch engines (Whisper REST, whisper.cpp) emit no interims so the local tier has none. This item is render + rate-limit + regression-test, not a new contract and not new propagation.

What is explicitly not changing?

The engine interface and the TranscriptionFrame record — both already support this; we add neither ISpeechToTextEngine nor TranscriptionFrame.IsFinal. The existing interim flow is preserved: no on/off suppression flag is introduced, so streaming engines keep emitting partials exactly as they do today. The agent and the hallucination filter already act on finals only — that stays true and is now regression-tested; interims remain display/turn-signal, never a turn.

2.Executive summary

VRT-004 is two workstreams over the existing speech abstractions and the Voxa.Speech.Azure package:

  • WS1 — Render & rate-bound the interims that already flow additive. Interim TranscriptionFrame(IsFinal:false)s already reach VoxaDiagnosticsHub and transports for streaming engines, and the agent already drops them. WS1 adds one genuinely-new knob: InterimMinInterval on SpeechToTextProcessor, which coalesces interim frames (latest partial in a window wins) so a chatty engine can't flood the bounded data channel. It does not introduce a suppression flag — interims keep flowing. The agent's finals-only behaviour already exists (AgentLoopProcessor matches { IsFinal: true }); WS1 locks it with a regression test (interims still never start a turn; the agent fires once, on the final), and adds a test that the existing interim flow is preserved.
  • WS2 — Pin the streaming-engine mapping; name the real engine gaps P0. Azure is a shipping streaming engine: AzureSpeechToTextEngine already raises the SDK's continuous-recognition events (Recognizing → IsFinal:false, Recognized → IsFinal:true) and its interims already propagate. WS2 therefore shrinks to a mapping regression test (Recognizing→interim, Recognized→final, ordered) plus noting interims already reach the pipeline. The real engine gaps are latency (a Deepgram sub-200 ms fast-follow) and the local streaming tier (an on-device Nemotron-class engine, VLS-007) — not "expose Azure interims". Deepgram is named, not built here (program Open Decision #1: lean Azure first, no new package).
Net effect. With the Azure streaming engine, interim TranscriptionFrames already flow to the hub and transports today; VRT-004 makes Studio's Talk view draw them as a live caption that refines word by word, coalesces their rate via InterimMinInterval, and regression-tests that the agent and hallucination filter still fire only on the settled final. The change preserves today's interim flow rather than gating it — and a guard test asserts both invariants (interims keep flowing; interims never reach the agent).

3.Goals, non-goals & budgets

Goals

  • Coalesce / rate-bound the interim TranscriptionFrame(IsFinal:false)s that SpeechToTextProcessor already forwards, via a new minimum emit interval, so a chatty engine can't flood the bounded data channel. (The interval is the one genuinely-new knob; it is additive and bounds churn while interims keep flowing.)
  • Render interims in Studio: draw the non-final TranscriptEvent the hub already publishes as a live caption (reusing the existing TranscriptEvent(Text, IsFinal) — no new diagnostic event type).
  • Regression-test the two invariants that already hold: (a) interims keep propagating to the hub/transports for a streaming engine; (b) interims never trigger a turn or feed the agent (the agent already matches finals only).
  • Pin the Azure event mapping (Recognizing→interim, Recognized→final) with a focused test; note that Azure interims already reach the pipeline.

Non-goals (this spec)

  • A local streaming STT model — its own follow-up (VLS-007); it would slot here later as a second streaming engine (the on-device analogue of Azure). Its interims would propagate through the same existing path — no new propagation work.
  • Streaming TTS / Cartesia — a separate P0 TTS bullet, unrelated to this STT path.
  • A new Voxa.Speech.Deepgram package — named as the lower-latency fast-follow, deferred (§12, Open Decision #1).
  • Any change to ISpeechToTextEngine or the TranscriptionFrame shape — both already support interims, which already flow.

4.Architecture context — where this plugs in

The granular chain is VAD → STT → TranscriptionFilter → [Stt tap] → agent → SentenceAggregator → TTS. Four facts about the existing tree make this item small — and reframe it from "add propagation" to "render + rate-bound + lock":

  • The processor already forwards interims unconditionally. SpeechToTextProcessor.ReadLoopAsync does await foreach (var t in _engine.ReadTranscriptsAsync(ct)) and pushes new TranscriptionFrame(t.Text, t.IsFinal, t.Language) for every result — interim and final — with no gate. So for a streaming engine an interim already leaves the processor today. The change is to coalesce those interim frames by a minimum interval — not to add a code path that pushes them, and not to add a flag that withholds them.
  • Azure already emits interims into this loop. AzureSpeechToTextEngine.OnRecognizing writes TranscriptionResult(text, IsFinal:false, …) and OnRecognized writes IsFinal:true. With Azure configured, the read loop above is already receiving and forwarding interims.
  • The diagnostics tap already maps interims. DiagnosticsTapProcessor at DiagnosticsTapScope.Stt maps TranscriptionFrame t => new TranscriptEvent(t.Text, t.IsFinal) — so an interim frame already becomes a TranscriptEvent(IsFinal:false) on the hub, with no new event type. Studio's caption only needs to render the non-final event it already receives.
  • The finals-only gates already exist — including at the agent. TranscriptionFilter only inspects TranscriptionFrame { IsFinal: true } and passes everything else through unchanged; the StageLatencyTracker keys its STT-final timestamp off TranscriptEvent { IsFinal: true }; and AgentLoopProcessor matches case TranscriptionFrame { IsFinal: true } — a non-final transcription frame falls to its default arm, which forwards it without starting a turn. So no new agent gate is needed; the remaining work is to lock this with a regression test — see §5.
Load-bearing ordering detail. In DefaultVoicePipelineComposer the STT diagnostics tap is inserted after TranscriptionFilter (so diagnostics report transcripts that actually drive the agent). The filter forwards interim frames unchanged today — exactly what we want for the caption (interims must reach the tap) — and an interim frame then reaches AgentLoopProcessor, which already ignores it (it matches { IsFinal: true } only; non-finals fall through its default arm, forwarded but not acted on). So the caption already sees interims and the agent already doesn't act on them. WS1 does not add a gate here; it regression-tests that this stays true (interims keep flowing to the tap; the agent fires exactly once, on the final).

5.WS1 — Interim propagation end-to-end

Component Voxa.Speech.Abstractions Risk Medium (preserve flow + lock agent-ignores-interims) Behaviour interims already flow; preserved Surface one new knob: InterimMinInterval

The processor's read loop already forwards every result — interim and final — unconditionally. WS1 adds a single additive knob, InterimMinInterval, that coalesces the interim frames it already emits (latest partial in each window wins) so a chatty engine can't flood the bounded data channel. There is no on/off suppression flag: interims keep flowing. The finals path is untouched.

src/Voxa.Speech.Abstractions/SpeechToTextProcessor.cs (additions — sketch) 
// One new, additive option. Interims already propagate; this only bounds their rate.
/// <summary>Minimum gap between forwarded interim (IsFinal:false) frames; the latest partial
/// in a window wins. Bounds interim churn on the bounded audio/data path. Finals are never
/// coalesced. Interims are NOT gated off — they flow exactly as today, just rate-limited.</summary>
public TimeSpan InterimMinInterval { get; init; } = TimeSpan.FromMilliseconds(150);

private async Task ReadLoopAsync(CancellationToken ct)
{
    if (_engine is null) return;
    long lastInterimTicks = 0;
    try
    {
        await foreach (var t in _engine.ReadTranscriptsAsync(ct).ConfigureAwait(false))
        {
            if (t.IsFinal)
            {
                // Finals: exactly as today — never coalesced; drive filter, agent, latency tracker.
                await PushFrameAsync(new TranscriptionFrame(t.Text, t.IsFinal, t.Language), ct)
                    .ConfigureAwait(false);
                continue;
            }

            // Interims already flow today; here we ONLY coalesce so partial churn can't flood
            // the bounded channel. No suppression — drop only empties and the in-window churn.
            if (string.IsNullOrWhiteSpace(t.Text)) continue;

            var now = Environment.TickCount64;
            if (now - lastInterimTicks < InterimMinInterval.TotalMilliseconds) continue;
            lastInterimTicks = now;

            await PushFrameAsync(new TranscriptionFrame(t.Text, t.IsFinal, t.Language), ct)
                .ConfigureAwait(false);
        }
    }
    catch (OperationCanceledException) { /* shutdown */ }
    catch (Exception ex)
    {
        _logger.LogError(ex, "SpeechToTextProcessor: STT engine read loop failed");
        await PushErrorAsync($"STT engine failed: {ex.Message}", ex, ct).ConfigureAwait(false);
    }
}

The finals-only behaviour at the agent (already exists — now regression-tested)

Interim frames are forwarded downstream so they reach the STT diagnostics tap (caption) and any transport that wants them, and they stop short of acting on the agent today: AgentLoopProcessor.ProcessFrameAsync matches case TranscriptionFrame { IsFinal: true } to start a turn, and a non-final transcription frame falls through to its default arm — forwarded (so the sink still completes) but not queued to the turn worker or appended to the agent's input. No new gate is added here. WS1's job is to lock this with a regression test so a future refactor can't let an interim start a turn.

src/Voxa.Core/Processors/AgentLoopProcessor.cs (the existing finals-only match — unchanged) 
protected override async ValueTask ProcessFrameAsync(Frame frame, CancellationToken ct)
{
    switch (frame)
    {
        // Finals only: this is what forwards the user bubble AND starts a turn.
        case TranscriptionFrame { IsFinal: true } t when !string.IsNullOrWhiteSpace(t.Text):
            await PushFrameAsync(frame, ct).ConfigureAwait(false);
            await _turnQueue.Writer.WriteAsync(t.Text, ct).ConfigureAwait(false);
            return;

        // Non-final TranscriptionFrames hit this arm: forwarded for the caption/transport,
        // never queued to the turn worker. This is the existing behaviour we regression-test.
        default:
            await PushFrameAsync(frame, ct).ConfigureAwait(false);
            return;
    }
}
Invariant — interims are not turns (already enforced, now regression-tested). An interim TranscriptionFrame must never (a) start an agent turn, (b) be appended to conversation memory, or (c) be counted by StageLatencyTracker as the STT-final edge. (a)/(b) already hold because AgentLoopProcessor matches { IsFinal: true } only and routes non-finals through its default forward-without-acting arm; (c) holds because the tracker keys on TranscriptEvent { IsFinal: true }. The hallucination TranscriptionFilter already ignores non-finals — leave it that way so partials still reach the caption tap. VRT-004 adds a test that pins all three against a future regression; it does not introduce the behaviour.

Surfacing the caption (diagnostics)

No diagnostics code changes: the interim TranscriptionFrame already maps to TranscriptEvent(Text, IsFinal:false) at the STT tap and already reaches the hub today. The only missing piece is the draw: Studio's Talk view renders the non-final event as a caption that is replaced in place on each update, then promoted to settled text when the matching final arrives. Because every publish is behind VoxaDiagnosticsHub.HasListeners, an unobserved session still allocates nothing.

Optional — interims as a richer smart-turn signal

Cross-reference (optional, can defer to VRT-002). The smart-turn seam shipped in PR #26 (SileroVadOptions.ConfirmTurnEnd) re-decides end-of-turn at the silence timeout from audio alone. The latest interim text is a strictly richer signal ("…what's the weather in" clearly is not turn-final). Feeding the most-recent partial to the classifier is a natural enhancement that composes with VRT-002's eager-STT decisions; it is out of scope for VRT-004's core (it would couple STT text into the VAD seam) and noted here only as the obvious next hook.

6.WS2 — Pin the streaming engine; name the gaps

Package Voxa.Speech.Azure Risk Low (already shipping streaming interims) Decision Azure ships it; Deepgram/local are the gaps New deps none

Azure is already a shipping streaming engine: AzureSpeechToTextEngine runs in continuous-recognition mode, maps both event kinds to TranscriptionResult, and its interims already propagate through the processor's read loop to the hub and transports. WS2 is therefore a regression test (pin the mapping) plus a note that interims already reach the pipeline — not "expose Azure interims" and not a rewrite. Per program Open Decision #1 (lean: Azure first, cheapest, no new package), no second provider is added here; the real engine gaps are latency and the local tier (below).

src/Voxa.Speech.Azure/AzureSpeechToTextEngine.cs (the mapping that already exists) 
// Recognizing → interim hypothesis (IsFinal:false). Already wired.
private void OnRecognizing(object? sender, SpeechRecognitionEventArgs e)
{
    if (!string.IsNullOrEmpty(e.Result.Text))
        _transcripts.Writer.TryWrite(
            new TranscriptionResult(e.Result.Text, IsFinal: false, _options.RecognitionLanguage));
}

// Recognized → settled utterance (IsFinal:true). Already wired.
private void OnRecognized(object? sender, SpeechRecognitionEventArgs e)
{
    if (e.Result.Reason == ResultReason.RecognizedSpeech && !string.IsNullOrEmpty(e.Result.Text))
        _transcripts.Writer.TryWrite(
            new TranscriptionResult(e.Result.Text, IsFinal: true, _options.RecognitionLanguage));
}

So the SDK event → TranscriptionResult mapping is the model for any streaming engine:

SDK / socket eventMaps to
Partial / interim hypothesis (Azure Recognizing; Deepgram is_final:false)TranscriptionResult(text, IsFinal:false, lang)
Settled utterance (Azure Recognized; Deepgram is_final:true / speech_final)TranscriptionResult(text, IsFinal:true, lang)
Error / cancellationComplete the transcript channel with the error (as Azure's OnCanceled does)
What WS2 actually changes. Effectively one thing: add a focused regression test asserting Azure's two event kinds produce IsFinal:false then IsFinal:true in order, pinning a mapping that already ships. Plus a one-line note that these interims already reach the composed pipeline. No SpeechRecognizer reconfiguration (continuous recognition is already how StartAsync runs) and no flag to thread — interims are not gated.
The real engine gaps (not "expose Azure interims"). Azure already streams interims, so the open work is elsewhere: (1) latency — a Deepgram WebSocket engine for sub-200 ms partials (named below, deferred); (2) the local streaming tier — an on-device Nemotron-class engine (VLS-007) so the offline tier emits interims at all (batch Whisper REST / whisper.cpp produce none). Both are engine work; neither needs propagation changes — their interims would ride the path Azure already uses.
Deepgram — named, not built. A new Voxa.Speech.Deepgram package (WebSocket streaming, sub-200 ms partials) is the obvious lower-latency alternative and the program's fast-follow. It would implement the same ISpeechToTextEngine and map is_final to TranscriptionResult.IsFinal identically. It is deferred (§12) to keep VRT-004 to one provider and no new dependency; the Deepgram SDK would live in its own provider package, never in Core.

7.Configuration reference

One new key on the STT processor (provider-neutral), surfaced through whichever streaming engine is selected. The provider that needs configuration captures the IConfiguration passed to AddVoxa — it does not call GetRequiredService<IConfiguration>() (that breaks the plain-ServiceCollection hosts: Studio, tests, CLI).

appsettings.json — new/affected keys 
{
  "Voxa": {
    "Stt": "Azure",                       // a shipping streaming engine — already emits interims (WS2)
    "Azure": {
      "SubscriptionKey": "...",
      "Region": "eastus",
      "RecognitionLanguage": "en-US"
    },
    "Stt:InterimMinInterval": "00:00:00.150" // NEW. coalescing window for the interims that already flow (default ~150 ms)
  }
}

The exact binding location (a shared STT options block vs. per-provider) is an implementation choice. There is no on/off interim flag: for a streaming engine, interims already propagate and stay that way. The contract is the coalescing window (rate-bound) plus the finals-only behaviour at the agent (already true, regression-tested). With InterimMinInterval unset it defaults to ~150 ms; the interims a streaming engine emits keep flowing, merely rate-limited.

8.Plan of action — phases & dependencies

  1. Phase A — WS1 coalescing + regression tests (fake engine). Add InterimMinInterval to SpeechToTextProcessor (coalesce the interims it already forwards); add the regression tests that interims keep flowing to the tap and that the agent fires once on the final (the agent already ignores interims — no code change there). Prove it with a fake streaming engine that yields interim-then-final. No network, no provider change.
  2. Phase B — WS2 pin the Azure mapping. Add the event-mapping regression test (interim before final, in order); note that Azure's interims already reach the composed pipeline. No flag to thread, no descriptor change — continuous recognition and interim emission already ship.
  3. Phase C — Studio caption surface. Render the non-final TranscriptEvent in the Talk view as a live, replaced-in-place caption, promoted to settled text on the final. Studio-only; uses the existing diagnostics stream, no new event type, voxa-design tokens for the surface.

One logical change per PR (CONTRIBUTING): Phase A, Phase B, Phase C are separate PRs, each with an [Unreleased] CHANGELOG.md entry. This spec is authored before any of its code, per the program's spec-before-code rule.

12.Out of scope / deferred

  • A local streaming STT model (VLS-007). The on-device analogue of Azure (speech-core's Nemotron-class, partials at RTF ~1.0) is its own engine item; it slots in here later as a second streaming engine, and once it produces interims they ride the same already-working propagation path — no new propagation work, and only the same InterimMinInterval coalescing applies.
  • Voxa.Speech.Deepgram (sub-200 ms partials). Named as the fast-follow; deferred so VRT-004 ships one provider with no new dependency (Open Decision #1). Same ISpeechToTextEngine, its SDK in its own package.
  • Cartesia / streaming TTS. A separate P0 TTS bullet; unrelated to this STT path.
  • Interims feeding smart-turn / eager STT. The richer-signal hook (§5) belongs with VRT-002; VRT-004 only makes the interim text available.

1.What is this? — a plain-English explainer

What does VLS-007 add, in one sentence?

A second on-device speech-to-text option — three NVIDIA-family ONNX models (Parakeet and two Nemotron variants) that are more accurate than the local whisper.cpp tier, and — for the first time on the offline path — can stream partial transcripts as you speak instead of only producing a final result once you stop.

Why now?

Voxa's local STT tier (VLS-001/VLS-002) is whisper.cpp, which is batch: it buffers a whole utterance and emits a single final transcript — no live captions, no early text. speech-core proves that Parakeet TDT v3 and the Nemotron streaming models run well on commodity hardware as ONNX graphs, and VLS-006 has just given Voxa a shared ONNX-Runtime host to plug them into. The STT contract was already shaped for streaming (ReadTranscriptsAsync yields interim + final); VRT-004 wired interim frames through the rest of the pipeline. This item supplies the missing piece: a local engine that actually emits those interims.

Does this change the STT contract or replace whisper.cpp?

No to both. ISpeechToTextEngine already has StartAsync / WriteAudioAsync / ReadTranscriptsAsync (interim + final) / StopAsync / FlushAsync — these engines implement that contract, they don't redefine it. whisper.cpp stays exactly as it is; this is an additional engine you select with Voxa:Stt = OnnxAsr. Pick whichever local tier you want per deployment.

What is the hard part?

The pre- and post-processing, not the inference. Each model needs a faithfully ported log-mel front-end, a SentencePiece tokenizer/vocab, and a correct neural-transducer decode (TDT for Parakeet, cache-aware streaming RNN-T for the Nemotron pair). That decode logic is fiddly and must be validated against speech-core's reference outputs and known transcripts — it is the long pole of this work, which is why batch Parakeet ships first to prove the host + descriptor before the streaming pair lands.

2.Executive summary

VLS-007 is one new opt-in package, Voxa.Speech.OnnxAsr, running on the VLS-006 shared ONNX-Runtime host, with three model families behind the existing ISpeechToTextEngine contract:

  • WS1 — Parakeet TDT v3 (0.6B) P6 batch. A FastConformer encoder + Token-and-Duration-Transducer (TDT) decoder, exported as two ONNX graphs (encoder + decoder_joint) with a 128-bin mel front-end. Highest accuracy; auto-detects language. Runs once per utterance on UserStoppedSpeaking / FlushAsync and emits TranscriptionResult(IsFinal: true) — the simplest decode path, so it lands first.
  • WS2 — Nemotron Speech Streaming (0.6B) P0 streaming. Cache-aware streaming RNN-T (three graphs: encoder + decoder + joint). speech-core's begin_stream / push_chunk / end_stream loop maps onto WriteAudioAsync → interim TranscriptionResult(IsFinal: false), finalised on FlushAsync/StopAsync. This is the local streaming engine VRT-004 deferred — it consumes VRT-004's interim propagation end-to-end. ~RTF 1.0 on CPU.
  • WS3 — Nemotron-3.5 ASR Streaming Multilingual (0.6B) P0 streaming. Same streaming shape, plus multilingual coverage and prompt-conditioning — a language/context hint selects an encoder prompt slot (mapped from TranscriptionResult.Language / a config prompt). ONNX-FP16 variant available.
Net effect. A developer who wants higher local accuracy sets "Stt": "OnnxAsr", "OnnxAsr": { "Model": "parakeet-tdt-v3" } and gets best-in-class on-device transcription; switching Model to nemotron-streaming turns on live partial captions on the fully offline path — the first time the local tier streams. whisper.cpp users change nothing, and CI stays offline and deterministic.

3.Goals, non-goals & budgets

Goals

  • Add a high-accuracy local STT tier (Parakeet TDT v3) selectable by config name, SHA-256-pinned like every other artifact, on the VLS-006 host.
  • Ship the first local streaming STT engine(s) — Nemotron streaming + Nemotron-3.5 multilingual — that emit interim TranscriptionResult(IsFinal: false) through VRT-004's propagation to live captions.
  • Implement the existing ISpeechToTextEngine contract faithfully — do not invent a new STT seam; the contract already streams.
  • Match the WhisperCpp descriptor/catalog/config shape so the new tier self-describes and resolves through VoxaModelCache exactly like whisper.cpp.
  • Keep the default meta-package CPU-only and offline-clean: the heavy 0.6B models are opt-in and trait-gated, GPU rides VLS-006/VLS-002 (never bundled).

Non-goals (this spec)

  • LiteRT (TFLite) backends — speech-core ships a LiteRT Nemotron twin; Voxa standardizes on ONNX Runtime via VLS-006. Out of scope.
  • Non-ASR ONNX models — DeepFilterNet3 (VLS-004), Sidon (VLS-008), the diarization models (VLS-005) are separate consumers of the same host.
  • On-device fine-tuning / training — inference only.
  • Replacing whisper.cpp — this is an additional engine; both local tiers coexist, selected per deployment.
  • A new OnnxEngine / session host — that is VLS-006; this item is its first ASR consumer.

4.Architecture context — where this plugs in

All three engines are vendor engines behind the existing ISpeechToTextEngine contract — exactly the seat WhisperCppSttEngine occupies — composed by the generic SpeechToTextProcessor. The processor is unchanged: it pipes AudioRawFrame PCM into WriteAudioAsync, calls FlushAsync on UserStoppedSpeakingFrame, and turns every TranscriptionResult the read loop yields into a TranscriptionFrame(Text, IsFinal, Language). The batch-vs-streaming difference is entirely inside the engine:

EngineShapeHow it uses the contract
parakeet-tdt-v3Batch (like whisper.cpp)WriteAudioAsync buffers PCM; FlushAsync/StopAsync run encoder + TDT decode over the buffered utterance and write one TranscriptionResult(IsFinal: true).
nemotron-streamingStreamingWriteAudioAsync feeds chunks into the cache-aware encoder loop; the read loop yields interim (IsFinal: false) as text accumulates; FlushAsync/StopAsync emits the authoritative (IsFinal: true).
nemotron-3.5-multilingualStreaming + promptAs above, plus a language prompt slot selected before/at stream start.
Mapping speech-core's STTInterface onto Voxa. speech-core's Parakeet and Nemotron ONNX impls satisfy STTInterface with transcribe() (batch) and the streaming quintet supports_streaming() / begin_stream() / push_chunk() / flush_stream() / end_stream() / cancel_stream(). Voxa's already-streaming contract is the natural target: begin_stream ≈ StartAsync, push_chunk ≈ WriteAudioAsync (its PartialResult ≈ an interim TranscriptionResult yielded from ReadTranscriptsAsync), flush_stream/end_stream ≈ FlushAsync/StopAsync, and cancel_stream ≈ the per-frame CancellationToken + DisposeAsync. We port the model math and state machine; we do not port the C++ interface — Voxa's contract is the interface.

5.WS1 — Parakeet TDT v3 batch engine

Package Voxa.Speech.OnnxAsr Host Voxa.Audio.Onnx (VLS-006) Risk Medium (TDT decode + mel port) Shape batch — final-only

Parakeet TDT v3 (0.6B) is a FastConformer encoder + Token-and-Duration-Transducer decoder, exported by NeMo as two ONNX graphs — encoder and decoder_joint — plus a SentencePiece vocabulary. It takes 16 kHz mono Float32 audio and produces text with language detection. Its shape matches whisper.cpp's exactly: the VAD gates audio so only speech reaches the engine, and SpeechToTextProcessor calls FlushAsync on UserStoppedSpeakingFrame — so the engine buffers PCM between flushes and transcribes once per utterance, final-only. That is why it ships first: same lifecycle as the proven whisper.cpp engine, just a different inference core.

src/Voxa.Speech.OnnxAsr/ParakeetSttEngine.cs (sketch — mirrors WhisperCppSttEngine's buffer/flush shape) 
// Batch on-device ASR via ONNX Runtime (VLS-007 WS1). FastConformer encoder + TDT decoder run on the
// shared VLS-006 host. Like WhisperCppSttEngine: buffer PCM between flushes, transcribe once per
// utterance, emit final-only. Graphs + SentencePiece vocab resolve as pinned VoxaModelArtifacts.
public sealed class ParakeetSttEngine : ISpeechToTextEngine
{
    public const int RequiredSampleRate = 16000; // 16 kHz mono — a hard model requirement, not a default

    public ParakeetSttEngine(OnnxAsrOptions options, VoxaModelCache cache, IOnnxModelHost host, ILogger? logger = null) { /* … */ }

    public async Task StartAsync(CancellationToken ct)
    {
        // Resolve encoder + decoder_joint + vocab from the catalog (cold-cache download), then load both
        // graphs on the VLS-006 host (process-wide cached sessions, like KokoroTtsEngine). Build the
        // SentencePiece detokenizer + language-token map from the pinned vocab sidecar.
    }

    public ValueTask WriteAudioAsync(ReadOnlyMemory<byte> pcm, CancellationToken ct)
    {
        // Append PCM16→float to the utterance buffer (grow ×2, like Whisper). No inference here.
        return ValueTask.CompletedTask;
    }

    public Task FlushAsync()
    {
        // Speech-end: queue ONE transcription of the buffered utterance off the data loop (chained,
        // serialized — exactly WhisperCppSttEngine.QueueTranscriptionLocked). Results arrive via the channel.
        return Task.CompletedTask;
    }

    public IAsyncEnumerable<TranscriptionResult> ReadTranscriptsAsync(CancellationToken ct) => _transcripts.Reader.ReadAllAsync(ct);

    // transcribe(): mel front-end → encoder Run → TDT greedy decode → detokenize → one final result.
    private async Task TranscribeAndEmitAsync(float[] samples, CancellationToken ct)
    {
        var mel  = ComputeLogMel(samples);                 // 128 mel bins, n_fft 512, hop 160, win 400, pre-emphasis 0.97
        var enc  = _host.Run(_encoder, mel);               // encoded_output[1, T, H], encoded_length
        var text = TdtGreedyDecode(enc);                   // token+duration transducer over encoder frames (blank_id = vocab_size)
        if (text.Length == 0) return;
        await _transcripts.Writer.WriteAsync(
            new TranscriptionResult(text, IsFinal: true, Language: _detectedLanguage), ct).ConfigureAwait(false);
    }
}
The TDT decode and mel front-end are the work. Parakeet's decoder is a Token-and-Duration Transducer: greedy decoding emits a token and a duration (frames to advance) per step, with a blank at vocab_size and a small set of duration bins. Get the front-end constants right (speech-core's reference: 128 mel bins, n_fft 512, hop_length 160, win_length 400, pre-emphasis 0.97, Slaney mel, log-floor) and the transducer loop right (duration-bin advance, blank handling, SentencePiece detokenization), or transcripts are subtly wrong. Validate against speech-core's outputs and known-clip transcripts (§12 verification). Language is read from the model's language token, not configured.

Descriptor + catalog entry (mirror WhisperCpp)

The descriptor is the WhisperCppDescriptors.Stt shape: a VoxaSttDescriptor(Name, ConfigSection, PreferredInputSampleRate, Validate, CreateProcessor) with a keyless Validate (unknown model name, offline cache-miss, device value) and a WarmUpAsync that resolves + loads the graphs on startup so the first caller pays no download. The catalog entry is several pinned files per model (graphs + vocab + config):

src/Voxa.Speech.OnnxAsr/OnnxAsrModelCatalog.cs (WS1 entry — sha256 + size from HuggingFace LFS, never hand-written) 
// <-- Each Sha256 + SizeBytes read from the model repo's LFS pointer at implementation time. -->
// Parakeet TDT v3 (0.6B) — NeMo ONNX export: encoder + decoder_joint + SentencePiece vocab.
["parakeet-tdt-v3"] = new OnnxAsrModel(
    Kind:    OnnxAsrKind.Batch,
    Files:
    [
        Entry("parakeet-tdt-v3/encoder.onnx",        "<sha256>", <bytes>),
        Entry("parakeet-tdt-v3/decoder_joint.onnx",  "<sha256>", <bytes>),
        Entry("parakeet-tdt-v3/vocab.txt",           "<sha256>", <bytes>),
    ]),
Do not fabricate hashes. The authoritative SHA-256 and exact byte length of every file — each ONNX graph, the tokenizer/vocab, the config — must be read at implementation time from the model's HuggingFace LFS pointer (oid sha256:<hash> / size <bytes>), exactly as the VLS-001/VLS-002 whisper entries were sourced. A hand-written or guessed hash is a bug, not a placeholder: the cache will either fetch the wrong file silently or fail verification forever.

6.WS2 — Nemotron streaming engine

Package Voxa.Speech.OnnxAsr Host Voxa.Audio.Onnx (VLS-006) Risk High (cache-aware RNN-T state) Shape streaming — interim + final

Nemotron Speech Streaming (0.6B) is cache-aware streaming RNN-T, exported as three ONNX graphs — encoder (carrying four rolled caches: pre_cache, cache_last_channel, cache_last_time, cache_last_channel_len), decoder (an LSTM with h/c state), and joint. speech-core drives it as a begin_stream / push_chunk / end_stream state machine over fixed mel windows, carrying the encoder caches across windows and running greedy RNN-T over every emitted encoder frame. This is the local streaming engine VRT-004 deferred. It maps onto Voxa's already-streaming contract directly:

speech-core STTInterfaceVoxa ISpeechToTextEngineBehaviour
begin_stream(sr)StartAsync(ct)Load the three graphs on the VLS-006 host; reset per-stream caches/LSTM state.
push_chunk(pcm) → PartialResultWriteAudioAsync(pcm, ct) → interim via ReadTranscriptsAsyncAccumulate PCM, run any full mel windows (cache-aware encoder → greedy RNN-T), and yield TranscriptionResult(IsFinal: false) as text grows.
flush_stream() / end_stream()FlushAsync() / StopAsync()Drain the trailing partial window and emit the authoritative TranscriptionResult(IsFinal: true) for the utterance.
cancel_stream()per-frame CancellationToken + DisposeAsyncAbandon in-flight decode on interruption (the existing cancellation machinery — no new model).
src/Voxa.Speech.OnnxAsr/NemotronStreamingSttEngine.cs (sketch — the begin/push/end loop on Voxa's streaming contract) 
// Cache-aware streaming RNN-T (VLS-007 WS2). One engine == one stream (the speech-core constraint).
// Consumes VRT-004's interim propagation: interim results become TranscriptionFrame(IsFinal:false)
// downstream; TranscriptionFilter + the agent act on finals only (interims are display/turn-signal).
public sealed class NemotronStreamingSttEngine : ISpeechToTextEngine
{
    public ValueTask WriteAudioAsync(ReadOnlyMemory<byte> pcm, CancellationToken ct)
    {
        AppendPcm16(pcm.Span);                 // accumulate float32
        while (HasFullMelWindow())             // e.g. 80 ms window; chunk_samples = (mel_frames-1)*hop_length
        {
            var window = NextMelWindow();
            var text   = RunWindow(window);     // encoder (cache-aware, rolls the 4 caches) → greedy RNN-T over emitted frames
            if (text.Length > 0)
            {
                _accumulated += text;
                // INTERIM — IsFinal:false. Throttle/coalesce so interim churn doesn't flood the pipeline (VRT-004 risk).
                _transcripts.Writer.TryWrite(new TranscriptionResult(_accumulated, IsFinal: false));
            }
        }
        return ValueTask.CompletedTask;
    }

    public Task FlushAsync()  // UserStoppedSpeaking → finalise
    {
        var tail = DrainPartialWindow();
        var final = (_accumulated + tail).Trim();
        if (final.Length > 0) _transcripts.Writer.TryWrite(new TranscriptionResult(final, IsFinal: true));
        ResetStreamState();                    // caches + LSTM h/c primed to blank for the next utterance
        return Task.CompletedTask;
    }
}
~RTF 1.0 on CPU — the partials are the point. Unlike Parakeet (faster-than-real-time batch), the streaming Nemotron runs at roughly real time on CPU; you don't choose it for raw throughput, you choose it for live partial captions on the fully offline path. The cache-aware encoder state (four rolled caches across windows) and the LSTM h/c carry are the correctness-critical, fiddly part — port the window math, the cache shapes, and the blank-priming from speech-core's reference and validate the emitted interims against a known clip.
Interims are display + turn-signal, never a turn. Per VRT-004, only IsFinal: true results drive a turn or reach the agent; TranscriptionFilter and the agent ignore interims (they render as live captions and may feed the smart-turn classifier). The engine must throttle/coalesce interim emission so per-window churn doesn't flood the pipeline's bounded channels.

7.WS3 — Nemotron-3.5 multilingual streaming engine

Package Voxa.Speech.OnnxAsr Host Voxa.Audio.Onnx (VLS-006) Risk High (RNN-T + prompt slot) Shape streaming + multilingual

Nemotron-3.5 ASR Streaming Multilingual (0.6B) is the same cache-aware streaming RNN-T as WS2 — three graphs, four rolled encoder caches, greedy RNN-T over emitted frames, caches carried across fixed 320 ms windows — with two additions: multilingual coverage (a much larger vocabulary) and prompt-conditioning. Unlike Parakeet, language is not auto-detected: the encoder takes a one-hot language_mask selecting a prompt slot, chosen from a languages.json sidecar (locale → slot; default en-US). So WS3 reuses WS2's engine body and adds a language-prompt selection step.

src/Voxa.Speech.OnnxAsr/NemotronMultilingualSttEngine.cs (sketch — WS2 streaming body + prompt slot) 
// Streaming multilingual RNN-T (VLS-007 WS3). Same begin/push/end loop as WS2, plus a prompt slot.
// The language hint maps from TranscriptionResult.Language semantics / a config prompt key.
public sealed class NemotronMultilingualSttEngine : ISpeechToTextEngine
{
    public async Task StartAsync(CancellationToken ct)
    {
        // Load encoder + decoder + joint + vocab + languages.json on the VLS-006 host.
        // Resolve the prompt slot from OnnxAsrOptions.Language (locale key → slot; unknown → "auto"/slot 0).
        _languageMask = OneHotPromptSlot(ResolveSlot(_options.Language)); // fed to the encoder each window
    }

    // WriteAudioAsync / FlushAsync identical to WS2, but each encoder Run also passes _languageMask.
    // Emitted finals carry the resolved locale as TranscriptionResult.Language so it flows to TranscriptionFrame.
}

The language hint is sourced the same config-capture way as the rest of OnnxAsrOptions — a Voxa:OnnxAsr:Language (or a more explicit prompt key) bound from the captured "Voxa" root section, never service-located. A resolved final result sets TranscriptionResult.Language so the locale propagates into TranscriptionFrame.Language downstream.

ONNX-FP16 variant. Nemotron-3.5 publishes an FP16 ONNX export (smaller, faster on capable hardware). Treat each precision/repo variant as a distinct pinned catalog entry with its own SHA-256 — the FP16 graphs are different files than FP32 and must be pinned separately (never assume one hash covers both). The default catalog entry chooses one precision explicitly; an alternate precision is a second named entry, not a runtime flag.
Larger vocab, same decode-fidelity bar. The multilingual vocabulary is an order of magnitude larger than the English Nemotron's; the SentencePiece detokenization, the language_mask shape, and the per-locale slot mapping must all be ported faithfully and validated against speech-core's multilingual reference outputs. Get the prompt slot wrong and you transcribe the right audio in the wrong language model.

8.WS4 — Licensing & model catalog

Concern license closure + pinned catalog Risk High (weight redistribution) Gate mirror the Kokoro license-closure test
Verify and record every model's license before pinning it. NeMo / Parakeet / Nemotron weights are typically CC-BY-4.0, but the exact terms of the specific ONNX export you pin must be confirmed and recorded at implementation time — license, attribution requirement, and redistribution terms per repo. If any model's weights are not cleanly redistributable, do not ship that model: either resolve it from the upstream repo at first run with attribution recorded, or drop that family from the catalog and note it here. This is the same discipline the Kokoro/GPL boundary enforces, and the same rule VLS-004 follows for DeepFilterNet3.

License-closure gate (mirror the Kokoro one)

The Kokoro package enforces its boundary mechanically with a DependencyGraphGateTests pair — one test asserts the engine assembly references no forbidden (GPL/AGPL) assembly, another scans the build output for forbidden natives. Voxa.Speech.OnnxAsr ships the equivalent so a license regression fails CI rather than shipping; CC-BY attribution obligations for the pinned weights are recorded alongside the catalog (a NOTICE/attribution string per model).

Binding follows the WhisperCppOptions.FromConfiguration precedent exactly: OnnxAsrOptions reads the Voxa:OnnxAsr section off the captured "Voxa" root passed to AddVoxa — never GetRequiredService<IConfiguration>(), so the plain-ServiceCollection hosts (Studio, tests, CLI) resolve. 10.Plan of action — phases & dependencies

  1. Phase A — Parakeet batch engine (proves the host + descriptor). Land Voxa.Speech.OnnxAsr with ParakeetSttEngine (batch), the mel front-end + TDT decode, OnnxAsrModelCatalog (the real Parakeet hashes), OnnxAsrOptions/OnnxAsrDescriptors (the WhisperCpp shape), the license-closure gate, and the offline + LocalModels tests. Batch is the simplest decode path, so it validates the VLS-006 host integration and the descriptor wiring before any streaming complexity. Depends on VLS-006.
  2. Phase B — Nemotron streaming engine (on VRT-004). Add NemotronStreamingSttEngine (the begin/push/end loop, cache-aware RNN-T, interim emission) + its pinned catalog entry + the interim-then-final tests. Depends on Phase A (the package + host integration) and VRT-004 (interim propagation).
  3. Phase C — Nemotron-3.5 multilingual streaming. Add NemotronMultilingualSttEngine (Phase B's body + the language-prompt slot), the multilingual catalog entry (+ FP16 variant as a separate entry), and multilingual round-trip tests. Depends on Phase B.

One logical change per PR (CONTRIBUTING) — A, B, C are at least three PRs. Each adds a [Unreleased] CHANGELOG entry and updates docs/local-speech.md. This spec is authored before any of them lands — the program's spec-before-code rule.

14.Out of scope / deferred

  • LiteRT (TFLite) backends. speech-core ships a LiteRT Nemotron twin; Voxa standardizes on ONNX Runtime via VLS-006. If a TFLite path is ever wanted it is a separate host + item, not this one.
  • On-device fine-tuning / adaptation. Inference only; training/adapting these models is out of scope.
  • Non-ASR Nemotron variants and other non-ASR ONNX models — DeepFilterNet3 (VLS-004), Sidon (VLS-008), Pyannote/WeSpeaker diarization (VLS-005) are separate consumers of the same VLS-006 host.
  • The shared ONNX host itself is VLS-006 (Voxa.Audio.Onnx) — this item consumes it; it does not build or modify it.
  • Streaming-engine GPU tuning beyond the VLS-006/VLS-002 device seam (e.g. CUDA encoder + CPU decoder split, as speech-core does) — a follow-up optimization once the CPU path is correct.
  • A batch Nemotron path. speech-core exposes transcribe() on the streaming models as a begin/push/end convenience; Voxa's streaming engines finalise on FlushAsync, which covers the same need — no separate batch wrapper.

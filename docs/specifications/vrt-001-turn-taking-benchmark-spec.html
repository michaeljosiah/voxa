VRT-001 — Turn-Taking Quality Benchmark (Full-Duplex-Bench Harness)

A new bench/Voxa.TurnTaking harness — peer to bench/Voxa.Benchmarks — that drives the composed voice pipeline through a turn-taking corpus and turns "does this change actually improve the conversation?" into an answerable, regression-gated number. It walks the Full-Duplex-Bench v1.0 corpus layout (arXiv 2503.04721, ASRU 2025), runs each sample through DefaultVoicePipelineComposer.Compose with mock or local engines, emits a per-sample JSON record + a response WAV, rolls those up to per-category latency percentiles, and scores the three cascade-fair categories against a checked-in baseline.json. It is the measurement foundation the rest of the VRT line is gated against — and it reuses the existing diagnostics timing substrate rather than inventing new instrumentation.

1.What is this? — a plain-English explainer

What does VRT-001 add, in one sentence?

A benchmark harness that plays scripted conversations into Voxa's real pipeline and measures how well it takes turns — when the bot starts talking after you pause, whether it handles a clean hand-off, and whether it stops when you interrupt — emitting one comparable number per category that a regression gate can guard.

Why now?

The next four VRT items all claim to improve conversation: smart-turn already shipped (PR #26), eager STT is next (VRT-002), barge-in/AEC after that (VRT-003), streaming captions (VRT-004). But Voxa today has only latency micro-benchmarks (bench/Voxa.Benchmarks) and a per-turn latency waterfall — no behavioural turn-taking benchmark. You cannot tell whether any of those changes actually help a conversation without one. So the program opens here: measure first, then every turn-taking change after week one gets an objective before/after verdict.

What does this not change?

Nothing in the runtime. This is a benchmark in bench/ — a console app plus a small library. It is not a NuGet package, it is not in Voxa.Core, and it adds no new instrumentation: it observes the existing VoxaDiagnosticsHub / StageLatencyTracker stage timings and the voxa.turn.ttfb metric the pipeline already produces. It drives the same DefaultVoicePipelineComposer.Compose path a real server or Voxa Studio drives — no parallel pipeline.

How does it stay hermetic and offline in the default lane?

The default run uses mock STT / LLM / TTS engines and a tiny checked-in fixture (a handful of samples). It downloads nothing, hits no network, and is deterministic — so it can be a fast xUnit smoke test in the normal CI lane. The real run (local Whisper / Kokoro engines, the full corpus) is gated behind [Trait("Category","LocalModels")], exactly like the rest of the speech tier.

2.Executive summary

VRT-001 ports speech-core's Full-Duplex-Bench workflow onto Voxa's frame/processor pipeline. speech-core integrates FDB v1.0 end-to-end — driver → per-sample JSON → summary CSV → category scoring → checked-in baseline → CI gate — and Voxa is a cascade, so the same three cascade-fair categories apply. The work is four loosely-sequential workstreams in a single new bench/Voxa.TurnTaking project:

  • WS1 — The driver new harness. A console app + library that walks the FDB v1.0 corpus layout, composes the pipeline via DefaultVoicePipelineComposer.Compose(IServiceProvider), feeds each sample's input WAV through it, subscribes to the diagnostics hub for stage timings, and writes <category>__<id>.json (per-stage + voice-to-voice timings) plus the response WAV.
  • WS2 — Engines (mock + local) P0. Mock engines make the default lane offline and deterministic; trait-gated local engines (Whisper/Kokoro) make the run real. Selected by CLI flag; mock is the default so the harness builds and runs with no models present.
  • WS3 — Summary roll-up new harness. Aggregate the per-sample JSON into a per-category p50/p90/p99 CSV (the same shape as bench/BASELINE.md's tables).
  • WS4 — Scoring + baseline gate P0. A direction-aware scorer over the three cascade-fair categories (turn-offset-rate + first-word latency) and a checked-in baseline.json the smoke test diffs against, so a regression in turn-taking quality fails the build.
Net effect. "Did eager STT / barge-in / streaming actually improve the conversation?" becomes a single command (dotnet run --project bench/Voxa.TurnTaking) producing a per-category score, and the mini-fixture smoke run is a CI gate that catches turn-taking regressions before they merge — without ever touching the network in the default lane.

3.Goals, non-goals & budgets

Goals

  • Make "does this improve turn-taking?" an answerable, regression-gated number — the prerequisite for VRT-002 / VRT-003 / VRT-004.
  • Drive the real composed pipeline (DefaultVoicePipelineComposer.Compose), not a bespoke benchmark pipeline, so the number reflects what ships.
  • Reuse the existing diagnostics timing substrate (StageLatencyTracker stages, voxa.turn.ttfb) — add no new instrumentation to the runtime.
  • Stay offline + deterministic in the default lane (mock engines, tiny checked-in fixture); real engines and the full corpus behind Category=LocalModels.
  • Emit FDB-shaped artifacts: per-sample JSON + response WAV → per-category percentile CSV → category score → baseline.json regression gate.

Non-goals (this spec)

  • The backchannel category — architecturally N/A for a cascade (see §5); logged as skipped, never scored.
  • Running the full corpus in CI — the gate runs on a mini fixture; the full run is manual / LocalModels.
  • An LLM-judge scorer — opt-in, a later follow-up (§13).
  • Changing any pipeline behaviour. VRT-001 only measures; the improvements are VRT-002+.
  • New instrumentation in Voxa.Core — the harness consumes the hub, it does not extend it.

Budgets

4.Architecture context — where this plugs in

VRT-001 is a consumer of three things Voxa already ships. It introduces no new seam — its entire job is to wire them together into an FDB workflow:

Existing surfaceHow VRT-001 uses it
DefaultVoicePipelineComposer.Compose(IServiceProvider)The transport-agnostic composition that returns a ComposedVoice (processor factories + InputSampleRate/OutputSampleRate). The harness builds a per-sample service scope, calls Compose, links the parts into a runnable pipeline, and feeds the sample audio in. Same path a WebSocket connection or Voxa Studio uses.
VoxaDiagnosticsHub + StageLatencyTrackerThe timing substrate. The harness registers a hub per session (scoped, same as a server connection), sets Voxa:Diagnostics:Enabled=true, and SubscribeAsynces to collect the derived StageLatencyEvents (vad_close, stt_final, agent_first_token, tts_first_byte) and the TurnEvent edges — no new instrumentation.
voxa.turn.ttfb / voxa.stage.latency (VoxaMetrics)The voice-to-voice and per-stage histograms are the canonical definition of "latency". The harness derives the same numbers per sample from the hub events so the JSON timings line up with what production reports.
Voxa.Testing — WavFile, WavFileSourceProcessor, WavFileSinkProcessorRead each sample's input WAV; feed it as AudioRawFrames (chunked, default 20 ms) followed by an EndFrame; capture the synthesized response as a WAV. Reused verbatim — no new audio I/O.
Invariant — drive the composed pipeline, observe the existing substrate. VRT-001 must not assemble its own VAD→STT→agent→TTS chain or add a private latency timer. It calls Compose and reads the diagnostics hub. If the harness and production disagree on a number, the harness is wrong by construction — which is the point: the benchmark measures what ships.
Plain ServiceCollection, not the Generic Host. Like Voxa Studio and the tests, the harness uses a plain ServiceCollection + AddVoxa so hosted services (eager model warm-up) never run and nothing touches the network before a sample asks for it. Per the config-capture rule, any provider that needs configuration captures the IConfiguration passed to AddVoxa — the harness builds that configuration in-memory (the mock provider names + a diagnostics flag).

5.The corpus & categories

Corpus Full-Duplex-Bench v1.0 Source arXiv 2503.04721, ASRU 2025 Layout per-category sample dirs Default fixture mini (checked in)

The harness targets the Full-Duplex-Bench v1.0 corpus (arXiv 2503.04721, ASRU 2025) — the same corpus speech-core integrates. FDB groups samples by turn-taking behaviour: each category is a folder of samples, each sample an input audio clip (the user side of a conversation, with a scripted pause/hand-off/interruption) plus reference metadata. The harness walks that layout: <corpus-dir>/<category>/<sample-id>/….

The three cascade-fair categories

FDB defines four behaviours; only three are meaningful for a cascade (STT→LLM→TTS, half-duplex today), and Voxa scores exactly those three:

CategoryWhat it probesDirection (good = …)
pause_handlingThe user pauses mid-thought (a within-turn silence) but is not done. A good system waits and does not barge in on the gap. Measures premature turn-taking.lower turn-offset-rate — fewer/later bot starts during the pause is better
smooth_turn_takingThe user finishes a turn cleanly. A good system responds promptly after end-of-turn — not too slow (dead air), not too eager.higher responsiveness — a faster, correctly-placed first word is better
user_interruptionThe user barges in while the bot is (notionally) speaking. A good system yields quickly. Measures interruption responsiveness.higher responsiveness — a faster yield / first reaction is better
Do not fake backchannel. FDB's fourth category, backchannel (the system emitting "mm-hm" / "right" while the user keeps the floor), requires a full-duplex model that talks and listens at once. A cascade structurally cannot do this — it is half-duplex by design (mic gated during playback). The harness must discover the category, log it as skipped with a one-line reason, and emit no score for it. Never synthesize a placeholder number, never run it through the cascade and pretend the result means something. A fabricated backchannel score is a lie in the report, not a TODO.
Do not fabricate a corpus. FDB ships via Google Drive, not a single tarball, and its license governs redistribution. Ship a tiny, real, checked-in fixture — a handful of genuinely short WAVs across the three categories (the fdb_mini analogue) for the offline smoke gate — and document the full-corpus fetch (the Drive link + the expected on-disk layout) in the harness README and --help. Do not generate synthetic "sample" audio and call it the corpus; the smoke fixture exists to exercise the plumbing, and its provenance must be real audio with a recorded source.

Per-sample loop

For each sample under the selected categories:

  1. Build a per-sample ServiceCollection + AddVoxa(configuration) with the chosen STT/LLM/TTS provider names (mock by default) and Voxa:Diagnostics:Enabled=true; register a fresh VoxaDiagnosticsHub in the scope.
  2. Call composer.Compose(scope.ServiceProvider) → a ComposedVoice; materialise its Parts into a pipeline with a WavFileSourceProcessor (the sample's input WAV) at the head and a WavFileSinkProcessor (the response WAV) at the tail.
  3. Start a SubscribeAsync reader on the hub before the run; collect TurnEvents and the derived StageLatencyEvents into a per-sample list.
  4. Run the pipeline to completion (the WavFileSourceProcessor emits the audio then an EndFrame; wait for the sink to drain), under a wall-clock cap so a wedged sample can't hang the run.
  5. Reduce the collected events to the timings (below), write <out-dir>/<category>__<sample-id>.json and the response WAV. On any exception, write the record with error populated and null timings — never drop a failed sample silently; a missing record is indistinguishable from a pass.

Per-sample JSON record schema

One file per sample, named <category>__<sample-id>.json (the FDB convention). All timings are milliseconds, derived from the hub's stage events; null when the stage didn't occur (e.g. the bot correctly stayed silent through a pause_handling gap):

<out-dir>/pause_handling__sample_0007.json 
{
  "sample_id":   "sample_0007",
  "category":    "pause_handling",
  "engines":     { "stt": "mock", "llm": "mock", "tts": "mock" },   // provenance of the run
  "timings_ms": {
    "stt":                            142.0,   // stt_final stage (gate close → final transcript)
    "llm":                             88.0,    // agent_first_token (final transcript → first delta)
    "tts":                             61.0,    // tts_first_byte (first delta → first synth audio)
    "ttft_first_audio_from_speech_end": 304.0,  // the voice-to-voice number == voxa.turn.ttfb
    "total_wall":                     1290.0    // sample start → pipeline drained
  },
  "transcripts": {
    "reference":  "so the thing is ... uh ... I wanted to ask",   // from the sample metadata, if present
    "hypothesis": "so the thing is I wanted to ask"               // the STT final transcript
  },
  "response_wav": "pause_handling__sample_0007.response.wav",
  "error":        null    // a string (message + stage) when the sample failed; timings then null
}
Timing provenance. stt / llm / tts map one-to-one onto the StageLatencyTracker stages stt_final / agent_first_token / tts_first_byte; ttft_first_audio_from_speech_end is the UserStopped-edge → first TtsChunkEvent delta, i.e. the same quantity voxa.turn.ttfb records. The harness does not time anything itself — it reduces the events the pipeline already publishes, so the JSON and the production histograms agree by construction.

7.Summary & scoring

WS3 — summary roll-up (per-category percentiles)

A summarize step reads every per-sample JSON in the out-dir and emits one CSV row per category with the p50/p90/p99 of each timing — the same percentile shape bench/BASELINE.md uses for the micro-benchmarks, so the two read alike. Failed samples (non-null error) are counted but excluded from the percentiles, and the count is reported:

<out-dir>/summary.csv 
category,n,errors,ttfb_p50,ttfb_p90,ttfb_p99,stt_p50,llm_p50,tts_p50,total_wall_p50
pause_handling,40,0,,,,142.0,88.0,61.0,1290.0        # ttfb blank when the bot correctly stayed silent
smooth_turn_taking,52,1,331.0,512.0,690.0,150.0,95.0,64.0,1402.0
user_interruption,33,0,288.0,470.0,640.0,,,,612.0
backchannel,0,0,,,,,,,                                # discovered, skipped — N/A for a cascade (§5)

WS4 — the scorer (direction-aware, per category)

A score step turns the per-category aggregates into the FDB-style metrics: a turn-offset-rate (TOR) and a first-word latency, interpreted per category. The crucial detail speech-core gets right and a naive scorer gets wrong is direction: the same raw quantity is good in one category and bad in another.

CategoryMetricBetter isRationale
pause_handlingTOR (fraction of samples where the bot started speaking during the within-turn pause)lowerBarging in on a thinking-pause is the failure; a perfect score is "never spoke during the gap".
smooth_turn_takingfirst-word latency after end-of-turn (the ttfb percentile)higher responsiveness (lower ms)After a clean hand-off the bot should respond; faster-and-correctly-placed wins.
user_interruptionresponsiveness to the barge-in (yield / first-reaction latency)higher responsiveness (lower ms)The bot should yield quickly; a faster reaction to the interruption wins.
backchannel——skipped; no score emitted (§5).
Why "responsiveness", not raw latency. For smooth_turn_taking and user_interruption the harness reports a direction-normalised score so "higher is better" holds across the report (e.g. a bounded inverse of the latency percentile), while keeping the raw ms in the CSV. pause_handling stays a rate where lower is better. This mirrors FDB's per-category interpretation and keeps the baseline diff legible: every scored category moves in one obvious direction.

The checked-in baseline gate

A baseline.json (per category: the TOR / latency the current pipeline achieves on the mini fixture) is committed to the harness directory. The smoke test (§11) runs the mock-engine harness over the fixture, scores it, and diffs against the baseline within a tolerance — a regression in turn-taking quality fails the build. Refreshing the baseline is a deliberate code change with a justification in the PR (a knob moved, a category improved), never a silent overwrite — the same discipline as bumping a model SHA-pin.

bench/Voxa.TurnTaking/baseline.json (mini-fixture baseline — illustrative numbers) 
{
  "fixture":   "mini",
  "engines":   "mock",
  "tolerance": { "tor": 0.05, "latency_ms_pct": 0.20 },   // regression beyond this fails the gate
  "categories": {
    "pause_handling":     { "tor": 0.00 },                // lower-better: must not regress upward
    "smooth_turn_taking": { "ttfb_p50_ms": 330.0 },       // must not regress slower beyond tolerance
    "user_interruption":  { "ttfb_p50_ms": 290.0 },
    "backchannel":        "skipped"
  }
}

8.Configuration / CLI reference

The console exposes a small flag set. Mock engines are the default for every engine slot so the harness builds and runs with no models and no network — the offline smoke path is the path of least resistance.

FlagDefaultMeaning
--corpus-dir <path>the checked-in mini fixtureRoot of the FDB-layout corpus (<category>/<id>/…). Omit to use the bundled mini fixture.
--out-dir <path>./fdb-outWhere per-sample JSON, response WAVs, summary.csv, and the score land.
--category <name>all threeRestrict to one cascade-fair category (pause_handling | smooth_turn_taking | user_interruption). backchannel is always skipped.
--limit <n>(none)Cap samples per category — for a fast smoke pass over a large corpus.
--stt <name>mockSTT provider name passed to AddVoxa (mock | a real provider, e.g. WhisperCpp in the LocalModels lane).
--tts <name>mockTTS provider name (mock | e.g. Kokoro).
--llm <name>mockAgent backing (mock deterministic responder | a real IChatClient provider).
examples 
# Offline smoke — mock engines, bundled mini fixture (what CI runs):
dotnet run -c Release --project bench/Voxa.TurnTaking

# One category, real local engines, the full fetched corpus (LocalModels lane):
dotnet run -c Release --project bench/Voxa.TurnTaking -- \
  --corpus-dir ./full-duplex-bench --category smooth_turn_taking \
  --stt WhisperCpp --tts Kokoro --llm openai --out-dir ./fdb-full
Mock defaults are load-bearing. If a real engine were the default, dotnet run (and the smoke test) would download models / hit the network on a clean checkout — breaking the offline default lane. Mock is the default precisely so the hermetic path is the zero-argument path; real engines are an explicit opt-in.

9.Plan of action — phases

Sequenced to get an end-to-end signal as early as possible, mirroring speech-core's M1–M4 progression (driver → real engines → summary → scoring/gate). Each phase is independently reviewable.

  1. Phase A — Mock-driver smoke. The project, the corpus walk, the per-sample loop over Compose with mock engines, the per-sample JSON + response WAV, and the bundled mini fixture. End state: dotnet run produces JSON for the three categories offline, and an xUnit smoke test asserts the records exist and parse. This is the spine — everything else hangs off it.
  2. Phase B — Real local engines. Wire --stt/--tts/--llm to the real providers so a LocalModels-gated run over the fetched corpus produces real transcripts and real timings. No new harness structure — just provider selection + the fetch doc.
  3. Phase C — Summary roll-up. The summarize step → per-category p50/p90/p99 summary.csv, error counts, backchannel logged-skipped.
  4. Phase D — Scoring + baseline gate. The direction-aware score step, the checked-in baseline.json, and the smoke test's baseline diff. End state: a turn-taking regression on the mini fixture fails the build — the gate VRT-002+ are measured against.

One logical change per PR (per CONTRIBUTING): Phases A–D are separate PRs, each with an [Unreleased] CHANGELOG entry. This spec lands before Phase A's code (spec-before-code).

13.Out of scope / deferred

  • The backchannel category. Architecturally N/A for a half-duplex cascade — logged skipped, never scored (§5). Revisit only if/when the local full-duplex speech-to-speech seam (VRT-005) gives Voxa a model that can talk and listen at once.
  • Full corpus in CI. The gate runs the mini fixture; the full FDB run is manual / LocalModels by design — corpus size and engine cost don't belong in the inner loop.
  • LLM-judge scoring. An opt-in semantic judge (did the bot's content make sense, not just its timing) is a later follow-up; VRT-001 scores turn-taking mechanics (TOR + latency) only.
  • Changing pipeline behaviour. VRT-001 is measurement only. The improvements it gates — eager STT (VRT-002), AEC / barge-in robustness (VRT-003), streaming interim captions (VRT-004) — are separate items, each required to cite a VRT-001 before/after delta. See docs/speech-core-parity-program.md.

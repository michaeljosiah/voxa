1.What is this? — a plain-English explainer

What does VLS-004 add, in one sentence?

An optional "clean up the mic" step that runs a small neural denoiser over the incoming audio before voice-activity detection and speech-to-text, so a noisy fan, a busy café, or a reverberant room doesn't wreck the transcript — without changing anything for users who don't turn it on.

Why now?

The local / edge tier is exactly where Voxa has been investing (whisper.cpp, Piper, Kokoro all run on-device), and it's exactly where there is no enhancement at all. Today the only signal conditioning is an RMS floor in the VAD — a crude gate, not noise reduction. Cloud STT vendors denoise server-side; the on-device path is on its own. speech-core wires DeepFilterNet3 as an EnhancerInterface for this; Voxa already runs ONNX models in-process (Silero VAD, Kokoro), so the same move ports cleanly.

Does this slow everyone down or change the default pipeline?

No. The seam ships with a null/passthrough default, so with enhancement off the composed pipeline is byte-identical to today (golden-tested). Turning it on adds a per-frame denoise pass — real CPU, real latency — which is why it is opt-in and its cost shows up directly in Studio's latency waterfall. You pay for it only when you ask for it.

What is explicitly not changing?

The frame contract (AudioRawFrame in, same length / sample-rate / channel layout out), the VAD and STT engines, the model cache, the offline/air-gap behaviour, and the descriptor wiring. The enhancer is a new stage inserted into the existing chain, not a rewrite of any existing stage. And Voxa.Core stays zero-dependency — the seam and the ONNX engine live in Voxa.Audio.* packages.

2.Executive summary

VLS-004 is two workstreams that mirror the rest of the Speech-Core Parity program's "define the seam, ship a default, defer/opt-in the heavy model" discipline:

  • WS1 — the IAudioEnhancer seam additive. A vendor-neutral interface (Enhance(pcm) → pcm, input-sample-rate contract, Reset) in a new Voxa.Audio.Abstractions, an AudioEnhancerProcessor that runs it per AudioRawFrame, a null/passthrough default, and a descriptor + registry slot (TryGetEnhancer) so an implementation drops in by config. The composer inserts the stage before the VAD, after the AEC stage (VRT-003). Off ⇒ byte-identical.
  • WS2 — the reference ONNX engine P1. A DeepFilterNet3 (license to be verified — see §6) denoiser in a new Voxa.Audio.Enhance package, running in-process on ONNX Runtime exactly the way SileroVadEngine and KokoroTtsEngine already do, with the model file SHA-256-pinned in a catalog and resolved through VoxaModelCache. Opt-in; the default reference set and pack output are unchanged.
Net effect. A developer running the local STT tier in a noisy room can set "Enhance": { "Engine": "DeepFilterNet3" } and recover transcription accuracy that a fan or a reverberant office was eating. Clean-room and cloud-STT users set nothing, get nothing extra, and CI stays offline and deterministic.

3.Goals, non-goals & budgets

Goals

  • Give Voxa a first-class, vendor-neutral enhancement seam (IAudioEnhancer) the same way VRT-003 gives it an AEC seam — interface + null default + chain wiring, no DSP/model commitment in the seam itself.
  • Ship one working reference ONNX denoiser (DeepFilterNet3, pending license closure) in an opt-in Voxa.Audio.Enhance package, SHA-256-pinned like every other artifact.
  • Keep the default composed pipeline byte-identical to today (passthrough default, golden-tested) and the default reference set unchanged (no new bundled natives beyond what the opt-in package pulls in).
  • Make the added per-frame cost visible — it lands as a stage in the diagnostics latency waterfall, measured against VRT-001's timings.

Non-goals (this spec)

  • Sidon-style offline restoration of voice-clone reference audio — that belongs with VVL-002 / Studio artifacts, not the live mic path. See §12.
  • GPU enhancement — if a model needs it, it rides the VLS-002 ONNX-Runtime/Whisper.net GPU patterns; not re-specced here.
  • Acoustic echo cancellation — that is VRT-003 (IEchoCanceller), the stage immediately upstream of this one. Enhancement cleans noise, not the bot's own voice fed back through the speakers.
  • Replacing the VAD's RMS floor or its pre-roll (VLS-001) — both stay; enhancement runs ahead of them.

4.Architecture context — where this plugs in

The full on-device capture chain, with the two Speech-Core Parity audio stages added, is:

mic → [AEC (VRT-003)] → [enhance (VLS-004)] → VAD → STT → TranscriptionFilter → agent → …

Both bracketed stages are opt-in and null-by-default. DefaultVoicePipelineComposer.Compose(IServiceProvider) is the single insertion point (the program doc names it the one place the AEC and enhancement stages go). In the current tree the VAD is composed first (step 1 — the o.Vad.Engine branch) and STT second; this spec inserts the enhancer immediately before that VAD block, after the AEC block VRT-003 introduces. It runs on raw AudioRawFrames, which is exactly what the VAD and SpeechToTextProcessor consume next — so order is: AEC removes the bot's echo, enhancement removes ambient noise, then a cleaner signal reaches detection and recognition.

Why before the VAD, not after. Denoising ahead of detection means the VAD's probability and RMS floor judge the cleaned signal — fewer noise-triggered false starts and fewer missed quiet onsets — and STT transcribes cleaned audio. Placing it after the VAD would denoise only what the VAD already (mis)gated. speech-core makes the same ordering choice (mic → AEC → enhance → VAD → STT).

5.WS1 — the IAudioEnhancer seam

Package Voxa.Audio.Abstractions (seam) + composer wiring Risk Low Network none Default NullAudioEnhancer (passthrough)

The seam is deliberately the same shape as VRT-003's IEchoCanceller and Voxa's existing engine interfaces: a small, single-thread-per-session contract that an ONNX or DSP backend can satisfy. It lives in Voxa.Audio.Abstractions (a new audio-abstractions package, or the one VRT-003 lands for IEchoCanceller — they should share it) so no provider package, and certainly not Voxa.Core, takes a dependency on a model runtime.

src/Voxa.Audio.Abstractions/IAudioEnhancer.cs (seam sketch) 
// Vendor-neutral spectral-enhancement seam (VLS-004). Mirrors speech-core's EnhancerInterface;
// satisfied by an in-process ONNX engine (Voxa.Audio.Enhance) or any future DSP. Per-session,
// single-threaded — the processor's data loop calls Enhance from one thread, like the VAD engine.
public interface IAudioEnhancer : IDisposable
{
    // The sample rate the model/DSP expects. The processor asserts inbound AudioRawFrame.SampleRate
    // matches this and throws a clear error at session start on a mismatch (no silent resample).
    int SampleRate { get; }

    // Clean one chunk of 16-bit mono PCM. MUST return the SAME byte length / sample rate / channel
    // layout it was given — this is signal conditioning, not a format change. May be called many
    // times per second on the audio hot path; keep it allocation-light (reuse buffers like Silero).
    ReadOnlyMemory<byte> Enhance(ReadOnlyMemory<byte> pcm);

    // Drop any internal/recurrent state between sessions (cf. SileroVadEngine.Reset()).
    void Reset();
}

// The default. Off == today: returns the buffer unchanged, holds no state. The composer adds
// NO processor at all when Engine=None, so this exists mainly for tests and explicit wiring.
public sealed class NullAudioEnhancer : IAudioEnhancer
{
    public int SampleRate { get; }
    public NullAudioEnhancer(int sampleRate) => SampleRate = sampleRate;
    public ReadOnlyMemory<byte> Enhance(ReadOnlyMemory<byte> pcm) => pcm;
    public void Reset() { }
    public void Dispose() { }
}

The processor is a thin FrameProcessor: it intercepts AudioRawFrame, replaces the PCM with the enhanced PCM (same envelope), and forwards every other frame downstream unchanged — including StartFrame/EndFrame, per the cross-cutting invariant (or the sink never completes). Long-running work honours the per-frame CancellationToken; enhancement is interruptible like any data-path stage.

src/Voxa.Audio.Abstractions/AudioEnhancerProcessor.cs (sketch) 
protected override async ValueTask ProcessFrameAsync(Frame frame, CancellationToken ct)
{
    if (frame is AudioRawFrame audio)
    {
        // Same length / rate / channels out — verified by the seam contract.
        var cleaned = _enhancer.Enhance(audio.Pcm);
        await PushFrameAsync(audio with { Pcm = cleaned }, ct).ConfigureAwait(false);
        return;
    }
    await PushFrameAsync(frame, ct).ConfigureAwait(false); // forward control + everything else
}

Registration follows the VoxaVadDescriptor pattern exactly — a self-describing record with a config-keyless Validate and a CreateProcessor(sp, root) factory — and the registry grows a TryGetEnhancer twin of TryGetVad. The composer's insertion is gated on the config name:

src/Voxa.AspNetCore/DefaultVoicePipelineComposer.cs (insertion — before the VAD block, after VRT-003's AEC block) 
// 0.6  Enhancement (VLS-004) — after AEC (VRT-003, step 0.5), before VAD (step 1).
// Engine=None (default) adds NOTHING, so the pipeline stays byte-identical to pre-VLS-004.
if (!string.Equals(o.Enhance.Engine, "None", StringComparison.OrdinalIgnoreCase)
    && _registry.TryGetEnhancer(o.Enhance.Engine, out var enhDesc))
{
    parts.Add(sp => enhDesc.CreateProcessor(sp, root)); // runs at inputSampleRate, like the VAD
}
// (an unknown non-None name fails fast in the options validator, like every other provider name)
Config-capture rule. The descriptor factory and WarmUpAsync receive the captured "Voxa" root section (the same one the composer already threads to STT/TTS) and must never GetRequiredService<IConfiguration>() — that breaks the plain-ServiceCollection hosts (Studio, tests, CLI). This is the identical rule the Kokoro descriptor follows.

6.WS2 — the reference ONNX implementation

Package Voxa.Audio.Enhance (new, opt-in) Risk Medium (model license + ORT coexistence) Runtime ONNX Runtime, in-process Model SHA-256-pinned via VoxaModelCache

The reference engine is DeepFilterNet3, run in-process on ONNX Runtime — the precedent is already in the tree twice. SileroVadEngine runs a small ONNX model with reused OrtValues and a single-thread session; KokoroTtsEngine runs a larger ONNX model resolved from the pinned catalog through VoxaModelCache, caching the InferenceSession process-wide. DeepFilterNetEnhancer combines both: catalog-resolved weights, a process-wide cached session, allocation-light per-frame inference.

Verify the license before writing a single line. This spec assumes DeepFilterNet3, but its weight/license terms must be confirmed permissive at implementation time (the DeepFilterNet code is dual MIT/Apache-2.0, but pre-trained-weight redistribution terms are the gating question — confirm the exact model artifact you pin is redistributable). If the weights are not cleanly permissive, do not ship them: substitute a permissively-licensed alternative — an MIT/Apache RNNoise or an NSNet2 ONNX export — and update this section. The chosen model and its license must pass the same license-closure gate test the Kokoro package already enforces (DependencyGraphGateTests: assembly-reference scan + output-file scan; see below).

In-process ONNX engine

The engine mirrors the existing two ONNX engines so it inherits their hard-won behaviour (process-wide weight sharing, off-the-data-loop inference, deterministic teardown):

src/Voxa.Audio.Enhance/DeepFilterNetEnhancer.cs (sketch — pin exact I/O names to the real ONNX export) 
// In-process denoise on ONNX Runtime. Weights load once per process and are shared by every
// connection — same ConcurrentDictionary<string, Lazy<...>> session cache as KokoroTtsEngine.
public sealed class DeepFilterNetEnhancer : IAudioEnhancer
{
    private static readonly ConcurrentDictionary<string, Lazy<InferenceSession>> Sessions = new();

    public int SampleRate { get; } = 48000; // DeepFilterNet3 operates at 48 kHz — see the rate note

    public DeepFilterNetEnhancer(EnhanceOptions options, VoxaModelCache cache, ILogger? logger = null) { /* … */ }

    // StartAsync-equivalent: resolve the pinned model (downloads on cold cache), build/fetch the
    // shared InferenceSession. Same shape as KokoroTtsEngine.StartAsync.
    public async Task StartAsync(CancellationToken ct) { /* cache.ResolveAsync(model, ct); GetOrAdd(...) */ }

    public ReadOnlyMemory<byte> Enhance(ReadOnlyMemory<byte> pcm)
    {
        // PCM16 → float, run the model, float → PCM16. SAME sample count out as in.
        // Reuse buffers on the hot path (cf. SileroVadEngine's pinned OrtValues).
    }

    public void Reset() { /* clear any recurrent/overlap state between sessions */ }
    public void Dispose() { /* session stays in the process-wide cache, like Kokoro */ }
}
Sample-rate reality. DeepFilterNet3 runs at 48 kHz, while the local STT tier (whisper.cpp) runs at 16 kHz. The seam's SampleRate contract forces this to be explicit: either (a) the engine resamples 16 kHz↔48 kHz internally and still returns a 16 kHz buffer of the same length to the processor, or (b) the composer runs enhancement at the model's native rate and a converter bridges to the VAD's rate. Pick one in implementation and document it; the seam itself stays rate-honest (it throws on an unannounced mismatch rather than denoising at the wrong rate). A permissively-licensed 16 kHz model (e.g. an NSNet2 export) would sidestep the conversion entirely — a point in its favour if the DeepFilterNet license check fails.

Pinned catalog

The model file is one VoxaModelArtifact in a tiny catalog, resolved through VoxaModelCache just like the Kokoro and whisper.cpp catalogs — the same (Id, DownloadUrl, Sha256, SizeBytes) record, the same offline-miss behaviour (a copy-pasteable provisioning message), the same first-run-download-then-offline lifecycle.

src/Voxa.Audio.Enhance/DeepFilterNetCatalog.cs (sketch) 
// <-- Sha256 + SizeBytes from the REAL artifact's metadata at implementation time; NEVER hand-written. -->
private static readonly VoxaModelArtifact Model = new(
    Id:          "deepfilternet/deepfilternet3.onnx",
    DownloadUrl: new Uri("<pinned model URL>"),
    Sha256:      "<sha256>",      // from the file's published checksum / LFS pointer
    SizeBytes:   <bytes>);
Do not fabricate the model hash. The SHA-256 and exact byte length must be read from the real artifact at implementation time — its repository checksum, release asset, or LFS pointer (oid sha256: / size) — exactly as the VLS-001 and VLS-002 entries were sourced. A hand-written or guessed hash is a bug, not a placeholder: it makes the cache either fetch the wrong file silently or fail verification forever.

License-closure gate (mirror the Kokoro one)

The Kokoro package already enforces its GPL boundary mechanically with DependencyGraphGateTests — one test asserts the engine assembly references nothing forbidden, another scans the build output for forbidden native files. Voxa.Audio.Enhance ships the equivalent: a test asserting the package's closure contains only permissively-licensed dependencies (no GPL/AGPL transitive, no native it isn't entitled to redistribute), so a license regression fails CI rather than shipping.

tests/Voxa.Audio.Enhance.Tests/DependencyGraphGateTests.cs (mirror of the Kokoro gate)

With Engine absent or None (the default), the composer adds no enhancement stage and the pipeline is byte-identical to today. Model resolves through the pinned catalog; an explicit ModelPath bypasses the cache (the descriptor concern, exactly like Voxa:Kokoro:ModelPath). All keys are parsed from the captured root section in EnhanceOptions.FromConfiguration, never service-located.

8.Plan of action — phases & dependencies

  1. Phase A — the seam + null default + processor + composer wiring (ships alone). Land IAudioEnhancer, NullAudioEnhancer, AudioEnhancerProcessor, VoxaEnhancerDescriptor, the TryGetEnhancer registry slot, and the gated composer insertion. Default Engine=None ⇒ passthrough golden proves byte-identical. No model, no network. Independently mergeable; this is the contract every implementation targets.
  2. Phase B — the reference ONNX engine + pinned catalog. Once the model + license are settled, add Voxa.Audio.Enhance (DeepFilterNetEnhancer, DeepFilterNetCatalog with the real hash, descriptor with WarmUpAsync, the license-closure gate test, and the Category=LocalModels round-trip test). Depends on Phase A's seam.

One logical change per PR (CONTRIBUTING). Phase A and Phase B are two PRs; each adds a [Unreleased] CHANGELOG entry and updates docs/local-speech.md. This spec is authored before either lands — the program's spec-before-code rule.

12.Out of scope / deferred

  • Offline restoration of voice-clone reference audio (Sidon-style). Cleaning up a recorded sample to improve a cloned voice is an artifact-processing concern that belongs with VVL-002 / Studio, not the live mic path. speech-core pairs DeepFilterNet3 (denoise, imported here) with Sidon (restore, deferred there).
  • GPU enhancement. If a heavier enhancement model needs it, it rides the VLS-002 ONNX-Runtime / Whisper.net GPU patterns (opt-in runtime package, device key) rather than re-inventing them — and only on demand.
  • Streaming / lookahead-free variants. The reference engine processes buffered chunks; a low-latency streaming enhancer (sub-frame lookahead) is a follow-up once the seam and a baseline model are proven.
  • Acoustic echo cancellation is VRT-003 (IEchoCanceller), the upstream stage — not this item.
  • A second enhancement backend (e.g. a classic noise-suppression DSP via P/Invoke) — the seam supports it, but one reference engine is the scope here.

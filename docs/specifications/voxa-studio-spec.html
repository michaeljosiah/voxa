VST-001 — Voxa Studio Specification

A desktop application — Voxa.Studio — that runs the real Voxa pipeline in-process against the developer's microphone and speakers, and gives them what a sample web page never can: a live VAD probability trace, a per-turn latency waterfall, a voice audition lab for the local TTS catalogs, a visual model-cache manager, and an appsettings.json composer driven by the provider registry. Built on a new pipeline diagnostics surface (VoxaDiagnosticsHub) that is independently useful to every Voxa host.

1.What is this? — a plain-English explainer

What is the feature, in one sentence?

A desktop app you launch on your development machine that lets you talk to a Voxa pipeline and watch it think — see exactly when the voice-activity detector decided you stopped speaking, what the transcriber heard, how long every stage took, and how each local voice sounds — without writing a line of code or touching a config file.

Why does Voxa need it?

Today, developing with Voxa means editing appsettings.json, restarting a server, opening a browser page, and inferring what happened inside the pipeline from log lines. When something is off — the bot cuts you off mid-sentence, a transcript is garbled, the reply feels slow — the pipeline is a black box. A real example from this project: the Silero VAD silently never detected speech for an entire release cycle (the v5 context-window bug). The fix took a bisection session with hand-written probe scripts; a live probability chart would have made the flat-zero line obvious in seconds.

What does it look like?

Four tabs:

  • Talk — pick a microphone and speaker, press Start, and have a conversation with the configured pipeline. Below the transcript: a scrolling VAD probability chart with the open/close threshold drawn in, markers where turns started and ended, and a per-turn waterfall bar showing how the response time split across VAD hangover → transcription → agent first token → first synthesized audio.
  • Voices — every Piper and Kokoro voice in the pinned catalogs, with a play button. Type a sentence, audition it across voices, A/B two engines side by side, and see real numbers (time-to-first-byte, real-time factor) measured on your hardware, not someone's benchmark blog.
  • Models — what the model cache actually contains: name, size, hash-verified status. Buttons to verify, prefetch everything for an air-gapped deployment, or purge. Today this directory is invisible unless you go digging in ~/.cache/voxa/models.
  • Config — build a pipeline by picking providers from dropdowns (the lists come from the live provider registry, so they can never drift from the code), then export the exact appsettings.json block to paste into your server.
Why would a user care?
  • Debugging becomes visual. "Why did it cut me off?" is answered by looking at the VAD trace, not by adding log statements and re-running.
  • Latency becomes diagnosable. voxa.turn.ttfb tells you the response took 1.4 s; the waterfall tells you which stage to fix.
  • Voice selection takes minutes, not restarts. Auditioning seven Piper voices today means seven config edits and seven server restarts.
  • Air-gap provisioning gets a button. "Prefetch all" + copy the cache directory = a provisioned offline box.
  • Zero accounts, zero keys. Thanks to VLS-001, the entire app works out of the box with WhisperCpp + Piper/Kokoro + the Echo agent. Cloud providers light up automatically if keys are present, but nothing requires them.
Is it Windows-only?

The UI is built with Avalonia, so the codebase is cross-platform from day one — the same choice keeps faith with the rest of Voxa, which runs everywhere .NET runs (and whose local-speech catalogs already serve Linux and macOS users). v1 ships and is tested on Windows: the audio capture/playback layer is implemented for WASAPI first, behind an IStudioAudioDevice seam sized so a macOS/Linux backend is an additive package, not a rewrite.

Why is half of WS0 not about the app at all?

The killer features (VAD trace, waterfall) need the pipeline to narrate what it is doing — structured events for VAD probabilities, turn boundaries, transcripts, and stage timings. That eventing surface (VoxaDiagnosticsHub) is deliberately specified as a framework feature, not a Studio internal: a server host can subscribe to the same stream for a /voxa/debug page or production observability (it is the roadmap's P7 "per-stage latency waterfall" item). Studio is simply its first — and best — renderer. This also keeps Studio honest: it consumes only public Voxa surface, making it a permanent dogfooding instrument for the framework's API.

3.Goals, non-goals & budgets

3.1 Goals

  • G1. A developer with no Voxa config and no API keys launches Studio, presses Start, and is in a spoken conversation (WhisperCpp + Echo + Piper) after the first-run model download.
  • G2. Every turn renders a VAD probability trace and a stage-latency waterfall accurate to the frame timestamps (Frame.PtsMicros).
  • G3. Every catalog voice (Piper + Kokoro) can be auditioned and A/B-compared without restarting anything.
  • G4. The model cache is fully manageable from the UI: inventory, verify, prefetch-all, purge.
  • G5. The Config tab exports an appsettings.json block that VoxaDefaultsGuard accepts verbatim on a server.
  • G6. The diagnostics surface is a framework feature any host can subscribe to — Studio holds no privileged access.

3.2 Non-goals

  • Not a production monitoring dashboard — OpenTelemetry (VoxaActivities/VoxaMetrics) remains the production story; the hub is a development/debug stream.
  • Not an end-user softphone; no call recording, contact lists, or telephony.
  • Not a cloud-key vault: cloud providers work if keys are already in user-secrets/env, but Studio does not store secrets itself.
  • No installer/MSIX in v1 — dotnet run --project apps/Voxa.Studio is the supported launch path.
  • No remote attach in v1 (subscribing to a server's hub over WebSocket is deferred; the hub API is shaped so this is additive).

4.Architecture — Studio hosts the pipeline in-process

Studio is a fourth host for the same pipeline the servers run — not a client of one. Where MapVoxaVoice("/voice") bridges the composed pipeline to a WebSocket, Studio bridges it to the local audio devices. The composed middle (VAD → STT → filter → agent → aggregator → TTS) is byte-identical because it comes from the same DefaultVoicePipelineComposer.

┌────────────────────────── Voxa.Studio (Avalonia) ──────────────────────────┐ mic (WASAPI capture, 20 ms) ──►│ StudioMicSource ─► [ composed parts: VAD → STT → … → TTS ] ─► StudioSpeakerSink │──► speakers │ │ │ DiagnosticsTapProcessor(s) │ │ │ ▼ │ │ └──────────── VoxaDiagnosticsHub ─► Talk view (trace, waterfall, log) │ └─────────────────────────────────────────────────────────────────────────────────┘ same composed parts, hosted by a server: WebSocketSource ─► [ … ] ─► WebSocketAudioSink (unchanged)

4.1 The compose seam

DefaultVoicePipelineComposer.Compose(HttpContext) is the only entry point today, and it uses the context solely for RequestServices. WS0 adds a transport-agnostic overload; the HTTP one becomes a forwarder. This is the entire change Studio needs to reuse composition additive:

public ComposedVoice Compose(IServiceProvider services)        // new — any host
public ComposedVoice Compose(HttpContext ctx) => Compose(ctx.RequestServices);  // existing callers unchanged

4.2 Why events, not frames or OTel

  • Not frames: a VadWindowEvent fires every 32 ms per session. Injecting those as pipeline frames would pollute the data stream every processor sees and contend with real audio for channel capacity. Diagnostics ride a side channel.
  • Not OTel: ActivitySource/Meter are aggregation-shaped (spans, histograms) and reach the consumer through exporters with seconds of latency. The Talk view needs an ordered, low-latency, lossy-by-design in-proc stream. The two coexist: StageLatencyProcessor publishes each measurement to both the hub (for Studio) and a new voxa.stage.latency histogram (for production, completing roadmap P7).

4.3 Session model

Studio runs at most one live Talk session plus N Voice-Lab synthesis jobs. Engines come from the same DI container AddVoxa() builds; the Piper process pool and Kokoro ONNX session are shared exactly as in a server, so Voice Lab audition exercises the same warm-pool code paths users run in production.

5.Workstreams (WS0–WS6) with code

WS0 — VoxaDiagnosticsHub: the pipeline event stream P0

Projects Voxa.Core, Voxa.Audio.SileroVad, Voxa.AspNetCore Depends on nothing Unblocks WS2 (and roadmap P7's /voxa/debug for any host)

A per-session hub with typed events. Publishing is free when nobody listens; when someone listens, the channel is bounded and drops oldest — the pipeline must never backpressure on a renderer.

src/Voxa.Core/Diagnostics/DiagnosticEvents.cs 
namespace Voxa.Diagnostics;

/// Base for all diagnostics events. SeqNo is hub-assigned and gap-free per session,
/// so a renderer can detect drops. PtsMicros aligns events with Frame.PtsMicros.
public abstract record DiagnosticEvent
{
    public long SeqNo { get; init; }
    public long PtsMicros { get; init; }
}

public sealed record VadWindowEvent(float Probability, double Rms, bool Voiced, bool GateOpen) : DiagnosticEvent;  // Voiced = prob AND energy passed — anchors vad_close
public enum TurnEdge { UserStarted, UserStopped, BotStarted, BotStopped, Interrupted }
public sealed record TurnEvent(TurnEdge Edge) : DiagnosticEvent;
public sealed record TranscriptEvent(string Text, bool IsFinal) : DiagnosticEvent;
public sealed record AgentDeltaEvent(string TextDelta) : DiagnosticEvent;
public sealed record TtsChunkEvent(int Bytes, int SampleRate) : DiagnosticEvent;
public sealed record StageLatencyEvent(string Stage, double Ms) : DiagnosticEvent;   // "vad_close", "stt_final", "agent_first_token", "tts_first_byte", "audio_out"
public sealed record PipelineErrorEvent(string Source, string Message) : DiagnosticEvent;
src/Voxa.Core/Diagnostics/VoxaDiagnosticsHub.cs 
/// Per-session diagnostics stream. Register as scoped (server: per connection;
/// Studio: per Talk session). Publish is allocation-free until the first subscriber.
public sealed class VoxaDiagnosticsHub
{
    // Volatile flag — every publish site guards on this before building an event object.
    public bool HasListeners { get; }

    // Non-blocking. Stamps SeqNo. Bounded channel per subscriber, BoundedChannelFullMode.DropOldest.
    public void Publish(DiagnosticEvent e);

    // Multiple concurrent subscribers each get an independent bounded stream (default capacity 4096).
    public IAsyncEnumerable<DiagnosticEvent> SubscribeAsync(CancellationToken ct);
}

WS0 publish sites

EventEmitted fromMechanism
VadWindowEventSileroVadProcessorNew SileroVadOptions.ProbabilityObserver (Action<float prob, double rms, bool voiced, bool gateOpen>?, carried on VoxaVadSettings so any VAD provider can honour it) invoked synchronously on the VAD thread after each window — the engine itself stays observer-free (it is not thread-safe and should not grow responsibilities). The composer wires the observer to the hub when one is resolvable.
TurnEvent, TranscriptEvent, AgentDeltaEvent, TtsChunkEvent, PipelineErrorEventDiagnosticsTapProcessorPass-through FrameProcessor (the TracingProcessor pattern) mapping UserStarted/StoppedSpeakingFrame, TranscriptionFrame, text deltas, AudioRawFrame (downstream), InterruptionFrame, ErrorFrame to events. The composer inserts taps after VAD/STT/agent/TTS when the hub has listeners.
StageLatencyEventStageLatencyTracker (inside the hub)Implementation refinement: rather than a pipeline processor reading PtsMicros (zero on frames created mid-pipeline), a tracker inside the hub's publish path derives stages from the hub-stamped timestamps of the anchor events (last voiced window → gate close → final transcript → first agent delta → first TTS chunk). Derived events ride the same ordered stream after their anchors, and every stage — tracker-derived or sink-published (audio_out) — is recorded on voxa.stage.latency (tag: stage), completing the roadmap P7 waterfall item for all hosts.
Wiring rule. All taps are inserted by DefaultVoicePipelineComposer only when Voxa:Diagnostics:Enabled is true (default: false for servers, forced true by Studio). A production pipeline with diagnostics off composes byte-identically to today's.

/// The only OS-specific surface in Studio. A macOS/Linux backend is a new implementation,
/// nothing else changes.
public interface IStudioAudioDevice : IAsyncDisposable
{
    IReadOnlyList<AudioEndpoint> CaptureEndpoints();
    IReadOnlyList<AudioEndpoint> RenderEndpoints();

    // 20 ms PCM16 mono frames, resampled by the device layer to sampleRate.
    IAsyncEnumerable<ReadOnlyMemory<byte>> CaptureAsync(AudioEndpoint mic, int sampleRate, CancellationToken ct);

    ValueTask StartRenderAsync(AudioEndpoint speaker, int sampleRate, CancellationToken ct);
    ValueTask RenderAsync(ReadOnlyMemory<byte> pcm, CancellationToken ct);
    ValueTask FlushRenderAsync();   // barge-in: drop queued device audio immediately
}

public sealed record AudioEndpoint(string Id, string DisplayName, bool IsDefault);
Sample-rate ownership. The composed pipeline announces its effective input/output rates (the same values the server's session envelope carries — e.g. 16000 in / 22050 out for WhisperCpp + Piper amy-low). The device layer owns resampling from/to hardware rates. Hardcoding 16 k/22.05 k anywhere in Studio is a bug — the rates change with provider config, exactly as the browser client learned in VLS-001.

WS2 — Talk view: live conversation + VAD trace + latency waterfall P0

Project apps/Voxa.Studio Depends on WS0, WS1
  • Controls: mic/speaker pickers, Start/Stop, provider summary chip (e.g. SileroVad · WhisperCpp · Echo · Piper/amy-low), first-use model download progress (via PrefetchAsync progress callback).
  • Transcript timeline: user/bot bubbles stamped from TranscriptEvent/AgentDeltaEvent; interruptions render as a cut marker (TurnEdge.Interrupted).
  • VAD strip chart: scrolling probability line from VadWindowEvent (~31 events/s), the configured ConfidenceThreshold as a horizontal rule, gate open/close shading, turn-edge markers. Renders the Silero context-bug class of failure as an unmissable flat line.
  • Waterfall: per completed turn, one stacked bar from StageLatencyEvents: VAD hangover → STT → agent first token → TTS first byte → audio out; total equals voxa.turn.ttfb within measurement tolerance. Hover for ms numbers; copy-as-text for bug reports.
  • Event log pane: raw scrolling event view with SeqNo, filterable by type — the "I need to see exactly what fired" escape hatch.

Rendering rule: view-models coalesce hub events into ≤ 30 fps batches on a render timer; the subscriber loop itself never touches the UI thread. Chart drawing uses Avalonia's own drawing API (no charting NuGet dependency for a strip chart and stacked bars).

WS2 acceptance

  • WS2-A1Headless test: feeding a recorded hub event sequence (golden JSON fixture from a real jfk.wav turn) yields a view-model state with 1 user turn, 1 bot turn, ≥ 25 VAD points/s, and a 5-segment waterfall.
  • WS2-A2Headless test: TurnEdge.Interrupted truncates the active bot bubble and the sink flush is requested (barge-in path).
  • WS2-A3Manual protocol (§9): live conversation on Windows hardware with WhisperCpp+Echo+Piper; VAD trace visibly tracks speech; waterfall totals ≈ perceived latency.

9.Verification protocol

  1. Build: dotnet build Voxa.slnx -warnaserror on Linux and Windows.
  2. Default suite: dotnet test --filter "Category!=LocalModels" — includes all hub, composer-golden, view-model (headless), device-resampler, and cache-inventory tests. No new network or hardware dependencies.

11.Out of scope / deferred

VLS-005 — Speaker Diarization (Voxa.Audio.Diarization)

A new capability category: batch/offline "who spoke when" labelling for multi-speaker and meeting audio. It composes two opt-in model seams (ISpeakerSegmentation + ISpeakerEmbedding) through a pure-C# DiarizationPipeline (constrained agglomerative clustering, no ML runtime of its own) and fills the already-present, always-null TranscriptionFrame.SpeakerId. The orchestration package carries no ONNX dependency; the reference models live in a separate .Onnx sub-package.

1.What is this? — a plain-English explainer

What does VLS-005 add, in one sentence?

It tells Voxa who spoke each part of a multi-speaker recording — putting a speaker 0 / speaker 1 label on every segment of a meeting clip — by filling the SpeakerId field that TranscriptionFrame has always carried but never populated.

Why now?

This is a whole capability category Voxa simply lacks. The STT contract already exposes TranscriptionFrame.SpeakerId (see src/Voxa.Core/Frames/DataFrames.cs), but nothing ever sets it — it is null on every frame today. Diarization is the missing producer. speech-core demonstrates the value on-device, and its design ports cleanly (see the next answer).

What is explicitly not changing?

The real-time pipeline. Diarization here is batch / offline and opt-in: it runs over a complete recording (the voxa transcribe --diarize path), not the live VAD → STT → agent → TTS loop. Live/streaming diarization is out of scope (§13). With --diarize absent, behaviour is identical to today.

What is the design key?

The orchestration — windowing, embedding extraction, clustering, label assignment — is pure C# with no ML-runtime dependency, exactly mirroring speech-core's core-pure DiarizationPipeline (its SegmentationInterface / EmbeddingInterface / DiarizerInterface sit in front of swappable backends). That lets the new Voxa.Audio.Diarization package stay dependency-light and unit-testable on synthetic embeddings, while ONNX lives only in an opt-in .Onnx sub-package — the same discipline that keeps Voxa.Core at zero external dependencies.

2.Executive summary

VLS-005 introduces one new package family and one new CLI flag:

  • WS1 — Seams + pure-C# pipeline additive. A new Voxa.Audio.Diarization package defining ISpeakerSegmentation, ISpeakerEmbedding, IDiarizer, a DiarizerConfig record, a DiarizedSegment result, and a pure-C# DiarizationPipeline that composes a segmenter + embedder + constrained agglomerative clustering. Zero ML-runtime dependency.
  • WS2 — Reference ONNX impls opt-in. A sibling Voxa.Audio.Diarization.Onnx package shipping a Pyannote-Segmentation-3.0 segmenter and a WeSpeaker-ResNet34 embedder, run in-process via ONNX Runtime (the Kokoro / Silero precedent), with weights SHA-256-pinned through VoxaModelCache.
  • WS3 — Consumer wiring additive. A --diarize flag on the existing voxa transcribe verb that runs the pipeline over the input WAV and emits a speaker-labelled transcript, populating TranscriptionFrame.SpeakerId for the batch path.
Net effect. voxa transcribe --diarize meeting.wav produces a speaker-labelled transcript ([speaker 0] … / [speaker 1] …) from a single multi-speaker recording — entirely offline once the (opt-in) models are cached. The default build and the live pipeline are unchanged.

3.Goals, non-goals & budgets

Goals

  • Fill the existing TranscriptionFrame.SpeakerId on the batch transcription path with diarized speaker labels.
  • Define vendor-neutral seams (ISpeakerSegmentation, ISpeakerEmbedding, IDiarizer) so any backend can plug in.
  • Keep the clustering orchestration ML-runtime-free and deterministic, so it is fully unit-testable offline on synthetic embeddings.
  • Ship a reference ONNX backend (Pyannote + WeSpeaker) as an opt-in sub-package, weights SHA-256-pinned via VoxaModelCache.
  • Expose a first consumer (voxa transcribe --diarize) that proves the end-to-end path.

Non-goals (this spec)

  • Real-time / streaming diarization. This is batch/offline over a complete recording; the live pipeline is untouched.
  • Speaker identification / enrollment against known voices (mapping speaker 0 → "Alice"). A follow-up once embeddings exist (§13).
  • Diarization-aware turn-taking, or feeding speaker labels into the agent loop.
  • Studio speaker-label UI — Studio could later render the labels Core produces; that is its own item (§13).
  • Any ONNX / ML-runtime dependency in the orchestration package (it lives only in .Onnx).

Budgets

4.Architecture context — composition & package layout

Diarization is a composition of two model-backed steps and one pure-orchestration step, mirroring speech-core's DiarizationPipeline (which composes SegmentationInterface + EmbeddingInterface + agglomerative clustering, with the orchestration itself ML-runtime-free):

full recording (16 kHz mono PCM)
        │
        ▼
ISpeakerSegmentation   // per-window speaker-activity (e.g. Pyannote Segmentation 3.0)
        │  SegmentationWindow[]   (active speech regions + local speaker tracks)
        ▼
ISpeakerEmbedding      // fixed-dim speaker vector per region (e.g. WeSpeaker ResNet34)
        │  float[] per region    (EmbeddingDim wide)
        ▼
DiarizationPipeline    // PURE C# — constrained agglomerative clustering, NO ONNX
        │  DiarizedSegment[] { start, end, speaker }
        ▼
TranscriptionFrame.SpeakerId   // the existing, previously-null field is now filled

The split keeps the heavy, license-bound model code isolated behind seams and out of the dependency-light core package:

PackageContentsDependencies
Voxa.Audio.DiarizationThe seams (ISpeakerSegmentation, ISpeakerEmbedding, IDiarizer), DiarizerConfig, DiarizedSegment, SegmentationWindow, and the pure-C# DiarizationPipeline (clustering + label assignment).No ML runtime. Plain BCL math; may reference Voxa.Speech.Abstractions for VoxaModelCache types if the seams expose cache-relative ids (kept minimal).
Voxa.Audio.Diarization.OnnxThe reference PyannoteSegmentation (ISpeakerSegmentation) + WeSpeakerEmbedding (ISpeakerEmbedding) impls, their descriptors, and a pinned DiarizationCatalog.ONNX Runtime, in-process — coexists with Kokoro / Silero (already present).
Through-line. This is the program's "define the seam, ship a default backend, keep the orchestration pure" pattern (§ "seams over engines" in the program doc). The pure pipeline is what makes the whole feature testable offline; the ONNX package is what makes it useful on real audio. Per the program doc, VLS-005 is independent of the VRT items and parallelizable.

5.WS1 — the diarization seams + pure-C# pipeline

Package Voxa.Audio.Diarization (new) Risk Medium (clustering correctness) Network none Surface additive (new package)

The seams

Two model-backed seams and one orchestrating seam. The segmenter and embedder are the only members an ONNX backend implements; the DiarizationPipeline is the concrete pure-C# IDiarizer.

src/Voxa.Audio.Diarization/Seams.cs (sketch) 
// Per-window speaker activity over the whole recording. Mirrors speech-core's SegmentationInterface.
// Implementations are model-backed (e.g. Pyannote) and live in the .Onnx package.
public interface ISpeakerSegmentation
{
    IReadOnlyList<SegmentationWindow> Segment(ReadOnlySpan<float> audio, int sampleRate);
}

// One model-window's speech activity: absolute time span + local per-track activity.
public sealed record SegmentationWindow(double Start, double End, IReadOnlyList<SpeechRegion> Regions);
public sealed record SpeechRegion(double Start, double End);

// A fixed-dimension speaker embedding for a span of audio. Mirrors EmbeddingInterface.
public interface ISpeakerEmbedding
{
    int EmbeddingDim { get; }
    float[] Embed(ReadOnlySpan<float> audio, int sampleRate);
}

// The orchestrator. The pure-C# DiarizationPipeline is the reference implementation.
public interface IDiarizer
{
    IReadOnlyList<DiarizedSegment> Diarize(ReadOnlySpan<float> audio, int sampleRate, DiarizerConfig config);
}

// Result: a contiguous span attributed to one clustered speaker (0-based, stable per run).
public sealed record DiarizedSegment(double Start, double End, int Speaker);

The config — defaults from speech-core

Tunables are gathered into a record so the threshold (the one knob that materially changes results) is exposed rather than buried. Defaults track speech-core's DiarizationPipeline values.

src/Voxa.Audio.Diarization/DiarizerConfig.cs (sketch) 
public sealed record DiarizerConfig
{
    public double Onset               { get; init; } = 0.5;   // speech-activity onset threshold
    public double Offset              { get; init; } = 0.3;   // speech-activity offset threshold
    public double MinSpeechDuration   { get; init; } = 0.3;   // seconds; drop sub-threshold blips
    public double ClusteringThreshold { get; init; } = 0.715; // agglomerative merge distance (speech-core default)
    public int    MinSpeakers         { get; init; } = 0;     // 0 = auto (let clustering decide)
    public int    MaxSpeakers         { get; init; } = 0;     // 0 = auto (unbounded)
}

The pipeline — pure C#, no ONNX

DiarizationPipeline takes an ISpeakerSegmentation and an ISpeakerEmbedding and does everything else itself (see the four steps below). No model runtime is referenced — clustering is plain vector math, which is what makes WS1 testable on synthetic embeddings in the default lane.

src/Voxa.Audio.Diarization/DiarizationPipeline.cs (sketch) 
public sealed class DiarizationPipeline : IDiarizer
{
    private readonly ISpeakerSegmentation _segmentation;
    private readonly ISpeakerEmbedding _embedding;

    public DiarizationPipeline(ISpeakerSegmentation segmentation, ISpeakerEmbedding embedding)
    {
        _segmentation = segmentation;
        _embedding = embedding;
    }

    public IReadOnlyList<DiarizedSegment> Diarize(
        ReadOnlySpan<float> audio, int sampleRate, DiarizerConfig config)
    {
        // 1. segment → speech regions (onset/offset, MinSpeechDuration)
        // 2. embed each region → float[] (EmbeddingDim wide)
        // 3. constrained agglomerative clustering by cosine distance,
        //    stop at ClusteringThreshold, honour Min/MaxSpeakers when > 0
        // 4. assign stable speaker ids, merge adjacent same-speaker regions
        // → DiarizedSegment[]   (ALL of this is plain C#; no ONNX here)
    }
}
Why this matters. Because steps 3–4 take float[] embeddings and emit DiarizedSegment[] with no I/O and no model, the clustering can be exercised by handing it hand-constructed embedding clusters (e.g. two tight groups → two speakers). That is the default-lane test surface — no download, fully deterministic — and the reason the orchestration must stay ML-runtime-free.

6.WS2 — the reference ONNX implementations

Package Voxa.Audio.Diarization.Onnx (new) Risk Medium (models, license) Network first-run download only Default opt-in (not in default build)

Two ONNX models behind the WS1 seams, run in-process via ONNX Runtime — the same pattern Kokoro and Silero already use (KokoroTtsEngine + KokoroCatalog; the descriptor/catalog shape is the precedent to copy):

SeamReference modelRole
ISpeakerSegmentationPyannoteSegmentation — Pyannote Segmentation 3.0 (ONNX export)Per-window speech-activity / local speaker tracks.
ISpeakerEmbeddingWeSpeakerEmbedding — WeSpeaker ResNet34 (ONNX export)Fixed-dimension speaker vector per speech region.

Each model is one entry in a pinned DiarizationCatalog, resolved/verified/cached by VoxaModelCache exactly like the Kokoro catalog — VoxaModelArtifact(Id, DownloadUrl, Sha256, SizeBytes), forward-slash cache ids, hashes compiled in as constants. Descriptors validate config (engine names, config values) and register the impls the self-describing way the program doc prescribes for new audio processors.

src/Voxa.Audio.Diarization.Onnx/DiarizationCatalog.cs (sketch) 
// <-- sha256 + sizeBytes from each model's upstream LFS pointer; never hand-written -->
private static readonly Dictionary<string, VoxaModelArtifact> Models = new(StringComparer.OrdinalIgnoreCase)
{
    ["pyannote-seg-3.0"] = new("diarization/pyannote-segmentation-3.0.onnx",
        new Uri("<pinned-url>"), "<sha256>", <bytes>),
    ["wespeaker-resnet34"] = new("diarization/wespeaker-resnet34.onnx",
        new Uri("<pinned-url>"), "<sha256>", <bytes>),
};
Do not fabricate hashes. The SHA-256 and exact byte length for each model must be read at implementation time from the chosen upstream's LFS pointer (the raw/main/<file> pointer carrying oid sha256:<hash> and size <bytes>), exactly as the VLS-001 / Kokoro entries were sourced. A guessed or hand-written hash is a bug, not a placeholder.
License-closure gate. Before pinning, confirm the redistribution license of each model's weights and the chosen ONNX export, and record it in the catalog doc-comment (as KokoroCatalog notes Apache-2.0 for the model and GPL-isolation for espeak). If a chosen weight is not redistribution-clean, select a permissive alternative or gate it behind a process/consent boundary — mirror the Kokoro license-closure discipline. This is a merge gate, not a follow-up.

7.WS3 — consumer wiring (voxa transcribe --diarize)

Package Voxa.Cli Risk Low Surface additive (new flag) Default off (flag absent ⇒ today's behaviour)

The existing TranscribeAsync verb (src/Voxa.Cli/VoxaCliRunner.cs) reads a 16 kHz mono 16-bit WAV and streams whisper transcripts. VLS-005 adds a --diarize flag (parsed by the verb's existing Parse(args) option splitter). When present, the verb resolves the diarization models (via the pinned catalog), runs the DiarizationPipeline over the same PCM, and attributes each transcript to the diarized segment whose time span it overlaps — setting TranscriptionFrame.SpeakerId on the batch path and prefixing each output line with the speaker label.

src/Voxa.Cli/VoxaCliRunner.cs (TranscribeAsync — added branch, sketch) 
var diarize = options.ContainsKey("diarize");   // `--diarize` is a flag, no value
// ... existing WAV read + format guard (16 kHz mono 16-bit) ...

if (diarize)
{
    // PCM16 → float[]; resolve pinned Pyannote + WeSpeaker via VoxaModelCache;
    // build DiarizationPipeline(segmentation, embedding) and run it once.
    IReadOnlyList<DiarizedSegment> segments =
        pipeline.Diarize(samples, wav.SampleRate, DiarizerConfig); // pure-C# orchestration

    // each whisper transcript is attributed to the overlapping DiarizedSegment.Speaker:
    //   new TranscriptionFrame(text, IsFinal: true, SpeakerId: $"speaker {seg.Speaker}")
    output.WriteLine($"[speaker {speaker}] {result.Text}");
}
else
{
    output.WriteLine(result.Text);   // unchanged path
}
Output format. With --diarize, each line is prefixed [speaker N] (0-based, stable within a run). Without it, output is byte-identical to today. The SpeakerId string is "speaker N" for the batch frame, matching the printed label.
Studio (out of scope here). Because Core now produces populated SpeakerId values, Studio could later colour/label speakers in a transcript view. Per the program's Core-vs-Studio boundary, Studio only surfaces Core signals; the Studio UI is a separate item (§13) and not built in VLS-005.

8.Configuration / CLI reference

CLI

voxa transcribe — affected usage 
voxa transcribe <file.wav> [--model <name>] [--language <lang>] [--diarize]

  --diarize    NEW. Run batch speaker diarization over the recording and prefix each
               line with [speaker N]; populates TranscriptionFrame.SpeakerId. Requires
               the diarization models (resolved/cached on first use). Absent ⇒ unchanged.

Config (optional overrides)

The CLI uses sensible defaults; a host (or the CLI via config) may override the engine choice and DiarizerConfig knobs under a Voxa:Diarization section. Providers that read this section capture the IConfiguration passed to AddVoxa — they must not call GetRequiredService<IConfiguration>() (breaks the plain-ServiceCollection hosts: Studio, tests, CLI), per the program's config-capture invariant.

appsettings.json — Voxa:Diarization (all optional) 
{
  "Voxa": {
    "Diarization": {
      "Segmentation": "pyannote-seg-3.0",    // segmentation engine / catalog id
      "Embedding": "wespeaker-resnet34",      // embedding engine / catalog id
      "ClusteringThreshold": 0.715,           // DiarizerConfig override (merge distance)
      "MinSpeakers": 0,                        // 0 = auto
      "MaxSpeakers": 0                         // 0 = auto
    }
  }
}

9.Plan of action — phases & dependencies

  1. Phase A — Seams + pure-C# pipeline + synthetic tests (ships alone). Create Voxa.Audio.Diarization with the seams, DiarizerConfig, DiarizedSegment, and the DiarizationPipeline clustering. Cover clustering with synthetic-embedding unit tests (deterministic, offline, no models). Independently mergeable — it is pure orchestration with no ONNX.
  2. Phase B — Reference ONNX impls + pinned models. Create Voxa.Audio.Diarization.Onnx with the Pyannote + WeSpeaker impls, descriptors, and a pinned DiarizationCatalog (real LFS-sourced hashes, license-closure recorded). A Category=LocalModels test downloads + verifies the models against the pins.
  3. Phase C — CLI verb. Add --diarize to transcribe: resolve models, run the pipeline, attribute transcripts to segments, populate SpeakerId, print [speaker N].

One logical change per PR (per CONTRIBUTING) — Phases A / B / C are three PRs. Each adds a [Unreleased] CHANGELOG entry; the spec lands before any code; warnings-as-errors stays clean.

13.Out of scope / deferred

  • Real-time / streaming diarization. VLS-005 is batch/offline over a complete recording; live labelling in the VAD → STT → agent → TTS loop is a separate effort.
  • Speaker identification / enrollment. Mapping speaker 0 to a named, known voice is a follow-up once embeddings exist (the program calls this out explicitly).
  • Studio speaker-label UI. Core now produces populated SpeakerId; rendering speaker colours/labels in Studio is its own item under the Core-vs-Studio boundary.
  • Diarization-aware turn-taking / agent context. Feeding speaker labels into the agent loop is not part of this capability.
  • Alternative diarization backends (other segmentation/embedding models) — the seams allow them; only the Pyannote + WeSpeaker reference pair is pinned here.

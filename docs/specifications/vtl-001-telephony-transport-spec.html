VTL-001 — Telephony Transport (Twilio Media Streams over the WebSocket seam)

A first-class telephony transport — phone calls in and out of a Voxa pipeline — built on the transport seam Voxa already has. Twilio (and later Azure ACS) stream call audio over a WebSocket, not WebRTC, so this is a new PipelineSource/PipelineSink pair speaking the vendor's JSON-envelope-over-WS framing plus a μ-law codec and a sample-rate bridge at the edge — the VAD → STT → agent → TTS chain is untouched, and there is no WebRTC stack, no ICE, no TURN.

1.What is this? — a plain-English explainer

What does VTL-001 add, in one sentence?

It lets a real phone call drive a Voxa voice agent: a caller dials a number, Twilio bridges the call audio to your server over a WebSocket, and Voxa runs its normal VAD → STT → agent → TTS pipeline on it — speaking back down the same call — with barge-in.

Isn't telephony a WebRTC thing? Didn't we decide not to do WebRTC?

That's the key insight that makes this cheap. Twilio Media Streams (and Azure ACS) deliver phone audio over a plain WebSocket — Twilio owns the WebRTC/SIP/PSTN side and hands you JSON messages whose payload is base64-encoded μ-law audio. So telephony needs no WebRTC stack in .NET (no SIPSorcery, no ICE, no DTLS-SRTP, no TURN server). It reuses the exact transport mechanism Voxa already uses for its native client: an ASP.NET WebSocket upgrade feeding a PipelineSource, and a PipelineSink writing back.

So what is actually new, if the transport mechanism already exists?

Three small things, all at the edge of the pipeline: (1) a new source/sink that speak Twilio's JSON-envelope framing instead of Voxa's native binary-PCM framing; (2) a μ-law (G.711) encode/decode — a 256-entry lookup table both ways; (3) a sample-rate bridge between the telephone's 8 kHz and whatever rate the composed pipeline announces (e.g. 16 kHz for Whisper). The whole STT/agent/TTS chain in the middle is byte-for-byte the same as today.

Why is this the right first transport to add (vs. browser WebRTC)?

Because phone agents are the highest-value, lowest-cost target: it's the canonical production use case for voice AI, it closes the "no telephony" gap the Pipecat comparison flagged as the single biggest functional gap, and it rides the seam we already have for a few days of work — versus multi-week, infra-heavy native WebRTC. Browser WebRTC stays deferred and, when needed, is best delivered by integrating a media server (LiveKit/Daily) rather than implementing the stack — a separate decision.

Does this change anything for today's (native-client / Studio) users?

No. It's an additive new package (Voxa.Transports.Telephony + Voxa.Transports.Twilio) and a new endpoint (MapVoxaTwilioVoice). The native WebSocket transport, Studio's in-process transport, the composer, and every processor are untouched. If you never reference the telephony package, nothing about your build changes.

What ships in VTL-001, and what is deferred?

This item is a spec only — no code lands with it. The spec defines: the shared telephony media-stream base, the Twilio codec + TwiML endpoint as the first vendor, the μ-law + resample edge, and the lifecycle/barge-in mapping. Deferred to follow-up items: Azure ACS as the second vendor (designed in §9), DTMF-as-frames, outbound/dial-out calls, and any SIP-direct path. Implementation is phased (§11) so Twilio ships first behind this spec.

2.Executive summary

Voxa's pipeline is PipelineSource → FrameProcessor… → PipelineSink, and the entire VAD/STT/agent/TTS chain is composed transport-agnostically by DefaultVoicePipelineComposer.Compose(IServiceProvider), which returns a ComposedVoice(Parts, InputSampleRate, OutputSampleRate). The native client transport (Voxa.Transports.WebSocket) is just two leaf processors — WebSocketAudioSource : PipelineSource and WebSocketAudioSink : PipelineSink — carrying raw PCM as binary WebSocket frames plus a typed JSON sidechannel. Telephony is the same shape with a different edge codec.

Twilio Media Streams connects a phone call to a wss:// endpoint (declared via TwiML <Connect><Stream>) and exchanges JSON text messages: inbound connected/start/media/dtmf/stop, outbound media/mark/clear. The audio is base64-encoded μ-law (G.711 PCMU), 8 kHz, mono, in 20 ms chunks. Three additive workstreams plus a second-vendor design:

  • WS1 — the shared base additive. A TelephonyMediaStreamSource : PipelineSource / TelephonyMediaStreamSink : PipelineSink pair in a new Voxa.Transports.Telephony package that own the WebSocket read/write loops, the bounded outbound queue with the same interruption-epoch purge as WebSocketAudioSink, and the sample-rate bridge — and delegate the wire format to a small codec seam. This mirrors the proven WebSocketSttEngine "shared base + thin vendor codecs" pattern from the STT work.
  • WS2 — the Twilio codec + endpoint P1. A TwilioMediaStreamCodec implementing the seam (parse media→PCM, serialize PCM→media, clear on barge-in, capture streamSid) and a MapVoxaTwilioVoice("/twilio") endpoint that returns TwiML on the webhook GET/POST and accepts the media WebSocket, composing the pipeline body via the same DefaultVoicePipelineComposer as UseDefaults().
  • WS3 — μ-law + sample-rate bridge P1. A branchless G.711 PCMU encode/decode (256-entry tables) and reuse of the existing, already-tested LinearResampler (promoted from Studio into a shared package) to bridge 8 kHz wire ↔ the pipeline's announced rate. When STT announces 8 kHz (a cloud "telephony" model), the resample is a no-op.
  • WS4 — lifecycle & barge-in mapping P1. Map Twilio start/stop to StartFrame/EndFrame, InterruptionFrame to a Twilio clear, and (optionally) mark for playout completion. Telephony has no acoustic echo loop (the caller hears the bot through the phone; Twilio never feeds bot audio back on the inbound track), so barge-in here needs no AEC — strictly simpler than open-speaker barge-in.
  • WS5 — Azure ACS (design, deferred). Same base, a AcsMediaStreamCodec that is even simpler (ACS streams 16-bit PCM at 16 kHz — no μ-law, no resample) and fits Voxa's existing Azure lean. Specced in §9; implemented as a follow-up item.
Net effect. A phone call drives the existing pipeline by adding one package and one endpoint. No WebRTC, no new infrastructure for the deployer beyond a Twilio number + webhook URL. The middle of the pipeline — every processor, the composer, the diagnostics — is unchanged and untested-against because it never sees the transport.

3.Goals, non-goals & budgets

Goals

  • Accept an inbound Twilio phone call and run the standard composed pipeline on it, speaking back down the call with barge-in — keyless-of-WebRTC, over the WebSocket seam Voxa already has.
  • Define a shared telephony media-stream base (source/sink + codec seam) so a second vendor (ACS) is a thin codec, not a second transport — mirroring WebSocketSttEngine.
  • Reuse DefaultVoicePipelineComposer.Compose(IServiceProvider) verbatim: the telephony endpoint composes the identical VAD → STT → … → TTS body that UseDefaults() does. Telephony is purely an edge skin.
  • Ship a correct, allocation-disciplined μ-law codec and reuse the existing tested resampler to bridge 8 kHz ↔ the announced pipeline rate.
  • Map call lifecycle and barge-in to Voxa frames faithfully: start/stop ↔ StartFrame/EndFrame, InterruptionFrame → Twilio clear, epoch-purge stale bot audio exactly as the native sink already does.
  • Honour every cross-cutting invariant: forward StartFrame/EndFrame; bounded DropOldest on the inbound audio path and bounded Wait on the outbound control path; the config-capture rule; Voxa.Core stays zero-dependency; one logical change per PR with a CHANGELOG entry; this spec lands before any code.

Non-goals (this spec / first item)

  • WebRTC of any kind. No native WebRTC (SIPSorcery), no browser P2P, no media-server integration (LiveKit/Daily). Those are separate, later decisions; telephony here is deliberately WebSocket-only because the vendors deliver it that way.
  • Outbound / dial-out calls and call control. VTL-001 handles an inbound media stream. Originating calls, transfers, conferencing, and recording are Twilio REST/TwiML concerns outside the media path — deferred.
  • DTMF-as-frames. The codec will parse the Twilio dtmf event, but mapping keypad digits to a first-class frame (there is no DtmfFrame today) is a follow-up; for now it is logged/ignored behind a seam hook.
  • A direct SIP path. Bridging SIP without a provider (e.g. via SIPSorcery) is explicitly out — it reintroduces exactly the stack telephony-over-provider lets us avoid.
  • A Studio surface. Studio hosts the in-process mic/speaker transport; it is not a softphone. No telephony UI.
  • Shipping ACS. ACS is designed (§9) but implemented as its own follow-up item.

4.Architecture context — where telephony plugs into the seam

The native HTTP host builds a connection like this (MapVoxaVoiceExtensions.HandleContextAsync): accept the WebSocket, build WebSocketAudioSource → the composed Parts → WebSocketAudioSink, run the PipelineRunner, and inject a SessionInfoFrame announcing the rates. Telephony is the same five steps with a telephony source/sink and a TwiML pre-step:

 PSTN ──▶ Twilio ──(wss: JSON + base64 μ-law 8 kHz)──▶ [ Twilio media WebSocket ]
                                                          │
   ┌──────────────────────────────────────────────────────┘
   ▼
 TelephonyMediaStreamSource  ── decode μ-law → resample 8k→Nk → AudioRawFrame ──▶ ┐
   (PipelineSource)                                                                 │
                                                                                    ▼
        VAD ──▶ STT ──▶ filter ──▶ agent ──▶ aggregator ──▶ TTS   (unchanged composed body)
                                                                                    │
 TelephonyMediaStreamSink  ◀── encode μ-law ◀── resample Nk→8k ◀── AudioRawFrame ◀┘
   (PipelineSink) ── JSON media/clear/mark ──▶ Twilio ──▶ PSTN

The middle is produced by the same composer. The only seam-level decisions are: how the source turns inbound bytes into AudioRawFrames, and how the sink turns outbound AudioRawFrames back into Twilio messages — which is exactly what a codec abstracts.

5.WS1 — the shared telephony media-stream base

Package Voxa.Transports.Telephony (new) Risk Medium (WS lifecycle/threading) Network inbound WebSocket only Surface additive (new source/sink + codec seam)

The base owns everything that is vendor-independent: the server WebSocket receive/send loops, the bounded outbound queue with the barge-in epoch purge, the resample bridge, and the frame mapping. The vendor supplies only the wire format through a tiny codec — exactly how WebSocketSttEngine owns the ClientWebSocket and delegates ParseMessage/BuildCloseMessage to each STT vendor.

src/Voxa.Transports.Telephony/ITelephonyMediaCodec.cs (new — sketch) 
namespace Voxa.Transports.Telephony;

/// <summary>
/// Per-vendor wire codec for a telephony media stream (Twilio Media Streams, Azure ACS, …). The shared
/// source/sink own the WebSocket, the outbound queue + barge-in purge, and the sample-rate bridge; the
/// codec owns ONLY the JSON/base64 framing and the vendor's negotiated media format.
/// </summary>
public interface ITelephonyMediaCodec
{
    /// <summary>The wire media format (e.g. μ-law 8 kHz mono for Twilio, PCM16 16 kHz for ACS). The base
    /// uses Rate to size the 8k↔pipeline resample and Encoding to pick μ-law vs. raw PCM.</summary>
    TelephonyMediaFormat WireFormat { get; }

    /// <summary>Parse one inbound text message into zero or more transport events: decoded near-end audio
    /// (still at wire rate/encoding — the base resamples), start/stop lifecycle, dtmf, or "ignore".
    /// Returns the raw wire audio bytes; the base does μ-law decode + resample so every codec stays trivial.</summary>
    TelephonyInbound Parse(ReadOnlySpan<byte> utf8Message);

    /// <summary>Serialize one chunk of outbound audio (already wire-rate, wire-encoding) into a vendor
    /// "media" message addressed to the active stream.</summary>
    byte[] BuildMedia(ReadOnlyMemory<byte> wireAudio);

    /// <summary>The vendor's "flush my buffered playout now" message (Twilio: clear). Sent on barge-in so
    /// the caller stops hearing now-stale bot audio almost immediately. Null ⇒ vendor has no flush.</summary>
    byte[]? BuildClear();

    /// <summary>Optional playout marker (Twilio: mark) to detect when the caller has heard a chunk. Null ⇒ unsupported.</summary>
    byte[]? BuildMark(string name);
}

The base source's read loop is the native source's loop with the codec spliced in. Note the inbound media on telephony arrives as text JSON (not binary), so unlike WebSocketAudioSource the audio path goes through Parse too:

src/Voxa.Transports.Telephony/TelephonyMediaStreamSource.cs (new — sketch, read loop core) 
var inbound = _codec.Parse(message.Span);
switch (inbound.Kind)
{
    case TelephonyInboundKind.Audio:
        // wire (μ-law 8k) → PCM16 8k → PCM16 Nk (announced input rate). One exact-size array → frame payload.
        var pcm8  = _muLaw.Decode(inbound.WireAudio.Span);              // μ-law → PCM16 @ 8 kHz
        var pcmIn = _bridge.ToPipeline(pcm8);                           // 8 kHz → InputSampleRate (no-op if equal)
        await IngestAsync(new AudioRawFrame(pcmIn, _inputRate, 1), ct).ConfigureAwait(false);
        break;
    case TelephonyInboundKind.Start:
        // streamSid captured inside the codec; optionally announce a SessionInfoFrame here.
        break;
    case TelephonyInboundKind.Stop:
        await IngestAsync(new EndFrame(), ct).ConfigureAwait(false);
        return;
    case TelephonyInboundKind.Dtmf:   // VTL-001: logged behind a hook; DtmfFrame is a follow-up.
    case TelephonyInboundKind.Ignore:
        break;
}
Why a base, not just a Twilio source. The user explicitly favours a shared base, and it is the right factoring regardless of how many vendors ship: the WebSocket lifecycle, the bounded outbound queue, the barge-in epoch purge, the resample bridge, and the EndFrame/runner-completion discipline are identical for Twilio and ACS — only the JSON shapes and the wire media format differ. Putting them in the base means the ACS codec (§9) is ~60 lines, exactly as AssemblyAISttEngine is a thin subclass of WebSocketSttEngine.
Threading & backpressure, copied from the native sink. The outbound queue is a BoundedChannel (Wait for control, the audio purged by epoch on InterruptionFrame), a single writer task drains it to the socket, and EndFrame completes the channel unconditionally so a caller hang-up (socket already closed) can't deadlock the data loop — the exact failure WebSocketAudioSink documents and guards. The base reuses that proven shape verbatim rather than re-deriving it.

The endpoint & TwiML

Twilio first hits your webhook (a normal HTTP request configured on the phone number) and expects TwiML in response; the TwiML's <Stream url> tells Twilio which wss:// to open for the media. The endpoint therefore has two faces on one helper:

src/Voxa.Transports.Twilio/MapVoxaTwilioVoice.cs (new — sketch) 
// 1) Webhook: return TwiML pointing Twilio at the media wss route.
endpoints.MapPost(pattern, ctx =>
{
    var wss = $"wss://{ctx.Request.Host}{pattern}/media";
    ctx.Response.ContentType = "text/xml";
    return ctx.Response.WriteAsync(
        $"<Response><Connect><Stream url=\"{wss}\"/></Connect></Response>");
});

// 2) Media stream: accept the WebSocket, compose the SAME pipeline body as UseDefaults(), run it.
endpoints.MapGet($"{pattern}/media", async ctx =>
{
    if (!ctx.WebSockets.IsWebSocketRequest) { ctx.Response.StatusCode = 400; return; }
    using var socket = await ctx.WebSockets.AcceptWebSocketAsync();

    var composed = ctx.RequestServices.GetRequiredService<DefaultVoicePipelineComposer>().Compose(ctx);
    var codec  = new TwilioMediaStreamCodec();
    var source = new TelephonyMediaStreamSource(socket, codec, composed.InputSampleRate);
    var sink   = new TelephonyMediaStreamSink(socket, codec, composed.OutputSampleRate);

    var b = Pipeline.Build().Source(source);
    foreach (var f in composed.Parts) b.Then(f(ctx.RequestServices));
    var pipeline = b.Sink(sink);

    await using var runner = new PipelineRunner(pipeline, ctx.RequestAborted);
    await runner.StartAsync(ct: ctx.RequestAborted);
    await runner.WaitAsync().WaitAsync(ctx.RequestAborted);
});
Same composer, same body. The media route resolves DefaultVoicePipelineComposer and calls Compose(ctx) — the identical call VoxaVoiceRoute.UseDefaults() makes. The processor list is whatever the host configured (STT/TTS/VAD/profile); telephony does not fork it. The only telephony-specific objects are the source, the sink, and the codec.
Security note. The webhook should validate Twilio's X-Twilio-Signature (HMAC over the request URL + params with your auth token) before returning TwiML, and the media route should be reachable only over TLS (wss). VTL-001 provides a hook for signature validation; enabling it is a host concern documented in the README. Without it, anyone who learns the URL can open a media stream.

7.WS3 — μ-law (G.711) & the sample-rate bridge

Codec G.711 PCMU (μ-law) Tables 256-entry, static Resampler existing LinearResampler (promoted) No-op case announced rate == 8 kHz

μ-law is a fixed, standardized companding of 14-bit PCM into 8 bits. Decode is a direct 256-entry table (byte → short); encode is the standard segment/quantize algorithm, precomputable into a table keyed on the top bits of the sample. Both are pure and allocation-free; the round-trip is lossy only to μ-law's quantization (exactly what the PSTN already imposes), so a decode→encode round-trip of μ-law-origin audio is a faithful identity for test purposes within one quantization step.

src/Voxa.Transports.Telephony/MuLaw.cs (new — sketch) 
public static class MuLaw
{
    private static readonly short[] DecodeTable = BuildDecodeTable();   // 256 → PCM16

    public static void Decode(ReadOnlySpan<byte> muLaw, Span<short> pcm)
    {
        for (int i = 0; i < muLaw.Length; i++) pcm[i] = DecodeTable[muLaw[i]];
    }

    public static byte EncodeSample(short pcm) { /* standard G.711 segment + quantize */ }
}

The bridge. Telephony is 8 kHz; the composed pipeline announces whatever STT/TTS resolve to (ComposedVoice.InputSampleRate / OutputSampleRate). The source resamples 8 kHz → InputSampleRate; the sink resamples OutputSampleRate → 8 kHz. Voxa already has a tested linear resampler — apps/Voxa.Studio/Audio/LinearResampler.cs with LinearResamplerTests — used to reconcile WASAPI mix formats. VTL-001 promotes it into a shared package (e.g. Voxa.Audio.Abstractions, which already hosts audio-edge types) so the framework transport and Studio share one tested implementation rather than duplicating it.

STT modelAnnounced input rateInbound bridge
WhisperCpp / most local + cloud models16 kHzresample 8 k → 16 k (band-limited; no new information, but matches the model's expected rate)
A cloud "telephony" model (e.g. Google telephony, Deepgram on 8 k)8 kHzno-op — run the pipeline natively at 8 k, best fidelity + least work
Linear is enough here. The PSTN is band-limited to ~3.4 kHz, so 8 kHz already captures the signal; upsampling to 16 kHz for an STT model is about matching the model's expected input, not recovering detail. A linear resampler is adequate and cheap; a higher-order (sinc) resampler is an optional later refinement and explicitly out of scope (§15).

8.WS4 — lifecycle & barge-in mapping

start/stop → StartFrame/EndFrame InterruptionFrame → Twilio clear Echo none (separate tracks) Epoch purge reused from native sink

Lifecycle

  • WebSocket open + connected: the PipelineRunner injects the StartFrame that fans out each processor's drain loop (same as the native route). connected itself is a handshake — ignored by the codec.
  • start: capture streamSid (needed to address every outbound message) and the customParameters (Twilio's mechanism to pass call metadata via <Parameter> in TwiML — the natural analog of Voxa's hello envelope). Optionally emit a SessionInfoFrame.
  • stop / socket close / caller hang-up: ingest EndFrame; the sink completes its outbound channel unconditionally (the native sink's anti-deadlock rule) and the runner drains to completion.

Barge-in — simpler on the phone than on a speaker

The output side already exists: Voxa fires an InterruptionFrame (or a UserStartedSpeakingFrame) on barge-in, and the native sink bumps an epoch and drops queued bot audio so the bot stops almost immediately. The telephony sink does the same epoch purge and additionally emits a Twilio clear so Twilio flushes whatever it has already buffered for playout toward the caller. That is the whole barge-in story for telephony:

InterruptionFrame ─▶ Interlocked.Increment(epoch)  // stale bot audio in the queue is dropped by the writer
                  └▶ send {"event":"clear","streamSid":…}  // Twilio drops what it already buffered
No AEC needed on a phone call. Open-speaker barge-in is hard because the mic hears the bot through the room — which is why VRT-003 built the IEchoCanceller seam and why Voxa is half-duplex on speakers. Telephony has no such loop: Twilio sends you only the inbound (caller) track and never mixes the bot's outbound audio back into it, so the STT never hears the bot. Barge-in is therefore just VAD on a clean caller track + clear — no echo canceller, no half-duplex gate. Telephony is the easier full-duplex environment, not the harder one.
Mark-based playout (optional refinement). Sending a Twilio mark after each utterance's audio and watching for the echoed inbound mark tells you when the caller has actually heard a chunk (vs. when you sent it). That sharpens "bot finished speaking" timing and barge-in accounting. VTL-001 wires the BuildMark hook but treats consuming marks as optional polish, not required for a working call.

9.WS5 — Azure ACS codec (second vendor; design)

Package Voxa.Transports.Acs (follow-up item) Wire PCM16 16 kHz (no μ-law) Resample none (rate already matches) Fit Voxa's existing Azure lean

Azure Communication Services Call Automation streams call audio over a WebSocket too, but as base64 PCM16 at 16 kHz mono wrapped in ACS's own JSON envelopes (AudioData in / StopAudio for barge-in out). That makes the ACS codec strictly simpler than Twilio's: no μ-law table, and if STT announces 16 kHz the resample bridge is a no-op. It also fits Voxa's stated Azure orientation (Speech / OpenAI / ACS already named in the design materials).

On the shared base, ACS is a thin codec — the same way AssemblyAISttEngine is a thin WebSocketSttEngine subclass:

  • WireFormat => Pcm16_16k (the base then skips μ-law decode/encode and, for a 16 kHz pipeline, skips resampling).
  • Parse handles ACS's AudioData / metadata / stop envelopes.
  • BuildMedia emits ACS outbound audio; BuildClear maps to ACS's stop-audio (barge-in).
  • The endpoint integrates with ACS Call Automation's media-streaming start instead of TwiML, but the WebSocket media loop is the base's.

ACS is designed here for completeness and to validate the base's vendor-neutrality, but it ships as its own follow-up item (its own PR, CHANGELOG entry, and live-validation caveat). VTL-001's deliverable scope is Twilio.

10.Configuration reference

Telephony is an opt-in endpoint, not a provider in the Voxa:Stt/Voxa:Tts sense — you add it by mapping a route, like the native MapVoxaVoice. The pipeline body still comes from the standard Voxa config section (STT/TTS/VAD/profile), unchanged.

Program.cs — mapping a Twilio voice endpoint 
builder.Services.AddVoxa(builder.Configuration);   // same providers/profile as any Voxa host
// …
app.UseWebSockets();
app.MapVoxaTwilioVoice("/twilio");                 // webhook (TwiML) + /twilio/media (WebSocket)
appsettings.json — optional telephony knobs (under Voxa:Telephony) 
{
  "Voxa": {
    "Stt": "WhisperCpp",         // the pipeline body is composed exactly as usual…
    "Tts": "Kokoro",
    "Telephony": {
      "Twilio": {
        "ValidateSignature": true,        // verify X-Twilio-Signature on the webhook (recommended)
        "AuthToken": "",                  // for signature validation (or via env / secrets)
        "PublicWssBaseUrl": ""            // override the wss host in TwiML when behind a proxy/tunnel
      }
    }
  }
}
KeyDefaultMeaning
Voxa:Telephony:Twilio:ValidateSignaturetrueVerify Twilio's request signature on the webhook before returning TwiML. Disable only for local tunnels where the signature URL won't match.
Voxa:Telephony:Twilio:AuthToken—Twilio auth token used for signature validation. Honour the config-capture rule; read from the captured IConfiguration / environment, never GetRequiredService<IConfiguration>().
Voxa:Telephony:Twilio:PublicWssBaseUrlrequest hostOverride the wss:// host embedded in TwiML when the public URL differs from Request.Host (reverse proxy, ngrok).
Config-capture rule. Any telephony option binding follows the same discipline as the composer, validator, and DefaultAgentFactory: capture the IConfiguration passed to AddVoxa; never service-locate IConfiguration (which breaks plain-ServiceCollection hosts — Studio, tests, CLI).

13.Verification protocol

LaneWhat runs
Default (offline) — codecParse a fixture of real-shaped Twilio envelopes (connected/start/media/dtmf/mark/stop); assert each maps to the right TelephonyInbound; assert BuildMedia/Clear/Mark emit the documented JSON addressed to the captured streamSid. No account, no network.
Default (offline) — μ-law & resampleμ-law decode/encode round-trip over all 256 codes (identity within one quantization step); LinearResampler 8 k↔16 k round-trip within a bounded error; no-op when rates are equal.
Default (offline) — base lifecycle & barge-inWith a FakeMediaCodec over a fake/in-memory WebSocket: synthetic media in produces AudioRawFrames at the announced input rate; bot AudioRawFrames out produce media messages; an InterruptionFrame bumps the epoch (stale queued audio dropped) and emits a clear; stop/socket-close yields EndFrame and the runner completes without deadlock.
Default (offline) — composition parityAssert the telephony media route composes its processor body via the same DefaultVoicePipelineComposer.Compose result as the native route for a given config (the middle of the pipeline is identical; only source/sink differ).
Manual / live (caveat lane)A real Twilio number → webhook (via a public URL / tunnel) → a real call: confirm two-way audio, barge-in (talking over the bot triggers clear and the bot stops), and clean hang-up. Not in CI (needs a Twilio account + PSTN). This is the load-bearing real-world check — see the caveat below.
Live-wire caveat (same discipline as the STT providers). Like the streaming STT vendors added on feat/stt-vendor-coverage, the Twilio (and ACS) wire protocol here is parser/spec-tested only until run against the real service. The envelope shapes are from Twilio's public Media Streams docs and are stable, but exact field presence, base64 framing, chunk cadence, and clear/mark timing must be validated on a real call before this is relied on in production. The manual lane above is mandatory before declaring Twilio "verified," and the README must carry the caveat.

15.Out of scope / deferred

  • WebRTC of any kind — native (SIPSorcery), browser P2P, or media-server integration (LiveKit/Daily). Telephony here is deliberately WebSocket-only because the providers deliver it that way. Browser WebRTC is a separate, later decision (and, when taken, likely a media-server integration rather than a .NET stack).
  • Azure ACS implementation — designed in §9, shipped as its own follow-up item on the same base.
  • Outbound / dial-out calls, transfer, conferencing, recording — Twilio REST/TwiML concerns outside the media path.
  • DTMF-as-frames — parsed-and-logged now; a first-class DtmfFrame (and IVR-style routing) is a follow-up.
  • Direct SIP — bridging SIP without a provider reintroduces the WebRTC/RTP/ICE stack telephony-over-provider exists to avoid.
  • Higher-order (sinc) resampling — linear is adequate for band-limited 8 kHz telephony; a better resampler is an optional refinement, not part of VTL-001.
  • A Studio telephony surface — Studio is not a softphone (per its spec's non-goals); no telephony UI.
  • Codec negotiation beyond μ-law/PCM16 — Opus or other Twilio/ACS formats are not handled; μ-law 8 k (Twilio) and PCM16 16 k (ACS) cover the providers' defaults.

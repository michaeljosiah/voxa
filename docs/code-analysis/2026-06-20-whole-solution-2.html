1.Executive summary

Overall health: Voxa remains in good shape. The Release build is warning-free, Voxa.Core stays dependency-clean, the documented hot-path invariants hold (the data loop is allocation-gated and regression-tested, the diagnostics hub is zero-cost-unobserved), and the previous audit's 13 findings are resolved on main (CQ-009's WAV-reader consolidation is the one remaining in-flight item). Nothing here blocks shipping.

The findings cluster into three themes. Theme 1 — "one transient fault, degraded for the whole session" is the meaningful one. Three independent code paths turn a single, recoverable fault into a session-long outage: a throwing interruption handler can permanently disable barge-in (CQ-014); one transient cloud STT failure permanently ends the transcript stream for a granular pipeline (CQ-015); and one transient voice-list failure breaks the entire Studio voices grid and is swallowed (CQ-016). Theme 2 — teardown hygiene: a handful of small, real resource/observability leaks (an abandoned timer, an unobserved read loop, an undisposed JsonDocument, a leaked CTS). Theme 3 — residual duplication: the WebSocket request lifecycle is copy-pasted, and the realtime codec/transport duplication first raised as CQ-006 still awaits its extraction (only the UTF-8 sub-bug was fixed).

Highest-value fixes are tiny: guard the two drain-loop bodies (CQ-014) and stop killing the STT channel on a transient failure (CQ-015). Each is a few lines and each converts a session-fatal fault into a recoverable one.

4.Detailed findings

CQ-014 — Unguarded exceptions in the processor drain loops disable interruptions / hang the sink MajorLikely

src/Voxa.Core/Processors/FrameProcessor.cs:130–156 (system loop) and :158–214 (data loop)
Dimension: Bad C# practices (over-narrow catch on a long-lived loop whose death is silent and consequential)

Evidence

The system loop deliberately wraps inFlight.Cancel() in a try/catch specifically so a fault can't kill the loop and disable interruptions — then leaves the two awaits immediately after it unguarded:

// FrameProcessor.cs:138-155 (RunSystemLoopAsync)
//  ... swallow it (the frame we wanted to preempt already completed) instead of letting it kill
//  this loop and disable interruptions for the rest of the session.
var inFlight = _currentFrameCts;
if (inFlight is not null)
{
    try { inFlight.Cancel(); }
    catch (ObjectDisposedException) { /* frame finished; CTS already replaced */ }
}
await OnInterruptionAsync(interruption, ct).ConfigureAwait(false);   // <-- unguarded
}
await ProcessFrameAsync(frame, ct).ConfigureAwait(false);            // <-- unguarded
}
catch (OperationCanceledException) { /* shutdown */ }
catch (ChannelClosedException) { /* shutdown */ }

The data loop has the same shape: the only catches are the preemption OperationCanceledException (line 196, with a when filter) and the loop-level OperationCanceledException/ChannelClosedException (lines 208–209). Any other exception from ProcessFrameAsync propagates out of the await foreach and ends the loop.

Why it matters

The system loop drains priority frames — InterruptionFrame (barge-in), speaking edges, ErrorFrame. If any subclass's OnInterruptionAsync or its system-frame ProcessFrameAsync throws a regular exception, the loop terminates and interruptions stop working for the rest of the session — the user can no longer cut the bot off. There are real overrides on this path (SpeechToSpeechProcessor.OnInterruptionAsync, EchoCancellerProcessor.OnInterruptionAsync, and system-frame handling inside the realtime/WebSocket processors). The inline comment three lines up proves the loop is meant to survive faults; the unguarded awaits are an oversight, not a deliberate fail-stop — that asymmetry is what makes this a bug.

In the data loop the consequence is different but also bad: StartFrame/EndFrame are forwarded from this same loop, so if ProcessFrameAsync throws on an earlier data frame the loop dies and the later EndFrame is never forwarded → PipelineSink.EndFrameObserved never completes → PipelineRunner.WaitAsync() only unblocks at the StopAsync grace timeout, and the original error is swallowed. That directly violates CLAUDE.md's "forward unhandled frames downstream (especially StartFrame/EndFrame, or the sink never completes)".

Recommendation

Wrap the per-frame body of each loop in a try/catch that treats OperationCanceledException/ChannelClosedException as loop-exit (as today) but, for any other exception, surfaces it via PushErrorAsync (an upstream ErrorFrame) and then continues (system loop) or exits deterministically (data loop) so the runner observes a PipelineFailedException rather than hanging. One bad handler must not silently disable interruptions or strand the sink.

Effort: trivial · Do-not-flag check: not the just-fixed disposal/CTS cluster — this is fault-resilience of an already-running session; the existing inline comment confirms the loop is meant to survive faults.

CQ-015 — A single transient transcription failure permanently kills the STT stream (OpenAI Whisper + Mistral) MajorCertain

src/Voxa.Speech.OpenAI/OpenAIWhisperEngine.cs:141–145 · src/Voxa.Speech.Mistral/MistralSpeechToTextEngine.cs:118–122
Dimension: Bad C# practices (exception handling that tears down a long-lived channel; documented-intent mismatch)

Evidence

// OpenAIWhisperEngine.cs:140-145 (Mistral is identical)
catch (OperationCanceledException) { /* session torn down mid-flush — teardown, not a transcription failure */ }
catch (Exception ex)
{
    _transcripts.Writer.TryComplete(new InvalidOperationException(
        $"OpenAI Whisper transcription failed: {ex.Message}", ex));
}

TryComplete(exception) permanently closes the unbounded transcript channel; ReadTranscriptsAsync is _transcripts.Reader.ReadAllAsync(ct), so it never yields again. The POST goes through raw _http.SendAsync(req, ct) with no retry (unlike the TTS path, which uses VoxaHttp.SendWithSingleRetryAsync), so a single 429/5xx/socket blip is enough.

That this is unintended is settled by the local engine, which makes the opposite choice and even claims the cloud engines match it:

// WhisperCppSttEngine.cs:356-362
catch (Exception ex)
{
    // Per-utterance failure must not kill the session: log and keep the engine alive for
    // the next utterance (parity with cloud engines' per-request failure handling).
    _logger.LogError(ex, "WhisperCpp STT: transcription failed for a {Seconds:F1}s utterance", ...);
}

Why it matters

In a granular pipeline the upstream SpeechToTextProcessor read loop sees the faulted enumerator, emits one ErrorFrame, and exits. For the rest of the call the user keeps talking and every subsequent utterance is silently dropped — no transcripts, no further errors, no recovery short of a new session. The "parity with cloud engines" comment is simply false today.

Recommendation

On a non-cancellation exception, log and return (drop just that utterance), mirroring WhisperCppSttEngine — do not TryComplete the writer. Keep the no-argument TryComplete() for the graceful StopAsync path. Optionally route the POST through VoxaHttp.SendWithSingleRetryAsync for parity with TTS.

Effort: trivial · Do-not-flag check: distinct from CQ-008 (that fixed cancellation threading — the OperationCanceledException arm here is already correct) and from CQ-009 (WAV); no test asserts the terminate-on-error behavior.

CQ-016 — A transient cloud failure breaks the whole Studio voices grid and is swallowed in Config MajorLikely

apps/Voxa.Studio/Services/VoiceCatalogService.cs:117–128 & :142–148 · escapes from ElevenLabs/Mistral ListVoicesAsync
Dimension: Bad C# practices (over-narrow catch defeating a designed isolation contract) · blast radius: the Studio app, not the runtime library

Evidence

The per-provider reconcile catches only VoiceProviderException:

// VoiceCatalogService.cs:117-128
try { live = await GetLiveAsync(ttsName, catalog, ct).ConfigureAwait(false); }
catch (VoiceProviderException ex)
{
    var stale = profiles.Select(p => new LibraryVoice(ToProviderVoice(p), VoiceState.Stale, p)).ToList();
    return new ProviderVoiceSet(ttsName, stale, ex.MissingApiKey, ex.Message);
}

But the catalogs throw unwrapped exceptions on transient failure — EnsureOkAsync only converts HTTP status errors:

// ElevenLabsVoiceCatalog.cs:34-38 (Mistral catalog is the same shape)
using var resp = await _http.SendAsync(req, ct).ConfigureAwait(false);   // HttpRequestException (DNS/TLS/socket) — unwrapped
await EnsureOkAsync(resp, "list voices", ct).ConfigureAwait(false);       // only this throws VoiceProviderException
using var doc = await JsonDocument.ParseAsync(stream, ...).ConfigureAwait(false); // JsonException — unwrapped

So a raw HttpRequestException/JsonException escapes ForProviderAsync. The grid builder has no isolation:

// VoiceCatalogService.cs:142-148
foreach (var name in _services.Registry.TtsNames.OrderBy(...))
    sets.Add(await ForProviderAsync(name, ct).ConfigureAwait(false));   // one raw throw aborts the whole loop

And in Config the call is fire-and-forget with no try/catch: public void RefreshCloudVoices() => _ = ReloadCloudVoicesAsync(); and a second _ = ReloadCloudVoicesAsync(); in OnSelectedTtsChanged (ConfigViewModel.cs:363, :409; body :365-393).

Why it matters

The design intends per-provider error isolation — ProviderVoiceSet carries an Error/MissingKey so the status strip can render one failed provider as an error chip while the rest load. A transient network blip on a single cloud provider instead: (a) aborts the entire AllAsync loop so the whole library grid fails to populate, and (b) in Config becomes an unobserved TaskScheduler.UnobservedTaskException — the picker just silently stays empty with no feedback. The narrow catch defeats the very resilience the Error field exists for.

Recommendation

Broaden the catch in ForProviderAsync to also degrade non-VoiceProviderException failures into an error chip, e.g. add after the existing catch:

catch (Exception ex) when (ex is not OperationCanceledException)
{
    var stale = profiles.Select(p => new LibraryVoice(ToProviderVoice(p), VoiceState.Stale, p)).ToList();
    return new ProviderVoiceSet(ttsName, stale, MissingKey: false, Error: ex.Message);
}

Keep OperationCanceledException propagating for cooperative cancellation. Secondarily, give ReloadCloudVoicesAsync's two fire-and-forget call sites an internal try/catch that surfaces a status string, so even an unexpected throw is visible rather than swallowed.

Effort: trivial · Do-not-flag check: not CQ-007/011; the VoiceProviderException-only contract and the unwrapped SendAsync/ParseAsync throw sites were both verified.

CQ-017 — PipelineRunner.StopAsync leaks an abandoned Task.Delay timer (and a possible unobserved fault) MinorCertain

src/Voxa.Core/Pipelines/PipelineRunner.cs:67–73
Dimension: Bad C# practices (fire-and-forget timer without cancellation)

Evidence

// PipelineRunner.cs:67-73
var grace = gracePeriod ?? TimeSpan.FromSeconds(5);
var graceTask = Task.Delay(grace, ct);
var winner = await Task.WhenAny(_completion.Task, graceTask).ConfigureAwait(false);
if (winner != _completion.Task)
{
    _cts.Cancel();
}

Why it matters

On the common path where the pipeline completes before the grace window, graceTask is abandoned but its underlying Timer keeps running for the full grace period (up to 5 s), holding state. If ct is cancelled, graceTask faults with TaskCanceledException and is never awaited — an unobserved task exception. It's once-per-StopAsync (not per-frame), so impact is small, but it's an avoidable textbook Task.Delay leak in lifecycle code.

Recommendation

Replace the manual WhenAny with a self-cleaning wait:

try { await _completion.Task.WaitAsync(grace, ct).ConfigureAwait(false); }
catch (TimeoutException) { _cts.Cancel(); }

That removes the dangling timer and the unobserved-fault path (or, if keeping WhenAny, drive the delay off a linked CTS cancelled in a finally).

Effort: trivial · Do-not-flag check: lifecycle path, but a concrete resource/observability hazard, not a style nit.

CQ-018 — WebSocketAudioSource's read loop is unobserved and not awaited on teardown MinorLikely

src/Voxa.Transports.WebSocket/WebSocketAudioSource.cs:19, 30
Dimension: Bad C# practices (unobserved Task; teardown ordering inconsistent with every sibling)

Evidence

// WebSocketAudioSource.cs:19,30 — field written, read by nothing
private Task? _readLoop;
...
protected override ValueTask OnStartAsync(StartFrame frame, CancellationToken ct)
{
    _readLoop = Task.Run(() => ReadLoopAsync(ct), ct);
    return ValueTask.CompletedTask;
}

Every other processor that spawns a read loop awaits it during teardown to observe faults and guarantee the loop stopped: SpeechToTextProcessor (DisposeAsyncCore), OpenAIRealtimeProcessor, AzureVoiceLiveProcessor, SpeechToSpeechProcessor — and the paired WebSocketAudioSink awaits its own _writerTask. This source is the only one that doesn't.

Why it matters

The receive loop guards ReceiveAsync (OCE → return, WebSocketException → ErrorFrame + return) but an exception thrown elsewhere in the loop — e.g. from a downstream IngestAsync, or a non-WebSocketException surfacing from receive — faults the task with no observer. And because DisposeAsync never awaits _readLoop, it can dispose _processorCts while the detached loop is still unwinding, so unlike its siblings the source gives no stop-before-dispose guarantee.

Recommendation

Override the documented extension point and await the loop, exactly like SpeechToTextProcessor:

protected override async ValueTask DisposeAsyncCore()
{
    if (_readLoop is not null) { try { await _readLoop.ConfigureAwait(false); } catch { } _readLoop = null; }
}
Effort: trivial · Do-not-flag check: a real robustness inconsistency vs. four sibling processors and the paired sink; the base does not await derived read loops.

CQ-019 — Per-tool JsonDocument in BuildSessionUpdate is never disposed (both realtime codecs) MinorCertain

src/Voxa.Services.OpenAIRealtime/Events/RealtimeEventCodec.cs:26 · src/Voxa.Services.AzureVoiceLive/Events/RealtimeEventCodec.cs:27
Dimension: Bad C# practices (undisposed IDisposable)

Evidence

var tools = options.Tools.Select(t => new
{
    ...
    parameters = JsonDocument.Parse(t.ParametersJsonSchema).RootElement.Clone(),
}).ToArray();

JsonDocument is IDisposable (it rents from a pooled buffer) and is discarded without Dispose(). The .Clone() already detaches the element, so the document is safe to dispose immediately.

Why it matters

One undisposed JsonDocument per configured tool leaks its pooled byte buffer back to the GC instead of the array pool. It's bounded and once-per-session (OnStartAsync), so impact is small — but it's a genuine, avoidable leak, and the .Clone() shows the author already intended the document to be short-lived.

Recommendation

Parse inside a using and clone within it (a small local loop or helper), or parse with JsonNode/JsonElement APIs that don't require disposal. Fixing once falls out naturally if the codec consolidation (§5) lands.

Effort: trivial · Do-not-flag check: a real undisposed IDisposable, not the CQ-002 transport-disposal finding.

CQ-020 — MapVoxaVoice duplicates the full WebSocket request lifecycle MinorCertain

src/Voxa.AspNetCore/MapVoxaVoiceExtensions.cs:103–215 (HandleContextAsync) vs. :217–293 (HandleAsync)
Dimension: DRY (duplicated handler lifecycle across 2 live sites)

Evidence

Both methods carry the same logger creation, the IsWebSocketRequest 400 guard, AcceptWebSocketAsync, the hello-reader try/catch with a PolicyViolation close (:143-155 vs :238-250), the source → factories → sink assembly, await using runner + StartAsync + WaitAsync(ct) with the OCE-when-cancelled swallow, and the identical ~15-line outer catch (Exception) → CloseAsync(InternalServerError) block (:199-214 vs :277-292). HandleContextAsync only adds two things: the pre-accept composition-error → HTTP 500 path (:123-135) and the session-rate source + SessionInfoFrame injection (:161-187).

Why it matters

Two copies of the connection lifecycle (hello handling, error-close semantics, cancellation) must be kept in lockstep; hardening one path silently misses the other. The public MapVoxaVoice(pattern, configure) overload routes to the leaner HandleAsync (line 67), so its callers never get the pre-accept-500 diagnostic the fluent path has — already a small behavioral divergence.

Recommendation

Have HandleAsync delegate to HandleContextAsync with a constant factory (_ => builder; the 500 branch is inert for an already-built builder), or extract the shared post-accept body (hello → assemble → run → close-on-error) into one private helper both call. No behavior change required.

Effort: moderate · Do-not-flag check: both entry points are live (fluent route + the configure overload used by the AspNetServer sample); not provider-registration boilerplate that legitimately varies.

CQ-021 — DictationSession.Start leaks the prior CancellationTokenSource on a re-start MinorLikely

apps/Voxa.Studio/Services/DictationSession.cs:43–63
Dimension: Bad C# practices (CTS handle leak)

Evidence

// DictationSession.cs:43-51
public void Start(AudioEndpoint microphone)
{
    ArgumentNullException.ThrowIfNull(microphone);
    if (State == DictationState.Recording) return;     // only guards while RECORDING
    ...
    _capture = new CancellationTokenSource();           // previous _capture never Disposed
    var ct = _capture.Token;

The Recording guard only blocks re-entry while recording. After a StopAndTranscribeAsync reaches Completed/Failed (or the early capture-failure return at :76-83), State != Recording, so the next Start allocates a fresh CTS while the previous one is still undisposed — and DisposeAsync (:132-138) only disposes the latest.

Why it matters

One CancellationTokenSource (which wraps a wait handle) leaks per start-after-non-stop cycle. Push-to-talk dictation is driven by a global hotkey and invoked repeatedly, so this accrues over a session in a long-lived service.

Recommendation

Dispose the prior source before replacing it, at the top of Start: _capture?.Dispose(); (the prior capture task is already finished/cancelled on any non-Recording state).

Effort: trivial · Do-not-flag check: DictationSession is IAsyncDisposable and is created/destroyed per dictation — unlike the app-lifetime VM singletons whose replace-without-dispose CTS pattern is the established (and acceptable) Studio convention.

CQ-022 — Dead codec builders in both realtime stacks; barge-in never sends response.cancel NoteCertain

src/Voxa.Services.OpenAIRealtime/Events/RealtimeEventCodec.cs:55–59 & :96–97 · same in the AzureVoiceLive twin
Dimension: Bad C# practices (dead code; latent-allocation trap; possibly-missing feature)

Evidence

A repo-wide search for BuildInputAudioBufferAppend (the string overload) and BuildResponseCancel finds only their definitions — zero callers in src/ or tests/ (the processors use BuildInputAudioBufferAppendUtf8, BuildSessionUpdate, BuildToolCallOutput, BuildResponseCreate).

// RealtimeEventCodec.cs:55-59  — superseded by the byte/UTF-8 variant
public static string BuildInputAudioBufferAppend(ReadOnlyMemory<byte> pcm)
{

Effort: trivial · Do-not-flag check: unused non-private methods aren't compiler-flagged; this has a latent-allocation and possibly-missing-feature implication, not a style nit.

CQ-024 — Agglomerative speaker clustering is O(n³) (full pair rescan per merge) NoteCertain (cost)

src/Voxa.Audio.Diarization/SpeakerClustering.cs:47–87
Dimension: Performance / algorithmic — on the offline diarization path

Evidence

// SpeakerClustering.cs:47-52 — ClosestPair is re-run every merge
while (members.Count > 1)
{
    ...
    var (a, b, best) = ClosestPair(centroids);   // O(n^2) pairwise scan ...
    ...                                           // ... inside the O(n) merge loop  => O(n^3)
}

ClosestPair (lines 69–87) is a full O(n²) double loop over centroids, called once per merge iteration; each CosineDistance is itself O(d) in embedding width, so the whole cluster is ~O(n³·d) in the number of speech regions n.

Why it matters

For a long, many-speaker recording with many short regions (the VLS-005 spec flags "large-ish recordings" as a risk), clustering time grows cubically. At n ≈ 30 it's nothing; in the high hundreds it becomes seconds. This is a batch/offline path (voxa transcribe --diarize; the spec states diarization here is not the live loop), so it does not affect real-time latency.

Recommendation

Only if profiling shows it matters: keep candidate pair distances in a heap and update incrementally after each merge, or use NN-chain linkage — ~O(n² log n)/O(n²). Not worth the added complexity until a real recording is measured slow; the determinism/tie-breaking behavior is correct and tested and must be preserved.

Effort: involved · Do-not-flag check: downgraded to Note per the audit's batch-path rule; the algorithm's output is correct, this is purely about scaling.

5.Confirmed-intentional & known-deferred

Each of these looked like a smell and was checked against the docs/tests/architecture; none is an ask.

  • Realtime codec/transport duplication (known — CQ-006 deferral). RealtimeEventCodec, IRealtimeApiTransport, and the WebSocketRealtimeApiTransport body are near-identical across Voxa.Services.OpenAIRealtime and Voxa.Services.AzureVoiceLive (the decode switch and the event-type string vocabulary are duplicated verbatim; the only differences are the options type and two error-label strings). This is the same duplication the prior audit raised as CQ-006 — only its UTF-8 receive-decode sub-bug was fixed; the extraction into a shared internal source was explicitly deferred. It remains the largest open DRY item and is a real drift risk (a protocol change must be made in two places), but it is already tracked, so it is recorded here rather than re-filed. If/when addressed: a small shared internal package or linked <Compile Include> with an options-view interface + error-label parameter.
  • WAV PCM16 reader consolidation (in-flight — CQ-009). OpenAIWhisperEngine/MistralSpeechToTextEngine still carry a private WrapPcmAsWav on main; the shared Pcm16Wav helper exists and the writer/reader migration is the open PR for CQ-009. Not re-reported.
  • AgentLoopProcessor's turn worker on its own CTS, not the session token. The worker is deliberately decoupled so an interruption (which fires the per-frame CTS) doesn't kill an in-flight turn; disposal bounds it via the DisposeAsyncCore join added for CQ-001. Intentional.
  • OnnxModelHost.EvictAll evict-during-load race. If EvictAll runs concurrently with a mid-construction Load on the same key, the in-flight session could be orphaned. The XML doc scopes EvictAll to single-threaded teardown ("tests, Studio's unload models … NOT per-connection teardown") — a documented non-scenario. Left as-is.
  • DefaultVoicePipelineComposer length. The long AEC→Enhance→VAD→STT→filter→agent→aggregator→TTS→far-end-tap method is the documented composition home; each stage is a distinct conditional insert, not duplicated logic. Not a god-service smell.
  • VoxaDefaultsGuard resolving IConfiguration from DI. Looks like a config-capture-rule violation, but it is a guarded nullable GetService that only runs under the Generic Host (where IConfiguration is always registered); plain-ServiceCollection hosts don't run hosted services. Honors the rule's intent.
  • Per-frame byte[] allocations in the WebSocket source, realtime audio deltas, and TTS chunking. These are inherent to the records-for-frames contract (the array becomes a frame payload with unbounded downstream lifetime — it must not be pooled). Not avoidable waste.

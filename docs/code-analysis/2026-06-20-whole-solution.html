1.Executive summary

Overall health: Voxa's core architecture is sound and well-documented: frame processors are isolated, vendor seams are intentionally testable, Voxa.Core stays dependency-clean, and the Release build is warning-free. Nothing in this audit blocks shipping by itself.

The meaningful issues cluster into three themes. First, processor teardown is too dependent on EndFrame, so normal request-abort disposal skips several derived/resource cleanup paths. Second, a few hot paths still allocate or copy on every frame despite the repo's explicit real-time allocation goals. Third, several provider packages have high-risk copy-paste twins, especially the OpenAI Realtime and Azure Voice Live protocol stacks.

The highest-value fixes are small: make FrameProcessor.DisposeAsync safely extensible, move OnEndAsync-only resource release into real disposal, remove the eager per-frame Frame.Id allocation, and extract the duplicated realtime wire protocol.

4.Detailed findings

[MAJOR] AgentLoopProcessor cleanup is hidden behind a shadowed DisposeAsync MajorCertain

src/Voxa.Core/Processors/AgentLoopProcessor.cs:336

Dimension: Bad C# practices: lifetime/resource bugs.

Evidence:

public new async ValueTask DisposeAsync()
{
    _processorCts.Cancel();
    _processorCts.Dispose();
    await base.DisposeAsync().ConfigureAwait(false);
}
public async ValueTask DisposeAsync()
{
    _processorCts.Cancel();
    ...
}

The base method above is src/Voxa.Core/Processors/FrameProcessor.cs:214; it is not virtual. The production disposer iterates IReadOnlyList<FrameProcessor> and calls the base-typed method:

foreach (var p in _pipeline.Processors)
{
    await p.DisposeAsync().ConfigureAwait(false);
}

Why it matters: AgentLoopProcessor's private CTS at AgentLoopProcessor.cs:46 is not cancelled/disposed when the pipeline runner disposes the chain. On normal request-abort teardown, the turn worker may remain governed by a CTS the runner never cancels, and every agent session leaks that CTS registration/object until finalization or process exit.

Recommendation: Make the base disposal path extensible and require derived cleanup to override it. The smallest safe change is to make FrameProcessor.DisposeAsync virtual and change this method to override; alternatively add a protected virtual DisposeCoreAsync called from the base method and move derived cleanup there.

Effort: trivial.

[MAJOR] Realtime transports leak on request-abort disposal MajorCertain

src/Voxa.Services.OpenAIRealtime/OpenAIRealtimeProcessor.cs:67

Dimension: Bad C# practices: lifetime/resource bugs.

Evidence:

protected override async ValueTask OnEndAsync(EndFrame frame, CancellationToken ct)
{
    if (_transport is not null)
    {
        await _transport.DisposeAsync().ConfigureAwait(false);
        _transport = null;
    }
}

The same pattern exists in src/Voxa.Services.AzureVoiceLive/AzureVoiceLiveProcessor.cs:62-68. The transport holds disposable resources:

private readonly ClientWebSocket _ws = new();
private readonly SemaphoreSlim _sendLock = new(1, 1);

The request path shows abrupt disconnects do not call StopAsync or inject EndFrame before disposal:

await using var runner = new PipelineRunner(pipeline, ct);
await runner.StartAsync(ct: ct).ConfigureAwait(false);
...
await runner.WaitAsync().WaitAsync(ct).ConfigureAwait(false);

Why it matters: A browser tab closing or request abort cancels MapVoxaVoice and then disposes the runner. Because OnEndAsync is data-loop lifecycle, it is not guaranteed to run during that disposal path. The underlying ClientWebSocket and SemaphoreSlim can remain undisposed for each aborted realtime session.

Recommendation: Move transport release into the actual disposal path as well as OnEndAsync. After fixing CQ-001, override the disposal hook and call a shared DisposeTransportAsync helper from both OnEndAsync and disposal.

Effort: moderate, because both realtime packages need the same change and tests should cover abrupt disposal without EndFrame.

[MAJOR] Granular speech engines and sessions are released only on EndFrame MajorCertain

src/Voxa.Speech.Abstractions/SpeechToTextProcessor.cs:60

Dimension: Bad C# practices: lifetime/resource bugs.

Evidence:

protected override async ValueTask OnEndAsync(EndFrame frame, CancellationToken ct)
{
    if (_engine is not null)
    {
        try { await _engine.StopAsync().ConfigureAwait(false); } catch { /* best-effort */ }
        await _engine.DisposeAsync().ConfigureAwait(false);
        _engine = null;
    }
}

The same OnEndAsync-only resource release exists at src/Voxa.Speech.Abstractions/TextToSpeechProcessor.cs:43-49 and src/Voxa.Services.SpeechToSpeech/SpeechToSpeechProcessor.cs:153-159.

Why it matters: Engines wrap non-trivial resources: whisper.cpp processors, sidecar processes, audio sessions, HTTP streams, model sessions, and background read loops. The server endpoint treats client disconnect as normal cancellation and lets await using var runner dispose the pipeline. Without an EndFrame, these engines/sessions are not deterministically stopped/disposed.

Recommendation: Centralize each processor's cleanup in an idempotent private method and call it from both OnEndAsync and the real disposal path. Test with a started pipeline disposed under cancellation without first sending EndFrame.

Effort: moderate.

[MAJOR] Every Frame eagerly allocates a ULID string even though only tracing reads it MajorCertain

src/Voxa.Core/Frames/Frame.cs:22

Dimension: Performance / hot-path issues.

Evidence:

public abstract record Frame
{
    public string Id { get; init; } = Ulid.NewUlid().ToString();
}

A source search found the only production read of frame ids here:

activity.SetTag("voxa.frame.id", frame.Id);

Why it matters: Voxa's hot path constructs AudioRawFrame, VAD frames, transcription frames, LLM token frames, TTS audio frames, and system/control frames continuously. This initializer executes for every new frame, allocating a 26-character string and computing a ULID before anyone knows whether tracing is enabled. The performance spec calls out real-time audio allocation pressure as directly harmful; this is now one of the few remaining pervasive allocations.

Recommendation: Make the id lazy or opt-in. Minimal option: back Id with a nullable field and generate it only on first read. If record equality semantics must not include a mutable lazy field, consider moving trace ids into tracing metadata instead of the base frame record.

Effort: moderate, because record equality/serialization expectations should be checked before changing the base record.

[MAJOR] Silero VAD shifts a List from the front on the per-audio-frame path MajorLikely

src/Voxa.Audio.SileroVad/SileroVadProcessor.cs:117

Dimension: Performance / hot-path issues.

Evidence:

for (int i = 0; i < sampleCount; i++)
{
    short s = BinaryPrimitives.ReadInt16LittleEndian(pcm.Slice(i * 2, 2));
    _samples.Add(s / 32768f);
}
...
_samples.CopyTo(0, window, 0, _windowSize);
_samples.RemoveRange(0, _windowSize);

Why it matters: ProcessFrameAsync runs for every inbound audio frame; the VAD window loop runs repeatedly during live audio. List<float>.Add per sample is tolerable in isolation, but RemoveRange(0, _windowSize) memmoves the remaining tail every VAD window. The repo's performance spec and CLAUDE.md both treat real-time audio as a hot path where steady-state allocations and avoidable copies matter.

Recommendation: Replace the front-removing list with a small ring buffer or read-index accumulator. Keep the reused _window scratch array; only change how pending samples are retained between windows.

Effort: moderate.

[MAJOR] OpenAI Realtime and Azure Voice Live duplicate the same protocol stack MajorCertain

src/Voxa.Services.OpenAIRealtime/Events/RealtimeEventCodec.cs:19

Dimension: DRY violations.

Evidence:

// OpenAI package
public static IEnumerable<Frame> Decode(JsonElement evt, bool botSpeaking, int outputSampleRate)

// Azure Voice Live package
public static IEnumerable<Frame> Decode(JsonElement evt, bool botSpeaking, int outputSampleRate)

The matching websocket transport receive loop is duplicated too:

sb.Append(Encoding.UTF8.GetString(buffer, 0, result.Count));
if (result.EndOfMessage)
{
    var msg = sb.ToString();
    sb.Clear();
    yield return msg;
}

The Azure transport's own doc comment says it is suitable for Azure Voice Live, Azure OpenAI Realtime, and OpenAI Realtime endpoints; only URL and auth header differ (src/Voxa.Services.AzureVoiceLive/Transport/WebSocketRealtimeApiTransport.cs:7-10).

Why it matters: Static diff showed the codec files differ by only 11 lines, while the transports have already drifted by 35 lines. Event parsing, interruption semantics, base64 handling, UTF-8 message reassembly, and error handling must be fixed in two packages by hand. The duplicated receive loop also decodes each websocket fragment separately, which can split a multi-byte UTF-8 sequence across receives before appending to the StringBuilder.

Recommendation: Extract a shared internal realtime protocol/transport helper package or shared source item parameterized by endpoint/auth/session options. Fix the receive path once by accumulating bytes through EndOfMessage and parsing UTF-8 bytes directly where practical.

Effort: involved.

[MAJOR] Studio mirrors production pipeline composition logic by hand MajorCertain

apps/Voxa.Studio/Services/BuilderChainCompiler.cs:31

Dimension: Business-logic placement & layering.

Evidence:

/// the live registry's descriptors, and (c) export artifacts. Mirrors
/// <see cref="DefaultVoicePipelineComposer"/> block-for-block so "run from canvas" is the same
/// pipeline a server would boot

The production composer owns effective rates and stage order:

var inputSampleRate  = stt.GetEffectiveInputSampleRate(root);
var outputSampleRate = tts.GetEffectiveOutputSampleRate(root);
...
parts.Add(sp => stt.CreateProcessor(sp, root));
...

Why it matters: The application service that owns default composition is DefaultVoicePipelineComposer. Studio's compiler duplicates that logic, plus code generation, inside the desktop app. Any new stage, rate invariant, diagnostics tap placement, agent memory change, AEC/enhance behavior, or smart-turn wrapping must be updated in multiple places or Studio's "run from canvas" and exported code will diverge from server behavior.

Recommendation: Extract a shared composition plan model from Voxa.AspNetCore that both the default composer and Studio compiler consume. Studio can still add canvas-specific graph validation/export, but stage ordering/rate/tap rules should come from one service.

Effort: involved.

[MAJOR] REST STT HTTP calls ignore pipeline cancellation MajorCertain

src/Voxa.Speech.OpenAI/OpenAIWhisperEngine.cs:158

Dimension: Bad C# practices: cancellation propagation.

Evidence:

using var resp = await _http.SendAsync(req).ConfigureAwait(false);
resp.EnsureSuccessStatusCode();

var json = await resp.Content.ReadFromJsonAsync<JsonElement>().ConfigureAwait(false);

The same pattern appears in src/Voxa.Speech.Mistral/MistralSpeechToTextEngine.cs:132-135.

Why it matters: These calls run when buffered utterance audio is flushed for transcription. If the user disconnects or an interruption cancels the pipeline, the request is not cancelled; it can wait until the shared HttpClient's 100-second timeout (src/Voxa.Speech.Abstractions/VoxaHttp.cs:24). That delays cleanup and can hold buffered audio and multipart content much longer than the session lifetime.

Recommendation: Thread the engine/session cancellation token into TranscribeAsync, SendAsync, and ReadFromJsonAsync. A minimal signature change is TranscribeAsync(byte[] wav, CancellationToken ct) called from the flush path using the engine CTS token.

Effort: moderate.

[MINOR] FrameProcessor.Start creates a linked CTS that is never disposed MinorCertain

src/Voxa.Core/Processors/FrameProcessor.cs:84

Dimension: Bad C# practices: lifetime/resource bugs.

Evidence:

var linked = CancellationTokenSource.CreateLinkedTokenSource(_processorCts.Token, externalCt);
var token = linked.Token;
_systemTask = Task.Run(() => RunSystemLoopAsync(token), token);
_dataTask = Task.Run(() => RunDataLoopAsync(token), token);

Why it matters: The linked CTS registers with externalCt and lives for the processor lifetime. Because it is local-only, disposal never unregisters it. This is one object/registration per processor per session; small individually, but avoidable in a pipeline framework.

Recommendation: Store the linked CTS in a field and dispose it in FrameProcessor.DisposeAsync after the loops have stopped.

Effort: trivial.

[MINOR] TracingProcessor builds per-frame activity names before checking whether tracing is observed MinorCertain

src/Voxa.Observability/TracingProcessor.cs:38

Dimension: Performance / hot-path issues.

Evidence:

using var activity = VoxaActivities.Source.StartActivity($"voxa.{_scope}.{frame.GetType().Name}");

{
    activity.SetTag("voxa.scope", _scope);

Why it matters: This processor is explicitly pass-through for every frame. The interpolated activity name and frame.GetType().Name are evaluated before StartActivity can return null for an unobserved source. If users leave the processor in a hot pipeline without an active listener, every audio/token frame still pays the name construction cost.

Recommendation: Check VoxaActivities.Source.HasListeners() before building the name, or use a constant activity name and set type/scope only when an activity exists.

[MINOR] TranscriptionFilter lowercases before a case-insensitive Contains MinorCertain

src/Voxa.Speech.Abstractions/TranscriptionFilter.cs:84

Dimension: Simplification / optimization.

Evidence:

var lower = trimmed.ToLowerInvariant();
for (int i = 0; i < SubstringBlocklist.Count; i++)
{
    if (lower.Contains(SubstringBlocklist[i], StringComparison.OrdinalIgnoreCase)) return true;
}

Why it matters: StringComparison.OrdinalIgnoreCase already ignores case, so the lowercased copy is dead work and allocates on every final transcript that reaches substring filtering.

Recommendation:

for (int i = 0; i < SubstringBlocklist.Count; i++)
{
    if (trimmed.Contains(SubstringBlocklist[i], StringComparison.OrdinalIgnoreCase)) return true;
}

Effort: trivial.

[MINOR] Descriptor defaults and shared HTTP-client resolution are repeated across providers MinorCertain

src/Voxa.Speech.OpenAI/OpenAISpeechDescriptors.cs:46

Dimension: DRY violations / simplification.

Evidence:

// Options record
public string ApiBaseUrl { get; init; } = "https://api.openai.com/v1";
public string TtsModel { get; init; } = "tts-1";

// Descriptor binder
ApiBaseUrl = s["ApiBaseUrl"] ?? "https://api.openai.com/v1",
TtsModel   = s["TtsModel"] ?? "tts-1",

The same descriptor pattern repeats in Azure, ElevenLabs, and Mistral descriptors. The shared HTTP-client resolver expression appears in seven sites, including:

private static HttpClient? ResolveHttpClient(IServiceProvider sp)
    => (sp.GetService(typeof(IVoxaHttpClientProvider)) as IVoxaHttpClientProvider)?.Resolve();

Why it matters: Changing a default in an options record does not change config-driven descriptor binding unless the duplicate literal is updated. Similarly, changing HTTP client resolution policy requires edits across seven provider files.

Recommendation: Bind into a new options instance first and override only provided values, or centralize provider descriptor binding helpers in Voxa.Speech.Abstractions. Add a tiny helper for IVoxaHttpClientProvider resolution.

Effort: moderate.

6.Quick wins

  1. Fix CQ-001 and CQ-010 together: make base disposal extensible and dispose the linked CTS. This unlocks safe fixes for the other lifetime issues.
  2. Add idempotent real-disposal cleanup to realtime and granular speech processors (CQ-002, CQ-003), with one abrupt-disconnect test.
  3. Remove eager frame id allocation (CQ-004) or gate it behind tracing.
  4. Thread cancellation through OpenAI/Mistral REST STT calls (CQ-008).
  5. Apply trivial hot-path cleanups: TracingProcessor.HasListeners gate and TranscriptionFilter lowercase removal (CQ-011, CQ-012).
  6. Schedule a focused refactor for duplicated realtime protocol code (CQ-006) before adding another realtime event type.

VST-002 — Voxa Studio 2.0 Design Brief

The design direction for Voxa Studio's next major iteration: a branded launch experience with an animated mark, a refined visual and motion system, dedicated STT and TTS playgrounds, a node-style pipeline builder on an interactive canvas, and a run & metrics workbench that turns the diagnostics hub's event stream into decision-grade performance data. This is a design document — it defines look, feel, flows, and feasibility; the engineering spec for each phase follows separately.

1.Purpose, audience & scope

Studio v1 (VST-001) proved the core idea: run the real pipeline locally and watch it think. v2 turns a developer instrument into a product. The audience widens from "the Voxa maintainer debugging VAD" to three personas:

PersonaThey come to…The view that serves them
The evaluator — deciding whether to adopt Voxahear the voices, feel the latency, see it work in 2 minutes without an accountSplash → Talk; TTS Playground
The builder — composing a bot for their productprototype a pipeline, pick providers, tune the trade-offs, export configPipeline Builder; STT/TTS Playgrounds
The tuner — owning latency/quality in productionmeasure stage timings, compare configurations, find the regressionRun & Metrics workbench

In scope: visual identity + motion system, splash, restructured shell, two focused playgrounds, the node canvas, the metrics workbench. Out of scope: non-Windows audio backends, cloud accounts/licensing, telemetry leaving the machine, and any pipeline topology the framework cannot actually run (see §8.3 — the canvas must never let users draw a lie).

2.Design principles

  • P1 — The signal is the brand. Voxa's subject matter is live audio and time. Every signature visual — the logo's pulse, edge flow on the canvas, the VAD trace, waterfall bars — is a representation of signal moving through time. No decorative graphics that aren't data.
  • P2 — Dark, calm, one accent. Near-black blue-grey surfaces, hairline borders, generous whitespace, a single cyan accent doing all the talking. Color beyond cyan is semantic only (stage colors, good/warn/bad). If a screen looks exciting while idle, it's wrong.
  • P3 — Motion has a job. Animation either (a) explains causality (a frame travelled from STT to the agent), (b) confirms state change (gate opened), or (c) brands a threshold moment (launch). Nothing loops while idle except the live data itself. Every animation respects reduced-motion.
  • P4 — Truthful instrumentation. Numbers shown are measured on this machine, from the real engines, via the same diagnostics surface servers use. No synthetic demo data, ever — an empty state explains how to produce data instead of faking it.
  • P5 — Two minutes to wow, keyless. The first-run path (splash → Talk → speak → see the waterfall) must work with zero accounts and show something a screenshot can't: the app hearing you in real time.
  • P6 — The UI is a registry client. Every list of providers, voices, models, and node types comes from the live registry and pinned catalogs. The design must assume these lists grow; nothing is hardcoded to today's seven Piper voices.

3.Brand identity — the mark, motion, color, type

3.1 The mark

The Voxa mark is a V cut by a waveform: two clean strokes forming the V, with three signal bars rising inside the cut — the moment a voice enters the pipeline. It reads at 16 px (titlebar), 96 px (splash), and as a single-color glyph for monochrome contexts. Below is a live prototype of the draw-on animation this brief specifies (refresh the page to replay; it honours prefers-reduced-motion):

VOXA STUDIO

Construction rules: stroke weight 7% of cap height; bar heights follow a 22/30/18 rhythm (a stylised "vox" energy contour, not random); minimum clear space = one bar width on all sides. The mark never rotates, skews, or fills with gradients.

Inter (shipped via Avalonia.Fonts.Inter) for UI; Cascadia Code for every number, id, and timestamp — if it's data, it's mono. Scale: 11/12 (dense data), 13 (body), 15 (section), 20 (view title), with the wordmark's letter-spaced caps reserved for brand moments (titlebar, splash, empty states).

4.Launch experience — the animated splash

The splash is a borderless 480×320 window, not a page inside the shell — it appears in <300 ms (before any Voxa service work) and does real work while it plays:

tOn screenBehind it
0–1.1 sMark draws on (motion.draw): the V strokes sweep inconfiguration load, AddVoxa container build
0.9–1.3 sThe three signal bars spring up in sequenceaudio device enumeration (WASAPI)
1.25–1.95 sWordmark fades in, letter-spacing settling from wide; beneath it, microcopy ticks through real init stages: providers · devices · cachemodel-cache inventory scan (no network)
≈2.0 sSplash cross-fades out; the main window appears with the mark already seated in the titlebar — the logo "lands" rather than disappearsshell composition
Hard rules. (1) The splash never adds time: it dismisses the moment init completes, even mid-animation — on a warm machine the full sequence may be cut at ~1.2 s. (2) A click skips it. (3) prefers-reduced-motion (Avalonia: platform a11y settings) swaps animation for a static mark + progress microcopy. (4) No network on the splash path, ever (P5/WS1-A2 invariant).

The same draw-on animation becomes the brand's loading primitive: the Talk view's "downloading models" state replays the bars at small scale as a determinate progress indicator (bar heights = download progress of the three largest artifacts).

6.STT Playground

Job: answer "how well and how fast does speech become text on this machine?" for any Whisper model — without composing a whole pipeline.

6.1 Layout & flow

  • Input strip: three sources, one active at a time — 🎙 live mic (push-to-talk or VAD-gated), 📄 drop a WAV/recording, 🔁 replay the bundled fixtures (jfk.wav ships in-repo). Model selector (catalog: tiny→small, quantized variants) with size/cached badges.
  • Transcript pane: finals appear as cards stamped with utterance duration and final-transcript latency (utterance end → final, the stt_final stage measured standalone). A waveform strip of the captured utterance renders above each card.
  • Accuracy harness: paste a reference text → live WER (word error rate) with insert/delete/substitute coloring on the diff. This is the feature that turns "tiny.en sounds fine" into "tiny.en is 8.1% WER on my accent; base.en is 4.9% and 240 ms slower" — exactly the trade-off the model selector exists for.
  • Side-by-side mode: run the same audio through two models (sequentially; one whisper context at a time) and diff the transcripts + timings.
Feasibility. Everything here drives WhisperCppSttEngine directly (the WS3 Voice-Lab pattern, STT edition). WER is a ~80-line Levenshtein on word tokens. The mic path reuses IStudioAudioDevice.CaptureAsync + the existing VAD processor for gating. No framework changes.

7.TTS Playground

Job: the v1 Voice Lab, matured from a list into a lab. Carries over: full catalog, real engines, TTFB/RTF, A/B pins, WAV export, talk-session device arbitration. New:

  • Waveform + take history. Each synthesis renders its waveform inline (amplitude envelope, custom-drawn) and lands in a per-voice take history — change the text, compare takes, replay any. The waveform is also the playback scrubber.
  • A/B/X blind test. The honest upgrade to A/B: X is randomly A or B; after the user votes, reveal. Settles "Kokoro is obviously better" with data, in keeping with P4.
  • Stress phrases. A curated phrase deck (numbers, dates, code identifiers, long clauses, homographs) one click away — the sentences that actually break TTS, replacing the pangram default.
  • Batch bench. "Synthesize this phrase deck on N voices" → table of TTFB/RTF percentiles per voice, exportable; feeds the same run format Metrics consumes (§9).

8.2 Interaction design

  • Palette → canvas: the node list is generated from the live registry + built-ins (Source, VAD engines, STT providers, TranscriptionFilter, Agent, SentenceAggregator, TTS providers, Sink, DiagnosticsTap). Drag onto the canvas, or use the always-valid "+" affordance on a dangling port which offers only type-compatible nodes — beginners never see an invalid wire.
  • Typed ports: a port carries a frame type (audio, transcription, agent-text, synth-audio). Incompatible drags snap back with a 120 ms shake and a one-line reason ("Whisper consumes audio frames; the Agent emits text"). Compatibility comes from a static frame-flow table per node type — the same truth the default composer encodes.
  • Inspector: selecting a node shows its options panel (the descriptor's config section rendered as form controls — the Config view's machinery, per node). Cached/uncached and validation states render on the node itself (badge + red outline with the validator's message).
  • Run: the canvas compiles to the same Pipeline.Build().Source().Then(…).Sink() chain Talk uses and starts a live session in place (§8.4). Export produces (a) the appsettings.json when the graph matches the default shape, or (b) generated C# composition code when it doesn't — both honest artifacts a server can run.
  • Canvas furniture: pan/zoom (pinch + wheel), snap-to-grid, auto-layout ("tidy"), minimap beyond ~12 nodes, undo/redo, save/load graph as JSON in the user profile.

8.3 The honesty constraint important

Voxa pipelines are a linear chain today. Pipeline links processors in sequence — there is no fan-out/fan-in in the runtime. The canvas therefore enforces single-in/single-out wiring in v2 (each port holds at most one edge), and the design language makes that feel intentional: the grid, the left-to-right flow, the "+" port affordance all communicate chain, not graph. Branching (e.g. parallel TTS for comparison, a tee into a recorder) is a framework feature first — if we want it, it gets its own spec (frame routing, backpressure semantics) and the canvas inherits it later. Designing the canvas as if branching existed would violate P4 with geometry instead of numbers.

8.4 Live mode — the canvas as instrument

When a graph runs, the canvas becomes the most literal "watch it think" surface in the app, driven entirely by real VoxaDiagnosticsHub events:

  • Edge flow: frame activity renders as particles flowing along the edge, colored by frame type and rate-limited to ~30 fps. Audio edges shimmer continuously while the gate is open; a transcript edge fires one bright pulse per final.
  • Node state: the active stage glows (the §3.3 stage color at 20% halo); per-node badges show live queue depth (FrameProcessor.QueuedFrameCount) and last-stage latency.
  • Inline turn ticker: the most recent waterfall renders as a slim strip along the canvas bottom; clicking a bar highlights the corresponding node.

9.Run & Metrics workbench

Job: turn sessions into evidence. A run is the unit of record: one configuration (a Builder graph or the active config) exercised by one input source for some duration, producing a metrics bundle.

9.1 Running

  • Sources: live mic; a WAV fixture; or a scripted session — a list of utterance WAVs (or text injected post-STT) replayed with configurable gaps, giving repeatable measurements. Scripted runs are the workbench's killer feature: same input, two configs, honest comparison.
  • During a run: a compact live header (turn count, last TTFB, error count) — the heavy charts populate on completion to keep the run itself unperturbed (the hub's drop-oldest contract still applies).

9.2 Reading

  • Charts: TTFB percentile card (p50/p95/max, delta vs a baseline run), per-turn stage stacks, per-stage trend over the run, RTF for the TTS leg, error/interruption counts. All custom-drawn, all exportable.
  • The takeaway line: each run renders one generated sentence naming the dominant stage and the obvious lever ("VAD hangover is 54% of p50 — lower StopDuration or enable smart-turn"). Rule-based, not an LLM; it's the tuner persona's "so what".
  • Persistence: runs serialize (config snapshot + event stream + computed stats) as JSON bundles under the user profile; the run list is a folder scan. Nothing leaves the machine (P4).

13.Open questions

  1. Branching pipelines — is fan-out/fan-in (tee-to-recorder, parallel TTS) wanted enough to spec as a framework feature? It changes the runtime, not just the canvas (§8.3).
  2. Run sharing — run bundles are local JSON; is "export → attach to a GitHub issue" enough, or is a viewer for foreign bundles worth building (support workflows)?
  3. Cloud STT/TTS in playgrounds — keys permitting, should cloud providers appear in the labs for cross-vendor comparison, or do we keep the labs local-only and leave cloud to Talk?
  4. Brand reach — does the animated mark extend to the README (animated SVG renders on GitHub), docs site, and NuGet icon in this phase, or stay app-only?
